## 1. Як працює навігація

Внутрішнє ядро DrivePickerSection — три координати:
`mode` (`myDrive` / `sharedWithMe` / `sharedDrives`), `folderId`, `sharedDriveCtx`.
При першому розкритті секції стартує `mode='myDrive'` і `folderId = 01_ОРИГІНАЛИ` поточної справи. Один `useEffect` слухає всі три координати і за зміни тригерить два паралельні запити: список вмісту папки (через `q='<id>' in parents and trashed=false`, `orderBy=folder,name`) і побудову breadcrumb (підняття по `parents[]` доки `parents.length === 0` або помилка). Перемикач джерела (chips) скидає `folderId` і `sharedDriveCtx`. Клік на breadcrumb — повертає на той рівень. Клік на папку — `setFolderId(item.id)`. Клік на файл — `onPick(item)` з тим самим контрактом що в мікро-TASK 4.1 (маркер `_isDriveSource` + `_driveId`). Правило #8 дотримане: жоден `q=` не містить кирилиці, фільтрація по іменах не використовується взагалі.

## 3. Інструкція адвокату для тестування

### Сценарій А — файл з 01_ОРИГІНАЛИ цієї ж справи (legacy з мікро-TASK 4.1)
1. Відкрий справу з налаштованою Drive-папкою (наприклад Кісельова).
2. Натисни «+ Додати документ».
3. Розкрий секцію «Або вибрати файл вже на Drive».
4. Перевір що breadcrumb показує `Мій Drive › 01_АКТИВНІ_СПРАВИ › Кісельова... › 01_ОРИГІНАЛИ`.
5. Тапни будь-який файл — він має зʼявитись у file-card у дропзоні.
6. Заповни поля, натисни «Додати документ».
7. Очікувано: документ додано, OCR і документна природа коректні (legacy pipeline).

### Сценарій Б — файл з іншої папки тієї ж справи
1. У тій самій модалці тапни breadcrumb «Кісельова...» (батьківська папка справи).
2. Список переключається на вміст папки справи.
3. Тапни папку `04_ПОЗИЦІЯ` (або іншу що містить файли).
4. Тапни будь-який файл всередині.
5. Перевір що файл вибрано і додано як документ.

### Сценарій В — файл з папки іншої справи
1. У breadcrumb тапни «01_АКТИВНІ_СПРАВИ».
2. Список показує всі справи.
3. Тапни папку «Брановський...».
4. Зайди в `01_ОРИГІНАЛИ` Брановського.
5. Тапни файл — він приєднається до справи Кісельова (поточної).
6. Перевір що документ додано і OCR пройшов.

### Сценарій Г — файл з кореня Мій Drive
1. У breadcrumb тапни «Мій Drive».
2. Список показує верхній рівень My Drive (включно з `01_АКТИВНІ_СПРАВИ`, `02_АРХІВ` тощо).
3. Якщо є будь-який файл прямо в корені — тапни його.
4. Або зайди в довільну іншу гілку Drive (НЕ обовʼязково `01_АКТИВНІ_СПРАВИ`) і вибери файл там.
5. Файл має приєднатись до поточної справи.

### Додатково — Поділилися зі мною
1. Тапни chip «Поділилися зі мною».
2. Список показує файли і папки які тобі розшарили.
3. Можеш тапнути файл напряму або зайти в розшарену папку і вибрати всередині.
4. Якщо нічого не розшарено — empty state «Папка порожня».

### Додатково — Спільні Drive
Chip «Спільні Drive» зʼявляється тільки якщо в тебе є хоч один Shared Drive. У соло-сетапі цього chip не буде.

## 4. Що НЕ зроблено і чому

- **Повноцінний `google.picker.PickerBuilder`** — поза скопом. Він потребує `developerKey` і пошуку. Поточна версія покриває 4 сценарії без додаткової інфраструктури.
- **Пошук по іменах** — навмисно немає. Правило #8 забороняє кирилицю в `q=`, а пошук без кирилиці безглуздий для українського корпусу. Якщо знадобиться — окремий мікро-TASK з client-side фільтрацією.
- **Пагінація >100 елементів на рівень** — обмеження `pageSize=100`, з повідомленням «Показано 100. Перейдіть у вкладену папку щоб уточнити». Адекватно для типового Drive адвоката, де папки рідко мають >100 файлів. Лоад-мор додамо коли реально буде заявка.
- **Створення нових папок з UI** — поза скопом, адвокат робить це у Drive напряму.
- **Render-гейт `driveFolderId && !state.file`** збережений — секція не рендериться якщо у каси немає Drive-папки. Це не блокує функціональність (без папки документ нема куди класти), але утримує тести в актуальному стані.
- **Юніт-тести для нової навігації** — не додавались, бо вимагають мокати `driveRequest` з кількома сценаріями (folder list, file metadata walk, drives list, sharedWithMe). Це 100+ рядків моків і тестів. Якщо адвокат побачить регресії — додамо в окремий мікро-TASK з продуманим harness.

## 5. Чи був зачеплений pipeline у `onSubmit`?

**Ні.** Файл-маркер `{ _isDriveSource: true, _driveId, name, size, type }` створюється у `handleDrivePick` тим самим способом що в мікро-TASK 4.1. CaseDossier:`onSubmit` отримує цей обʼєкт без змін, гілка `if (file?._isDriveSource)` працює. OCR pipeline, корекція `documentNature`, `lastOcrAt`, `executeAction` для `update_document` — все недоторкане.
