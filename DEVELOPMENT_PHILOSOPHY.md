# 🤖 AI-FIRST ARCHITECTURE — ГОЛОВНИЙ ПРИНЦИП

Legal BMS — система народжена AI для людей.

## Суть принципу

AI тут не надбудова над системою — AI і є система. Те що видно в інтерфейсі — картки справ, календар, дашборд — це точки входу для людини до системи яка всередині є AI. Інтерфейс — вікно в цю систему, не сама система.

Орієнтир: Anthropic. Claude не "прикручений" до інтерфейсу — він і є продукт. Інтерфейс це спосіб з ним взаємодіяти. AI присутній скрізь — не як фіча, а як інфраструктура. Відчуття зсередини: це не надбудова, це частина цілого.

Legal BMS в ідеалі: адвокат відкриває систему і відчуває що розмовляє з розумною системою яка знає його справи, розуміє контекст, готова діяти. Картка справи — AI який знає цю справу. Календар — AI який розуміє розклад. Дашборд — AI який бачить пріоритети.

## Що це означає для розробки

**1. Логіка первинна.**
Кожна нова функція проектується як набір даних і операцій з ними. UI — потім, як зручне вікно до цієї логіки. Коли логіка досконала — UI виходить інтуїтивно.

**2. Усі ACTIONS доступні агентам.**
Якщо нову дію не можна виконати через executeAction — це баг архітектури. Кожна функція системи має бути доступна агенту.

**3. Голос як командний інтерфейс.**
Адвокат командує голосом — агент виконує. Реалізується через транслітерацію голосу в текст (нативний мікрофон браузера або телефону) — адвокат надиктовує, текст передається агенту. Пряме голосове управління без тексту — майбутня ціль. Будь-яку команду, дію, запит на аналіз має бути можливо виконати голосом.

**4. UI — рівноправний інтерфейс там де він природніший.**
Документи, календар, перегляд матеріалів справи, числа і дати — зручно читати з екрану, а не сприймати на слух. UI тут не допоміжний — він основний інтерфейс для роботи з контентом. Голос і UI рівноправні, обидва служать людині, кожен там де природніший.

**5. Дані структуровані для AI-обробки.**
Не "красиво відображаються", а "зручно парсяться агентом". Агент має отримати будь-які дані системи без перешкод.

**6. Принцип дублювання інтерфейсів.**
Будь-яка дія в системі доступна двома рівноправними шляхами — через UI і через агента в чаті. Додати засідання: форма в календарі або команда агенту. Створити справу: кнопка в реєстрі або голосова команда агенту. Це не надмірність — це архітектурна вимога. Якщо дія доступна тільки через UI і недоступна агенту — реалізація неповна. Якщо доступна тільки агенту без UI — теж неповна.

## Перевірка для нової функції

Перед початком розробки запитай:

✅ Чи можна виконати цю функцію через діалог з агентом?
✅ Чи всі дії функції покриті ACTIONS і доступні через executeAction?
✅ Чи можна віддати команду на виконання голосом (через текст)?
✅ Чи може агент перевірити результат без UI?

Якщо хоч на одне питання відповідь "ні" — функція не AI-first. Перепроектувати перед реалізацією.

## 🧬 ОСНОВНИЙ ПРИНЦИП — ЕМБРІОН З ПОВНИМ ДНК

Legal BMS — живий організм. Зараз він на стадії ембріону:
- **Зовні:** соло-практика з одним адвокатом
- **Всередині:** повна архітектура для SaaS з чотирма типами організацій (solo / bureau / association / firm), багаторівневими ієрархіями, командами справ, permissions, аудитом, білінгом

Ембріон людини маленький, але має повне ДНК дорослого тіла. Печінка не з'являється «ззовні» при розвитку — вона завжди була в ДНК і просто розгортається в потрібний момент.

**Так само Legal BMS:**
- Зараз багато речей — заглушки що повертають true (бо один користувач)
- Структури даних — повні, з усіма потрібними полями для майбутнього
- Перевірки прав — викликаються (готові прийняти реальну логіку)
- Audit log — реально пише (для одного користувача поки що)
- Білінг — тихо збирає дані (хоча UI ще немає)

Коли система розгорнеться в SaaS — це **не «додавання нового», а розгортання того що вже є**. Заглушки замінюються на реальну логіку. Користувачі додаються в `users[]`. Tenants — в `tenants[]`. Ніяких архітектурних переробок.

## ПРАВИЛО НАРОДЖЕННЯ МОДУЛЯ

Кожен новий модуль/функція **проектується ОДРАЗУ** з урахуванням трьох вимірів:

```
ПРИ НАРОДЖЕННІ МОДУЛЯ ЗАКЛАДАЄМО:
═══════════════════════════════════════════

1. SAAS-ГОТОВНІСТЬ
─────────────────────────────────────────
✅ tenantId на всіх даних
✅ userId на всіх діях (createdBy/updatedBy)
✅ createdAt / updatedAt timestamps
✅ Ізоляція через checkTenantAccess
✅ Структури підтримують multi-tenant

2. MULTI-USER ГОТОВНІСТЬ
─────────────────────────────────────────
✅ Permissions checking через executeAction
✅ Team awareness (для справ — case.team[])
✅ Дефолти permissions за ролями
✅ External access support (для зовнішніх адвокатів)
✅ shareType (private / internal / external)

3. BILLING ГОТОВНІСТЬ
─────────────────────────────────────────
✅ activityTracker.report() в значущих точках
✅ Категорії time_entries (case_work / admin / system)
✅ subCategory для деталізації
✅ Інтеграція з master_timer state machine
✅ Якщо викликає AI — паралельне ai_usage логування

4. AI ГОТОВНІСТЬ
─────────────────────────────────────────
✅ resolveModel(agentType) для виклику AI 
   (не hardcoded модель)
✅ logAiUsage() з повним context
✅ Tool Use якщо новий модуль (не JSON ACTIONS)
═══════════════════════════════════════════

НЕ "потім додамо" — а ОДРАЗУ при народженні.
```

### Чому це критично

**Якщо забути закласти при народженні** — потім доведеться:
- Робити міграцію даних (додавати tenantId до існуючих записів)
- Переписувати логіку модуля
- Тестувати все наново
- Можливо ламати існуючу функціональність

**Якщо закласти одразу** — модуль готовий до:
- Multi-user одного дня (просто активуємо заглушки)
- SaaS-комерціалізації (просто додаємо tenants)
- Білінгу (дані вже зібрані за весь час)

**Дешевше зараз, болюче потім.**

## ПИТАННЯ ДЛЯ КОЖНОГО НОВОГО TASK

При роботі над **будь-яким** новим функціоналом перевір:

**AI-FIRST (найважливіше):**
1. **AI-aware?** Чи можна виконати цю функцію через діалог з агентом І через UI? Обидва шляхи мають працювати.
2. **Action-aware?** Усі дії функції покриті ACTIONS і доступні через executeAction?
3. **Voice-aware?** Чи можна віддати команду голосом (через транслітерацію в текст)?
4. **Verifiable?** Чи може агент перевірити результат без UI?

**SAAS / MULTI-USER / BILLING:**
5. **Tenant-aware?** Нова сутність прив'язана до tenantId?
6. **User-aware?** Зрозуміло хто створив, оновив, видалив?
7. **Audit-aware?** Критичні дії пишуться в auditLog[]?
8. **Permission-aware?** Дія проходить через executeAction з перевірками?
9. **Future-aware?** Передбачені хуки для майбутніх ролей, команд, прав?
10. **Data-rich?** Збираються всі дані які можуть знадобитись потім?
11. **Billing-aware?** Якщо є активність адвоката — інструментована через activityTracker?
12. **Model-aware?** Якщо викликає AI — через resolveModel, з logAiUsage?

Якщо хоч одне питання має відповідь "ні" — зараз є можливість додати потрібні поля/перевірки/хуки. Робимо одразу.

## АРХІТЕКТУРНІ ШАБЛОНИ

### 1. Planka Picatinny (Provider Pattern)

```
─────────────────────────────────────────
ПРИКЛАД: ocrService.js
─────────────────────────────────────────

ocrService.js (фасад)
  ↓ автоматично використовує:
ocr/documentAi.js (основний)
ocr/claudeVision.js (fallback)
ocr/pdfjsLocal.js (локальний)

Додавання нового провайдера = один новий файл.
Решта системи не змінюється.
─────────────────────────────────────────
```

**Застосування:**
- ✅ OCR (документи через Document AI / Claude Vision / pdf.js)
- ⏸ Storage (Drive / R2 / BYOS-Drive — заглушка `tenant.storage.provider`)
- ⏸ AI Provider (Anthropic / OpenAI / Grok — майбутнє через `aiService.js`)
- ⏸ Messaging (Telegram / WhatsApp — майбутнє)

**Принцип:** один фасад, кілька реалізацій, додавання нової реалізації — один файл.

### 2. Архіваріус (executeAction)

```
─────────────────────────────────────────
ВСІ ЗМІНИ ДАНИХ ПРОХОДЯТЬ ЧЕРЕЗ ОДНУ ФУНКЦІЮ
─────────────────────────────────────────

executeAction(agentId, action, params)
  ↓
ОДНЕ МІСЦЕ ДЕ:
- Перевіряються permissions
- Викликається handler
- Пишеться audit log
- Інструментується billing
- Зберігається на Drive

Кожна нова дія = додавання в реєстр ACTIONS
Кожен новий агент = додавання в PERMISSIONS
─────────────────────────────────────────
```

**Виняток:** UI-функції addCase/closeCase/restoreCase/destroy_case з прямим writeAuditLog.

### 3. Стільниковий принцип (Module Isolation)

```
─────────────────────────────────────────
КОЖЕН МОДУЛЬ — ОКРЕМА КОМІРКА
─────────────────────────────────────────

src/components/[ModuleName]/
├── index.jsx              ← компонент
├── helpers.js             ← локальні хелпери
└── styles.module.css      ← локальні стилі (опціонально)

ЗВ'ЯЗОК З App.jsx:
- Отримує props (cases, updateCase, ...)
- Викликає функції з props для змін
- Власний state — тільки UI

ЯКЩО МОДУЛЬ ПАДАЄ:
- ErrorBoundary ловить помилку
- Показує "Модуль тимчасово недоступний"
- Решта системи продовжує працювати
─────────────────────────────────────────
```

### 4. Single Source of Truth

```
─────────────────────────────────────────
ОДНЕ ДЖЕРЕЛО ПРАВДИ ДЛЯ КОЖНОГО ТИПУ ДАНИХ
─────────────────────────────────────────

Cases → cases[] в App.jsx state
Notes → notes[] в App.jsx state
TimeEntries → time_entries[] в App.jsx state

ВИКЛЮЧЕННЯ (свідомі):
- agentHistory — 3-tier cache (Drive + localStorage + state)
  Документовано як валідна архітектура
- pinnedNoteIds — допустима денормалізація

ЯКЩО ЗНАХОДИТЬ ДУБЛЮВАННЯ:
1. Знайти source of truth
2. Видалити дублікати
3. Оновити доступ через єдине джерело
─────────────────────────────────────────
```

## ФІЛОСОФІЯ БІЛІНГУ — 13 ТЕЗ

(Зведено з адмін-чату «CRM і Білінг». Формує всю архітектуру білінгу.)

1. **Метафізика робочого дня.** Система бачить день як цілісний відрізок часу. Кожна хвилина куди-небудь приписується. Незаповненого часу не лишається.

2. **Внутрішнє ≠ зовнішнє.** Внутрішній облік — повний, до хвилини. Зовнішнє (для клієнта) — фільтроване, фіксована сума за українською моделлю.

3. **CRM не модуль, а зріз.** Той самий time_entries[], hearings[], notes[], переглянутий через призму клієнта.

4. **Інструментація зараз, UI потім.** Дані накопичуються з самого початку. Через рік увімкнули UI — є рік історії.

5. **Голос як основа, не фіча.** Скрізь мікрофон. Адвокат не повинен бути секретарем у себе самого.

6. **Семантичне розуміння переходів.** Система не фіксує клік — розуміє що клік означає в контексті адвокатської роботи.

7. **Категоризовані таймери замість простих старт/стоп.** Кожна категорія знає свою тривалість, чи billable, чи прив'язується до справи.

8. **Master timer + субтаймери = архітектура.** Майстер тікає весь день. Субтаймери розкладають час по конкретних діях. Сума субтаймерів = час майстра.

9. **Idle Detection — бонус, не основа.** На Chrome додатковий контроль. На Safari/iPad — стандарти і голосові команди.

10. **Облік працює навіть коли адвокат поза системою.** Засідання у календарі — система знає що це година + дорога. Telegram-чат з клієнтом — це client_communication.

11. **Білінг як орган.** Завжди є архітектурно. Адвокат вирішує наскільки бачити: повний / спрощений / класичний режим UI.

12. **Звіти як живі зрізи.** Не зберігаються готовими, формуються в момент запиту з потоку логів. 4 рівні: резюме, акт, AI-narrative, сирі логи.

13. **Розмовна аналітика.** Сторінка аналітики не з фільтрами, а порожнє полотно + чат з агентом.

## ПРИНЦИПИ AGENTNOY ARHITEKTURY

### Гібридний підхід — один розмовний + декілька технічних

**Розмовний агент** (для адвоката):
- Один на модуль (агент дашборду, агент досьє)
- Sonnet з повним контекстом
- Обличчя для адвоката, через нього вся взаємодія
- Внутрішньо викликає технічні агенти через tools

**Технічні під-агенти** (без діалогу):
- Document Processor агент — нарізка, OCR
- Context Generator агент — формує case_context.md
- Naming агент — назви документів за стандартом
- Кожен має вузький контекст і структурований output

**Принцип:** Технічна робота окремо, розмова окремо. Це SRP, чистий UX, дешева експлуатація.

### Tool Use vs JSON ACTIONS

**Старі агенти на JSON ACTIONS:**
- QI agent (працює стабільно)
- Dashboard agent (стабільний, точкові фікси промптів якщо треба)
- Dossier agent в режимі чату (тільки спілкується)

**Нові модулі на Tool Use:**
- Document Processor v2
- Context Generator
- Canvas-конструктор документів
- Майбутні агенти

**Перший модуль на tool use закладає інфраструктуру** (`toolUseRunner.js`, `toolDefinitions.js`). Наступні переюзовують.

### AI Provider Abstraction (на майбутнє)

Зараз: Anthropic — єдиний провайдер. Жорстко в 10 точках.

**В майбутньому** (через 6-12 міс при SaaS-комерціалізації):

```
src/services/ai/
├── aiService.js          ← фасад
├── providers/
│   ├── anthropic.js
│   ├── openai.js
│   └── grok.js
└── adapters/
    ├── messageAdapter.js
    └── toolAdapter.js
```

Дозволяє Premium-юзеру обрати GPT-4 замість Sonnet для агента досьє.

**Окремий TASK на 1-2 дні роботи** коли стане потрібно.

Кожен TASK з суттєвими змінами **перевіряє актуальність CLAUDE.md** в зачеплених розділах.

**Три варіанти дій:**

**A) Оновити тільки зачеплені розділи** в рамках TASK
**B) Оновити весь CLAUDE.md** (розширення scope)
**C) Залишити CLAUDE.md як є**, створити запис у `recommended_task_claude_md_audit.md` для майбутнього TASK

**За замовчуванням Варіант C** — мінімальне втручання.

### Знахідки накопичуються в окремих файлах

- `bugs_found_during_<task_name>.md` — баги виявлені під час впровадження TASK
- `recommended_task_claude_md_audit.md` — рекомендації для майбутнього CLAUDE.md Audit
- `LESSONS.md` — інституційна пам'ять (не змінювати без явного TASK)

## ПРАВИЛА ВЗАЄМОДІЇ З АДВОКАТОМ

### 1. Чесна критика переважає над згодою

Адвокат явно просив уникати sycophancy. Якщо щось архітектурно неправильно — сказати, з аргументами.

### 2. Технічний матеріал — для прослуховування

Адвокат краще засвоює структурований аудіо-дружній виклад, не щільний читабельний текст. Короткі речення, чітка структура.

### 3. Одна тема за раз

Не комбінувати кілька питань в одну відповідь.

### 4. Код залишати в відповідях

Навіть якщо адвокат не читає код — код потрібен для Claude Code. Залишати.

### 5. Пакетна обробка багів

Баги збирати під час тестування, виправляти всі разом, не точково.

### 6. Контекст між сесіями

Завжди використовувати memory для зафіксації архітектурних рішень. Контекстні файли для нових тематичних чатів.

## ОДНОЗНАЧНІСТЬ — КОЖНЕ ІМ'Я ВИРАЖАЄ ОДИН НАМІР

```

БАГИ ВИНИКАЮТЬ КОЛИ ОДНЕ ІМ'Я ПОЧИНАЄ
ОЗНАЧАТИ ДВІ РЕЧІ. ДИСЦИПЛІНА — ПАУЗА
ПЕРЕД ТИМ ЯК НАШАРУВАТИ НОВИЙ СЕНС НА
ІСНУЮЧЕ ІМ'Я.
─────────────────────────────────────────
```

### Що насправді сталося з skipCache

Прапор `options.skipCache` починав з одним сенсом — «не читай старий кеш, перезапусти OCR». Згодом до умови запису додалось `&& !options.skipCache`. Здавалось логічним: «якщо не використовуємо кеш — значить взагалі не торкаємось». Тести зелені. Code review нічого не зловив.

Це не помилка типу «погано назвав змінну». Помилка глибша — **не була зроблена пауза перед розширенням сенсу**. Якби автор додавання запитав «а що `skipCache` зараз означає?», побачив би: вже існує одне рішення (читання), нашаровується друге (запис). Два рішення на одному імені — тонкий баг який пройде тести і зламається в реальній експлуатації.

Принцип не «називай точніше». Принцип — **перед тим як розширити існуюче ім'я новим сенсом, зупинись і спитай чи це справді той самий намір**. Якщо ні — нове ім'я, не розширення. Дешевше створити нову сутність ніж потім розплутувати накопичені сенси.

### На трьох рівнях

**Прапор / параметр** — одне рішення. `skipCacheRead`, `skipCacheWrite`, не `skipCache` з подвійним сенсом. Якщо потрібні два рішення — два прапори.

**Поле структури** — одна відповідальність. `documentNature` про вміст файлу, `status` про lifecycle. Не складати в одне поле що змішує lifecycle і вміст («active», «scanned», «archived» в одному enum).

**Функція** — одна дія. `extractText` витягує. `writeCache` пише. Не `extractAndMaybeWriteCache` що залежно від прапора робить одне або обидва — композиція є задачею caller'а, ця явність і є цінність.

### Операційне правило

Коли вводиш новий прапор / поле / параметр — **на місці оголошення одне речення про єдиний сенс**. Не в commit-повідомленні. Не в окремому README. Прямо там:

```js
// options.skipCacheRead — пропустити пошук готового тексту в 02_ОБРОБЛЕНІ.
// Запис свіжого результату відбувається завжди коли є непорожній текст.
async function extractText(file, options = {}) { ... }
```

Якщо не можеш сформулювати сенс одним реченням без «АБО» / «а також» / «інколи» — абстракція зібрана неправильно, розділи на дві.

Це не про багато коментарів. Це про **момент паузи** — коли ти змушений сформулювати сенс словами, двозначність ловиться до того як потрапить у код.

### Що це НЕ означає

Не мінімізм. Не «короткі назви». `userWantsFreshOcr` довша за `forceFresh`, але читається як речення — лиши довшу. Дефенсивні перевірки, валідація на кордонах, говорящі коментарі — все лишається.

Принцип — про **однозначність елемента в момент коли його зустрічає читач**. Читач — наступна сесія Claude, інший розробник, ти сам через три місяці. Той хто не знає твого контексту в момент написання. Якщо такій людині сенс зрозумілий з імені і коментаря-визначення — мета досягнута.

### Історичний контекст (необов'язкове)

Peter Jennings (Microchess, 1976, KIM-1, 924 байти) і David Horne (1K ZX Chess, 1982, ZX81, <1024 байти) написали повноцінні шахи у менш ніж кілобайт. Жоден з них не міг дозволити прапор з двома сенсами — фізично не вистачило би місця на гілку «якщо інакше». Дисципліна виникала з обмеження.

У нас обмежень немає — гігабайти, вільні. Тому дисципліна має бути **свідомою**. Легше нашарувати сенс на існуюче ім'я ніж створити нове, саме тому це правило має бути письмовим.
