## 1. Що зроблено

Додано компонент `PdfJsViewer` (canvas + textLayer через pdf.js) як альтернативу Google Drive iframe для PDF документів у режимі «Скан». Drive `iframe preview` блокує текстовий шар (захист від скрейпінгу), через що адвокат не міг виділяти текст для копіювання. Тепер для PDF (mimeType=`application/pdf` або `*.pdf`) рендериться повноцінний HTML-документ: canvas з зображенням сторінки + невидимий шар `<span>` поверх з координатами кожного слова. Виділення тексту працює як у звичайному вебі — мишею, пальцем на тачскрін, потім стандартне копіювання. Lazy-render через `IntersectionObserver` (±500 px rootMargin) — для 50-сторінкових документів адвокат бачить placeholder-блоки, які замінюються на повний рендер при scroll'і. Зум через тулбар з кнопками −/+ (від 50% до 300%, крок 25%).

pdfjs підтягується через **динамічний `import('pdfjs-dist')`** всередині компонента — інакше юніт-тести `DocumentViewer.test.jsx` ламаються в jsdom (pdf.mjs читає `DOMMatrix` на module-level). У браузері динамічний import резолвиться миттєво — Vite попередньо бандлить chunk, тому затримки немає.

## 2. Логіка вибору рендеру

Розгортається у `ScanContent` (DocumentViewerContent.jsx):

| Тип файлу | Шлях рендеру |
|-----------|---------------|
| `image/*` (JPG, PNG, HEIC, WebP) | `<img src="https://drive.google.com/uc?export=view&id=...">` (без змін) |
| PDF (`application/pdf` або `*.pdf`) | `PdfJsViewer` з canvas + textLayer + zoom |
| Інше (DOCX, XLSX, …) | Drive iframe `https://drive.google.com/file/d/<id>/preview` (без змін) |

**Fallback на iframe:** якщо `PdfJsViewer` не зміг завантажити PDF (мережа, ZIP-замаскований PDF з ЄСІТС, пошкоджений файл) — викликається `onFatalError`, parent state `pdfJsFailed=true`, і ScanContent рендерить старий Drive iframe. Адвокат бачить preview без виділення замість порожнього екрану. State `pdfJsFailed` скидається при зміні `document.id`/`driveId` — інший документ отримує свіжий шанс на pdfjs.

## 4. Інструкція адвокату для тестування

### А — Текстовий PDF (Ухвала, Рішення суду)
1. Відкрий справу з документом який точно містить текстовий шар (наприклад скачана з ЄСІТС ухвала).
2. У досьє → вкладка Документи → відкрий документ. Режим за замовчуванням `Скан` (для scanned) або одразу контент (для searchable — там без перемикача).
3. Документ має зарендеритись через pdf.js (зверху тулбар з зумом і кількістю сторінок, не Drive iframe).
4. Тапни і протягни пальцем по тексту — рядок виділяється синім (var(--color-accent)).
5. Скопіюй (контекстне меню браузера або Ctrl/Cmd+C) — у буфері має бути виділений текст.
6. Перевір зум: `+` збільшує до 300%, `−` зменшує до 50%, поточне значення показано у тулбарі.

### Б — Сканований PDF (Адвокатський запит з печаткою)
1. Відкрий сканований документ (тепер `documentNature=scanned`, перемикач Скан/Текст видимий).
2. У режимі **Скан** — pdf.js рендер працює як у А, але виділення дасть мало (бо в скан-PDF текстовий шар або відсутній, або коротенький — лише там де реально є embedded text).
3. Перемкни на **Текст** — побачиш OCR-текст з 02_ОБРОБЛЕНІ (як було раніше).
4. Кнопки Footer (Перерозпізнати, Обговорити, …) працюють як раніше.

### В — Зображення (РНОКПП, скриншот)
1. Відкрий документ з mimeType `image/jpeg` або `.png`.
2. Має зарендеритись через `<img>` (як раніше) — без тулбара pdf.js.
3. Виділення тексту неможливе (це картинка) — це нормально.

## 5. Що НЕ зроблено і чому

- **Маркер/нотатки на тексті** — поза скопом. Зараз тільки виділення для копіювання. Маркер через анотації — окремий мікро-TASK з зберіганням у `documents_extended.json`.
- **Прибрати «стіну тексту» у режимі Текст для searchable** — поза скопом, вказано у завданні. Якщо адвокат потребує — окремий мікро-TASK.
- **Юніт-тести з реальним PDF rendering** — jsdom не підтримує canvas/DOMMatrix, тому повний інтеграційний тест неможливий без Playwright. Існуючі 422 тести зелені — DocumentViewer.test.jsx покриває mode-toggle і footer-кнопки, які не залежать від pdf.js рендеру.
- **Кешування PDF arrayBuffer між переключеннями** — не реалізовано. При повторному відкритті того самого документа `getDocument` ловить файл знову з Drive. Drive має HTTP кеш, тому реально 304. Якщо стане проблемою — додамо in-memory кеш по driveId.
- **Continuous scroll з пагінатором** — не зроблено окремий пагінатор; всі сторінки в одному скролі (lazy). Для типових судових документів 5-30 сторінок це зручніше за пагінатор. Якщо адвокат скаже що для 100+ сторінок незручно — додамо пагінатор.
- **Зміна `extractText` / `ocrService`** — не торкався. Це окремий шар для текстової копії в 02_ОБРОБЛЕНІ.
- **Footer кнопки** — не торкався. Вони працюють як раніше.
