## А. Питання які я ставив раніше (з обірваного промпту)

- (C) Soft banner у preview без блокування

### A2. Drag-and-drop у preview

- **(A)** `@dnd-kit/core` + `@dnd-kit/sortable` — ~15 KB gzip, lazy-loaded, touch + a11y, стандарт у React. Працює на iPad. **— Рекомендую**

- (B) Тільки 3+. Для 2 використовуємо порядок вибору адвоката без агента. Економія.

### A4. Стиль комітів

- **(A)** Інкрементальні (7 комітів як у TASK_B розділ 14). Push після кожного. **— Рекомендую (TASK B так і описує)**

## Б. Архітектура (мої рішення які хочу зафіксувати)

### Б1. Модульна структура converter

TASK каже "розширити converterService.js функцією `mergeImagesToPdf`". Існуючий pattern — окремий файл під кожен converter.

- **(A)** Фасад `convertImagesToPdf` у `converterService.js` делегує у НОВИЙ модуль `src/services/converter/multiImageToPdf.js`. Консистентно з docxToPdf/htmlToPdf/imageToPdf. **— Рекомендую**
- (B) Уся логіка у `converterService.js` (як буквально каже TASK)

### Б2. OCR concurrency

- **(A)** Послідовний OCR з прогрес-баром 1/N → 2/N → ...→ N/N. Document AI має rate limit (10 RPM concurrent), 50 паралельних викликів → 429 errors. Стабільніше + точний UI. **— Рекомендую**
- (B) Паралельно з обмеженням 3-5 одночасно (швидше, ~3× прискорення для 50 фото)
- (C) Повністю паралельно через Promise.all (ризик rate limit)

### Б3. JSON output Sonnet парсинг

- **(A)** System prompt каже `RESPOND ONLY WITH VALID JSON, NO PROSE`. Парсимо `JSON.parse(response)`. Fallback — regex `\{[\s\S]*\}` і повторна спроба. Якщо знову fail — пропускаємо сортування (адвокат бачить порядок вибору). **— Рекомендую**
- (B) Tool use API (більш надійно, але dragon: TASK явно каже "Простий JSON output, НЕ tool use")

### Б4. Preview thumbnails generation

- **(A)** `URL.createObjectURL(jpegBlob)` після rotation correction. Швидко. `revokeObjectURL` при закритті modal. **— Рекомендую**
- (B) `canvas.toDataURL('image/jpeg', 0.5)` (зменшені base64) — повільніше але менша memory pressure

## В. UI/UX питання які залишились нерозкритими

### В1. Що в preview thumbnails показуємо — оригінальне фото чи з виправленою орієнтацією?

- **(A)** З виправленою орієнтацією — те що піде у фінальний PDF. Адвокат бачить точно що отримає. **— Рекомендую**

### В2. Чи показувати OCR text fragment під thumbnail?

- (A) Так, перші 80 символів тексту під кожним thumbnail. Адвокат візуально верифікує що OCR розпізнав правильно. Більший grid row.
- **(B)** Ні, тільки thumbnail + badge "Сторінка X (warning)". Простіше, компактніше. Можна додати hover/tap tooltip з повним текстом. **— Рекомендую**

### В3. Якщо адвокат у preview видалив ВСІ зображення

- **(A)** Кнопка "Створити PDF" disabled. Toast "Додайте хоча б одне зображення". **— Рекомендую**
- (B) Закрити modal автоматично, повернути на стартовий екран

### В4. Кнопка "Назад" з preview

- **(A)** Назад на екран вибору файлів — preview state очищується, OCR результати втрачаються. **— Рекомендую (просто)**
- (B) Назад зберігає OCR результати, тільки дозволяє додати/видалити файли. Складніше — стан preview переплетений з file selection.

### В5. Чи можна додати ще файли після preview?

- (A) Так, "Додати ще зображень" кнопка у preview. OCR ranних зображень кешується, нові OCR'аться. Знову викликаємо агента.
- **(B)** Ні — щоб додати треба повернутись на старт і вибрати все знову. Простіше, рідкісний use case. **— Рекомендую**

### В6. Можна змішувати джерела (пристрій + Drive)?

- (A) Так — стартовий екран має обидва селектори, файли збираються у один список перед OCR
- **(B)** Ні — вибирається ОДНЕ джерело за сеанс. Простіше UX. Якщо треба змішувати — другий сеанс. **— Рекомендую**

### В7. Розмір thumbnails у preview grid

- **(A)** Grid 3 колонки на mobile (планшет), 4-5 на desktop. Thumbnail ~120×160 px. **— Рекомендую**
- (B) 2 колонки великі (200×280)
- (C) 6 колонок маленькі (80×110)

### В8. Auto-suggest назви документа з OCR

- (A) Так — після OCR першої сторінки витягуємо заголовок (перший рядок текстом >5 chars) і пропонуємо як назву. Адвокат може змінити.
- **(B)** Ні, поле порожнє за замовчуванням, адвокат заповнює вручну. Простіше, не плутає адвоката. **— Рекомендую**

## Г. OCR + Agent поведінка edge cases

### Г1. Якщо OCR падає на одному з зображень

- **(A)** Продовжуємо з рештою. Зображення позначається у preview як "OCR failed" warning з можливістю видалити. Текст для нього у .txt і .layout.json — порожній рядок. **— Рекомендую**
- (B) Зупиняємо весь процес, показуємо помилку, дозволяємо retry

### Г2. Якщо OCR повертає empty text для якогось зображення (фото без тексту, наприклад печатка)

- **(A)** Передаємо у agent з text=`""`. Агент розуміє що це сторінка без OCR-визначеного тексту, і використовує тільки структурні дані (розмір, EXIF). Зображення лишається у склейці. **— Рекомендую**
- (B) Pre-filter перед агентом (виключаємо empty-OCR зображення з вибору, але вони лишаються у preview)

### Г3. Якщо agent повертає невалідний order (duplicate indices, missing N, length != N)

- **(A)** Fallback на порядок вибору адвоката (без агента). Записуємо warning у logs. Адвокат бачить toast «Не вдалось проаналізувати порядок автоматично — встановіть вручну». **— Рекомендую**
- (B) Robust normalization (відрізати duplicates, доповнити missing) і використовуємо

### Г4. Token budget для агента при 50 зображеннях

50 фото × ~500 chars OCR тексту = ~25K input tokens.

- **(A)** Truncate text per image до 800 chars. Зберігаємо колонтитули (top/bottom), середину обрізаємо. Дешевше + швидше + достатньо для сортування. **— Рекомендую**
- (B) Повний text без truncation — точніше, але токени множаться
- (C) Перший рядок + останній рядок кожної сторінки (мінімум)

### Г5. Max output tokens агента

- **(A)** 1500. Для 50 зображень: order array (50 чисел), warnings (~5-10 об'єктів), missing string. Достатньо. **— Рекомендую**
- (B) 4000 (більше запасу)

## Д. Orientation correction edge cases

### Д1. Document AI повертає нестандартний angle (45°, 17°)

- **(A)** Округлюємо до найближчого з [0, 90, 180, 270]. Адвокатські фото зазвичай повертають на 90/180/270. **— Рекомендую**
- (B) Ігноруємо корекцію якщо angle != 0/90/180/270

### Д2. Якщо HEIC → JPEG конверсія втрачає EXIF orientation

heic2any зазвичай зберігає або «вирівнює» orientation. Тоді Document AI визначить як 0.

- **(A)** Не турбуємось — heic2any uniform-ить orientation. Document AI визначає по контенту (текст спрямований вверх). **— Рекомендую**
- (B) Додатково перевіряємо EXIF до конвертації і манульно обертаємо

### Д3. Фото landscape A4 — звичайне поведінкове

- **(A)** Document AI визначає orientation на основі тексту. Якщо текст спрямований вертикально на горизонтальному фото — повертаємо. Якщо фото справді landscape (наприклад план) — лишаємо. **— Рекомендую**
- (B) Hard-detect landscape (width>height) і завжди розглядаємо як landscape PDF сторінку
