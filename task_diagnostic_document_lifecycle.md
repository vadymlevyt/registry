## НАШІ ГІПОТЕЗИ (НЕ ДОГМИ)

Ми **підозрюємо** наступне, але це треба **перевірити в реальному коді**:

### Гіпотеза 1: Документи у Кісельова і Брановський — orphans

Адвокат раніше створював `case_context.md` для цих справ. У процесі контекст-агент читав файли з `01_ОРИГІНАЛИ`, робив OCR, зберігав текстові копії в `02_ОБРОБЛЕНІ`. **Але записів у `case.documents[]` для цих файлів немає** — бо контекст-агент не додає записи в реєстр, він тільки читає файли.

Коли адвокат потім додає той самий документ через "+ Додати документ" у Матеріалах:
- Система намагається завантажити файл у `01_ОРИГІНАЛИ` → файл з такою назвою вже існує → колізія → "Failed to fetch"?
- Або запис створюється, але без правильного `driveId` бо upload не вдався
- Або запис створюється, але посилання на `02_ОБРОБЛЕНІ` не правильне (NFC normalization?)

**Перевір:**
- Подивись `case.documents[]` для Кісельової і Брановського у registry_data.json (через console.log або через UI debug)
- Скільки записів? Скільки файлів реально в `01_ОРИГІНАЛИ`?
- Якщо є записи — чи мають вони коректні `driveId`, `documentNature`, посилання на копію в `02_ОБРОБЛЕНІ`?

### Гіпотеза 2: Видалення "повністю" не видаляє файли з Drive

Адвокат видалив документ з опцією "повністю". У реєстрі запис зник. Але файли в `01_ОРИГІНАЛИ` і `02_ОБРОБЛЕНІ` лишилися.

**Перевір:**
- Знайди ACTION `delete_document` у `App.jsx`
- Подивись що робить mode `'full'` — чи реально викликає Drive API для видалення файлів?
- Чи є виклики `gapi.client.drive.files.delete()` або аналогічні?
- Якщо немає — чому? Це навмисна логіка чи пропущена?

### Гіпотеза 3: Декілька шляхів створення файлів і записів — не синхронізовані

У системі ймовірно є **декілька різних шляхів** які пишуть файли в Drive і записи в реєстр:

- Document Processor (drag-n-drop) → пише в `01_ОРИГІНАЛИ` і додає в `case.documents[]`
- Контекст-агент при створенні `case_context.md` → пише в `02_ОБРОБЛЕНІ` (для своєї роботи), **не додає в реєстр**
- AddDocumentModal "+ Додати документ" → пише в `01_ОРИГІНАЛИ` і додає в `case.documents[]`
- Drag-n-drop у CaseDossier (стара логіка з TASK 1 patch) → пише в `01_ОРИГІНАЛИ` і додає в `case.documents[]` через `update_case_field`?

**Перевір:**
- Знайди **ВСІ** точки в коді які пишуть файли через Drive API в папки `01_ОРИГІНАЛИ` і `02_ОБРОБЛЕНІ`
- Знайди **ВСІ** точки які додають записи в `case.documents[]` через `add_document` або інший механізм
- Склади **карту** — які точки синхронізовані (пишуть і файл і запис), які тільки файл, які тільки запис
- Які точки використовують повну канонічну схему документа з TASK 1, а які — застарілу

### Гіпотеза 4: PDF rendering через Drive iframe не працює без auth

ScanContent рендерить через `https://drive.google.com/file/d/{id}/preview` — iframe Drive. Можливо Drive вимагає авторизації, а наш iframe не має cookies сесії, тому показує порожнє.

**Перевір:**
- Що реально рендериться в iframe? Подивись DevTools на справі Кісельова — який src, які запити, які помилки CORS/CSP
- Чи Drive iframe працює коли driveId дійсний і документ shared?
- Чи може Drive iframe працювати коли документ приватний (тільки в адвоката)?

Ймовірно тут потрібен альтернативний рендер для приватних файлів — pdf.js з blob URL.

### Гіпотеза 5: AddDocumentModal date picker не зачеплено

TASK 10.1 переписав native `<select>` на власний dropdown в Select компоненті. **Але date input** (`<input type="date">`) теж native — браузер показує свій календар (Android, iOS, Windows mають різні).

**Перевір:**
- В AddDocumentModal використовується `Input type="date"` — це native HTML date input
- Native date input показує системний календар (на Android — Android picker)
- Чи планувалось замінити на власний DatePicker компонент? Це не було в scope TASK 10.1.

### Гіпотеза 6: Архів — текст карток "з'їжджає"

Візуальна проблема. Імовірно Checkbox + Card + Buttons не вміщаються або не вирівнюються правильно при певній ширині.

**Перевір:**
- ArchiveView.jsx і ArchiveView.css — як організована картка
- Що відбувається на ширині лівої панелі (~280px)?
- Скріншот показує "П." замість "Позовна заява" — це truncate або overflow?
