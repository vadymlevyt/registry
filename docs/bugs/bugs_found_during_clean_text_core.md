**Дата:** 2026-06-01

## 1. (РОЗЧИНЕНО) DP cleaned-MD не персиститься для нарізаних документів

**Початкова знахідка (перша редакція 3.1):** тумблер «Очистити для читання»
чистив текст у `extractV3` ДО нарізки, а `splitDocumentsV3` для slice-документів
писав сирий per-page `.txt` → очищений MD відкидався (борг #16/#42).

**РОЗЧИНЕНО (поправка 2026-06-01, нова філософія адвоката):** очистка стала
ПОСТ-КРОКОМ — `cleanDocument` підключено ОСТАННІМ кроком у `splitDocumentsV3`
(після `writeProcessedArtifacts`, коли `.txt`+`.layout` у 02 готові) по кожному
готовому `scanned`-документу. Очистка строго ПІСЛЯ нарізки/склейки на роз'єднаних
документах → конфлікту меж нема, MD персиститься для slice і image_merge. Борг
#16 у `tracking_debt.md` позначено `✅ РОЗЧИНЕНО TASK 3.1`. Цей пункт лишається
як історичний слід — стандингового боргу більше немає.

## 2. Винос геометрії layout у спільний модуль (новий борг #43)

`cleanTextService.js` тримає локальну копію-дзеркало `blockBox`/`orderedBlocks`/
footer-евристики з `pageMarkers.js` (там вони module-private). Два місця з
геометрія-математикою = мʼяка напруга #11. Не виносив зараз: копія мала і чиста,
третього споживача ще нема (Rule of Three). → `tracking_debt.md #43` (тригер:
третій споживач geometry-хелперів або загальний рефактор DP-пайплайна).
