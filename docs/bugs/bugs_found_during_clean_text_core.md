# Знахідки під час TASK 3.1 (clean_text core)

**Дата:** 2026-06-01

## 1. DP cleaned-MD не персиститься для нарізаних документів (вже існуюча умова)

**Симптом (по коду, НЕ регресія 3.1):** Тумблер DP «Очистити для читання»
(`cleanForReading:true`) у `extractV3.js` рахує очищений текст і виставляє
`processedText`/`textFormat='md'` на `ctx.files[]`. Але `splitDocumentsV3.js`
для **нарізаних** документів **свідомо** бере сирий per-page `_text`
(`sliceProcessedArtifacts`, рядки 552-555) і пише `.txt` (`format:'txt'`
хардкод у `writeProcessedArtifacts`). Тобто очищений MD з `extractV3`
**не доходить до Drive** для slice-документів — Haiku-токени витрачаються,
а результат відкидається. У рідкому non-paged whole-file fallback очищений
текст пишеться, але теж як `.txt` без `textFormat`-метаданих.

**Чому НЕ виправлено в 3.1:**
- `flow_clean_text.md` і `splitDocumentsV3.js:552-555` явно **відкладають**
  інтеграцію clean+нарізка («окреме рішення адвоката — Варіант 1/2/3, доки
  сирий slicing не підтверджено»).
- Parent-заборона: не чіпати `splitDocumentsV3` понад потрібне; паралельна
  DP-сесія в сусідніх файлах (ризик гонки).
- 3.1 скоуп — ядро + консолідація AI-промпту + viewer + schema. Консолідація
  досягнута (inline-дубль `aiCleanText` → ядро); персистенція DP незмінна.

**Наслідок для адвоката зараз:** реальний шлях очистки наявних сканів дають
кнопки 3.2 (`cleanDocument` пише `.md` + переміщає `.txt` + видаляє layout).
DP-тумблер на нарізаних томах поки не зберігає `.md` (як і до 3.1).

**Тригер вирішення:** TASK після підтвердження page-precise slicing на реальній
справі — рішення Варіант 1/2/3 (чистити slice-текст / чистити post-persist /
чистити whole-file перед нарізкою). → `tracking_debt.md`.

## 2. Винос геометрії layout у спільний модуль

`cleanTextService.js` тримає локальну копію `blockBox`/`orderedBlocks`/
footer-евристики — дзеркало `pageMarkers.js` (там вони module-private).
Не виносив у спільний `layoutGeometry.js` зараз: торкатися робочого DP-пайплайна
під час паралельної DP-сесії = ризик гонки; копія мала і чиста. → `tracking_debt.md`
(тригер: третій споживач geometry-хелперів — Rule of Three).
