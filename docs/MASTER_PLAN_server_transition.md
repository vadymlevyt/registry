## 0. СЕРВЕРНА ЛІНЗА (наскрізний критерій)

Кожну роботу перепрочитуємо крізь один критерій — **що з нею стається на сервері**:

- **ПЕРЕЖИВАЄ** (портативне, переноситься як специфікація): CSS/дизайн-система, бізнес-логіка,
  block-модель Canvas, чисті детерміновані функції (паспорт/`pageMarkers`, `caseNoKey`, `sourcePolicy`,
  `documentFactory`, strip-layout), ACTIONS-контракт, DI-порти.
- **РОЗЧИНЯЄТЬСЯ** (сервер замінює безкоштовно): шар даних (`registry_data.json` full-replace → рядки+запити),
  оркестрація стану в `App.jsx`, Drive-обвʼязка Canvas (`_index.json`, lazy load/unload), 3-tier
  localStorage-кеші, in-memory job/resume-стан.
- **ПЕРЕРОБЛЯЄТЬСЯ** (rebuild під нову модель): секрети/проксі, важкий compute (PDF/OCR/canvas/AI) →
  worker, конкурентність (last-write-wins → транзакції+локи), білінг-ліміти (порадчі → примусові).

**Прецедент деферу** (`handoff_2026-06-23_metadata_a5_decisions_and_deferral.md`): «джерело даних → БД = rework».
Тому клієнтську передбудову того, що переїде в БД (метадані A5, Canvas-сховище, селективний контекст),
**свідомо не робимо** — це подвійна робота. Доктрину серверної лінзи винесено в `DEVELOPMENT_PHILOSOPHY.md`.

## 1. ПОРЯДОК СЕРВЕРНОЇ МІГРАЦІЇ (фази + блокери B1-B16)

**Хребет теми:** `docs/audits/audit_dp_server_migration_readiness.md` — 16 browser-bound блокерів,
3 з них безпеково-критичні (B1/B2/B3). Найцінніший актив: **DI-seam'и вже є** (`drivePort`,
`workerClient`, `callApi`/`callAgent`, фабрики `createActions`/`createDocumentPipeline`) —
міграція = підставити server-impl у наявні порти, а не переписувати споживачів.

### Напрям стеку (рішення/спайк — Фаза 0)
**PostgreSQL + один TS-бекенд + окремий worker для важких джоб + керований auth.**
Платформа: **Supabase** (менше сервісів: Postgres+RLS+Auth+Realtime+Storage «з коробки») **або**
Neon + окремий auth. **Окремий compute-шар обовʼязковий** — serverless не тягне 20-хв OCR/нарізку
(realtime-ліміти). Портативність — через стандартний Postgres + власний TS-репозиторій (не lock-in
у пропрієтарні примітиви). Спайк Фази 0: обрати платформу, підняти локальний dev-цикл з hot-reload +
deploy пакетом (окупається негайно, ще до серверу — `admin_opinion_pre_server_work` §9).

### Фаза 1 — Auth + проксі секретів (закриває B1/B2/B3; система стає ПРОДАВАНОЮ) — *rebuild*
Головний SaaS-блокер. До нього multi-tenant неможливий (клієнт тримає робочі ключі).
- **B1** Anthropic-ключ (`claude_api_key` у localStorage, `anthropic-dangerous-direct-browser-access`)
  → усі AI-виклики через backend-проксі; клієнт ключа не бачить.
- **B2** Google Document AI (Bearer-токен адвоката з браузера) → server service-account; клієнт шле байти.
- **B3** Drive OAuth access-token (`levytskyi_drive_token`) → server тримає refresh-token per-tenant; Drive I/O на бекенді.
- **Наслідок:** білінг-ліміти (`subscription.limits`) стають **примусовими**, а не порадчими (§4 Billing).
- Заголовки `anthropic-dangerous-direct-browser-access` і прямі `fetch` зникають (інвентар точок — audit §2.3, готовий чек-лист).

### Фаза 2 — Postgres + RLS (міграція `registry_data.json` → таблиці) — *rebuild шару даних*
- `registry_data.json` (full-replace PATCH) → нормалізовані таблиці; `cases`/`documents`/`hearings`/
  `time_entries`/`ai_usage`/`auditLog` — рядки. Оркестрація `App.jsx` **розчиняється** у запити.
- **RLS = ізоляція за замовчуванням + явний шар** (взірець уже спроектовано для Canvas —
  `canvas_server_data_model.md`): видно там, де `owner_id=ти` АБО є рядок shares, усе в межах `tenant_id`.
- Закриває **B15** (agentHistory без localStorage-tier), **B61** (конкурентний запис): full-replace →
  транзакції+рядкові локи; дедуп `case_no` стає атомарним (серверний арбітр), не in-memory.
- **Реляційна схема-насіння** (`consultation_metadata_graph_ontology.md`): 9 сутностей first-class,
  Party/Court/Judge поки вільний текст → **person-registry** як важіль графа (борг #84, слот у схемі, збір даних defer).

### Фаза 3 — важкий compute на worker (B4-B11, B16) — *rebuild запуску, логіка переживає*
Логіка вже середовище-агностична за DI-портами (audit §4) — переїзд = server-impl, не переписування споживачів.
- **B4** worker PDF split/merge (pdf-lib середовище-агностичний → Node); **B5** image→PDF (sharp+pdfkit);
  **B6** HEIC (libheif/sharp); **B7** DOCX (LibreOffice headless); **B8** HTML (headless Chromium);
  **B9** стиснення сканів (pdfium+jpeg); **B10** фото-склейка (sharp/OpenCV); **B16** ZIP-unzip (fflate у Node).
- **B11** claudeVision — dormant + гілка `extractMetadata` мертва → спершу видалити (leanness), не мігрувати мертве (audit §7.1).
- DocAI на бекенді відкриває природну точку **інструментації OCR-витрат per-tenant** (зараз невидимі).

### Фаза 4 — job-стан/resume + прибрати ефемерність (B12-B14) — *rebuild стану*
- **B12** OCR resume-store (in-memory Map) · **B13** jobProgressStore (RAM) · **B14** resume-асиметрія
  (durable лише нарізка; склейка/додати/ZIP — React state) → server job-queue зі статусом; клієнт поллить.
- Нарізка вже наполовину server-ready (`jobState.json` на Drive — взірець); поширити на всі сценарії.
- Тут же закриваються борги **#82** (кнопка «Скасувати» + abort зовнішніх викликів) і **#83**
  (жорсткий таймаут/видимий стан завислого AI-виклику) — на стабільному серверному каналі це справжній фікс класу.
- **Відкрите питання** (audit §7.6): чи durable resume потрібен на всі сценарії, чи дешевий re-run прийнятний — продуктове рішення.

### Порядок і що паралельно
**Canvas і Billing/CRM UI зав'язані лише на хребет (Фази 1-2), НЕ на DP-compute (Фази 3-4)** →
після Фаз 1-2 можуть іти **паралельно** з Фазами 3-4. DP вважається **готовим до переїзду** (A7 ✅ 2026-06-24) —
його **не допилюємо** до серверу, переносимо як є за DI-портами.

## 2. РЕФАКТОРИНГ (Вісь C) — що з міграцією, що після

- **`App.jsx` (~5681) переважно РОЗЧИНЯЄТЬСЯ** на сервері (оркестрація стану/hydration/persist →
  бекенд+запити). Тому масштабний спліт App.jsx «заради спліту» — низький пріоритет; природно
  розсмокчеться при Фазі 2. Виносити зараз лише те, що переживає (QuickInput/AnalysisPanel як компоненти).
- **`CaseDossier` (~3079) / `Dashboard` (~2710) ПЕРЕЖИВАЮТЬ** (UI-логіка) → спліт **разом з дизайном
  Рівень 2** (один прохід = стилі + структура, `handoff_2026-06-27_ui_design_unification`), не окремо.
- **Дизайн Рівень 0 — можна ЗАРАЗ** (не чекає серверу): звести два конкуруючі набори токенів
  (App.css короткі ↔ tokens.css довгі) в один SSOT через аліаси `--text: var(--color-text)`.
  Рівень 1 (dark-theme + per-tenant брендинг) — SaaS-гачок, після хребта.
- Дрібне за тригером редагування: `ImageMergePanel` вже зроблено (2822→773); правило #12 «файл >800 → спліт».

## 3. ДОПИЛЮЄМО vs РОБИМО НАНОВО — ТАБЛИЦЯ

| Фіча / шар | Вердикт | Одне речення чому (серверна лінза) |
|-----------|---------|-------------------------------------|
| Document Processor (нарізка/склейка/додати/ZIP) | **на сервер як є** | A7 ✅; логіка за DI-портами переживає, browser-прив'язка — лише 5 точок реалізації портів |
| Секрети (Anthropic/DocAI/Drive) | **наново (проксі)** | клієнт не може тримати робочі ключі під multi-tenant — переробляється |
| `registry_data.json` full-replace | **наново (Postgres+RLS)** | шар даних розчиняється у рядки+запити; full-replace = втрата даних при конкуренції |
| Оркестрація `App.jsx` (hydration/persist/save) | **розчиняється** | стан переїжджає на бекенд; не рефакторити «наперед», зникне при Фазі 2 |
| Важкий compute (PDF/OCR/canvas/конвертери) | **на сервер (worker)** | той самий код у Node за фасадами; browser-bound лише спосіб запуску |
| Білінг-інструментація (25 точок) | **наново (event-stream)** | ручне дублювання логів → один `recordActivity` emitter, проєкції (борг #86) |
| Метадані A5 (author/date/category) | **наново на сервері** | джерело тексту → БД = rework; невидиме автозаповнення = серверна модель |
| Селективний контекст (повний/вижимка/не-вкл + бюджет) | **на сервері** | `participation` = персистентне поле (не інференс), звірити з `isKey` (борг #87) |
| Canvas-сховище (`_index.json`/lazy load) | **розчиняється** | у БД це рядок+запит; block-модель/тули/редактор переживають як спека |
| 3-tier agentHistory (localStorage-tier) | **наново (server-cache)** | per-browser tier під multi-user = функц. баг; лишити server+Drive |
| claudeVision OCR-fallback + extractMetadata | **видалити перед міграцією** | dormant/мертве — не мігрувати мертве (leanness) |
| Дизайн-токени / CSS | **переживає (Рівень 0 зараз)** | незалежне від шару даних; звести SSOT можна негайно |
| CaseDossier/Dashboard спліт | **переживає (з Рівнем 2)** | один прохід стилі+структура; не окремо від дизайну |
| B2 (міграція fetch-точок на `callAgent`) | **зараз/принагідно** | шов уже є (B1 ✅ `callAgent`); чат-агенти QI/Dashboard/Dossier — не чіпати |

## 4. НОВІ МОДУЛІ — коли і в якому порядку

### Canvas — конструктор документів (на хребет Фаз 1-2; дата-модель уже є)
Серверна дата-модель **збережена**: `canvas_server_data_model.md` (5 таблиць + RLS: `canvas_documents`/
`_versions`/`_shares`/`_attached`/`_sessions`; ізоляція за замовчуванням + явний шар shares; body-as-jsonb).
Візії v1.0/v1.5 — `context_canvas_constructor_v1.md`, `canvas_context_model_v1_5.md`. **Переживає:** block-модель,
тули (нові ACTIONS через той самий `executeAction`), редактор-фасад, експорт `.docx`/PDF, шаблони-прапорець,
прив'язані в'юер-документи. **Синергія:** mermaid-діаграми (борг #50), ґрунтовані на структурованих полях
(`parties`/`processParticipants`/`movementCard`). Спільне редагування — **опційна фіча**, не ядро.

### Billing / CRM — новий модуль (підготовка вже збирається)
- Візія: **RETAIN `context_billing_crm_v1_5.md`** (найповніша; 13 тез уже в `DEVELOPMENT_PHILOSOPHY.md`).
  Архітектура подій→білінг: `recommended_activity_event_billing_architecture.md`. Модель $/токен для
  tariff matrix + `subscription.limits`: `TASK_ai_cost_analysis_and_forecast.md`.
- **Серверна форма:** один потік фактів (event-sourcing) → `time_entries[]` і `ai_usage[]` = **проєкції**,
  а не паралельні структури з ручною інструментацією (борг #86). Ліміти стають **примусовими** після Фази 1.
- CRM — не модуль, а **зріз** того самого потоку (теза 3). Storage-quota foundation — зі стиснення
  (`TASK_file_tools_compression_doctrine`: 1800px/JPEG 0.7 ≈ 2 МП/стор).

### Кандидати з зіпів (оцінка: парк / план)
- **Telegram-інгест/канал** — 🟢 **план** (обовʼязкова функція, слова адвоката). RETAIN
  `context_telegram_integration_v2.md`: 3 модулі (клієнтський канал / внутрішні групи / enterprise-групи
  з фільтром конфіденційності). Архітектурний бонус: downscale фото **на вході** (серверно) закриває
  борги #40/#41 у корені. Потребує бекенда → після Фази 1. Batches API (−50%, борг #45) теж вмикається бекендом.

  розвідка→банк ідей→бібліотека→генерація→відеопайплайн→автопостинг; 3 ітерації ($0 / $47 / $94 міс).

- **Main-agent + інституційна пам'ять** — 🅿️ **парк** (амбіція Ітерації 3). RETAIN
  `architecture_agents_hierarchy_v1.md`: Opus головний над горизонтальними; 3 шари памʼяті; `logDecision()`
  як точка збору досвіду; RAG (pgvector) після 50+ документів. Enabling-крок — інфра `logDecision` (борг #88).
  Гібридна агентна архітектура (розмовний + технічні під-агенти) уже в `DEVELOPMENT_PHILOSOPHY.md`.
