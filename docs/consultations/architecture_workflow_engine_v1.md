# ARCHITECTURE: Workflow Engine — створення справ і призначення адвокатів

**Дата фіксації:** 2026-04-28
**Версія:** 1.0
**Статус:** архітектурне бачення підсистеми, не TASK для негайного виконання
**Контекст:** базис для multi-tenant Legal BMS, готовність до масштабування з соло до великих юрфірм

---

## КОНЦЕПЦІЯ І ФІЛОСОФІЯ

### Що це таке

**Workflow Engine** — це система гнучких правил які визначають:

1. **Хто може створювати справи** в Legal BMS
2. **Чи потрібен акцепт** для створення справи (хто акцептує)
3. **Хто може призначати адвокатів** на справи
4. **Чи потрібен акцепт** для призначення (хто акцептує)
5. **Що робить адвокат** коли його призначили (підтверджує отримання, відмовляється)

Все це налаштовується **на рівні юр.особи** (tenant) і не вимагає переписування коду при зміні організаційної структури.

### Філософія — гнучкість на рівні даних, простота на рівні UI

Це **класичне правило архітектури** SaaS:

> **Дані гнучкі і розширювані. UI простий і цілеспрямований.**

Чому це важливо.

**Антипатерн SaaS-розробки:** спочатку зробити складний workflow engine з графічним редактором правил, через рік виявити що ним користуються 3% користувачів, видалити, переписати простіше.

**Уникнення антипатерну:** закладати **гнучкість на рівні моделі даних** (поля в `tenant.case_creation_rules` і так далі), а **на рівні UI** робити прості і прямі форми. Коли ринок покаже що workflow реально потрібен у складнішому вигляді — UI робиться за тиждень, бо дані вже готові.

### Принцип «закласти зараз, активувати пізніше»

Зараз АБ Левицького — це соло-практик. Workflow Engine для нього — простий випадок (керівник створює, без акцепту). Жодних складнощів.

**Але архітектура** має закладати моделі даних які підтримують **майбутні сценарії**:
- Партнер в АБ → акцепт для рядових адвокатів
- АО з кількома адвокатами → робота кожного зі своїми справами
- Велика юрфірма з куратором практик → трирівнева ієрархія

Якщо закласти зараз — потім просто додаємо налаштування. Якщо не закласти — переробляти структуру даних = міграція БД, проблеми з існуючими даними, тиждень-два роботи.

## ЕВОЛЮЦІЯ КОНЦЕПЦІЇ В ОБГОВОРЕННЯХ

### Перший варіант — централізована модель

**Тільки керівник заводить справи.** Адвокат-виконавець підхоплює готову.

**Переваги:** контроль, жодна випадкова справа не потрапить у систему.

**Недоліки:** бутилгорло — адвокат чекає на керівника. Якщо керівник у суді/у відпустці — справи стоять.

### Другий варіант — децентралізована модель

**Адвокат сам заводить, керівник підтверджує.**

**Переваги:** гнучкість, прозорість через історію.

**Недоліки:** треба робити робочий процес підтверджень.

### Третій варіант — комбінована модель (фінальний вибір)

Засновник запропонував:

- Адвокат заводить справу-чернетку (`status: draft`)
- Тобі приходить сповіщення «Кульчицький створив чернетку справи X, потребує підтвердження»
- Ти переглядаєш, акцептуєш → справа стає активною
- Або відмовляєш → справа архівується з причиною
- В чернетку можна вже додавати документи, нотатки, працювати — статус `draft` означає що справа не йде у звітність бюро і не моніториться через Court Sync

Це **збалансоване** рішення. Гнучкість + контроль.

### Розширення на куратора практики

Засновник додав сценарій великої юрфірми з куратором практик:

> «Якщо це велика юридична компанія, можна поширити сюди трьохступеневу структуру. Тобто справу може завести і керівник, і куратор практики, і рядовий адвокат. У випадках якщо завожу її я, або керівник практики — він же не означає і виконавця. Або двох виконавців. Двох адвокатів на справу.»

> «У трьохступеневому варіанті мабуть, коли керівник практики призначає адвоката, створює справу — я теж мабуть маю схвалити. І коли він призначає адвоката на справу, я теж мабуть маю схвалити цей момент. Це з метою контролю все ж таки.»

Тобто **трирівнева ієрархія** (head → curator → lawyer) з різними правилами акцептів для кожного рівня.

### Принципова позиція засновника

Засновник правильно сформулював:

> «Зараз на цьому йти заморочуватись, детально виписувати, ніяк воно має бути. В принципі, загальну архітектуру воно лягає просто.»

Тобто **закладемо гнучкість на рівні даних**, а конкретні правила — налаштовуються пізніше при реальному масштабуванні. Не намагаємось зараз продумати всі можливі варіанти і прописати їх у коді.

## МОДЕЛІ ДАНИХ

### `tenant.case_creation_rules`

Налаштування workflow конкретного tenant (АБ, АО, юрфірми):

```javascript
tenant.case_creation_rules = {
  by_head: "auto_approve",         // керівник створює без акцепту
  by_curator: "auto_approve" |     // куратор автоматично
              "head_approval_required",  // або з акцептом керівника
  by_lawyer: "head_approval_required" |          // акцепт тільки керівника
             "curator_approval_required" |       // акцепт тільки куратора
             "head_or_curator_approval_required" // акцепт будь-кого старшого
}
```

### `tenant.assignment_rules`

Налаштування призначення адвокатів на справи:

```javascript
tenant.assignment_rules = {
  by_head: "auto_approve",
  by_curator: "head_approval_required" | "auto_approve"
}
```

### `case.approval_workflow`

Стан акцепту конкретної справи:

```javascript
case.approval_workflow = {
  status: "draft" | "pending_approval" | "active" | "rejected",
  created_by: lawyer_id,
  created_at: timestamp,
  approval_required_from: ["head" | "curator" | "none"],
  approved_by: [
    { lawyer_id, role, timestamp, comment }
  ],
  rejected_by: { lawyer_id, role, timestamp, reason } | null
}
```

### `case.lawyers[]`

Хто веде справу і в яких ролях:

```javascript
case.lawyers = [
  {
    lawyer_id: "kulchytsky",
    role: "lead" | "co-lead" | "support",
    added_by: "head",
    added_at: timestamp
  }
]
```

Адвокатів у справі може бути:
- **1** (соло)
- **2** (стандарт АБ — керівник + виконавець, що дає основу для маркетингу «два адвокати»)
- **3+** (складна велика справа з командою)

### `case.lawyer_assignments[]`

Історія всіх призначень з акцептами:

```javascript
case.lawyer_assignments = [
  {
    id: "...",
    lawyer_id: "kulchytsky",
    role: "lead",
    proposed_by: "curator_xx",
    proposed_at: timestamp,
    status: "pending" | "approved" | "rejected" | "withdrawn",
    approved_by: "head" | null,
    approved_at: timestamp | null,
    rejected_by: null,
    rejected_at: null,
    rejected_reason: null,
    withdrawn: false,
    acceptance_confirmed_at: null  // коли адвокат підтвердив що прийняв до виконання
  }
]
```

### `users[]` — ролі і tenant

```javascript
users[i] = {
  lawyer_id: "kulchytsky",
  tenant_id: "ab_levytskoho",
  full_name: "Кульчицький А.Б.",
  role: "head" | "practice_curator" | "lawyer",
  practice: "criminal" | "civil" | "administrative" | "any",  // якщо curator
  active: true,
  joined_at: timestamp,
  left_at: null
}
```

## РОЛІ В СИСТЕМІ

### `head` — Керівник tenant

**Можливості:**
- Створювати справи (`auto_approve`)
- Призначати адвокатів на справи
- Бачити **всі** справи tenant
- Налаштовувати правила (`tenant.case_creation_rules`, `tenant.assignment_rules`)
- Управляти користувачами (додавати, видаляти, змінювати ролі)
- Делегувати свої акцепт-права іншому користувачу (фаза 4+)

**Хто це:**
- В АБ — керуючий адвокат-керівник
- В АО — керуючий партнер
- В юрфірмі — керуючий партнер

### `practice_curator` — Куратор практики (опціонально)

**Можливості:**
- Створювати справи в межах своєї практики (можливо з акцептом)
- Призначати адвокатів на справи в межах своєї практики (можливо з акцептом)
- Бачити справи **своєї практики**
- Не бачить справи інших практик і не може ними керувати

**Хто це:**
- Партнер відповідальний за криминальну/цивільну/адміністративну практику в великій юрфірмі
- В АБ і малих АО зазвичай **відсутній**

### `lawyer` — Рядовий адвокат

**Можливості:**
- Створювати справи (потребує акцепту)
- Підтверджувати отримання справи яку йому призначили
- Відмовлятись від призначення з причиною
- Бачити **тільки свої справи** (де він в `case.lawyers[]`)
- Може мати додаткове право «огляд» на конкретні справи (керівник додав як спостерігача)

**Хто це:**
- В АБ — адвокат-виконавець (партнер договору АБ)
- В АО — рядовий адвокат-учасник
- В юрфірмі — асоційований юрист

## СЦЕНАРІЇ ЗА РОЗМІРОМ ОРГАНІЗАЦІЇ

### Сценарій 1 — Соло (АБ Левицького зараз)

Один користувач — Левицький В.А. з роллю `head`.

```javascript
tenant.case_creation_rules = {
  by_head: "auto_approve"
}
tenant.assignment_rules = {
  by_head: "auto_approve"
}
```

**Як працює:**
- Левицький створює справу — одразу `status: active`
- Призначає себе як `lead` — одразу

**UI:** проста кнопка «Створити справу». Без чекбоксів акцептів. Без сповіщень.

**Це режим за замовчуванням** для будь-якого нового tenant. Тільки коли додається другий користувач — пропонується перейти на режим з акцептами.

### Сценарій 2 — Малий АБ з партнером

Двоє користувачів — Левицький (`head`) і Кульчицький (`lawyer`).

```javascript
tenant.case_creation_rules = {
  by_head: "auto_approve",
  by_lawyer: "head_approval_required"
}
tenant.assignment_rules = {
  by_head: "auto_approve"
}
```

**Як працює:**

*Сценарій 2А — Левицький створює справу:*
1. Натискає «Створити справу»
2. Заповнює дані
3. Призначає Кульчицького як `lead`, себе як `co-lead` (або тільки одного)
4. Справа одразу `active`, призначення одразу `approved`
5. Кульчицький отримує сповіщення «Вас призначено вести справу X»
6. Кульчицький переглядає, натискає «Прийняв до виконання» або «Відмовляюсь»

*Сценарій 2Б — Кульчицький створює справу:*
1. До нього прийшов клієнт, паперові матеріали
2. Кульчицький натискає «Створити справу-чернетку»
3. Заповнює дані, додає документи, нотатки
4. Натискає «Подати на підтвердження»
5. Левицький отримує сповіщення «Кульчицький створив чернетку справи X»
6. Левицький переглядає, натискає «Підтвердити» або «Відмовити»
7. Якщо «Підтвердити» — справа `active`
8. Якщо «Відмовити» — справа архівується з причиною

**UI:**
- Левицький: кнопка «Створити справу» + список «Потребує підтвердження» (на дашборді)
- Кульчицький: кнопка «Створити справу-чернетку» + список «Мої чернетки» з статусом

### Сценарій 3 — АО середнього розміру

Керуючий партнер + 3-5 адвокатів. Структура як Сценарій 2, але:
- Керуючий партнер більш зайнятий, не може швидко відповідати на акцепти
- Можна додати **другого керівного адвоката** який теж може акцептувати

```javascript
users[i].role = "head"  // обидва керівні партнери
```

В обидвох — `auto_approve` для створення. Будь-який з них може акцептувати справу від рядового адвоката.

**Сповіщення:** обом керівним партнерам. Хто перший побачить — той акцептує. Інший бачить що вже акцептовано і не дублює дію.

### Сценарій 4 — Велика юрфірма з куратором

10-30+ адвокатів. Структура:
- Керуючий партнер — `head`
- Партнери-куратори практик — `practice_curator` (кримінальна, цивільна, адміністративна)
- Рядові адвокати — `lawyer`

```javascript
tenant.case_creation_rules = {
  by_head: "auto_approve",
  by_curator: "head_approval_required",  // куратор створив — керівник схвалює
  by_lawyer: "head_or_curator_approval_required"  // адвокат — будь-хто старший
}
tenant.assignment_rules = {
  by_head: "auto_approve",
  by_curator: "head_approval_required"  // куратор призначив — керівник схвалює
}
```

**Як працює:**
- Адвокат створює чернетку → керівник АБО куратор підтверджує
- Куратор створив справу → керівник підтверджує
- Куратор призначив адвоката на справу → керівник підтверджує

**Принцип засновника:**
> «Має бути контроль все ж таки. Але не перевантажувати купою оцих свалювань. Можна все зав'язати на мені, якщо це вже дуже велике об'єднання, тоді просто обертаю цю функцію на інших керуючих партнерів.»

Тобто при дуже великих обсягах:
- Або все на керуючому партнерові
- Або делегування на кількох керуючих партнерів (кожен з своєю практикою як `head` для своїх справ)

Це налаштовується **через дані**, без зміни коду.

### Сценарій 5 — Корпоративний юридичний департамент

Юрвідділ великого банку чи корпорації. Структура:
- Керівник юр.відділу — `head`
- Старші юристи — `practice_curator` (для різних напрямків — корпоративне, трудове, договірне)
- Рядові юристи — `lawyer`

Особливості:
- Робота не з ордерами адвокатів, а як представництво юр.особи
- Інтеграція з АРІ ЕС (Канал 2) — справи через корпоративний кабінет
- Жорсткіший контроль — практично всі справи з акцептом

```javascript
tenant.case_creation_rules = {
  by_head: "auto_approve",
  by_curator: "head_approval_required",
  by_lawyer: "head_approval_required"  // тут тільки керівник, не куратор
}
```

## ПІДТВЕРДЖЕННЯ ВИКОНАВЦЕМ ОТРИМАННЯ СПРАВИ

### Що це

Окремий від акцепту створення механізм — коли адвоката **призначили** на справу, він повинен **підтвердити** що прийняв до виконання.

Це **не акцепт призначення** (то справа керівника). Це **підтвердження від виконавця** що він взявся.

### Як працює

1. Левицький призначає Кульчицького як `lead` на справу X
2. Кульчицький отримує сповіщення «Вас призначено вести справу X. Прийняти до виконання?»
3. Кульчицький переглядає матеріали (досьє вже є з документами і даними)
4. Натискає **«Прийняв до виконання»**
5. В системі фіксується `acceptance_confirmed_at`
6. Тепер справа повністю «його» — він відповідає за подальші дії
7. Якщо не натискає **N днів** (наприклад, 3 робочих) — повідомлення керівнику «Кульчицький не підтвердив отримання справи X»

### Чому це важливо

**Юридична відповідальність:** в АБ керівник відповідає за справи. Якщо адвокат отримав справу, але не підтвердив, не знає що йому її призначили — справа може зависнути. Дедлайн пропущено = дисциплінарна відповідальність.

**Прозорість:** видно хто коли отримав, хто пасивний.

**Підстраховка від «забув»:** нагадування виштовхує адвоката взяти справу або відмовитись.

### Підмножина — відмова від справи

Якщо адвокат **не може** взяти справу (конфлікт інтересів, перевантаження, особисті обставини):
- Натискає «Відмовляюсь від справи»
- Заповнює причину (вільний текст)
- Справа повертається керівнику для переназначення
- Справа залишається `active` в системі, але без активного виконавця
- В UI керівника — банер «Справа X залишилась без виконавця, причина: ...»

## СЦЕНАРІЇ ПЕРЕДАЧІ І ЗМІНИ ВИКОНАВЦЯ

### Передача справи від одного адвоката до іншого

Кульчицький веде справу. Іде у відпустку на 3 тижні. Треба тимчасово передати Корнієнку.

**Опція А — Тимчасове призначення:**
- Левицький додає Корнієнка як `co-lead` з міткою «тимчасово до 25.05.2026»
- Кульчицький залишається `lead` але неактивний
- Корнієнко отримує доступ до справи на цей період
- Після 25.05.2026 — Корнієнка автоматично прибирається

**Опція Б — Постійна передача:**
- Левицький змінює роль Кульчицького на `support` (зберігає історію)
- Призначає Корнієнка як новий `lead`
- Корнієнко підтверджує отримання
- Кульчицький бачить справу в архіві своїх (для історії), але без активних дій

### Звільнення адвоката

Кульчицький звільняється з АБ.

**Що відбувається:**
- Левицький в адмін-панелі ставить `users[Кульчицький].active = false`
- Доступ Кульчицького відключається
- Його сесія видаляється
- Всі справи де він `lead` — перевіряються:
  - Якщо є `co-lead` — той стає `lead`
  - Якщо немає — справа в стані «без активного виконавця», банер для керівника

**Дані:**
- Все що Кульчицький завів в Legal BMS — залишається власністю АБ
- Чернетки які він створив, але не були акцептовані — переходять до керівника для рішення (акцептувати або архівувати)
- Аудит-журнал зберігається

### Зміна ролі адвоката

Кульчицький росте по службі — стає партнером.

- Левицький змінює `users[Кульчицький].role` з `lawyer` на `practice_curator`
- Якщо `case_creation_rules.by_curator: "auto_approve"` — Кульчицький тепер може створювати справи без акцепту
- Призначається `practice` для нього («криминальна» наприклад)
- Бачить тепер всі справи своєї практики, не тільки свої

## ЯК ЦЕ ВПЛИВАЄ НА ІНШІ МОДУЛІ

### Зв'язок з Court Sync

```
Поки case.status = "draft" — справа НЕ у whitelist Court Sync
   - Court Sync не моніторить її
   - Не шукає документи в кабінетах адвокатів
   - Не оновлює дані з ЄСІТС

Тільки status = "active" → справа в whitelist
   - Court Sync моніторить
   - Документи з кабінетів синхронізуються
   - Email-фільтр Gmail оновлюється з номером справи
```

Це **захист** від ситуації коли адвокат створив чернетку в Legal BMS, але клієнт ще не підписав договір з бюро. Поки `draft` — система не торкається ЄСІТС.

### Зв'язок з білінгом

```
Time entries автоматично створюються тільки для status = "active" справ
   - Якщо адвокат працює над draft — час фіксується в "general" або "internal_admin"
   - Коли draft → active, можна перенести записи в справу

Інвойси формуються тільки для активних
Аналітика не враховує draft справи
```

Коли draft → active:
- Time entries від моменту створення можуть автоматично переноситись (опціонально, з ракрита адвокатом «застосувати з моменту створення»)
- Або починаються свіжі від моменту акцепту

### Зв'язок з досьє

В досьє чернетки справ — повний функціонал:
- Можна додавати документи
- Створювати нотатки
- Працювати з агентом досьє (він не знатиме що це чернетка — для нього просто справа)

Просто маркер `draft` показує що справа ще не активна. Чернетки видимі тільки тим хто їх створив + тим у кого є акцепт-роль (head, curator).

### Зв'язок з системним промптом v2

При генерації `case_context.md` (`TASK_system_prompt_v2.md`):
- Метадані містять `lawyer_id` і `tenant_id`
- Якщо справа multi-source — у промпті фігурує «Команда адвокатів: ...»
- Агент знає хто клієнт, хто веде справу

Кожна дія по справі і призначеннях логується:

```javascript
case.activity_log = [
  {
    timestamp: "2026-04-28T14:23:00Z",
    actor: "kulchytsky",
    action: "case_created_as_draft",
    details: { name: "...", number: "..." }
  },
  {
    timestamp: "2026-04-28T15:30:00Z",
    actor: "head",
    action: "case_approved",
    details: { from_status: "draft", to_status: "active", comment: "OK, починай" }
  },
  {
    timestamp: "2026-04-28T15:31:00Z",
    actor: "head",
    action: "lawyer_assigned",
    details: { lawyer: "kulchytsky", role: "lead" }
  },
  {
    timestamp: "2026-04-28T16:00:00Z",
    actor: "kulchytsky",
    action: "assignment_accepted",
    details: { case_id: "..." }
  },
  {
    timestamp: "2026-05-01T09:00:00Z",
    actor: "head",
    action: "lawyer_added",
    details: { lawyer: "kornienko", role: "co-lead", reason: "відпустка Кульчицького" }
  }
]
```

Це **аудит-журнал** який видно керівнику. Прозорість без шпигування — тільки факти дій.

### Хто бачить аудит

- `head` — повний аудит всіх справ tenant
- `practice_curator` — аудит справ своєї практики
- `lawyer` — аудит тільки **своїх** справ (де він в `case.lawyers[]`)

## РОЗШИРЕННЯ В МАЙБУТНЬОМУ

### Делегування акцептів

**Сценарій:** керівний партнер у відпустці на тиждень. На цей час **передає право акцептувати** заступнику.

**Технічно:**

```javascript
tenant.delegations = [
  {
    from: "head",
    to: "curator_civil",
    permissions: ["approve_case_creation", "approve_assignment"],
    valid_from: "2026-05-01",
    valid_until: "2026-05-08",
    reason: "відпустка"
  }
]
```

При перевірці права акцептувати:
1. Чи у користувача є базова роль з цим правом?
2. Якщо ні — чи є делегування на цей період?
3. Якщо так — дозволити, але в логи писати «акцептував curator_civil за делегуванням від head»

Реалізація — **фаза 4+**. Зараз не критично.

### Email і Telegram сповіщення про акцепти

Сповіщення тільки in-app зараз. Пізніше — email на адресу керівника + Telegram-повідомлення.

```javascript
notification = {
  user_id: "head",
  type: "case_approval_pending",
  case_id: "...",
  channels: ["in_app", "email", "telegram"],  // де доставити
  urgency: "normal" | "high",
  created_at: timestamp,
  read_at: null,
  acted_on_at: null
}
```

Якщо в `urgency: high` (наприклад, адвокат подав на акцепт справу яка має дедлайн через 3 дні) — крім in-app, шлеться Telegram з фразою «Терміново, дедлайн 3 дні».

### Кастомні ролі

Якщо в дуже великих юрфірмах потрібні специфічні ролі (наприклад, «практика-помічник», «менеджер з клієнтів») — додаємо в модель `users.role`. Але **не зараз** — поки тільки head/curator/lawyer.

Для розширення — додавання типу ролі в enum + правила що ця роль може робити в `tenant.permissions[role][action]`.

### Багатокрокові акцепти

Зараз — одноразовий акцепт (підтвердив/відмовив). Якщо потрібні багатокрокові (підтвердив куратор → підтвердив керуючий партнер → активація) — це фаза 5+.

```javascript
case.approval_workflow = {
  status: "pending_approval",
  steps: [
    { order: 1, role: "curator", required: true, approved_by: "curator_x", at: timestamp },
    { order: 2, role: "head", required: true, approved_by: null, at: null }
  ],
  current_step: 2
}
```

### Workflow для конкретних типів справ

Іноді одні типи справ потребують більше контролю ніж інші. Наприклад:
- Криминальні справи — обов'язкова акцепт-перевірка
- Цивільні споживчі — auto_approve
- Корпоративні — обов'язкова з підвищеним контролем

```javascript
tenant.case_creation_rules.per_type = {
  criminal: {
    by_lawyer: "head_and_curator_required"  // обов'язково обидва
  },
  civil_consumer: {
    by_lawyer: "auto_approve"  // довіряємо
  }
}
```

Фаза 5+. Поки не критично.

## ФАЗИ ВПРОВАДЖЕННЯ

### Phase 1 (зараз — для соло засновника)

**Що реалізовується:**
- Поля `tenant_id`, `lawyer_id` в моделях даних
- Простий sole-user mode (`head`, `auto_approve`)
- UI без акцептів — справа одразу `active`

**Що **не** реалізовується:**
- Workflow акцептів
- Чернетки
- Сповіщення
- Підтвердження виконавцем

Це **закладання фундаменту**. Структура даних готова до всього майбутнього, але UI простий.

### Phase 2 (коли візьмеш партнера в АБ)

**Активувати workflow акцептів:**
- Поле `case.approval_workflow` починає використовуватись
- Адвокат-виконавець може створити чернетку
- Керівник отримує сповіщення на акцепт
- Список «Потребує підтвердження» в UI керівника
- Підтвердження виконавцем отримання справи

**UI:**
- Кнопка «Створити справу» — для керівника одразу активна
- Кнопка «Створити чернетку» — для адвоката-виконавця, з примусовим акцептом
- Дашборд показує `pending_approval` справи
- Сповіщення про нові акцепти
- Сповіщення адвокату про нові призначення

### Phase 3 (АО з куратором)

**Розширити:**
- Додати роль `practice_curator`
- Поле `users.practice` для прив'язки до практики
- Налаштування `by_curator` в правилах
- Видимість справ за практикою

**UI:**
- Куратор бачить справи своєї практики в окремому view
- Може створювати справи з/без акцепту
- Окремі сповіщення для куратора і керівника
- Окремий дашборд для куратора

### Phase 4 (велика юрфірма)

**Додаткові функції:**
- Делегування акцептів
- Email + Telegram сповіщення
- Розширений аудит з фільтрами
- Багатокрокові акцепти (якщо потрібно)
- Workflow per case type

### Phase 5 (multi-tenant SaaS)

Один екземпляр Legal BMS обслуговує **багато юр.осіб**:
- Кожен tenant ізольований
- Свої налаштування правил
- Свої користувачі
- Свої справи
- Білінг per tenant

Це **зовсім інший рівень складності**. Поки не зараз.

**Передумови для Phase 5:**
- БД повністю готова до multi-tenant (поле `tenant_id` всюди ще з Phase 1)
- Бекенд авторизація з контекстом tenant
- Stripe-інтеграція для білінгу клієнтів
- Адмін-панель для управління tenant'ами
- SLA, моніторинг, support

## КРИТИЧНІ ПРИНЦИПИ

### 1. Гнучкість на рівні даних, простота на рівні UI

Налаштування workflow живуть у `tenants[]`, не в коді. Зараз UI простий — кнопка «створити», без акцептів. Коли потрібно — UI розширюється за тиждень.

### 2. Закласти зараз, активувати пізніше

Модель `tenant_id` і `case.approval_workflow` має бути в коді **зараз**, навіть якщо не використовується. Інакше переробляти структуру даних і робити міграції — болісно.

### 3. Self-control механізми

Історія всіх акцептів зберігається в `case.activity_log`. Видно хто, коли, що зробив. Підстраховка від «забув підтвердити» — нагадування. Підстраховка від «не пам'ятаю хто робив» — аудит.

### 4. Зв'язок з Court Sync

`status = "draft"` — справа невидима для Court Sync. Тільки `active` моніториться. Це захист від ЄСІТС-моніторингу справ які ще не оформлені.

### 5. Зв'язок з білінгом

Time entries і інвойси тільки для `active` справ. Чернетки не йдуть у звітність бюро.

### 6. Прозорий аудит

Кожна дія логується з timestamp, actor, action, details. Це не шпигунство — це підстраховка від суперечок «я не пам'ятаю хто що зробив».

### 7. Видимість справ за роллю

- `head` — всі справи tenant
- `practice_curator` — справи своєї практики
- `lawyer` — тільки свої справи

Це **юридична необхідність** — рядовий адвокат не повинен бачити справи інших адвокатів бюро (адвокатська таємниця між колегами теж важлива).

### 8. Self-healing при помилках

Якщо адвокат випадково створив справу яка не має сенсу (тестова, дублікат) — її можна:
- Архівувати з причиною (`status: "archived"`, не видаляти)
- В аудиті зберегти запис «створено-архівовано»
- Якщо була `active` — перевірити що time entries правильно відрегульовані

## ОСТАННЯ ДУМКА

Workflow Engine — це **інвестиція в майбутнє** яка коштує мало зараз і **багато економить** потім.

Якщо засновник через рік візьме партнера — все, що треба зробити, це **налаштувати правила в `tenant.case_creation_rules`** і додати другого користувача. Якщо через 3 роки трансформувати в АО з куратором — додати роль і налаштувати ще одне правило. Якщо через 5 років продавати як SaaS іншим юр.фірмам — структура вже multi-tenant, треба тільки додати фронтенд для самообслуговування і Stripe-інтеграцію.

Це **не складно** реалізувати зараз — кілька полів у моделях даних і простий UI. Складно буде потім переробляти якщо не закладено.

Тому: **архітектурна гнучкість зараз, простота використання залишається тією ж самою**. Користувач не бачить складності, але якщо вона потрібна — є.

Це і є дотримання класичного SaaS-принципу: «**не плати за гнучкість** яка тобі поки не потрібна, але **отримай її безкоштовно** коли стане потрібна».
