## 0. Роль адмін-сесії і воркфлоу

Адмін-сесія ЄСІТС: пише спеки/TASK, валідує їх з кодом, координує виконавців,
веде обговорення з адвокатом, дає пропозиції й інформацію для рішень, планує
роботу. **Сама код у проді не пише** (крім документації).

**Узгоджений воркфлоу (2026-05-27):**

```
1. Адмін пише спеку → docs/tasks/TASK_<id>_<slug>.md
2. Адмін показує спеку адвокату → затвердження (одне речення «ок»)
3. ОКРЕМА сесія-виконавець бере затверджену спеку і реалізує
4. Адмін звіряє результат з кодом (read-only), пише звіт/нотатку
```

Адмін **не** реалізує сам, навіть дрібне — це робить виконавець за спекою.
Координую обидва треки (див. §1): Track A (полігон) тут; Track B (розширення) —
в окремому чаті/репо, я лише координую, не пишу його код.

## 1. ДВА ТРЕКИ МОДУЛЯ ЄСІТС (ключова рамка)

Court Sync = синхронізація з кабінетом ЄСІТС (cabinet.court.gov.ua). Дві гілки:

- **Track A — полігон.** Claude for Chrome (AI-агент у браузері) обходить кабінет
  за промптом. **Не для щоденної роботи** — на ньому відточуємо логіку обходу,
  фільтри, дані, UX, структуру envelope. Має фундаментальні межі (compacting,
  ліміти токенів, природа чату — див. §4).
- **Track B — продукт.** Власне Chrome-розширення (детермінований код, без
  compacting). Робочий інструмент майбутнього. Окремий репо `legal-bms-extension`,
  окремий чат Claude Code, розробка переважно з iMac (Chrome dev mode).

**Стратегія:** Track A відточує логіку → переноситься у код Track B.
**Ембріон Track B уже в Legal BMS** (TASK 0.4): `window.LegalBMS` API,
`scenarioProcessor` через DI, `hashRouter` (#/court-sync/import), entitlements +
tariffMatrix.

**Етапи Track B (зі SPEC_extension_court_sync.md):** Етап 0 = каркас (manifest v3,
background, content-script handshake через window.LegalBMS, popup-скелет,
extension token, entitlements) → Етап 1 обхід однієї справи → Етап 2 список з
фільтрами → Етап 3 handshake+submit → Етап 4 popup UI (вибір обсягу, форма точки
рішення) → Етап 5 публікація. Підхід **А — каркас першим**, паралельно полігон
відточує обхід. Чат розширення НЕ обходить кабінет на Етапі 0, НЕ дублює логіку
Legal BMS (використовує window.LegalBMS).

> `SPEC_extension_court_sync.md` та `ecits_envelope_final.json` створені в чаті
> полігону (outputs тієї сесії), у цьому репо їх може не бути — за потреби
> попросити адвоката докласти.

## 3. Відкритий баг (діагностика виконана 2026-05-27)

**BUG-1 — read-after-write через заморожений снапшот.** ✅ Діагностика
ЗАВЕРШЕНА. Повний звіт: `docs/diagnostics/report_diagnostic_ecits_state.md`.

Суть: `getCases: () => cases` (`App.jsx`, `ImportTab.jsx`) — заморожений
immutable-снапшот рендеру; записи йдуть через `setCases(prev=>…)` у живий стан.
Усередині одного прогону читання не бачить записів → `update_case_ecits_state
failed: Справу case_XXX не знайдено`. Тести маскують дефект (мутабельний масив +
fake-хендлер з безумовним success).

Наслідки сьогодні: переважно косметичні (per-case ecitsState UI не читає; дедуп
тримається на ключі, проставленому при створенні). Ризики: фейкові помилки в
ResultCard лякають; латентний ризик дублів усередині прогону. **Лагодити треба
незалежно від вибору ключа дедупу.**

## 4. Природа чату і compacting (засаднича знахідка — для всіх промпт-TASK)

Це концептуальний інсайт адвоката, зафіксувати назавжди. Стосується будь-якого
TASK на промпт Track A.

- **Модель чату = повідомлення → відповідь.** Завершене повідомлення = очікування
  реакції. Тому інструкція «виписуй проміжний JSON і продовжуй» конфліктує сама з
  собою: щоб виписати, треба завершити повідомлення → чекати → але ніхто не
  відповідає. Claude **відкладає** проміжні журнали до кінця.
- **Compacting СТИРАЄ деталі (→ null), а не псує.** Коли Claude **примушують**
  дати значення зі стертого — він **реконструює = вигадує**. Коли дозволяють null
  — чесно ставить null. Два режими; формулювання «не вигадуй — з джерела або null»
  тримає в чесному.
- **Подвійне навантаження пам'яті:** тримати накопичувальний результат + поточну
  роботу → пам'ять переповнюється швидше → compacting раніше → галюцинації.
  «Виписати-в-чат-і-забути» краще за «тримати-в-пам'яті».

**Робочий метод Track A (єдиний надійний, підтверджений):**
**порції по 4 справи + зупинка з питанням «продовжувати?» + фінал зі звірки
журналів-повідомлень.** Журнал стає природним завершенням повідомлення — Claude
охоче його виписує свіжим (з ecitsCaseId і датами).

**Наслідок для стратегії:** це найсильніший аргумент за Track B — детермінований
код пише дані одразу після кожної справи, без compacting і без природи чату.

## 6. Зафіксовані рішення

**Р-1. Дедуплікація — за номером справи `case_no`, не за ЄСІТС-кодом.**
(Уточнення адвоката, 2026-05-27.) Номер справи унікальний, постійний, завжди є в
е-суді; **усі провадження** (апеляція/касація/оскарження ухвал) живуть під **одним**
номером. Отже `case_no` = ключ розрізнення справ. 32-hex `ecitsState.caseId` =
посилання на провадження/картку в кабінеті, НЕ ключ дедупу. **Засторога:** `case_no`
перед порівнянням НОРМАЛІЗУВАТИ (пробіли, суфікс, регістр, розділювачі).
*Підсилено handoff:* логіка активних справ оперує `case_no`, інстанція не має
значення → один `case_no` справді охоплює кілька проваджень.

**Р-2. ecitsState НЕ прибирати** (відхилено «Варіант A»). Лишається носієм
посилання на джерело + sync-метаданих; просто не ключ дедупу.

**Р-3. Воркфлоу адмін↔виконавець** — §0.

**Р-4. Track A = полігон, Track B = продукт.** Логіку відточуємо на A, переносимо
в код B. Полігон лишається активним поки розширення в розробці.

**Р-5. Робочий метод Track A — порції 4 + зупинки + журнали** (§4). Будь-який
промпт-TASK має це поважати, не ламати.

## 7. Відкриті питання для адвоката (перед фікс-TASK)

- **П-1.** ЄСІТС 32-hex код — per-case чи per-proceeding? Handoff підсилює гіпотезу
  «per-proceeding» (кілька інстанцій під одним `case_no`). Якщо так — `ecitsState.
  caseId` має дозволяти кілька кодів і/або переїхати на рівень `proceeding`.
  Підтвердити по тому, як кабінет перелічує апеляцію/касацію (окрема картка з
  власним URL?).
- **П-2.** Правила нормалізації `case_no`: суфікс `-ц`/`-к`/`-а` — частина ключа
  чи відкидати? Регістр, пробіли/нерозривні пробіли, розділювачі?
- **П-3.** Засідання без часу (`time:null`, кейс Нестеренка). Handoff §9.3.7 дає
  лід: такі дати беруться з **протоколів**; пропозиція — брати дати лише з
  повісток/внесення дат, не з протоколів. Рішення: (а) фільтрувати джерело дат у
  промпті; (б) як зберігати в коді — пропускати / дефолтний час + позначка / лише
  дата.

## 9. Структура правильного envelope (контракт scenarioProcessor)

```json
{
  "envelopeVersion": 1,
  "scenarioId": "ecits_import_cases_and_hearings",
  "scenarioVersion": 1,
  "producedAt": "ISO-datetime",
  "producedBy": { "provider": "claude_for_chrome", "providerVersion": "..." },
  "data": {
    "ecitsAdvocate": { "fullName": "...", "cabinetIdentifier": null },
    "stats": { "totalCasesInCabinet": 50, "filtered": 11, "withHearings2026": 7 },
    "cases": [ { "ecitsCaseId", "case_no", "court", "category", "advocateRole",
                 "primaryParty", "primaryPartyFullName", "cabinetUrl", "hearings": [] } ],
    "warnings": [ "рядок", "рядок" ],
    "skipped": [ { "case_no": "...", "reason": "..." } ]
  }
}
```

**КРИТИЧНО:** `warnings` — масив **рядків**, НЕ об'єктів (інакше React error #31,
модуль падає в заглушку). `validateEnvelope` вимагає `envelopeVersion===1`,
`scenarioId==='ecits_import_cases_and_hearings'`, наявність `data.cases[]`.
Проблема Track A: Claude for Chrome у фіналі видає неповний envelope (без
scenarioId/обгортки, warnings як об'єкти) → ручне переобгортання. Це лагодить
TASK 0.4.5.
