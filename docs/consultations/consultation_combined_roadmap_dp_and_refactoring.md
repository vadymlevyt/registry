## 1. Три осі — навіщо ставити їх в один процес

```
─────────────────────────────────────────────────────────────────────────
  ВІСЬ A: ПРОДУКТОВА — DP від поточних 40% точності до повної візії
─────────────────────────────────────────────────────────────────────────
  Що рухає: візія `consultation_dp_product_vision.md`
            Адвокат закидає → AI пропонує → ★ адвокат править план ★ → виконує
            + три ітерації диригента (Ітерація 1 → 2 → 3)
            + принцип «Cheap before Expensive»
  Без неї:  DP лишається box з лотереєю, користі мінімум

─────────────────────────────────────────────────────────────────────────
  ВІСЬ B: ХАОС — 10 точок неузгодженості з diagnostic
─────────────────────────────────────────────────────────────────────────
  Що рухає: 10 точок з `diagnostic_dpv2_call_trace_and_chaos.md`
            (C1: tool use; C2: AI helpers у Provider; C3: strip RAM;
             C4: AddModal уніфікація; C5: мертвий persistStage;
             C6: _temp плоска; C7: AI usage діра; C8: lazy import)
  Без неї:  кожен новий TASK дубль хаос, технічний борг росте

─────────────────────────────────────────────────────────────────────────
  ВІСЬ C: РЕФАКТОРИНГ — великі файли → split
─────────────────────────────────────────────────────────────────────────
  Що рухає: `consultation_large_files_refactoring_roadmap.md`
            ImageMergePanel (✓ виконано), AddDocumentModal, CaseDossier,
            Dashboard, App.jsx
  Без неї:  читання великих файлів стає дорогим, нові розробники тонуть
```

**Чому ці три осі НЕ можна вести як три окремі плани:**

- **DP без хаос-cleanup** = накопичуємо борг швидше ніж розгрібаємо. Нові тумблери лежать поруч з мертвим persistStage і дубльованими AI helpers.
- **Хаос-cleanup без DP-функцій** = марна робота. Адвокат не бачить нічого, лише «всередині стало чистіше».
- **Refactor без DP/хаосу** = переїзд коду без розв'язання структурних проблем (DrivePicker дублюється, AddModal паралельний з DP).
- **DP без refactor** = додаємо UI плану у вже-перевантажений CaseDossier (3260 рядків).

Тобто три осі — це **один процес**, де кожен TASK може зачіпати одну, дві або всі три осі **одночасно**.

## 4. Заборонені перетинання (не робити в окремих TASK'ах)

Наступні **не повинні** виконуватись як окремі TASK'и — їх **об'єднати** в одному:

- ❌ ImageEditor винос окремо від DP-UI image-merge: Rule of Three без реального споживача = overengineering. У TASK 1 разом.
- ❌ AddModal split окремо від DP-C4: переробка AddModal без виправлення дублювання з DP — марна. У Фазі 4 разом.
- ❌ cleanText винос окремо від модалки + DP: треба переписати ОБА споживачі одночасно. У TASK 3 разом.
- ❌ contextGenerator у двох місцях (DP + Огляд): один сервіс, два споживачі. У TASK 2 разом.

## 6. Стратегія «Широким фронтом»

Це назва підходу адвоката: одночасно рухаємо вперед:
- **DP** до візії (нові функції — image-merge UI, contextGenerator, cleanText)
- **Файлову структуру** (виносимо в спільне: ImageEditor, imageDocument, DrivePicker, contextGenerator)
- **Усуваємо дублювання** модалка ↔ DP (одне джерело для image-merge, cleanText, DrivePicker, contextGenerator)

Кожен TASK Фази 1 робить **кілька цих речей одночасно**. Не послідовно. Жодний TASK не «тільки cleanup без видимої користі» — кожен дає **видимий прогрес** для адвоката + **закриває хаос** + **спрощує структуру**.

Це і є **здоровий організм** у дії — система росте у всіх вимірах одночасно, а не один напрямок за рахунок іншого.
