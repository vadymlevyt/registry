**Дата:** 2026-06-04
**Для кого:** адмін-сесія, що керує впровадженням стиснення (пише спеки/таски, контролює виконавців).
**Чому зараз:** паралельна сесія **вже будує стиснення** (модалки, `standaloneCompressor`, тумблер). Ризик реальний — щоб не прикрутили **слабкий старий компресор**. Цей файл — щоб адмін-сесія зорієнтувалась правильно і давала вірні вказівки.

## 🚨 ГОЛОВНИЙ РИЗИК (читати першим)

У коді є **дві різні речі під словом «стиснення», і їх плутають**:

1. **`compressionService.compressPdf` (поточний) — СЛАБКИЙ.** Робить лише `pdf-lib doc.save({useObjectStreams:true})` — структурний re-save. **Виграш 1-2%.** Це legacy зі старого DP (TASK 1 salvage). **НЕ той рушій.** Зараз на ньому сидять `standaloneCompressor`, `CompressFilesModal`, worker `compressPdf`.
2. **Реальний рушій (downscale) — у стенді `public/lab/pdf-recompress.html`.** Рендер кожної сторінки → JPEG (1800px/0.7) → **pdf-lib-перебудова**. Це він дає 200→79 МБ. **Його ще НЕ перенесено в застосунок.**

> **ВКАЗІВКА ВИКОНАВЦЯМ:** прикручувати **реальний downscale-рушій зі стенда**, а НЕ `compressionService.compressPdf` (re-save). Слабкий re-save можна лишити окремо або викинути, але **він не є функцією стиснення для адвоката**.

## ЩО ТАКЕ ПРАВИЛЬНЕ СТИСНЕННЯ (суть)

PDF на основі зображень: **розпакувати → витягти зображення сторінок → стиснути зображення (downscale до стелі пікселів + JPEG-якість) → знову запакувати в PDF.** Вага PDF — у растрах сторінок; зменшуємо растри — зменшуємо файл.

### Скоуп — ТІЛЬКИ scanned (на основі зображень)
- **Працює лише для PDF, що мають усередині зображення** (скани/фото). 
- Текстові (searchable) PDF — **НЕ стискаємо** (вага в тексті/векторах, растрів нема). Це **scanned-guard** (§4.1 doctrine): відмовити + чесне повідомлення.
- «scanned» = «на основі зображень» — одне й те саме.

### 🔑 КРИТИЧНО: рушій МУСИТЬ давати придатний до нарізки PDF
Перебудова — **через pdf-lib (кожна сторінка — власні ресурси)**, **НЕ через jsPDF**.
**Чому (вже зафіксовано, не загубити):** jsPDF кладе всі зображення в **один спільний `/Resources`** на всі сторінки → конвеєрна нарізка (`copyPages`) тягне **ВСІ** зображення в **кожен** чанк → чанк ≈ весь файл → `documentAi` 40 МБ → падіння / «1 документ» / порожнє `02`. Доведено node-експериментом (`copyPages([0])` ≈ повний файл). **Стенд уже виправлено на pdf-lib саме через це** (коміт `b76ba04`, doctrine §3.2).
→ Стиснення на вході DP **і потрібне саме для того, щоб потім нарізати** (легші, правильно структуровані сторінки). **Не пропустити цей зв'язок.**

## 🔁 ПОВЕДІНКА З НЕ-ПРИДАТНИМИ ФАЙЛАМИ — pass-through, нічого не падає

Стиснення застосовується **вибірково, по типу файлу**, всередині будь-якого пакета:
- стискаються **тільки**: зображення (фото) і **скановані PDF** (на основі зображень);
- **усе інше** (HTML, DOC/DOCX, PDF із текстовим шаром, інші формати) — **ігнорує** функцію: проходить **як є**.

**Сценарій:** адвокат додає мішаний пакет (HTML + DOC + зображення + сканований PDF + текстовий PDF) і вмикає тумблер «стиснути». Результат:
- стискаються лише зображення та скановані PDF;
- решта проходить незмінною;
- **пайплайн НЕ падає, не зупиняється, не зависає** — непридатний файл просто пропускається.

Система **сама знає**, що стискати лише такі файли, і запускає процедуру **тільки по них**. «Стиснути» = «застосуй там, де доречно», а НЕ «зламайся на DOCX». Це стосується і DP-тумблера, і Інструментів.

**Повідомлення адвокату (UX):** має бути зрозуміла підказка/тост, що стиснути можна **лише скановані PDF та зображення (або PDF, утворені із зображень)**; інші файли функція не чіпає — щоб адвокат не дивувався, чому DOCX «не стиснувся» (це не баг, це за дизайном).

> Критерій scanned↔інше — **один** по всій системі (та сама детекція, що guard в Інструментах і `documentNature`).

## ДВА МІСЦЯ ЕКСПОЗИЦІЇ (рушій один, поведінка різна)

| | **Інструменти** (вкладка) | **Document Processor** (тумблер) |
|---|---|---|
| Що | standalone «Стиснути файл(и)», 1 або багато | тумблер «стиснути перед обробкою» |
| Рівень | **адвокат обирає пресет** (Слабкий/Середній/Сильний, §4.1) — як на стенді | **фіксований Середній (1800/0.7)**, без вибору |
| Навіщо | свідома разова дія (передати/розшарити) | технічна нормалізація на вході (розвантажити OCR/нарізку, зробити slice-able) |

**Пресети (§4.1):** Слабкий ~2200px/0.8 · Середній **1800/0.7** (дефолт=стандарт) · Сильний ~1600/0.65 (не нижче підлоги 1400/0.6). Параметри — одна константа, UI лише обирає пресет.

## ДЕ ЩЕ ЗАСТОСОВУЄТЬСЯ / ДЕ ВЖЕ Є (звірено в коді)

- **Склейка фото (image-merge):** ✅ **уже має автоматичний downscale** (`imageDocument/downscaleImage.js`, крок 1.5 у `prepareImagesForMerge`, після HEIC→JPEG, перед OCR). Окреме стиснення там **не потрібне**. **Caveat #40:** фото **з Drive-джерела** downscale **пропускає** (тільки device-upload). 
- **Одиночне зображення → PDF (`converter/imageToPdf.js`):** ⚠️ **НЕ стискається** — jsPDF, JPEG **quality 0.92**, без downscale. **Прогалина:** сюди стиснення теж варто застосувати (адвокат це хоче).
- **DP-тумблер `compressAll`** («Стиснути всі файли пакета»): ⚠️ **UI-ONLY, не під'єднаний** (`compressAll` не споживається в пайплайні). Логіки нема — її і робить паралельна сесія.
- **Окремої верхньої вкладки «Інструменти» НЕМА:** `CompressFilesModal` і `RecognizeTextModal` живуть **модалками в DocumentProcessor**, не як top-level вкладка.
- **OCR-інструмент** зберігає лише `.txt` (не DOC).

## ЧЕК-ЛИСТ ДЛЯ АДМІН-СЕСІЇ (що проконтролювати у виконавців)

1. ☐ Рушій = **downscale зі стенда** (render→JPEG→pdf-lib), **НЕ** `compressPdf` re-save.
2. ☐ Перебудова **pdf-lib** (власні ресурси/стор.) — інакше зламається нарізка (40 МБ).
3. ☐ **scanned-guard** — стискаємо лише PDF із зображеннями; текстові відхиляємо.
4. ☐ DP-тумблер `compressAll` — **під'єднати** до входу пайплайну (перед нарізкою/склейкою/додаванням), **фіксований Середній**.
5. ☐ «Інструменти» — пресети (3 рівні); окрема вкладка (або хоча б узгодити з §7 doctrine).
6. ☐ Одиночне зображення→PDF — застосувати стиснення (зараз 0.92 без downscale).
7. ☐ Склейку **не чіпати** (там downscale уже є); хіба закрити #40 (Drive-source).
8. ☐ Рушій ОДИН (Rule of Three) — Інструменти + DP + (опційно конвертер) тягнуть зі спільного, без дублів.
