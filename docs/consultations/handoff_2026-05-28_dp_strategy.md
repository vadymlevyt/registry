# Handoff — DP strategy «Широким фронтом» (2026-05-28)

**Призначення:** передача контексту з сесії розробки цілісної стратегії руху DP до повної продуктової візії. Контекст попередньої сесії переповнюється — наступна сесія продовжує з цієї точки.

**Стратегія в одному реченні:** одночасно рухаємо DP до візії (нові функції), реформуємо файлову структуру (виносимо в спільне), і усуваємо дублювання модалка ↔ DP — це **один процес**, не три окремі плани.

## 2. СТРАТЕГІЯ «ШИРОКИМ ФРОНТОМ»

Це назва підходу, який вибрав адвокат. Три осі робіт йдуть **одночасно** через кожен TASK:

```
ВІСЬ A — ПРОДУКТОВА        ВІСЬ B — ХАОС           ВІСЬ C — РЕФАКТОРИНГ
DP до повної візії         10 точок неузгодженості великі файли → split
з 3 ітерацій               у поточному коді       (адмін-чат веде)
```

Кожен TASK може зачіпати одну, дві або всі три осі **одночасно**. Цей принцип критичний:
- НЕ робити «тільки cleanup без видимої користі для адвоката»
- НЕ робити «нову функцію без розгрібання хаосу що стосується її»
- НЕ робити «винос у спільне без реального споживача (Rule of Three)»

## 3. ПЕРШИЙ ФРОНТ — серія з 4 TASK'ів (Фаза 1 з combined roadmap v3)

Це **поточна стартова точка**. До цього всі попередні фази (фундаментні tool use, видимість помилок, max_tokens, image-merge у модалці) вже виконані.

```
TASK_1: image_merge_unify          ← НАСТУПНИЙ TASK для підготовки
─────────────────────────────
  - DP отримує image-merge UI з можливістю N документів
  - Винос components/ImageEditor/ + services/imageDocument/
  - Розширення deterministicRoute (всі фото → пропустити Triage)
  - Toggle «Просто додати файли» у Зоні 2
  - .txt пишеться для text-layer PDF теж
  3-5 днів. Найбільший TASK першого фронту.

TASK_2: context_generator_unify
─────────────────────────────────
  - Тумблер «Оновити case_context» у DP стає функціональним
  - contextGenerator.js → CaseDossier/services/ (за refactor §6.6)
  - Вкладка Огляд CaseDossier + DP тягнуть з одного місця
  2-3 дні.

TASK_3: clean_text_unify
────────────────────────
  - aiCleanText виноситься з Provider у services/dp-ai/cleanTextClient.js
  - Модалка + DP тягнуть звідти
  - Закриває C2 (для cleanText) + C7 (логування AI usage)
  1-2 дні.

TASK_4: drive_picker_unify
──────────────────────────
  - DrivePicker → components/UI/DrivePicker/
  - DP + AddModal тягнуть звідти (закриває реальну дублікацію)
  1-2 дні.
```

**Чому в такому порядку:** TASK 1 — найбільший, дає видиму користь (нова функція у DP). TASK 2 і TASK 3 — інкрементальні. TASK 4 — окремий простий cleanup.

**Можна паралельно** через окремі сесії якщо адвокат запитує (бо TASK'и незалежні), або послідовно однією сесією.

## 4. ЯК ВЗАЄМОДІЯТИ З АДВОКАТОМ

### Ролі
- **Адвокат** — продуктовий диригент. Приймає **продуктові** рішення (що робити, в якому порядку).
- **Ти (Claude Code)** — технічний виконавець-радник. Реалізуєш TASK'и, попереджаєш про ризики, пропонуєш альтернативи.

### Принципи комунікації
- **Чесна критика переважає над згодою.** Якщо адвокат пропонує щось архітектурно неправильне — сказати з аргументами.
- **Технічний матеріал — для прослуховування.** Адвокат краще засвоює структурований аудіо-дружній виклад. Короткі речення, чітка структура.
- **Одна тема за раз** — не комбінувати кілька питань.
- **Лаконічно.** Адвокат явно просив короткі відповіді коли контекст великий.

### Що адвокат любить
- Конкретні рекомендації з обґрунтуванням
- Таблиці порівняння
- Чесні «не знаю / поки не перевіряв»
- Підтвердження що його розуміння правильне (коли так)
- Поправлення коли його розуміння неправильне (з поясненням)

### Що адвокат НЕ любить
- Загальні відповіді без файл:рядок посилань
- Гіпотези подані як факти
- Передчасні оптимізації / overengineering
- Слово «error» у видимих користувачеві текстах (нейтральним тоном)
- Sycophancy (підтакувати без аналізу)

### Стандартний робочий цикл для TASK'у
1. Адвокат каже «готувати TASK X»
2. Ти читаєш потрібні файли, перевіряєш реальний код
3. Готуєш TASK-специфікацію у `/tmp/TASK_X.md` (не в репо!)
4. Передаєш через `SendUserFile` адвокату на review
5. Адвокат править або затверджує
6. Затверджений TASK передається **ОКРЕМІЙ сесії-виконавцю** (не тобі)
7. Виконавець реалізовує + комітить + push
8. Адвокат тестує на планшеті, повідомляє результати
9. Якщо проблеми — діагностика, нові TASK'и

### Файли НЕ в репо
- TASK-специфікації для виконавця → `/tmp/`
- Великі обговорювальні документи перевіряються адвокатом перед комітом → `/tmp/`
- Тільки після його згоди — комітимо у `docs/`

### Push у main
За `CLAUDE.md` правило №1:
- Тільки .md → FF + push origin HEAD:main без підтвердження (якщо тести зелені)
- Зміни коду → треба підтвердження адвоката одним реченням ПЕРЕД push

### Гілка
`claude/*` (harness видасть автоматично). Ця сесія була на `claude/smart-triage-degenerate-plan-Xxg1n`. Нова сесія матиме свою.

## 6. КЛЮЧОВІ КОНЦЕПЦІЇ З ПРОДУКТОВОЇ ВІЗІЇ

Якщо не прочитаєш `consultation_dp_product_vision.md` — рішення будуть неправильні. Тут — найважливіше для швидкого orienting.

### Три ітерації диригента
- **Ітерація 1** (поточний етап): людина-оркестратор, окремі сценарії (нарізка / image-merge / PDF-merge / unpack / стиснення)
- **Ітерація 2**: людина-оркестратор з UI-комбінацією галочок
- **Ітерація 3** (повна візія): AI-оркестратор пропонує комплексний план → людина править → виконує

Кожна ітерація — самодостатня, накладається як шар.

### Modifier vs основна дія
- **Основні дії**: розпакувати, конвертувати, склеїти, нарізати
- **Modifier'и**: стиснути, очистити текст, розкласти по провадженнях, оновити case_context
- Modifier'и **застосовуються до** основних дій, не самостійні

### Cheap before Expensive
- Маршрутизація сценарію — детермінована (MIME, text layer) де можливо
- AI підключається ВСЕРЕДИНІ сценарію для деталей (порядок фото, дублі, межі)
- AI НАД сценаріями — тільки в мікс-кейсах і у Ітерації 3

### Toggle «Просто додати файли»
- OFF (default): AI Triage перевіряє ВСІ PDF на нарізку
- ON: Triage-нарізка пропускається, кожен PDF = 1 документ
- Що залишається: OCR + .txt + метадані + класифікація

## 10. ОСТАННЯ ПОРАДА

Адвокат **довіряє системі**. Він не питає кожну дрібницю. Він задає **архітектурні питання** і чекає **архітектурних відповідей**.

Якщо ти бачиш що архітектурне рішення поверне DP до візії — пропонуй. Якщо бачиш що рішення створює борг — попереджай.

Це не реактивна позиція «що скажуть — те й зроблю». Це **активна співпраця** з юристом який експертно розуміє свій бізнес але потребує тебе як **архітектора** його технічної системи.
