# Sliding window / multi-window Triage для великих томів — як серверна фіча

Дата: 2026-06-02
Статус: консолідація + рішення (відкладено в борг як серверну фічу)
Призначення: зібрати в одне місце розкидану ідею «різати том на вікна для
різника» і зафіксувати висновок — це **серверна** фіча, не клієнтська.

> Цей документ **консолідує** згадки з трьох ширших документів (вони
> присвячені іншим темам і **не редагувались**): джерела перелічені нижче.
> Тут — тільки тема sliding window / multi-window Triage, без інших ниток.

---

## 1. Що це за ідея

Поточний різник (Triage / детектор меж документів) обробляє весь том **одним
проходом**: будує компактний адаптивний «паспорт» сторінок і одним викликом
Haiku пропонує план меж (`pageMarkers.js`, `RICH_PASSPORT_MAX_PAGES_DEFAULT=70`
→ ≤70 стор. багатий профіль, >70 компактний).

**Sliding window** — альтернатива для дуже великих томів: різати том на
**вікна** (наприклад, по N сторінок), пропускати кожне вікно через різник
окремо, а потім **зшивати** результати в один план. Документи, що
перетинають межу вікна, переносяться як «відкритий хвіст» у наступне вікно
(нахлест/overlap на стику дає контекст для впізнавання продовження).

## 3. Чому досі НЕ зроблено

1. **Потреба для норми вже закрита іншим шляхом.** Масштаб 200-250 стор. (норма
   тому) тягне **однопрохідний компактний паспорт** (~50-70K токенів, у зоні
   якості Haiku) — `docs/tasks/TASK_smart_triage_passport_scale_and_text.md`.
   Тобто sliding window для ≤250 стор. не потрібен.
2. **Пайплайн щойно стабілізувався.** Додавати багатовіконну логіку у щойно
   виправлений потік — невиправданий ризик на той момент.
3. **Ризик серйозних багів сконцентрований на стиках** (див. §4) — а саме цей
   клас помилок (загублена/не туди підшита сторінка) найдорожчий на юридичних
   документах.

## 4. Уся складність — у зшиванні через межу вікна

Сегментувати том на вікна тривіально. Небезпека вся тут:

- **Документ перетинає межу.** Починається наприкінці вікна №1, триває у
  вікні №2; вікно №2 бачить «продовження без шапки» → ризик загубити сторінки
  або завести фантомний документ.
- **Нахлест лікує впізнавання, але породжує дублі** — ті самі сторінки
  пропонують два сусідні вікна → потрібен дедуп.
- **Кожне вікно — окремий виклик LLM** → більше місць незгоди моделі на стику
  («тут кінець» vs «тут початок»); реконсиляція мусить це мирити детерміновано.

**Інфраструктура під це частково вже існує** (знижує ризик «з нуля»):
- `src/services/documentBoundary/multiFileReconstructor.js` — `openTails`:
  перенос «відкритих хвостів» документа через стик, **між файлами** (той самий
  патерн, лише одиниця «файл» → «вікно»).
- `triageStage.js` → `resolveOverlaps` — детермінований дедуп перекритих
  діапазонів у плані (готовий механізм проти дублів від нахлесту).

## 5. Чому це має бути СЕРВЕРНА фіча (висновок цієї сесії)

Ідея лягає на сервер **краще**, ніж у поточний клієнтський вигляд:

1. **Безпека стиків доступніша.** На сервері можна дозволити собі повноцінну
   реконсиляцію меж (окремий дешевий прохід по стиках + нахлест + дедуп) без
   обмежень RAM/латентності клієнта — тобто головний ризик §4 пом'якшується
   саме там.
2. **Немає лімітів браузера** (RAM/CPU планшета, крихкість при багатьох
   паралельних викликах, вкладка мусить лишатись відкритою).
3. **Будується один раз правильно** — серверно-нативно, замість крихкої
   клієнтської версії, яку однаково довелося б переписувати при переході.

**Вердикт:** на клієнті зараз — ні (потреба ≤250 стор. закрита, ризик не
виправданий). Серверно — варто. Послідовність: **серверний перехід → потім
sliding window як серверна фіча**, не навпаки.

## 6. Відкриті проектні питання (вирішуються при побудові)

- Розмір вікна (скільки сторінок за вікно).
- Розмір нахлесту (скільки сторінок перекриття на стику): мало — не впізнає
  продовження; багато — більше дублів і токенів. (Історично сперечались, але
  числа не зафіксовані.)
- Стратегія зшивання/дедупу: як мирити незгоду сусідніх вікон на межі;
  виявлення розривів (gap) і перекриттів.
- Чи лишати однопрохідний паспорт як шлях за замовчуванням для ≤N стор. і
  вмикати вікна лише понад поріг.

## 7. Цінність vs ціна (тверезо)

**Цінність** (тільки реальні аргументи):
- Запас за межі того, що тягне однопрохідний компактний паспорт (дуже великі
  томи понад ~250-300 стор.).
- Точність різника на великих томах — коротший контекст = менше дрейфу моделі.

**Ціна:** коректність зшивання стиків (§4) + тест-матриця (документ повністю
в одному вікні / перетинає 2 вікна / 3 вікна / дедуп нахлесту / виявлення
розриву). Орієнтовно ~1-1.5 тижні роботи, більшість — у коректності стиків і
тестах, не в коді.
