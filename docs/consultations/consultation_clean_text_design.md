# Дизайн функції очищення тексту (clean text → Markdown) — для TASK 3

**Дата:** 2026-05-30
**Статус:** продуктовий дизайн адвоката, зафіксований ПЕРЕД написанням TASK-спеки
**Призначення:** clean_text — не просто «винести cleanText зі спільного місця» (як було
в roadmap), а повноцінна функція з кількома узгодженими рішеннями. Цей файл — джерело

**Зв'язок:** roadmap §2 TASK_3 (clean_text_unify); патерн спільного джерела як у

---

## 1. СУТЬ ФУНКЦІЇ

Сирий OCR-текст (`.txt` у `02_ОБРОБЛЕНІ`) містить сміття розпізнавання, ламане
форматування, артефакти. Очищення робить з нього **нормальний читабельний документ
у Markdown** (`.md`): вирівняне форматування, прибране сміття, відновлена структура
(абзаци, відступи). Для людини сирий .txt читати незручно; .md — зручно, і у viewer
показується гарно.

**Консервативність — головне правило:** AI чистить ФОРМАТУВАННЯ, НЕ лізе в юридичний
зміст. Не виправляє цитати, суми, дати, імена, формулювання. **Краще лишити сміття,
ніж змінити зміст.** (Реальний кейс — див. §6 «звернути увагу».)

- **Спільний AI-клієнт** `services/dp-ai/cleanTextClient.js` (roadmap §6.6 / dp_reuse).
  Логування AI usage всередині (C7), як sortImageDocument/contextGenerator.
- **DP-точка:** тумблер «Очистити для читання» (вже існує в Зоні 2) — у процесі
  обробки нових документів чистить їхній текст.
- **Огляд-точка (НОВЕ):** кнопка «Очистити тексти» поруч зі «Створити контекст» —
  для справ де матеріали ВЖЕ є, але тексти брудні; почистити retroactively без

- **Робимо ОДРАЗУ правильно** (на відміну від контексту, який виносили постфактум):
  спільне джерело з народження, обидві точки тягнуть з нього.

## 3. КЛЮЧОВЕ РІШЕННЯ — формат і місце результату

### Результат: .txt → .md (НЕ перезапис .txt, НЕ окремий _clean.txt)
- Очищений текст зберігається як **Markdown** (`<basename>_<driveId>.md`) у `02_ОБРОБЛЕНІ`.
- **Viewer показує .md** (форматований, гарний) — для цього функція й існує.
- Markdown бо: структурує (заголовки, абзаци, списки), людино-читабельний сирим,
  у viewer рендериться як нормальний документ.

### Сирий OCR .txt — НЕ зберігаємо (рішення адвоката, ~60/40)
- Концептуально правильно було б зберегти оригінал OCR для звірки, АЛЕ: використається
  в одиничних випадках, вигода мінімальна проти вартості зберігання/складності.
- **Рішення: сирий .txt не зберігається окремо** після очищення (заміщується .md).
- **Страховка замість зберігання (§5):** якщо адвокат сумнівається в конкретному
  документі — може **перезапустити OCR на ОДНОМУ документі** (через агента досьє),
  отримати свіжий сирий текст і звірити. Тобто оригінал відтворюється за потреби,
  не зберігається постійно.

## 4. LAYOUT — паливо для якісного форматування, потім видаляється

- `.layout.json` (`02_ОБРОБЛЕНІ`) містить `pageStructure` на рівні **paragraphs/blocks**
  (підтверджено: `ocrService.js:138` — «на рівні paragraphs/blocks»; стрипляться лише
  image/tokens). Тобто метадані абзаців/структури Є — годяться для відновлення
  форматування при очищенні.
- **Layout = паливо очищення:** clean text використовує `.layout.json` (абзаци, відступи,
  структура) щоб AI правильно відформатував .md.
- **Після повного циклу layout БІЛЬШЕ НЕ ПОТРІБЕН:** коли документ пройшов обробку
  (нарізка використала layout для меж; очищення використало для форматування) — layout
  виконав роль. Зараз layout НЕ видаляється автоматично — **TASK 3 має додати видалення
  `.layout.json` після очищення** (документ готовий, лишається тільки .md-текст).
  Перевірити: чи layout ще комусь потрібен після clean (нарізка вже відпрацювала на
  своєму етапі) — якщо ні, видаляти.

## 5. СТРАХОВКА — re-OCR одного документа за сумнівом

Замість зберігати сирі .txt — функція «перевірити документ»:
- Адвокат читає очищений .md у viewer, щось здається не так (не танцює з оригіналом).
- Дає вказівку **агенту досьє** → агент запускає **OCR тільки на ЦЬОМУ документі** →
  свіжий сирий текст → адвокат+агент звіряють чи є помилка.
- **UI-точка:** кнопка на панелі viewer (зверху/знизу), коли відкрито окремий документ /
  коли перемикач на «Текст». Аналог дублювання інтерфейсів (агент + UI).
- Це окремий під-функціонал — **можливо НЕ в scope першого TASK 3**, але закласти
  концептуально (борг/наступний крок). Головне зараз — clean text + .md + layout cleanup.

## 6. «ЗВЕРНУТИ УВАГУ» — AI помічає, але НЕ міняє

Реальний кейс (тест адвоката у звичайному Claude): в ухвалі на початку «я захисник,
брав участь у засіданні», у кінці «задовольнити клопотання захисника [іншого]». Claude
**помітив** розбіжність і виправив — АЛЕ це була НЕ помилка: адвокат брав засідання,
а клопотання формально заявив попередній адвокат обвинуваченого (між засіданнями
змінився захисник). Тобто AI «виправив» правильний текст.

**Висновок-вимога:**
- AI при очищенні **НЕ виправляє зміст** навіть якщо здається помилкою. Лишає як є.
- Але **відмічає у список «звернути увагу»** (як у нарізці є «звернути увагу») — окремий
  вихід де AI каже «тут щось привернуло увагу» БЕЗ зміни документа.
- Результат очищення повертає не лише .md, а й `attentionNotes[]` (що AI помітив) —
  показується адвокату десь у результатах процесу (як у DP Зона 3 «Потребує уваги»).
- Принцип: **краще лишити сміття, ніж змінити зміст всупереч оригіналу.**

## 7. ЩО ЦЕ ЗНАЧИТЬ ДЛЯ TASK 3 (розширення проти roadmap)

Початковий roadmap: «винести aiCleanText зі спільного місця, модалка+DP тягнуть».
Тепер TASK 3 ширший:
1. Спільний `cleanTextClient.js` (AI-клієнт, консервативний промпт, C7-логування).
2. Формат результату: **.txt → .md** у `02_ОБРОБЛЕНІ`, viewer показує .md.
3. Сирий .txt не зберігається (заміщується .md).
4. Layout як паливо форматування → **видалення `.layout.json` після очищення**.
5. DP-тумблер «Очистити для читання» (існує) → через спільний клієнт.
6. **Кнопка «Очистити тексти» в Огляді** (НОВА точка, retroactive, N-документів, прогрес).
7. `attentionNotes[]` — AI відмічає що помітив, НЕ міняючи зміст; показ у результатах.
8. (Можливо окремо/борг) re-OCR одного документа за сумнівом + UI-кнопка у viewer.

**N-документів нюанс:** clean у Огляді проходить ПО ВСІХ документах справи — дорого
(N AI-викликів), прогрес «чищу 12 з 43». На відміну від контексту (1 файл).

**Перетин з контекстом:** чистий текст → кращий контекст. Контекст-генератор уже бере
текст документів; якщо є .md — бере .md. Не ускладнювати, але врахувати порядок.
