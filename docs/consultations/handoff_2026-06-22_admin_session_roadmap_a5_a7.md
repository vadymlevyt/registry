## 1. ЩО ЗАКРИЛОСЯ ОСТАННІМ (щоб не передіагностовувати)

- **Контекст-генератор не читав HTML/DOCX→PDF — ✅ ВИРІШЕНО** (коміт `14930be`). Корінь: `pdfjsLocal.extract` роутив stored PDF за **розширенням імені** (`originalName="X.html"`) у HTML-гілку → `TextDecoder(win-1251)` на байтах PDF → бінарне сміття в `case_context.md`. Фікс: `mimeType` авторитетніший за розширення. **Деталі + урок:** `docs/diagnostics/diagnostic_context_generator_html_pdf_text.md` (§12 — справжній корінь; §11 — хибні гіпотези).
- **§7.1 «тексти» / борг #49 — ✅ ЗАКРИТО** (коміт `0bc28eb`). Searchable DOCX/HTML читаються прямо з текстового шару PDF; `.txt`/side-car не потрібні. Лишок — лише можлива разова міграція старих `.txt`-only сканів (борг **#43**).
- **A3 «чесні написи форматів» — ✅ ЗАКРИТО** (коміт `efdc804`). Прибрано з UI заявлені-але-неіснуючі формати (XLSX/PPTX/RTF/ODT/RAR/7z) у DP dropzone-hint і `accept` інлайн-форми досьє; TXT/MD лишені (самі є текстом). Конвертери для них — постпродакшн.

**Раніше в цьому циклі (вже в карті):** A1 (конвеєр→чиста нарізка), A2 (тумблери), A4-ядро (пропорційна щільність паспорта), B1 (`callAgent` парасоля), Model Picker Фаза 1, фікс застарілих ID моделей (404).

## 5. ОДИН УРОК ДЛЯ ДОВІРИ ДО КАРТИ (інформація, не лекція)

Попередній цикл по контекст-генератору дав **двічі хибний корінь** до того, як знайшли справжній: спершу «subset-шрифти/ToUnicode» (з чистого код-рідингу, без відтворення), потім «стейл/транзієнт, бага немає». Справжня причина (routing за розширенням імені) знайшлась лише при відтворенні кінець-у-кінець на реальних файлах. Записано як урок у `diagnostic_context_generator_html_pdf_text.md` (§11-§12). Контекст для наступної сесії: до висновку «корінь такий» варто мати відтворення, а не лише прочитаний код — особливо коли симптом стосується реальних даних на Drive.
