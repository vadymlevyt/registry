# Рекомендація: Activity-події як єдине джерело, білінг — похідна проєкція

> **Статус:** дизайн-нотатка (консультація), НЕ реалізація. Готує ґрунт під
> майбутній TASK Billing UI / Billing v2. Жодного коду не змінено цією нотаткою.
> Контекст народження: діагностична сесія DP-регресії (червень 2026) показала,
> що тимчасовий діаг-логер (`src/services/documentPipeline/diagLogger.js`) дає
> точні, дешеві, структуровані факти про роботу системи — і це наштовхнуло на
> питання, чи не так само має бути влаштований і білінг.

---

## 1. Проблема поточного підходу

Білінг зараз збирається через **~25 вшитих точок** `activityTracker.report(...)`,
розкиданих по модулях (App/Dashboard/CaseDossier/QuickInput/Notebook/
DocumentProcessor), кожна в `try/catch`, з ручною категоризацією на місці. Це:

- **крихко** — додав модуль/дію, забув вставити точку → роботи немає в білінгу;
- **зв'язано** — правила білінгу (категорії, `billFactor`, `billable`) вшиті в
  модулі; зміна правила = редагування багатьох місць;
- **дублює сенси** — `time_entries[]` (час адвоката) й `ai_usage[]` (токени для
  оператора SaaS) інструментуються паралельно, з ризиком розсинхрону (саме тому
  CLAUDE.md прямо забороняє дублювати поля між ними).

Інтуїція адвоката: «не вигадувати велосипед — нехай збираються звичайні логи, а
білінг-модуль їх аналізує». Архітектурно це **event sourcing**: один потік
фактів як SSOT, білінг — **похідна проєкція**. Напрям правильний.

---

## 2. Три шари — не змішувати (правило #11)

Ключове розрізнення. Зараз вони злиплі; їх треба розчепити на три сутності з
одним сенсом кожна:

| Шар | Сенс | Приклад | Властивості |
|-----|------|---------|-------------|
| **Debug-логи** | діагностика для розробника | `dp_diag_*.json` | best-effort (можна губити), шумні, тимчасові |
| **Activity-події** | бізнес-факт «що сталося» | `{дія: документи_нарізано, справа: X, сторінок: 285, тривалість: 18хв}` | **durable**, структуровані, метадані-only, SSOT |
| **Білінг** | інтерпретація фактів у ціну | рядок рахунку «опрацювання матеріалів справи» | проєкція; правила цінотворення живуть ТУТ і лише тут |

**Білінг виводиться з activity-подій, НЕ з debug-логів.** `diagLogger` — це
debug-шар (він навіть ковтає помилки запису — для діагностики ок, для грошей ні).
Але він **довів патерн**: точно, дешево, структуровано, лише метадані.

## 3. Запропонована модель

### 3.1 Один тонкий емітер

Усі модулі викликають **одну** функцію замість 25 вшитих варіацій:

```
recordActivity({
  type,            // доменна подія: 'documents.sliced' | 'case.opened' |
                   //   'ocr.completed' | 'hearing.attended' | 'agent.called' | ...
  tenantId, userId,// SaaS-контекст (ембріон з повним ДНК — завжди присутні)
  caseId?,         // над чим працювали (непрямий доказ роботи над справою)
  at,              // RFC3339
  durationMs?,     // тривалість, де є сенс
  facts: { ... },  // метадані-only: pageCount, fileCount, bytes, model,
                   //   inputTokens/outputTokens, module, ... — НІКОЛИ зміст
})
```

Це append-only потік (Drive: місячна ротація, як `time_entries`; або окремий
`activity_events[]`). Модулі НЕ знають про білінг — лише констатують факт.

### 3.2 `time_entries[]` і `ai_usage[]` стають ПРОЄКЦІЯМИ

Замість двох паралельно інструментованих структур — обидві **виводяться** з
потоку подій:

- `time_entries[]` ← проєкція подій з `durationMs` у час адвоката (з
  категоризацією/`billFactor` за правилами білінг-шару).
- `ai_usage[]` ← проєкція подій `agent.called` у токени/вартість (для оператора
  SaaS). Той самий факт (`agent.called`) живить обидві проєкції — **без
  дублювання інструментації** (дублювання зникає за визначенням, бо джерело одне).

### 3.3 Білінг-модуль інтерпретує

Окремий модуль читає потік і будує **рядки рахунку**, зокрема:

- пряма юридична робота (підготовка позиції, засідання);
- **адмінробота як окремий рядок** — «опрацювання/нарізка матеріалів справи,
  285 стор.». Це не безпосередньо юридична праця, але без неї юридичної не буває;
  вона має вартість і йде в рахунок (рішення адвоката, `billable`/`billFactor`);
- **контемпоральний запис роботи** — «02.06, опрацьовано Том 3 справи Гідрохіт» —
  це класичне обґрунтування рахунку перед клієнтом/судом. Activity-потік дає його
  природно.

Не кожна подія йде в рахунок безпосередньо — але все це **матеріал, з якого
формується вартісна величина**, і непрямий доказ, над чим адвокат працював.

## 4. Обмеження (щоб не зробити «велосипед криво» вдруге)

1. **Не логувати все підряд.** Емітити **доменні події** (значущі факти), не
   кожен виклик функції. Сире вичерпне логування — важке і шумне.
2. **Конфіденційність — залізно.** Тільки метадані (числа, назви, тривалості,
   токени). **Ніколи зміст документів.** `diagLogger.sanitize()` (ріже рядки
   >200 символів) — мінімальний зразок дисципліни; для білінг-фактів обов'язково.
3. **Durability для грошей.** Факти, з яких рахуються гроші, не можна губити
   «best-effort». Потрібна гарантована фіксація або видимі прогалини (на відміну
   від diag-логера, що свідомо ковтає помилки).
4. **Ідемпотентність проєкцій.** Перерахунок `time_entries`/`ai_usage`/рахунку з
   того самого потоку має давати той самий результат (повторний прогін безпечний).
5. **SaaS з народження.** `tenantId`/`userId` у кожній події; ротація й ліміти —
   як у наявних `time_entries`/`ai_usage`.

---

## 5. SAAS IMPLICATIONS

Один потік подій на tenant — природний носій для multi-user (хто з команди що
зробив), для лімітів підписки (проєкція «спожито за період») і для оператора
SaaS (`ai_usage` як проєкція). `caseAccess`/permissions не змінюються — події
лише фіксують факт, доступ перевіряє `executeAction` як і раніше.

## 6. BILLING IMPLICATIONS

Це і є фундамент білінгу: категорії (`case_work`/`admin`/`hearing_*`/…),
`billable`, `visibleToClient`, `billFactor` переїжджають з 25 вшитих точок у
**один білінг-шар-інтерпретатор**. Адмінробота (нарізка томів = підготовка
матеріалів) стає першокласним рядком рахунку. Історичні події можна
**переінтерпретувати** при зміні тарифів (re-projection) — чого вшита
інструментація не вміє.

---

## 7. Міграційний шлях (без ламання)

Не виривати поточний білінг. Еволюція:

1. Ввести `recordActivity` як **тонку обгортку поверх наявного**
   `activityTracker` (фасад) — нуль зміни поведінки.
2. Поступово перевести 25 точок на доменні події (по модулю за раз).
3. Переписати `time_entries`/`ai_usage` як проєкції потоку (спершу паралельно,
   звіряючи з поточними значеннями).
4. Винести правила цінотворення в окремий білінг-шар.
5. Коли проєкції збігаються зі старими даними — прибрати стару інструментацію.

`diagLogger` лишається як **debug-шар** (окремий сенс), або стає першим
споживачем тієї ж event-моделі — на розсуд TASK.

---

## 8. Що вирішити у власне TASK Billing

- Сховище потоку: окремий `activity_events[]` у registry чи окремий Drive-файл з
  ротацією (як `_archives/time_entries_YYYY-MM.json`)?
- Гарантія фіксації (durability) при offline/збоях — черга + повтор?
- Стабільна схема події (версіонована, як `schemaVersion`).
- Перелік доменних типів подій (закритий enum, правило #11 на кожен).
- Чи `diagLogger` зливається в цю модель, чи лишається окремим debug-інструментом.

---

**Підсумок:** інтуїція правильна — один потік фактів, білінг як проєкція, чистіше
за 25 вшитих точок. Головна засторога: розчепити **debug-логи / activity-події /
білінг** на три сутності (один сенс кожній) і не годувати білінг debug-логами.
Реалізація — окремий TASK; ця нотатка фіксує напрям, поки він не за горами.
