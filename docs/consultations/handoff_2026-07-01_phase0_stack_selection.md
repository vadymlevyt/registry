## 0. ЯК ТОБІ ПРАЦЮВАТИ (тон — критично)

- **Веди, не вивалюй.** Власник — гуманітарій зі структурним мисленням, але **не технар**. Пояснюй **кожен крок простими словами + навіщо він**. Мінімум жаргону; вжив термін — одразу поясни.
- **Крок за кроком, не все одразу.** Одне рішення / одна дія за раз, підтвердження, далі. Не кидай стіну тексту.
- **Готовий до реальних дій:** створення акаунтів, конфіг, перший деплой — проведи як за руку («натисни тут», «встав це», «ось що воно робить»).

- **Доступність понад усе** (умова власника): якщо для кроку треба тижнями читати мануали — знайди простіший шлях. AI (ти) — і є те, що робить це можливим соло.

## 2. ЩО ВЖЕ ВИРІШЕНО (не перевідкривати — це фундамент)
- **Напрям стеку:** **PostgreSQL + один TypeScript-бекенд + окремий worker для важких джоб + керований (managed) auth.**
- **Кандидати платформи:** **Supabase** (Postgres+RLS+Auth+Storage «з коробки» — **менше сервісів**, лідер) **або** **Neon** (Postgres) + окремий auth (Clerk) + сховище (R2). Обидва — стандартний Postgres.
- **Обмеження власника (незмінні критерії вибору):** соло + не-інженер; ціль — **200 тенантів самотужки**; **доступність понад усе**; **портативність** (стандартний Postgres + власний TS-репо, не lock-in); **менше сервісів** (кожен зайвий = нічне чергування); **auth готовий, не руками**; **окремий compute-шар обовʼязковий** (serverless не тягне 20-хв OCR/нарізку).
- **Модель даних:** реляційний **скелет** (легке = таблиці/колонки з індексами) + **JSONB** для важкого (тіла документів, лейаути з `02_ОБРОБЛЕНІ`, переписка як `messages`, важкі метадані). **НЕ все JSONB.** Шари глибини на **одному** Postgres: реляц + JSONB + граф (розшир. **Apache AGE**) + вектор (**pgvector**, під RAG/онтологію) — доклацуються за потребою, не наперед.
- **Гроші:** будувати/відлагоджувати на **безкоштовних тарифах**; платити — на комерційному старті. Домен (~$10-30/рік) — єдина рання фіксована плата.
- **Dev-workflow зберігається:** Codespaces/локально; **push → авто-деплой за хвилини** (як зараз GitHub Pages); пристрій (планшет/iMac/телефон) = **вікно**, важке крутиться в хмарі; єдине нове — міграції БД (обережно, але швидко, через гілкування бази).

## 3. ЩО ФАЗА 0 МАЄ ВИРІШИТИ (з власником, по одному, з рекомендацією)
1. **Платформа:** Supabase vs Neon+окреме. *(Рек.: Supabase — найменше сервісів для соло.)*
2. **Compute-шар важких джоб:** де крутиться worker + черга. *(Рек.: Render/Railway фон-worker + Upstash Redis/BullMQ.)*
3. **Хостинг фронту + домен `legalbims.com.ua`:** фронт лишається React/Vite (Cloudflare Pages/Netlify/GitHub Pages), бекенд — на Render; купити домен у реєстратора, під'єднати DNS.
4. **Репозиторій:** GitHub лишається (код), хостинг читає з нього на push.
5. **Free-tier план:** що безкоштовно, «засинання» проєкту на free, коли переходити на Pro (~$25 Supabase + ~$7-14 Render).
6. **Auth-провайдер** (якщо не Supabase Auth): Clerk. **Захист портативності:** дзеркалити юзерів у власну таблицю `users`.

## 4. ЩО ФАЗА 0 МАЄ ЗРОБИТИ (спайк — підняти СКЕЛЕТ, не мігрувати)
- Провести власника через **створення акаунтів** (Supabase-проєкт, Render, домен) — покроково.
- Підняти **порожній dev-цикл:** мінімальний TS-бекенд («hello») + під'єднаний Postgres + hot-reload локально/у Codespace + **перший deploy** (push → live).
- **Довести цикл** «змінив код → за хвилини live» (щоб зберігся звичний темп).
- Все — на free-tier.
