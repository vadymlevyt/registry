# Спостереження адвоката по флоу Document Processor

**Дата:** 2026-05-29
**Контекст:** після перегляду схеми `docs/mermaid/document_processor_flow.md`
адвокат поставив два питання по логіці DP. Обидва — слушні, зафіксовані тут як
кандидати на майбутні TASK'и / уточнення дизайну.
**Тип:** консультація / спостереження, НЕ TASK.

## Спостереження #1 — «Просто додати файли» не має йти через правку плану

### Що помітив адвокат
На схемі гілка тумблера «Просто додати файли» (skipPdfSlicing) вела через крок
«адвокат править план» (Зона 3). Питання: якщо адвокат свідомо сказав «просто
додай ці PDF як є» — навіщо йому крок редагування плану?

### Вердикт: адвокат правий
Сенс тумблера — **«не думай, поклади як є»**. Проводити його через гейт правки
плану суперечить самій ідеї. Це була помилка **схеми**, не коду.

### Як насправді в коді (станом на 1C)
- `skipPdfSlicingPlan` (`triageStage.js`) будує детермінований план (кожен файл =
  1 документ, add_as_is / image_merge solo), AI Triage пропускається.
- Нарізка PDF зараз працює на `autoConfirm:true` — кроку правки плану **взагалі ще
  немає** (він заплановий на Фазу 5). Тобто фактично тумблер ON вже додає напряму,
  без редагування. Поведінка коду правильна — помилка була лише в діаграмі.

### Дія
- ✅ Схему виправлено: гілка тумблера оминає Зону 3, веде через OCR → підсумок
  «що додано» → у справу.
- ⚠️ **Важливо на Фазу 5:** коли зʼявиться реальний гейт правки плану для нарізки —
  переконатись що тумблер ON його **оминає** (інакше повернемо цей самий конфлікт).
  Записати у вимоги TASK'у Фази 5.
- Можливо доречний легкий **підсумковий екран** «додано N файлів» (звіт, не
  редагування) — для відчуття контролю. Дрібний UX, не пріоритет.

## Спостереження #2 — стискання, можливо, варто робити ПЕРЕД OCR, а не в кінці

### Що помітив адвокат
Зараз «стиснути» — модифікатор у **кінці** пайплайну (для економії місця на Drive).
Адвокат пропонує: скани часто **надлишкової якості**; стискання **на вході** зробило
б файли меншими → OCR і системі легше/швидше працювати. Приклад: iLovePDF тисне PDF
на ~80%, і вони все одно нормально розпізнаються.

### Вердикт: інтуїція правильна, але потребує заміру перед рішенням
**Чому розмір має значення (за нас):**
- Document AI має ліміт ~40 МБ — великі файли доводиться різати на шматки.
- Великі файли — повільніше, дорожче, важче для планшета (RAM).
- Скани часто 600 dpi там, де для тексту достатньо 200-300 dpi — надлишок.

**Нюанси, які треба продумати (проти поспіху):**
- Не перетиснути **нижче** межі читабельності тексту. Золота середина — ~200-300 dpi.
- Рішення: стискати **робочу копію для OCR**, а **оригінал** зберігати у повній
  якості? Чи стискати і фінал, що лягає у справу? Дві різні політики.
- Не перевірено **на нашому конкретному пайплайні** (Document AI), наскільки саме
  пришвидшується і чи якість тримається. Потрібен замір на реальних сканах.

### Дія
- 📋 Кандидат на окремий TASK / під-крок: **pre-OCR компресія** (downsample до
  200-300 dpi для робочої копії, оригінал недоторканий).
- Перед закладанням — **мінімальний замір**: взяти 2-3 важкі скани, стиснути,
  прогнати через наш OCR, порівняти швидкість і якість розпізнавання vs оригінал.
- Тригер: при роботі над модулем стискання АБО при Фазі якості великих томів
  (`TASK_dp_quality_large_volumes` у combined roadmap).
- На схемі позначено як **відкрите питання** (жовтий вузол COMP).
