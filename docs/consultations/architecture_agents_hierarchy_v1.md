## 1. ФІЛОСОФІЯ

Система розвивається як помічник адвоката який з часом набирається досвіду.
На старті — нічого не знає про практику конкретного бюро.
Через рік — знає типові ситуації, процесуальні звички, тактичні підходи,
стиль комунікації, вподобання адвоката.

Цей досвід не губиться між сесіями — він накопичується в інституційній пам'яті
і використовується кожним агентом системи.

Ключовий принцип: система не «розумнішає» сама по собі.
Вона запам'ятовує рішення адвоката і вчиться діяти відповідно до них.

## 2. ІЄРАРХІЯ АГЕНТІВ

```
ГОЛОВНИЙ АГЕНТ (Opus)
— Інституційна пам'ять
— Маршрутизація між модулями
— Тижневий аналіз і оновлення пам'яті
— Відповіді на питання що виходять за межі одного модуля
        ↓
┌──────────┬──────────┬──────────┬──────────────────────┐
│    QI    │ Дашборд  │  Досьє   │    Content Hub       │
│  агент   │  агент   │  агент   │ ┌────────────────┐   │
│          │          │          │ │Агент розвідки  │   │
│ Вхідні   │ Розклад  │ Одна     │ │Агент генерації │   │
│ дані,    │ засідань,│ справа   │ │Агент відео     │   │
│ повістки,│ дедлайни │ глибоко  │ │Агент публікації│   │
│ файли    │          │          │ └────────────────┘   │
└──────────┴──────────┴──────────┴──────────────────────┘
```

### Принципи ієрархії:

**Горизонтальні агенти не спілкуються між собою напряму.**
QI не знає про дашборд. Дашборд не знає про досьє.
Всі запити що виходять за межі модуля — через головного агента.

**Головний агент — диспетчер, не виконавець.**
Він не генерує контент і не змінює дані.
Він маршрутизує, координує і накопичує знання.

**Спільний стан — єдиний канал обміну даними.**
cases[], notes[], calendarEvents[], ideas[] — живуть в App.jsx.
Агенти читають звідти і пишуть тільки через функції App.jsx.

## 3. ІНСТИТУЦІЙНА ПАМ'ЯТЬ

### Де живе:
`institutional_memory.json` — Google Drive, окремий файл.
Не сирі дані — оброблені висновки і патерни.

### Три шари:

**Шар 1 — Процесуальні патерни**
Що зазвичай відбувається в певних категоріях справ.
Типові кроки, документи, строки, послідовність дій.
Джерело: Cases Registry + Case Dossier.

Приклад:
```json
{
  "category": "civil_leasing",
  "pattern": "Запит до банку завжди перший крок",
  "avg_response_days": 30,
  "success_rate_with_early_request": 0.78,
  "learned_from": ["Голобля", "Салун", "Квант"]
}
```

**Шар 2 — Тактичні рішення**
Які аргументи спрацювали, які ні.
Які позиції були успішними в конкретних судах.
Що писалось в Position Constructor і чим закінчилась справа.
Джерело: Case Dossier + Position Constructor.

Приклад:
```json
{
  "court": "Дарницький районний суд",
  "tactic": "Клопотання про забезпечення позову з основним позовом",
  "result": "granted",
  "alternative_tested": "окреме клопотання",
  "alternative_result": "denied",
  "confidence": 0.85
}
```

**Шар 3 — Профіль і вподобання**
Як адвокат працює з системою.
Які рішення приймає в контенті.
Стиль комунікації і типові запити.
Джерело: всі модулі + Content Hub.

Приклад:
```json
{
  "content_preferences": {
    "telegram_facebook_together": 0.87,
    "tiktok_usage": 0.12,
    "rejects_overly_technical": true,
    "preferred_publish_times": ["monday_09:00", "thursday_18:00"]
  },
  "communication_style": "голосові команди вранці, текст ввечері",
  "typical_queries": ["що горить сьогодні", "засідання на тижні"]
}
```

### Як оновлюється:
Opus запускається через n8n раз на тиждень (неділя 23:00).
Читає все що відбулось за тиждень.
Формує нові висновки і додає в пам'ять.
Надсилає тижневий звіт адвокату в Telegram.

## 4. МАРШРУТИЗАЦІЯ МІЖ МОДУЛЯМИ

### Приклади маршрутизації:

**Запит з дашборду що виходить за межі:**
«Розкажи детально по справі Бабенко»
→ Агент дашборду: «Це глибоке питання — відкриваю досьє»
→ Головний агент: навігує в CaseDossier
→ Агент досьє підхоплює з повним контекстом справи

**Ідея для контенту з реєстру:**
Адвокат натиснув [💡] в картці лізингової справи
→ QI передає в Content Hub
→ Агент розвідки Content Hub перевіряє:
  чи є свіже рішення ВС по лізингу?
→ Якщо є — пропонує тему з додатковим контекстом

**Тижневий аналіз головного агента:**
Opus читає: нові справи, засідання, контент, рішення адвоката
→ Оновлює institutional_memory.json
→ Кожен агент при наступному запуску читає оновлену пам'ять

## 5. ТЕХНІЧНА АРХІТЕКТУРА (заготовка для реалізації)

### Структура даних:

```javascript
// App.jsx — додати до спільного стану
const [institutionalMemory, setInstitutionalMemory] = useState({
  proceduralPatterns: [],  // Шар 1
  tacticalDecisions: [],   // Шар 2
  profilePreferences: {}   // Шар 3
});

// Функція оновлення пам'яті (викликається головним агентом)
const updateMemory = (layer, data) => {
  setInstitutionalMemory(prev => ({
    ...prev,
    [layer]: [...prev[layer], data]
  }));
  // + зберегти в Drive
};
```

### System prompt головного агента:

```
Ти — головний агент Legal BMS адвокатського бюро Левицького.
Твоя роль: координація між модулями і накопичення досвіду системи.

Ти маєш доступ до:
- Всіх справ і їх стану
- Інституційної пам'яті (три шари)
- Поточного стану всіх модулів

Ти НЕ виконуєш оперативні задачі.
Ти маршрутизуєш, аналізуєш і вчишся.

При тижневому аналізі:
1. Переглянь всі події тижня
2. Виділи патерни і висновки
3. Онови institutional_memory.json
4. Сформуй звіт для адвоката
```

### Принцип підключення до існуючої системи:

Головний агент підключається через App.jsx мінімально:
- Отримує props: всі спільні стани + функції оновлення
- Запускається через окремий тригер (n8n або кнопка)
- Не втручається в роботу модулів — тільки читає і координує

Це і є принцип стільника: головний агент — окрема комірка
яка знає про всіх але не залежить від жодного.

## 6. ВИМОГИ ДО НОВИХ МОДУЛІВ

Кожен новий модуль при розробці повинен:

1. **Логувати рішення адвоката** в стандартному форматі:
```javascript
logDecision({
  module: 'content_hub',
  action: 'approve_post',
  context: { topic, platforms, time },
  result: 'approved'
});
```

2. **Читати релевантну частину пам'яті** при ініціалізації:
```javascript
const relevantMemory = institutionalMemory
  .filter(m => m.category === currentCategory);
```

3. **Передавати головному агенту** запити що виходять за межі:
```javascript
if (queryScope === 'cross-module') {
  routeToMainAgent(query, currentContext);
}
```

4. **Бути ізольованим** — падіння модуля не впливає на пам'ять
   і не блокує інші модулі.

## 8. ЗНАННЯ СФОРМУЛЬОВАНІ ЛЮДИНОЮ — НАЙЦІННІШЕ ДЖЕРЕЛО

### Принцип:
Система збирає факти автоматично.
Але знання які може сформулювати тільки адвокат —
найцінніше що є в базі. Їх не можна отримати з даних.

### Три джерела знань (за цінністю):

1. ЛЮДСЬКІ СПОСТЕРЕЖЕННЯ (найцінніше)
   Нотатки адвоката позначені ⭐
   «Суддя X в ситуації Y поводиться так»
   «Ця тактика в цьому суді не працює»
   Формулює тільки людина — система зберігає і використовує

2. АВТОМАТИЧНІ ДАНІ (висока кількість)
   Факти, події, рішення, хронологія
   Система збирає сама без участі адвоката

3. ЗОВНІШНІ ЗНАННЯ (загальний контекст)
   Судова практика, публікації, зміни закону
   n8n збирає, Opus аналізує

### Механізм фіксації людських спостережень:

В будь-якому місці системи:
  Нотатка → кнопка [⭐ Важливе спостереження]
  
  Міні-форма:
  - Про що: суддя / суд / тактика / клієнт / процес
  - Контекст: яка справа / яка ситуація
  - Зберегти в базу знань

Opus обробляє і формує структурований запис:
{
  "type": "human_insight",
  "source": "advocate_observation",
  "subject": "...",
  "insight": "...",
  "confidence": "high/medium/low",
  "confirmed_by": [...],
  "applicable_when": "...",
  "date_observed": "...",
  "cases_context": [...]
}

## 9. ГОЛОВНИЙ АГЕНТ — ПОВНІ МОЖЛИВОСТІ

### Три шари знань при відповіді на питання:

1. База знань бюро (внутрішнє)
   — Інституційна пам'ять (патерни, тактики)
   — Людські спостереження адвоката (⭐ нотатки)
   — RAG пошук по документах справ

2. Веб-пошук (зовнішнє)
   — Свіжа судова практика
   — Зміни законодавства
   — Публікації і аналітика

3. Загальні знання Opus
   — Юридична база
   — Процесуальні норми

При відповіді завжди розмежовує джерела:
«З твого досвіду...», «З практики ВС...», «З відкритих джерел...»

### Навички які розвиваються а не прописуються:

Головний агент не має жорстких правил типу
«якщо X то Y». Він має контекст і вчиться:

- Читає як адвокат формулює питання
- Запам'ятовує які відповіді були корисними
- Адаптує стиль і глибину відповіді
- З часом передбачає що буде потрібно далі

Це досягається через:
- logDecision() на кожну взаємодію з агентом
- Opus аналізує ці взаємодії тижнево
- Оновлює профіль комунікації в пам'яті

## 10. ВІКНО СПІЛКУВАННЯ З ГОЛОВНИМ АГЕНТОМ

### Що це:
Окремий чат-інтерфейс з агентом який знає все.
Не QI (для нових даних), не агент модуля (для конкретної задачі).
Це «розумний партнер» для обговорення тактики,
аналізу ситуацій, отримання рекомендацій.

### Приклади запитів:
«Яка тактика по ОГД в адміністративних судах?»
«Що ти знаєш про суддю Коваленко в Печерському?»
«Порівняй справи Голобля і Бабенко — що спільного?»
«Скільки в середньому тривають мої лізингові справи?»
«Допоможи сформулювати позицію по справі Рубан»

### Збереження результатів розмови:

Після кожної розмови — одна кнопка з варіантами:
[📋 В конструктор позицій]
[📝 Зберегти як нотатку]
[📄 Створити документ]
[💾 В базу знань]
[Нічого не зберігати]

Нічого не губиться — кожна цінна думка
може бути перетворена на артефакт системи.

### Зв'язок з іншими модулями:

Розмова про справу Рубан →
  [В конструктор позицій] →
    Блоки Position Constructor заповнюються
    висновками розмови

Тактичне спостереження →
  [В базу знань] →
    Структурований запис human_insight
    доступний всім агентам системи

## 11. RAG — ПОШУК ПО ДОКУМЕНТАХ СПРАВ

### Проблема:
Документів стає багато. Перечитувати все — неефективно.
Потрібен швидкий пошук «схожого за змістом».

### Рішення — векторна база:

Кожен документ при завантаженні:
  Текст → Embedding (числовий відбиток змісту)
  Зберігається у векторній базі

При запиті агента:
  Питання → Embedding питання
  Пошук найближчих за змістом документів
  Знайдено за секунди → передано Opus

### Що індексується:
- Документи справ (завантажені через QI або Досьє)
- Правові позиції з Position Constructor
- Нотатки позначені ⭐
- Огляди практики ВС

### Технічний стек (заготовка):
- Embeddings: Anthropic Embeddings API або OpenAI
- Векторна база: Supabase pgvector (безплатний tier)
  або Pinecone (хмарний, простий старт)
- Інтеграція: через n8n або напряму з Legal BMS

### Коли реалізовувати:
Після накопичення 50+ документів в системі.
До цього — достатньо пошуку по тексту.
