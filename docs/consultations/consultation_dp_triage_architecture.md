# DP v2 — Архітектурна пропозиція: «кинь усе, DP сам розбереться»

**Дата:** 18.05.2026 · **Ревізія:** 1.1 (додано вартісну модель text-first)
**Тип:** Консультація / архітектурний план (БЕЗ коду, БЕЗ TASK) — на запит
адвоката після root-cause розслідування
**Передумова:** адвокат пояснив суть задачі простими словами; потрібен план
як зробити DP таким, плюс варіанти і рекомендація.
**Ревізія 1.1:** адвокат підняв питання оптимізації витрат — з'ясовано що
Document Block платно рендерить кожну сторінку як зображення (Vision), і
що Document AI вже повертає і **персистить** структуру сторінки (абзаци,
блоки, орієнтація, dimension). Triage переведено на **text-first через цю
структуру + Haiku**, vision — лише вузький fallback. Див. §4 «Вартісна
модель» і оновлені §6/§7/§8.

---

## 1. Суть задачі (одним абзацом)

Адвокат кидає у DP **будь-який набір файлів** (фото з айфону, великий скан-
PDF, ZIP з суду, кілька PDF-фрагментів одного документа, .p7s-підписи —
усе разом або по черзі). DP **сам дивиться у вміст** (через AI, як дивився
б адвокат очима) і вирішує що з кожним зробити. На виході —
`01_ОРИГІНАЛИ/` з готовими PDF, **один PDF = один логічний документ**;
службові файли окремо/відкинуті. **Один вхід, без режимів і кнопок.**

## 2. Що ВЖЕ є в репо (інвентар — більшість будівельних блоків готова)

| Блок | Файл | Що вміє | Для сценарію |
|---|---|---|---|

| **Multi-file реконструкція** | `documentBoundary/multiFileReconstructor.js` | склейка фрагментів одного документа з кількох файлів (openTails) | складний #4 |

який дивиться на змішаний вхід і вирішує **маршрут для кожного файла/групи**.

## 3. Корінь проблеми поточної архітектури

DP v2 має **жорстке припущення** замість «дивитись очима»:

- `detectBoundariesV3`: `files > 1 → реконструкція`, `files == 1 →
  detectSingle`. Це і є «жорстке правило за кількістю файлів», яке адвокат
  прямо назвав неправильним.
- BUGFIX `shouldReconstruct: >=1` зробив гірше — погнав single-file у
  multi-file реконструктор (промпт «пакет/хвости», не «наріж цей PDF»).
- §4.4 «після convert+OCR усе = текст» — зручно для коду, але у поточному
  вигляді **викидає структуру** яку OCR уже порахував. Перший DP працював
  бо **надсилав PDF Клоду цілим** (Document Block — Клод бачить сторінки).
  АЛЕ: Document Block платно **рендерить КОЖНУ сторінку як зображення**
  (Vision, ~1.5-2К токенів/стор. понад текст) — це і робить його надійним
  на сканах, і це **домінуюча стаття витрат DP**. Правильний синтез (див.
  §4 «Вартісна модель»): візуальну роботу вже зробив Document AI на боці
  Google і **зберіг структуру** — повторно платити Клоду за пікселі не
  треба.

**Немає компонента «диспетчер/тріаж»** — кроку де AI дивиться на весь
набір і будує план: що з кожним зробити.

## 4. Запропонована архітектура: **AI Triage / Dispatcher**

Один новий «мозок» між нормалізацією і нарізкою. Лягає на ІСНУЮЧИЙ
диригент **без його зміни** (як stageOverride для `DETECT_BOUNDARIES` —
саме там зараз живе рішення «що це за вхід»).

### Конвеєр (концептуально)

```
кинули все → INTAKE(unpack: ZIP розпак, .p7s убік, sidecar)
           → CONVERT(кожен не-PDF → PDF; зображення → 1-стор. PDF, з міткою "image")
           → [OCR/прев'ю: streaming chunk-OCR + thumbnail 1-ї/останньої стор.]
           → ★ TRIAGE (новий мозок): AI дивиться на ВСІ артефакти і будує
                      ЄДИНИЙ ПЛАН з маршрутом для кожного
           → CLASSIFY (category/author) → EXTRACT (clean text)
           → CONFIRM (зараз auto; UI-підтвердження — DP-6)
           → PERSIST: виконати маршрут кожного запису плану відповідним
                      ІСНУЮЧИМ обробником → 01_ОРИГІНАЛИ
           → EMIT
```

### Що бачить Triage (вхід у промпт диспетчера)

Для кожного артефакту (після unpack+convert+OCR) — компактний **текстовий
структурний паспорт**, зібраний зі вже наявного `pageStructure` /
`.layout.json` (нуль image-токенів):
- ім'я, mime/походження (pdf / був-image / був-docx), розмір, к-сть сторінок;
- **per-page дайджест**: текст сторінки + геометрія першого/останнього
  блоку (короткий центрований заголовок? футер з номером?), `orientation`,
  `dimension` (A4-скан vs фото-пропорції), прапори `tables`/`formFields`,
  детект скидання нумерації — все це Document AI **уже порахував і ми
  персистимо** (див. §4 «Вартісна модель»);
- крос-файлові сигнали: спільний timestamp/послідовні імена (IMG_001..),
  «обрив» тексту в кінці одного PDF і продовження на початку іншого;
- **vision лише як вузький fallback**: тільки ті сторінки, де Document AI
  повернув ~нуль і тексту, і структури (справжнє фото/чистий скан без
  шару) — їм image-блок реально потрібен; решта йде текстом.

### Що Triage повертає — єдиний план з маршрутами

Розширення наявного формату плану (`{documents, unusedPages}` вже є —
додаємо `route` на запис):

| route | Що означає | Виконавець (вже існує) |
|---|---|---|
| `add_as_is` | готовий single-doc PDF (типово з е-суду) | passthrough → upload 01_ОРИГІНАЛИ |
| `slice` | один PDF з багатьма документами | `documentBoundary.detectBoundaries` — переведений на структурний паспорт (text-first), vision-fallback лише для сторінок без шару |
| `image_merge` | група фото = сторінки одного документа | `imageSortingAgent`+`orientationCorrector`+`cropHelper`+`imageRenderer` |
| `fragment_reconstruct` | один документ розрізаний по кількох PDF | `reconstructAcrossFiles` (openTails) |
| `signature_sidecar` | .p7s/.sig | вже зроблено в `unpack` (`ctx.signatures[]`) |
| `to_fragments` | обкладинка/штамп/порожнє | → `03_ФРАГМЕНТИ` (вже є в splitDocumentsV3) |
| `discard` | сміття | відкинути + decision |

`CONFIRM` (зараз autoConfirm; повний propose→confirm UI — DP-6) лишається
гейтом. `PERSIST` (splitDocumentsV3) стає **диспетчером виконання**: для
кожного запису плану викликає відповідний існуючий обробник. Диригент і
9 стадій — недоторкані (усе через stageOverrides + поле `route` у плані).

**Ключ:** «дивитись очима» НЕ означає «платити Клоду за пікселі». Document
AI **уже подивився** (відрендерив і структурував кожну сторінку на боці
Google) і ми **зберігаємо результат**. Triage використовує цю структуру
**як текст**; vision Клода — лише там де структури об'єктивно нема (чистий
скан/фото). Те що працювало в першому DP (Клод бачить структуру сторінки)
зберігається; те що його робило дорогим (повторний рендер пікселів) —
прибирається.

### Вартісна модель — text-first через структуру Document AI

**Факт із коду:** `documentAi.js:209-211` повертає `pageStructure` = **весь
оригінальний об'єкт сторінки** (paragraphs, blocks з bbox-геометрією,
`layout.orientation`, `dimension`, tables, formFields, мови, якість).
`ocrService.js:139` при серіалізації в `.layout.json` викидає **рівно 2
поля**: `image` (base64-PNG рендер — ~7 МБ/стор., **саме за це платиться
Vision**, надлишковий бо оригінал у 01_ОРИГІНАЛИ) і `tokens` (по-літерні
координати — не треба). **Уся структурна метадані зберігається** — і в
пам'яті, і в `.layout.json` (тобто доступна і на resume).

Сигнали промпта пошуку меж лягають на цю метадані **як текст**:

| Сигнал | Джерело (вже є, безкоштовно текстом) |
|---|---|
| нові заголовки | геометрія першого блоку сторінки (короткий центрований абзац) |
| нова нумерація | текст футер-блока скидається на 1 |
| зміна типу | зміна тексту + поява/зникнення `tables`/`formFields` |
| орієнтація/формат | `layout.orientation` + `dimension` (фото vs A4-скан) |
| печатки/підписи | `visualElements`+`imageQualityScores` — **частковий**, тут vision-fallback |

3 з 4 ключових сигналів повністю відновлюються зі збереженої структури.
Економія: на текстовому PDF е-суду зникає ~1.5-2К image-токенів/стор. —
типово **5-15× дешевше**; + Triage/slice на **Haiku** (структурна задача,
~1/3 ціни Sonnet). Vision лишається тільки на сторінках-фото без шару.

## 5. Перевірка: усі сценарії проходять ОДНИМ шляхом

| Сценарій | Маршрут від Triage |
|---|---|
| **A** 5 фото однієї сторінки | усі → `image_merge` (1 група) → sort+rotate+crop+merge → 1 PDF |
| **B** PDF 100 стор. зі справою | `slice` → detectBoundaries → N документів |
| **C** ZIP: PDF+JPG+.p7s | unpack: .p7s→sidecar; PDF→`add_as_is`; JPG(и)→`image_merge` або `add_as_is` |
| **D** ZIP: документ+додатки+.p7s | unpack; ухвала→`add_as_is`; фото-додатки→`image_merge`; .p7s→sidecar |
| **Складний мікс** (1+2+3+4 разом) | один Triage-прохід: великий PDF→`slice`; фото-група→`image_merge`; ZIP-вміст розкладається; PDF-фрагменти→`fragment_reconstruct`; .p7s→sidecar — **усе за один захід, без режимів** |

Жодного різного UI/кнопок — адвокат кидає все, Triage маршрутизує.

## 6. Варіанти реалізації (2-3) + рекомендація

### Варіант 1 — **Smart Triage** (один AI-диспетчер → існуючі обробники) ⭐ рекомендований

AI-крок будує єдиний план з `route`; PERSIST виконує кожен маршрут наявним
обробником.
- **+** переюз ~85% коду; один вхід; диригент незмінний; точно
  відповідає словам адвоката («AI дивиться і вирішує»); природно тягне
  A-D і складний; нарощується інкрементально (спершу PDF-маршрути, далі
  image/fragment).
- **−** промпт Triage складний (гетерогенний вхід); крос-файлове виявлення
  фрагментів — найважче; ризик хибного маршруту → потрібен «перегляд
  плану» (DP-6 UI) як страхувальна сітка. (Токен-ризик знятий вартісною
  моделлю §4: вхід text-first зі структури, vision — вузький fallback.)

### Варіант 2 — **Двопрохідний** (1: класифікувати кожен файл; 2: згрупувати)

- **+** простіші per-file промпти, паралельно, детермінованіше/тестованіше.
- **−** два AI-проходи; роль файла часто видно лише в контексті інших
  (PDF-фрагмент сам по собі = «цілий документ») → 1-й прохід помиляється;
  найважче (групування фрагментів) усе одно лишається у 2-му.

### Варіант 3 — **Гібрид: дешеві евристики + AI лише для неоднозначного**

Очевидне — правилами (single малий PDF→add_as_is; ZIP→unpack;
.p7s→sidecar; послідовні IMG_*→image group); AI лише там де неясно.
- **+** найдешевший, найшвидший для частих простих (A,C,D).
- **−** евристики = ті самі «жорсткі правила» які адвокат відкинув;
  крихке на нерегулярних реальних даних; суперечить «AI дивиться і
  вирішує».

**Рекомендація:** **Варіант 1** як цільова архітектура (відповідає
філософії AI-first і словам адвоката), із трьома уточненнями по витратах:
**(а)** вхід Triage/slice — **text-first структурний паспорт** зі вже
персистованого `pageStructure`/`.layout.json` (§4 «Вартісна модель»),
vision — вузький fallback лише для сторінок без текстового шару; **(б)**
Triage/slice на **Haiku** (структурна задача, ~1/3 ціни Sonnet — звірити
`resolveModel('documentProcessor')`); **(в)** тонка детермінована сітка
безпеки з Варіанта 3 для абсолютно однозначних дешевих випадків (.p7s,
ZIP, явний 1-стор. фото) — не «режими», а невидима пре-фільтрація. Повний
propose→confirm перегляд плану = DP-6 UI (вже закладено в архітектуру).

## 7. Супутні фікси (виявлені root-cause розслідуванням — фолдимо сюди)

1. **`fetch` без timeout** (`toolUseRunner.js:352`) → будь-який AI-виклик
   може підвісити весь pipeline назавжди. **Обов'язково** AbortController.
2. **`shouldReconstruct: >=1`** (поточний main) — відкотити: Triage замінює
   жорсткий gate повністю.
3. **Resume не підключений до UI** — Provider при mount має сканувати Drive
   на незавершені job_state і пропонувати «продовжити».
4. **`resolveModel('documentProcessor')` тягне Sonnet** — для структурної
   задачі пошуку меж достатньо **Haiku** (~1/3 ціни); узгодити ключ/
   agentType при wiring (це і є важіль «в» з §6).
5. **Текст у промпт `slice(0,50000)` без маркерів сторінок** — замінити на
   **per-page структурний паспорт** (текст + дайджест геометрії блоків +
   orientation + dimension + прапори), зібраний зі збереженого
   `pageStructure`. Це і дає сигнал меж, і прибирає Vision-вартість, і
   знімає 50К-обрізку (паспорт компактний по конструкції).

## 8. Оцінка обсягу

**Переюзується (готове, ~85%):** unpack, converterService, detectBoundaries
(slice), reconstructAcrossFiles, imageSortingAgent + orientationCorrector +
cropHelper + edgeDetection + imageRenderer, classifyV2, extractV3,
splitDocumentsV3, streamingExecutor, диригент.

**Нове / суттєва доробка:**
- **Білдер структурного паспорта** зі `pageStructure`/`.layout.json`
  (per-page дайджест: текст + геометрія блоків + orientation + dimension +
  прапори + детект скидання нумерації) — нова чиста функція, тестована:
  ~3-4 год.
- Triage-стадія + промпт-дизайн (вхід — паспорт, не пікселі) — **ядро**:
  ~8-11 год (трохи менше ніж раніше — вхід тепер текстовий і стабільний).
- PERSIST як диспетчер маршрутів (під'єднати image_merge / reconstruct /
  add_as_is / slice до плану): ~6-9 год.
- Детермінована сітка безпеки (.p7s/ZIP/явні випадки): ~2-3 год.
- Супутні фікси (fetch timeout, revert shouldReconstruct, resume-wire,
  model agentType): ~4-6 год.
- Тести (unit стадій + інтеграційні A-D + складний мікс, моки AI): ~6-8 год.

**Разом орієнтовно ~29-41 год** (1 великий TASK у 4-5 фаз; фази
самодостатні — можна зупинятись). Білдер паспорта — окрема рання фаза
(розблоковує і Triage, і дешевший slice незалежно).

**Ризики:**
- **Якість Triage-рішень** на реальних нерегулярних даних — головний
  ризик; пом'якшення: структурний паспорт (Document AI уже структурував),
  DP-6 перегляд плану перед нарізкою, детермінована сітка для очевидного.
- **Втрата суто-графічного сигналу** (печатка/підпис без OCR-тексту) у
  text-first — вузький випадок; пом'якшення: vision-fallback саме на
  сторінки де Document AI повернув ~нуль структури (рідко в е-суді);
  `imageQualityScores`/`visualElements` дають частковий сигнал і текстом.
- **Токени/вартість** — **знято вартісною моделлю §4** (нуль image-токенів
  на текстових PDF + Haiku); лишковий ризик низький.
- **Крос-файлове виявлення фрагментів** (сценарій #4) — найважче для AI;
