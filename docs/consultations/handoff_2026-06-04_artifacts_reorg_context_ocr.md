**Дата:** 2026-06-04
**Тип:** handoff для **адмін-сесії** (керує процесом, пише спеки/таски для сесій-виконавців; сама код не пише, крім дрібних розвідок).
**Призначення:** звести в один логічний напрям три пов'язані речі — відмову від `.txt`, селективний контекст-генератор і керований OCR. **Тут — детальна проблематика і кінцевий результат. Конкретних варіантів рішення НЕ пропоновано навмисно** (це продукт адмін-сесії).

## 0. Роль цієї сесії

1. Прочитати `CLAUDE.md` + `DEVELOPMENT_PHILOSOPHY.md`.
2. **Термінологія (узгоджено з адвокатом, тримати скрізь):**
   - **OCR / розпізнавання** — картинка → текст (скан/фото, тексту в файлі нема).
   - **Зчитка** — витяг наявного текстового шару з цифрового PDF (нічого не розпізнається).
   - **Отримати текст** — парасолька (система сама обирає OCR чи зчитку).
3. Продукт сесії — **набір узгоджених спек/тасків** (послідовність, залежності, кожен з SAAS/BILLING-секціями і тестами), а не код.
4. Працювати за правилом №1 (гілки). Перед спеками — **залізти в код самій** і перевірити стан (нижче — стартова мапа, не заміна власної перевірки).

## 1. КІНЦЕВИЙ РЕЗУЛЬТАТ, ЯКОГО ХОЧЕ АДВОКАТ (три в одному)

Три цілі мають скластися в **ОДИН логічний алгоритм**, а не три розрізнені фічі:

1. **Відмова від `.txt`.** Текстові файли не створюються. Їх **повністю заміняє `.layout.json`**, який лишається після OCR і містить **той самий текст** (+ геометрію). `.txt` — дубль, не потрібен. Для **searchable**-документів текст дістає спеціальна функція (зчитка) на вимогу і передає агентам/іншим частинам — **без зберігання дубля**.

2. **Узгоджений контекст-генератор** — однакова, логічна поведінка в **усіх** місцях, де він є/планується (Огляд справи, Document Processor), **масштабована** (не «лопатить» усю справу) і **керована** (вибіркова участь документів — повний / вижимка / не включати).

3. **Гнучкий і керований OCR.** Адвокат визначає, **які документи розпізнаються** (→ створюються їх копії-лейаути в `02_ОБРОБЛЕНІ` + ідуть у контекст), а **які просто переносяться в `01_ОРИГІНАЛИ` без артефактів у `02`** — що **не заважає** розпізнати їх пізніше окремо: **по одному / усі одразу / вибрати кілька**.

**Зв'язок усіх трьох:** один «важіль» на документі (концепт із попередньої консультації) має керувати разом: робити OCR→лейаут / тільки вижимку / нічого + рівень участі в контексті. Відмова від `.txt` прибирає дубль; керований OCR вирішує, чи взагалі робити лейаут; контекст-генератор споживає результат вибірково.

## 2. ДЕТАЛЬНА ПРОБЛЕМАТИКА (стан коду — звірено 2026-06-04)

### 2.1 Поточна модель артефактів
- **`extractText`** (`src/services/ocrService.js:355`): (1) читає **`.txt`-кеш** першим (`checkCache`, ~361) → якщо є, повертає з `.txt`; (2) ланцюг провайдерів (`pdfjsLocal` — зчитка text-layer; `documentAi` — OCR сканів; `claudeVision` — фолбек); (3) пише **і `.txt`** (`textCacheFileName`, ~457), **і `.layout.json`** (`layoutCacheFileName`, ~465).
- **`.layout.json`** (`serializeLayout`, ~163): `{schemaVersion, provider, generatedAt, pages:[...]}`. `stripHeavyFields` викидає лише `image` + `tokens`, але **зберігає `page._text`** + геометрію (boundingPoly). → **`.txt` = строгий дубль конкатенації `page._text` із лейаута.**

### 2.2 Хто читає `.txt` (усі точки, які доведеться перевести на лейаут/зчитку)
- `getCachedText(file)` → `.txt` (`checkCache`): **DocumentViewerFooter** (`DocumentViewerFooter.jsx:81`), **clean-text adapter** (`cleanTextDriveAdapter.js:37` `fetchRawText`).
- `getCleanOrRawText(file)` → `.md` інакше `.txt`: **DocumentViewerContent** (`:194`).
- `extractText`/`extractTextBatch` → `.txt`-кеш першим: **контекст-генератор** (`contextGenerator.js:497`).
- `getCachedLayout(file)` → `.layout.json`: **clean-text КРОК 1** (`layoutToMarkdownDraft` читає `page._text`). → тобто **читання з лейаута вже існує і працює**.

### 2.3 Едж-кейси, де `.txt` НЕ можна просто прибрати
- **DOCX/HTML-конвертовані** (`converter/converterService.js`): текст від mammoth/innerText → `writeExtractedTextArtifact` пише `.txt`, `documentNature='searchable'`, **Document AI НЕ кличеться, `.layout.json` НЕ створюється**. → для них `.txt` — **єдине** сховище тексту. Прибрати наосліп = втратити текст. **Потрібне окреме рішення.**
- **Searchable native PDF** (цифровий, e-court): текст у шарі PDF; `pdfjsLocal` зчитує на вимогу. Зараз `extractText` усе одно кешує `.txt`. → дубль, але прибирати треба узгоджено зі зчиткою на вимогу.

### 2.4 OCR при нарізці — обов'язковий і транзитний
- `streamFile` (`streamingExecutor.js:103`) **завжди** чанк-OCR (для визначення меж). Навіть **add_as_is** («Просто додати файли», `triageStage.js:85 skipPdfSlicingPlan`) проходить через `streamFile` → OCR + запис артефактів. → **Зараз НЕМА шляху «додати документ у `01_ОРИГІНАЛИ` без OCR і без `02`-артефактів».** Це треба створити.

### 2.5 On-demand розпізнавання — існує частково, не як система
- `RecognizeTextModal` (`DocumentProcessorV2/modals/RecognizeTextModal.jsx`): один файл → `extractText({skipCache:true})` → текст у viewer + опції збереження. Re-OCR існуючого: viewer `onReprocess` → `extractText(skipCache)`. → точки розрізнені; **немає керованого «обрати документи (1/усі/кілька) і розпізнати»** (це зв'язок із мультивибором, п.4).

### 2.6 Контекст-генератор — читає ВСЕ
- `generateCaseContext` (`contextGenerator.js`): джерело = `caseData.documents` (канонічний SSOT) → `extractTextBatch` по всіх документах (читає `.txt`-кеш або OCR-ить) → склейка всього тексту → AI. **Читає тексти ВСІХ документів.** Виклики: вкладка Огляд + DP-подія `DOCUMENT_BATCH_PROCESSED`. → не масштабується; розрослій справі переповнить контекстне вікно (час/вартість/якість). Концепт розв'язку (рівні участі «важіль» + бюджет по сторінках + інкремент) — у попередній консультації адвоката, **узгодити сюди**.

## 3. НАПРЯМКИ (простір для спек — НЕ готові рішення)

Адмін-сесія опрацьовує ці напрями і розкладає на таски. Варіанти **навмисно не вибрані**.

- **A. Єдине канонічне джерело «отримати текст».** Після відмови від `.txt` визначити одну точку, звідки будь-який споживач (viewer, контекст, агент, clean-text) бере текст: скан → з лейаута (`page._text`); searchable → зчитка на вимогу; DOCX/HTML → див. B. Усі читачі `.txt` із п.2.2 переводяться на неї.
- **B. Доля тексту DOCX/HTML** (єдиний випадок без лейаута) — окреме рішення напряму.
- **C. Керований OCR і його взаємодія з обов'язковим OCR нарізки.** Розмежувати «транзитний OCR для меж» (завжди) від «персистентний лейаут-артефакт» (за вибором). Як вибір виглядає для шляху з нарізкою vs «просто додати».
- **D. Шлях «у `01_ОРИГІНАЛИ` без `02`-артефактів» + відкладене розпізнавання** (по одному / усі / вибрані) — зв'язок зі спільним компонентом вибору (TASK 3.3).
- **E. Селективний контекст-генератор** — рівні участі (повний / вижимка / не включати), поріг і бюджет **по сторінках** (`Σ pageCount`), інкрементальне оновлення; узгодити Огляд і DP в одну логіку.
- **F. Єдиний «важіль»** на документі, що зв'язує C+D+E: один вибір керує OCR→лейаут / вижимка / нічого + рівнем у контексті.

## 6. ОБМЕЖЕННЯ / ЩО НЕ ЗЛАМАТИ

- **Нарізка** завжди має достатньо тексту для визначення меж (транзитний OCR не прибирати).
- Не втратити доступ до тексту для: viewer (.md/текст), агента досьє, контексту, clean-text.
- **DOCX/HTML текст не загубити** (єдине сховище — `.txt`).
- **Схема:** новий «важіль»/OCR-стан документа → ймовірний bump `schemaVersion 11` + ідемпотентна міграція (правило #6). Нове поле — один сенс (правило #11).
- **SAAS/BILLING (правило #10):** OCR = `ai_usage` + вартість Document AI; **керований OCR = пряма економія токенів/часу/Drive-квоти**; вижимки = Haiku; тариф може лімітувати глибину контексту.
- Відмова від `.txt` — фактично зміна формату артефактів справи → перевірити чи потрібна разова **міграція наявних `.txt`** (читати лейаут; старі `.txt` лишити чи прибрати) + бекап перед міграцією.
