## СУТЬ ІДЕЇ (як описав адвокат)

Як у GitHub: `.md` із блоком ```` ```mermaid ```` рендериться як **візуальна блок-схема**. Зробити це
у в'ювері Legal BMS. Схеми ілюструють:
- **рух справи** (timeline/flowchart);
- **дерево проваджень**;
- **зв'язки між суб'єктами**: корпоративні між юрособами; між фізособами; сторони у справі.

**Механіка через агента досьє (а не просто кнопку):**
- Кнопка у в'ювері «згенерувати схему» сама по собі **слабка** — не дасть потрібного ефекту.
- Краще — **спеціальний режим агента**: адвокат тисне відповідну кнопку (рух справи / дерево
  проваджень / зв'язки суб'єктів / **вільна тема**) → агент отримує **контекстний файл-шаблон з
  інструкціями** (малювати в Mermaid, що саме малювати, типові приклади) → одразу розуміє завдання
  → **задає серію уточнюючих питань** (як це робила сесія-виконавець) → формує `.md` з mermaid-кодом
  → рендериться у в'ювері як схема/алгоритм.
- Набір: типові шаблони + вільна тема (адвокат пояснює без шаблону).

**Де:** створення/редагування — **Canvas** (зручніше); перегляд — **в'ювер** (рендер); зберігання —
`.md`-файли.

## ОЦІНКА (chat-сесія, 2026-06-02)

**Здійсненність — висока.**
- Рендер Mermaid: або `mermaid.js` lazy-load (патерн `html2pdf`/`heic2any` — важкі залежності через `import()`),
  або розширити наш `MarkdownRenderer` на блок ```` ```mermaid ````. Репо **вже масово** використовує
  Mermaid у `docs/mermaid/` — формат знайомий.
- Агент-генерація: **природний фіт для LLM** (Mermaid — це текст; моделі добре його пишуть).
  Шаблон-driven підхід (окремий контекст-файл per тип схеми) — розумно: фокусовані інструкції + приклади
  → агент точніше розуміє намір. Дзеркало патерну «спеціалізований режим/tool use».

**Цінність — висока.** Візуалізація складних справ (рух, дерево проваджень, граф зв'язків) — реально
корисна адвокату для розуміння й презентації.

**🔑 Ключове підсилення (синергія, якої адвокат прямо не назвав):**
**Ґрунтувати схеми на СТРУКТУРОВАНИХ даних справи**, а не лише на розмові. У схемі справи (v7 ECITS-каноніка)
вже є: `parties[]`, `processParticipants[]`, `proceeding.composition` (склад суду), `movementCard` (картка
руху), `hearings[]`, `alternativeSources`. Тобто:
- **дерево проваджень** і **рух справи** можна будувати з `proceedings`/`movementCard`/`hearings` —
  агент малює з **реальних даних**, не вигадує;
- **зв'язки суб'єктів** — з `parties`/`processParticipants`.
Це робить схеми **точними**, а не приблизними, і дає сильну синергію з уже наявними ECITS-полями.
Free-form-тема лишається для випадків поза структурою.

**Застереження:**
- `mermaid.js` — зовнішня залежність (репо свідомо уникає bloat — власний `MarkdownRenderer` без npm).
  Рішення: lazy-`import()` (як heic2any), щоб не роздувати основний бандл. Або вузький власний рендер
  для підмножини Mermaid (складніше — Mermaid багатий).
- Це **окремий чималий TASK** (Canvas + mermaid-рендер + агент diagram-mode + шаблони + ґрунтування на даних).

## ДЕ РЕАЛІЗУВАТИ (узгоджено з інстинктом адвоката)

- **Створення/редагування** → модуль **Canvas** (планований). AI-first: кнопки типів схем + вільна тема;
  агент отримує шаблон-контекст per тип → уточнює → генерує `.md`(mermaid).
- **Перегляд** → **в'ювер** (рендерить ```` ```mermaid ```` як SVG; lazy-load mermaid.js).
- **Зберігання** → `.md`-файли (з mermaid-кодом) — як артефакт справи (узгодити: окремий тип/папка).

## КОЛИ

Коли візьметься **TASK Canvas-конструктор документів** (роадмап) — закласти diagram-mode як його частину,
з шаблонами схем і ґрунтуванням на структурованих полях справи (ECITS-каноніка). Рендер Mermaid у в'ювері
можна закласти й раніше (невеликий самостійний крок: ```` ```mermaid ```` → SVG).
