## 1. КРИТИЧНА КОРЕКЦІЯ ПЕРЕДУМОВИ (звірено з кодом)

Handoff §2.1 описує **стару** модель артефактів. Фактичний стан інший:

### 1.1 `.txt` і `.layout` більше НЕ співіснують як дубль
`ocrService.js:561-583` (V2-A2):
```js
const hasPageStructure = Array.isArray(finalPageStructure) && finalPageStructure.length > 0;
// .txt пишемо ⇔ layout ВІДСУТНІЙ
if (finalText && finalText.trim().length > 0 && !hasPageStructure) { writeArtifact(.txt) }
if (hasPageStructure) { writeArtifact(.layout.json) }
```
Тобто для **scanned** (Document AI / фото-склейка) пишеться **тільки** `.layout.json`; `.txt` **не створюється**. Три точки запису gated (підтверджено `report_task_v2a2_core.md`).

### 1.2 No-layout `.txt` — це НЕ дубль, а єдине сховище
`pdfjsLocal.js:8` — «без pageStructure за дизайном». Отже:
- **Searchable native PDF** (цифровий e-court) → `pdfjsLocal` → текст без layout → `.txt` **єдине** джерело.
- **DOCX/HTML** (`converterService.js`) → mammoth/innerText → `.txt`, `documentNature='searchable'`, Document AI НЕ кличеться, layout НЕ створюється → `.txt` **єдине** джерело.

**Прибрати ці `.txt` наосліп = втратити текст.** Це не «дубль», який handoff обіцяв прибрати.

### 1.3 Канонічна точка тексту вже існує
`getDocumentText(doc, caseData)` (`ocrService.js`, V2-A2) — єдина точка ВІРНОГО тексту: scanned layout → `page._text`; інакше `.txt`; НІКОЛИ не digest. Споживачі вже переведені: `DocumentViewerFooter`, `cleanTextDriveAdapter.fetchRawText`, `contextGenerator` (helper-first). Тобто **напрям A handoff здебільшого готовий** — лишилось доревізувати останні прямі читачі `.txt` і вирішити долю searchable-зчитки.

### 1.4 Контекст-генератор уже уніфіковано
`report_task_context_generator_unify.md` (DONE 2026-05-30): спільний `contextGenerator.js`, обидва споживачі, event-тригер `DOCUMENT_BATCH_PROCESSED`, джерело = SSOT `caseData.documents`, stale-guard (`TASK_context_stale_guard`). **Відкрите — тільки селективність** (читає ВСЕ, без бюджету: `contextGenerator.js` `generateCaseContext` → `extractTextBatch` по всіх документах).

### 1.5 Керований OCR і «add без 02-артефактів» — справді відсутні
`streamingExecutor.js:103` `streamFile` **завжди** чанк-OCR; навіть add_as_is (`triageStage.js:85`) проходить екстракт/персист → шляху «в `01_ОРИГІНАЛИ` без `02`» **немає**. `RecognizeTextModal` — лише один файл; керованого «обрати (1/усі/кілька) і розпізнати» **немає**. **Це — справжнє нове ядро TASK'ів.**

### 1.6 Підсумок переформулювання цілей

| Ціль handoff | Фактичний стан | Що реально лишається |
|---|---|---|
| 1. Відмова від `.txt` | Дублі вже прибрано (V2-A2). No-layout `.txt` — єдине сховище, прибирати не можна | (а) разова чистка **legacy** дублів; (б) рішення по searchable-зчитці; (в) явно зафіксувати DOCX/HTML `.txt` як легітимний |
| 2. Узгоджений контекст | Уніфікація DONE | **Селективність**: рівні участі + бюджет сторінок + інкремент |
| 3. Керований OCR | Відсутній | Повністю новий: керований OCR + deferred-стан + відкладене розпізнавання |

### T1 — Доревізія канонічної точки «отримати текст» (напрям A)

**Скоуп.** `getDocumentText` уже існує і вже єдина точка. Завдання: (1) аудит усіх прямих читачів `.txt`/`getCachedText` з §2.2 handoff; (2) перевести залишкові на `getDocumentText`; (3) для searchable native PDF — увʼязати «зчитку на вимогу» (re-read text-layer через `pdfjsLocal` з оригіналу на Drive) як свідому опцію, а не сайд-ефект кешу. **Без зміни схеми.**
**Не робити.** Не чіпати DOCX/HTML `.txt` (T2). Не прибирати layout-читання (вже працює).
**SAAS.** Без нових полів; читач не розрізняє джерело — інкапсульовано в `getDocumentText`.
**BILLING.** Зчитка text-layer — 0 токенів (детермінована); якщо опційно тригерить OCR — `ai_usage` як зараз.
**Тести.** unit: `getDocumentText` повертає вірний текст для scanned(layout)/searchable(.txt)/DOCX(.txt); жоден читач не повертає digest. Регрес: viewer, clean-text adapter, contextGenerator helper-first.
**Залежності.** Немає.

### T2 — Доля тексту DOCX/HTML + searchable `.txt` (напрям B) — рішення-led

**Скоуп.** Зафіксувати політику для no-layout `.txt` (єдиний випадок без layout).
**Рекомендація (на ревʼю).**
- **DOCX/HTML:** `.txt` **ЗБЕРІГАЄМО** як легітимне єдине сховище тексту. Конвертований PDF може бути растровим → повторний OCR був би втратним і платним. Документуємо `.txt` як норму для `documentNature='searchable'` без layout. (Альтернатива — синтезувати мінімальний `.layout.json` з тексту — це gold-plating, YAGNI.)
- **Searchable native PDF:** `.txt` поки **ЗБЕРІГАЄМО** (re-зчитка дешева, але файл дрібний; вигода маргінальна). Перехід на чисту зчитку-на-вимогу — окрема опц. оптимізація, не зараз.
**Наслідок.** «Повна відмова від `.txt`» як самостійна ціль **відпадає**: мета звужується до «без ДУБЛІВ `.txt`», що V2-A2 уже досяг. У CLAUDE.md — оновити опис моделі артефактів (no-layout `.txt` = легітимний).
**SAAS/BILLING.** Без змін.
**Тести.** unit: DOCX/HTML додавання → `.txt` присутній, layout відсутній, `getDocumentText` віддає його. Жодного шляху, що мовчки прибирає no-layout `.txt`.
**Залежності.** Немає. Виконати рано (розблоковує формулювання решти).

### T3 — Керований OCR: транзит-OCR vs персист-layout + шлях «01_ОРИГІНАЛИ без 02» (C + D-частина1)

**Скоуп.** Розмежувати два сенси, які зараз злиті:
- **Транзит-OCR для меж** (`streamFile`) — завжди, ефемерний, НЕ артефакт. Не чіпати.
- **Персистентний layout-артефакт** — за вибором адвоката.
Додати маршрут: **«додати в `01_ОРИГІНАЛИ` без `02`-артефактів»** (deferred). add_as_is перестає примусово писати `02`.
**Схема (bump, напр. v12).** Нове поле документа — **один сенс** (правило #11):
`document.ocrStatus` — *чи створено персистентний текст/layout-артефакт у `02_ОБРОБЛЕНІ` для цього документа*.
enum: `recognized` (layout або `.txt` персистовано) | `deferred` (лежить в `01_ОРИГІНАЛИ`, `02` порожньо) | `not_applicable` (searchable digital, текст у самому PDF). default для існуючих = `recognized` (усі мають артефакти). Міграція `migrateToVersion12` — ідемпотентна, бекап.
> Імʼя `ocrStatus` НЕ плутати з `documentNature` (searchable|scanned — природа вмісту) і `status` (active|archived — lifecycle). `ocrStatus` — про **наявність персистентного артефакта розпізнавання**.
**ACTIONS.** Розширити `add_documents`/`add_as_is` параметром режиму (recognize-now | defer). `update_document` allowlist отримує `ocrStatus`.
**SAAS.** Поле успадковує `tenantId` справи; `updatedBy` через executeAction. Tenant-тариф може лімітувати кількість/глибину розпізнавань (точка в `entitlementsService`).
**BILLING.** **Пряма економія:** `deferred` = нуль OCR-вартості (Document AI + токени) до явного вибору. Розпізнавання → `ai_usage` (agentType `document_parser`) + `activityTracker.report` на дію адвоката.
**Тести.** unit: міграція v12 (default recognized, ідемпотентність). integration: add у defer → файл у `01`, `02` порожньо, `ocrStatus='deferred'`, нуль `ai_usage`; add recognize-now → артефакти + `ocrStatus='recognized'`. Транзит-OCR меж не зламано (нарізка має текст).
**Залежності.** Немає (фундамент для T4, T6).

### T4 — Відкладене розпізнавання «обрати і розпізнати» (D-частина2)

**Скоуп.** На спільному компоненті вибору (3.3): дія **«Розпізнати вибрані»** (1/усі/кілька) над `deferred`-документами → `extractText` по кожному → запис `02`-артефактів → `ocrStatus: deferred→recognized` → тригер релею контексту (T5/relay-trigger). AI-first дзеркало: ACTION `recognize_documents(docIds[])`, доступний агенту (правило дублювання інтерфейсів).
**SAAS.** Permission на масове розпізнавання (тариф-ліміт через entitlements). `court_sync_agent`/інші — за allowlist.
**BILLING.** Кожен документ — `ai_usage` + `activityTracker` (категорія `case_work`). Масовий вибір = передбачувана пакетна вартість (показати оцінку до запуску — UX-гачок для тарифу).
**Тести.** integration: вибір 1/усіх/кількох → лише обрані розпізнані, `ocrStatus` оновлено, контекст-релей спрацював; `ai_usage` рівно по обраних. Агентний шлях `recognize_documents` еквівалентний UI.
**Залежності.** T3 (deferred-стан) + TASK 3.3 (компонент вибору).

### T5 — Селективний контекст-генератор (напрям E)

**Скоуп.** Генератор уже єдиний — додати **керованість**:
- **Рівні участі документа в контексті** — нове поле `document.contextParticipation`: `full` (повний текст) | `digest` (короткий зміст, F10/Haiku) | `none` (не включати). default `full`.
- **Бюджет по сторінках** `Σ pageCount`: поріг; при перевищенні — нудж адвокату знизити частину до `digest`/`none` (а не мовчазне обрізання). Політика дефолту — experimental (`review after 1 month`).
- **Інкрементальне оновлення** (debt #29) — опційно: оновлювати лише дельту документів. Якщо складно — лишити повну регенерацію (поточну), інкремент окремим тиком.
**Схема (bump).** `contextParticipation` — один сенс: *рівень включення документа у `case_context.md`*. НЕ плутати з `ocrStatus` (T3) — документ може бути `recognized` але `contextParticipation='none'`. Міграція ідемпотентна, default `full`, бекап.
**SAAS.** Поле під справою (tenantId успадк.). Тариф може лімітувати max сторінок у контексті / доступність `digest` (Haiku-точка).
**BILLING.** `digest` через Haiku (`resolveModel('textCleaner'/'summary')`) — дешевше за повний. Бюджет = пряме обмеження токенів контексту. `logAiUsage` agentType `case_context_generator` як зараз.
**Тести.** unit: відбір документів за `contextParticipation`; бюджет Σ pageCount тригерить нудж, не обрізає мовчки; `digest` кличе Haiku. integration: Огляд і DP дають однакову селективну поведінку (єдиний шлях).
**Залежності.** Спільний генератор (готовий). Може йти паралельно T3. Узгодити з F10 («короткий зміст») як джерелом `digest`.

### T6 — Єдиний «важіль» (напрям F) — keystone

**Скоуп.** UI-контрол на документі (дім — **Фаза 5, Zone 3 правки нарізки**, за §4 handoff), що одним вибором ставить **пресет** над двома полями:
- `ocrStatus` (T3): розпізнати→layout / відкласти / n/a
- `contextParticipation` (T5): full / digest / none
**КЛЮЧОВА АРХІТЕКТУРНА РЕКОМЕНДАЦІЯ (правило #11).** «Один важіль» — **тільки на рівні UI** (пресет, що пише ОБИДВА поля). На рівні **даних — два ортогональні поля**, бо це два різні питання (чи є артефакт розпізнавання ≠ як глибоко документ бере участь у контексті). Зливати їх в одне поле = два сенси на одне імʼя → тонкі баги (кейс `skipCache`). Пресети можна назвати по-адвокатськи: «Повністю» (recognized+full), «Лише згадка» (recognized+digest), «Не аналізувати» (deferred+none) тощо — мапінг пресет→2 поля задокументувати на місці оголошення.
**SAAS/BILLING.** Сукупно з T3/T5; важіль = головна точка адвокатського контролю вартості (що розпізнавати + що в контекст).
**Тести.** unit: кожен пресет пише очікувану пару (`ocrStatus`,`contextParticipation`); зміна пресету ідемпотентна. integration: важіль у Фазі 5 змінює обидва, тригерить розпізнавання/реґен контексту відповідно.
**Залежності.** T3 + T5.
