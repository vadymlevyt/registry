# Multi-user / SaaS Activation — живий бриф (нотатки на потім)

**Призначення:** зібрати в ОДНОМУ місці питання й вимоги до активації
multi-user / SaaS, що накопичуються ДО того, як ця робота почнеться. Коли
стартує TASK/чат «Multi-user Activation» — це готовий бриф, не треба пригадувати.

**Статус:** НЕ на часі (рішення власника 2026-06-11: «не на часі, хоча й не за
горами»). Поки система фактично соло (один адвокат, одне сховище) — нічого з цього
не реалізуємо превентивно (YAGNI). Поповнюється в міру появи питань.

**Тригер активації:** окремий чат/TASK «Multi-user Activation» (CLAUDE.md
«Заплановано»). До нього конкурентний мультиюзер НЕ підтримується.

---

## 1. Конкурентність запису (питання сесії розширення, 2026-06-11) — tracking_debt #61

Поточний чесний стан (по коду):
- `registry_data.json` — ОДИН файл на тенант; `driveService.writeRegistry` робить
  **повну заміну** (PATCH `uploadType=media`) **без ETag/If-Match/optimistic-locking/
  merge** → паралельні сесії = **останній перемагає = втрата даних**.
- Дедуп за `case_no` — у пам'яті в межах сесії (`getCases()`), **не атомарний** між
  юзерами → двоє юзерів одного тенанту можуть створити **дубль** тієї ж справи.
- NB: це INTER-сесійна проблема, окрема від виправленої INTRA-сесійної React-batching
  гонки (casesRef).

**Що має зробити активація:**
- full-replace → **optimistic-concurrency** (версія/ETag + merge) АБО **серверний
  store з транзакціями**;
- **атомарний** `case_no`-дедуп (серверний арбітр / lock);
- визначити storage-модель: спільне на тенант з **координацією** записів.

## 2. Інші активаційні закладки (вже в коді/CLAUDE.md — зібрано тут)

- **`caseAccess[]`** — заглушка індексу `{caseId,userId,tenantId,caseRole,addedAt,
  expiresAt,permissionsHash}`; «активується у Multi-user Activation» (CLAUDE.md).
- **`checkRolePermission`** (`permissionService`) — зараз заглушка (true для
  bureau_owner); активувати реальну рольову логіку.
- **`checkTenantAccess`/`checkCaseAccess`** — активні базово; перевірити повноту під
  реальних кількох юзерів (team membership, externalAccess validUntil).
- **`tenant.storage.provider`** (`drive_legacy`) — провайдер-патерн для BYOS/R2/
  серверного сховища (planka Picatinny). Зміна сховища (п.1) лягає сюди.
- **`tenant.subscription`** (limits/current/alerts) + **entitlements** —
  активувати ліміти/тарифи реально (зараз self_hosted).
- **`ecitsCabinetIdentifier`** (user) — multi-user dedupe для ЄСІТС (хто синхронізує).
- **Білінг UI** (CLAUDE.md «через 6+ міс») — окремо.
- **AI Provider Abstraction** (CLAUDE.md «6-12 міс при SaaS-комерціалізації») — окремо.

## 3. Принцип

Структури несуть повне ДНК (поля готові), активація = «розгортання заглушок у
реальну логіку», НЕ переписування (DEVELOPMENT_PHILOSOPHY «ембріон»). АЛЕ
**storage-конкурентність (п.1) — це справжня зміна архітектури сховища**, не
заглушка; її закласти свідомо в активаційному TASK, не косметикою.
