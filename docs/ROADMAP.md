### 7.2 Керований OCR 🟡 (детально — частково реалізується в TASK 4)
**Чесна модель (виправлено):** щоб **нарізати** через агента, треба текст УСЬОГО документа — агент шукає межі по тексту, тож при нарізці OCR **повний і неминучий** (Document AI, по чанках). Часткового OCR «тільки для меж» не буває. Тому керований OCR економить НЕ на нарізці, а на **рішенні нарізати чи ні**.

**Три режими обробки на додаванні (вибір адвоката, per-батч/документ):**
1. **Повний OCR** (`full`, Document AI) — для нарізки (обов'язково) і для документів, що йдуть у контекст повністю. Лейаут у `02_ОБРОБЛЕНІ`, доступний агенту/пошуку/контексту.
2. **«Без OCR»** (Claude Vision, **перші 1-2 сторінки → метадані**) — НЕ повне розпізнавання: Vision читає лише 1-2 перші сторінки і заповнює канонічні метадані (`date`/`category`/`author`/`name` + extended summary). Файл лежить **тільки в `01_ОРИГІНАЛИ`**, артефактів у `02` **нема**, але **рендериться у в'ювері** (гортати можна). Це і є чесна легка форма (окремого Document-AI-partial-режиму не робимо).
3. **Відкласти → повне розпізнавання пізніше** — через в'ювер «Розпізнати» (повний) або (Фаза 5) на нарізку; вибрати документи **1 / усі / кілька** через спільний `useSelection` (той самий, що bulk-delete).

**Стан:** реалізується в **TASK 4** (`docs/tasks/TASK_4_ingest_drivepicker_unify.md` — власник 2026-06-04 згорнув «керований OCR» сюди, бо це одне питання — додавання файлів). Етапи A-C (спільний інгест, «просто додати» на всі типи+комбо, усунення дубль-шляху) — ✅; **етап D** («без OCR» = Vision-2-стор.→метадані, розетка `ocrMode`) — ⬜ наступний. Білінг: «без OCR» = дешевий Vision-виклик, `ai_usage` agentType `metadataExtractor`.

**Дім вибору режиму:** тумблер у модалці/DP («просто додати» + `ocrMode`); повторне розпізнавання наявного документа — Фаза 5 (проміжний екран нарізки).

### 7.3 Активний (селективний) контекст-генератор ⬜ (детально)
**Проблема:** `generateCaseContext` читає тексти **всіх** документів → з ростом справи переповнить контекстне вікно (час/вартість/якість). **Рішення — «важіль» участі на документі:**
- **Повний** — весь текст у нарис (ядрові: обвинувальний акт, ключові ухвали, договір);
- **Вижимка** — лише короткий зміст+результат (процедурні: протокол впізнання); джерело вижимки — **F10 «короткий зміст»**;
- **Не включати** — лише факт наявності, без вмісту (довідки/дублі/графіка) — **без видалення зі справи**.
Плюс: **бюджет по сторінках** (`Σ pageCount`, не по штуках) — генератор не переповнюється; **інкрементальне оновлення** — новий документ дописується точково, не перечитуючи всю справу. UX: мала справа → мовчки; велика → модалка/крок вибору. **Один важіль** зв'язує §7.2+§7.3: керує OCR→лейаут / вижимка / нічого + рівнем у контексті.

### 7.7 Горизонтальні зв'язки між досьє + позначення «томів» ⬜ (записано в борг #52)
Виникло при дизайні §7.3 (активний контекст). **Проблема:** великий слідчий том (напр. 335 стор.), доданий у судову справу (де вже є ухвали/засідання з ЄСІТС — напр. Нестеренко), перемішує динаміку судових актів. **Дві ідеї:**
- **позначення «том»** на додаванні — флаг/метадані, щоб томи відрізнялись у списку матеріалів (окреме сортування / пізніші дати / візуальне виділення, не зливались із судовою хронологією);

Це НЕ одна справа — різні провадження (різні суди/час), свідомо поєднані для спільного/підтягнутого контексту. Обмін файлами між досьє для контексту — окрема пізніша історія.

## ▶ TASK 4 — AddModal: ingest-unify (C4) + DrivePicker split (§8)

> **СПЕКА: `docs/tasks/TASK_4_ingest_drivepicker_unify.md`** (2026-06-04, рішення власника).
> Scope РОЗШИРЕНО проти опису нижче: окрім C4+§8 додано **DP-сценарій «просто додати»**

> **режим «без OCR»** (Claude Vision 2 стор. → метадані) і **тумблер «стиснути»**.
> Це одне питання (додавання файлів) — одна спека, 5 етапів (A-E). Опис нижче — історичний (вужчий).

**Осі:** A+B (C4) + C (§8). **Місце в карті:** Фаза 4 (пересунута одразу після Ф1). **Оцінка:** ~2-3 дні (C4) + ~0.5 (§8). **Залежність:** TASK 1 (виконано) — патерн уже доведений.

### Суть і чому саме зараз
Модалка «Додати документ» = дві частини:
1. **склейка фото** — ✅ уже уніфікована з DP (TASK 1; спільні `ImageEditor/`, `imageDocument/`).
2. **додавання звичайного документа** (отримати текст → у справу) — **досі свій окремий шлях**, не через DP-конвеєр.

DP робить те саме, тільки **пачкою**; модалка — **одним файлом**. Пачка з одного елемента — вироджений випадок пачки. Тобто патерн уже доведений склейкою — лишилось застосувати до інгесту.

### Що входить
**C4 — звести трубу обробки в одну:**
- модалка додає документ через `pipeline.run({ files:[одинФайл] })` (streamingExecutor), а не власним `runOcrWithRetryUI`;
- прибрати `runOcrWithRetryUI` (дубль-шлях);
- один конвеєр на модалку і DP → одиночне додавання отримує RAM-чанки, resume, ту саму нарізку й однакові артефакти у `02`.

**§8 — спліт DrivePicker (косметика):**
- винести з `AddDocumentModal.jsx` (1039 рядків) у `AddDocumentModal/DrivePicker/*`: `DrivePickerSection`, `SourceSwitcher`, `Breadcrumb`, `DriveList`, `DriveListItem` + helpers. Сам `AddDocumentModal` (~320 рядків) лишається.

### Межі (що НЕ робити)
- **НЕ зливати UI модалки в DP.** Уніфікуємо лише бекенд-трубу; модалка лишається легкою (швидко додав один файл — побачив результат).
- **НЕ затягувати «керований OCR» (§7.2)** — це окремий майбутній TASK. C4 лише зводить шлях; вибір «розпізнавати/ні/відкласти» сюди не входить.
- **§8 — не зливати два DrivePicker'и одразу.** Спершу винести модальний; чи зводити з `DocumentProcessorV2/DrivePicker.jsx` у спільний `components/UI/DrivePicker/` — окремий мікро-TASK після виносу (стане видно, чи це той самий патерн).

### На що зважати
- **Артефакти змінились (clean_text V2-A2):** єдина точка тексту — `getDocumentText`; `.txt` пишеться лише коли **нема** лейаута; DP-пост-крок очистки прибрано. Уніфікований шлях має **поважати цю реальність** — не відроджувати `.txt` для сканів.
- **Для адвоката поведінка не змінюється** — це переважно внутрішнє зведення; UI модалки той самий. Тести фіксують: одиночне додавання працює як раніше, але через спільний конвеєр.
- **SAAS/BILLING:** інструментацію додавання не загубити при переході на `pipeline.run` (`activityTracker` / `ai_usage`).
