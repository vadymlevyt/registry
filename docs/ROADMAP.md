## 7. РОБОТА, ЩО ВИНИКЛА ПО ХОДУ (поза первісним roadmap)

### 7.1 Реорганізація артефактів — відмова від `.txt` 🟡 (частково вже в коді)
**Ідея:** `.txt` — дубль; той самий текст лежить у `.layout.json` (`page._text`). Для searchable текст дістає зчитка на вимогу.
**Вже зроблено (через clean_text V2-A2):**
- ✅ `.txt` НЕ пишеться коли є layout (скан → лише лейаут; `splitDocumentsV3.js:673`).
- ✅ `getDocumentText` — єдина точка вірного тексту (layout→`page._text`, інакше `.txt`); тягнуть viewer, contextGenerator, DP, cleanText, ocrService.
- ✅ DP-пост-крок очистки прибрано.
**Лишилось:** едж DOCX/HTML (текст без лейаута — єдине сховище `.txt`); можлива разова міграція старих `.txt`.
*Handoff:* `handoff_2026-06-04_artifacts_reorg_context_ocr.md` (підрізати — частина вже готова).

### 7.2 Керований OCR ⬜ (детально — що мається на увазі)
Зараз OCR при нарізці **обов'язковий**, і навіть «просто додати» все одно OCR-иться; шляху «в оригінали без обробки» нема. **Хочемо:** адвокат на стадії додавання/правки вирішує per-документ:
- **розпізнати** → створюється лейаут-копія в `02_ОБРОБЛЕНІ`, документ доступний для контексту/агента/пошуку;
- **не обробляти** → документ просто в `01_ОРИГІНАЛИ`, без артефактів у `02`;
- **відкласти** → розпізнати пізніше окремо: **по одному / усі / вибрані** (через спільний `useSelection` — той самий, що bulk-delete).
Розрізняємо: **транзитний OCR для меж** (завжди, коли є нарізка) vs **персистентний лейаут-артефакт** (за вибором). Дім вибору — Фаза 5 (правка нарізки) + модалка/реєстр.

### 7.3 Активний (селективний) контекст-генератор ⬜ (детально)
**Проблема:** `generateCaseContext` читає тексти **всіх** документів → з ростом справи переповнить контекстне вікно (час/вартість/якість). **Рішення — «важіль» участі на документі:**
- **Повний** — весь текст у нарис (ядрові: обвинувальний акт, ключові ухвали, договір);
- **Вижимка** — лише короткий зміст+результат (процедурні: протокол впізнання); джерело вижимки — **F10 «короткий зміст»**;
- **Не включати** — лише факт наявності, без вмісту (довідки/дублі/графіка) — **без видалення зі справи**.
Плюс: **бюджет по сторінках** (`Σ pageCount`, не по штуках) — генератор не переповнюється; **інкрементальне оновлення** — новий документ дописується точково, не перечитуючи всю справу. UX: мала справа → мовчки; велика → модалка/крок вибору. **Один важіль** зв'язує §7.2+§7.3: керує OCR→лейаут / вижимка / нічого + рівнем у контексті.

### 7.4 «Інструменти» (стиснення + розпізнавання) ⬜
З doctrine #39 (`TASK_file_tools_compression_doctrine.md`): окрема верхня вкладка — «Стиснути файл» (3 пресети, guard тільки для сканованих, складання pdf-lib) + «Розпізнати текст» (OCR по оригіналу). Стенд `public/lab/pdf-recompress.html` уже фікснуто на pdf-lib.

### 7.5 Telegram-бот інгест (#41) ⬜
Фото боту → downscale на вході → у справу. Потребує бекенда; закриває в корені межу #40.

### 7.6 Compression doctrine + фікс стенда ✅
jsPDF (спільний `/Resources`, ламав нарізку) → pdf-lib (власні ресурси/стор.). Стиснення ідентичне, нарізка чиста.
