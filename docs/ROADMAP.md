### 7.2 Керований OCR ✅ (спрощено в add-files rework)
> **ОНОВЛЕНО (rework):** «Без OCR» спрощено — це **просто без розпізнавання** (без Vision, без артефактів у `02`): файл лягає в `01_ОРИГІНАЛИ` з базовими метаданими (назва з форми/імені), видимий у в'ювері; розпізнати — пізніше через «Розпізнати». Vision-екстракт метаданих (1-2 стор.) **прибрано** з цього шляху (рішення власника — у першій ітерації не потрібен, був джерелом проблем). Повний OCR (`full`, дефолт) — Document AI → layout у `02`. Опис нижче — історичний (плани Vision-метаданих).

**Чесна модель (виправлено):** щоб **нарізати** через агента, треба текст УСЬОГО документа — агент шукає межі по тексту, тож при нарізці OCR **повний і неминучий** (Document AI, по чанках). Часткового OCR «тільки для меж» не буває. Тому керований OCR економить НЕ на нарізці, а на **рішенні нарізати чи ні**.

**Три режими обробки на додаванні (вибір адвоката, per-батч/документ):**
1. **Повний OCR** (`full`, Document AI) — для нарізки (обов'язково) і для документів, що йдуть у контекст повністю. Лейаут у `02_ОБРОБЛЕНІ`, доступний агенту/пошуку/контексту.
2. **«Без OCR»** (Claude Vision, **перші 1-2 сторінки → метадані**) — НЕ повне розпізнавання: Vision читає лише 1-2 перші сторінки і заповнює канонічні метадані (`date`/`category`/`author`/`name` + extended summary). Файл лежить **тільки в `01_ОРИГІНАЛИ`**, артефактів у `02` **нема**, але **рендериться у в'ювері** (гортати можна). Це і є чесна легка форма (окремого Document-AI-partial-режиму не робимо).
3. **Відкласти → повне розпізнавання пізніше** — через в'ювер «Розпізнати» (повний) або (Фаза 5) на нарізку; вибрати документи **1 / усі / кілька** через спільний `useSelection` (той самий, що bulk-delete).

**Стан:** ✅ зроблено в add-files rework (`docs/tasks/HANDOFF_task4_rework_add_files_standalone.md`). «Просто додати» = окремий сервіс `addFiles`, режими `full` (Document AI → `02`) і `none` («без OCR» = без розпізнавання, лише `01`+метадані). Vision-метадані з «без OCR» **прибрано** (спрощення). `ocrMode` живе в опціях прогону (тумблер модалки/DP).

**Дім вибору режиму:** тумблер у модалці/DP («просто додати» + `ocrMode`); повторне розпізнавання наявного документа — Фаза 5 (проміжний екран нарізки).

**Гранулярність — БАТЧ зараз, per-документ відкладено (борг #56):** TASK 4 D реалізував «без OCR» на рівні ВСЬОГО набору DP (один тумблер). Per-документний вибір (у списку доданих файлів обрати, який осіарити, який без OCR) — окрема фіча: per-рядковий тумблер + `ocrMode` на кожен item. Не в TASK 4 (завершується E).

### 7.3 Активний (селективний) контекст-генератор ⬜ (детально)
**Проблема:** `generateCaseContext` читає тексти **всіх** документів → з ростом справи переповнить контекстне вікно (час/вартість/якість). **Рішення — «важіль» участі на документі:**
- **Повний** — весь текст у нарис (ядрові: обвинувальний акт, ключові ухвали, договір);
- **Вижимка** — лише короткий зміст+результат (процедурні: протокол впізнання); джерело вижимки — **F10 «короткий зміст»**;
- **Не включати** — лише факт наявності, без вмісту (довідки/дублі/графіка) — **без видалення зі справи**.
Плюс: **бюджет по сторінках** (`Σ pageCount`, не по штуках) — генератор не переповнюється; **інкрементальне оновлення** — новий документ дописується точково, не перечитуючи всю справу. UX: мала справа → мовчки; велика → модалка/крок вибору. **Один важіль** зв'язує §7.2+§7.3: керує OCR→лейаут / вижимка / нічого + рівнем у контексті.

**Уточнення (адмін-звірка 2026-06-09, правило #11):** «один важіль» — це **UI-пресет над ДВОМА ортогональними значеннями**, не одне поле:
- **(a) стан розпізнавання** — чи є артефакт у `02` (керується `ocrMode`; зараз runtime/інференс, НЕ в схемі);
- **(b) рівень участі в контексті** — Повний / Вижимка / Не включати.
Документ може бути `full`-розпізнаний, але «Не включати» — і навпаки (вижимка не потребує повного лейаута). Зливати в одне поле = два сенси на одне імʼя (кейс `skipCache`). На рівні даних: (a) інфериться, **(b) має бути персистентним полем документа** (рівень участі рішенням адвоката не відновлюється інференсом → likely bump v12 + ідемпотентна міграція, default «Повний»). На старті спека — **звірити рівень «Повний/ключовий» з наявним `isKey`** (не плодити дубль-поле; можливо нове enum поглинає `isKey` або лягає поверх). Деталі: `docs/consultations/admin_review_2026-06-09_reconcile_roadmap7.md`.

Це НЕ одна справа — різні провадження (різні суди/час), свідомо поєднані для спільного/підтягнутого контексту. Обмін файлами між досьє для контексту — окрема пізніша історія.

## Ключові терміни (для не-айтішника)
- **Конвеєр / диригент** — механізм, що проводить документ через станції по черзі.
- **Стрім-мотор** — носить великий том частинами через розпізнавання (щоб влізло в пам'ять).
- **Triage** — станція, де AI складає план нарізки.
- **Паспорт тому** — стислий посторінковий конспект для AI (краї тексту + сигнали меж).
- **layout** — розкладка: текст кожної сторінки + геометрія блоків.
- **callAgent** — майбутня єдина «парасоля» над усіма викликами AI з авто-обліком.
- **PERSIST** — момент, коли план виконується: файли на Drive, записи в реєстрі.
- **tracking_debt #N** — реєстр свідомо відкладеного з тригером «коли робити».
