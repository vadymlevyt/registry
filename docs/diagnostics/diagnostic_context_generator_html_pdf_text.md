# Diagnostic / Handoff — контекст-генератор не читає PDF, створені з HTML (і, ймовірно, з DOCX)

**Дата:** 2026-06-21 (діагноз), 2026-06-22 (справжній корінь + фікс)
**Статус:** ✅ RESOLVED (2026-06-22). **Справжній корінь — §12.** Симптом РЕАЛЬНИЙ і живий (відтворено на свіжому `case_context.md` 2026-06-22). Але причина — НЕ шрифти (гіпотеза §3 хибна, §11) і НЕ стейл (припущення §11.3 теж хибне): `pdfjsLocal.extract` роутив stored PDF за **розширенням імені** (`originalName="X.html"`) у HTML-гілку → `TextDecoder(win-1251)` на байтах PDF → бінарне сміття. Фікс: явний `mimeType:'application/pdf'` тепер пріоритетніший за розширення.
**Зачеплені модулі:** `ocr/pdfjsLocal.js` (корінь), `ocrService.getDocumentText/extractTextLayer`, `contextGenerator`.
**Severity:** high — ЄСІТС HTML→PDF (ключові процесуальні документи) не потрапляли у `case_context.md`.

> **⚠️ ЧИТАТИ §12 (справжній корінь) + §11 (хибні гіпотези — урок).** Дві хиби
> по дорозі: (1) §3 — код-рідинг без емпірики дав «шрифти/ToUnicode» (невірно);
> (2) §11.3 — після спростування §3 поспішний висновок «стейл/транзієнт, бага
> немає» (теж невірно). Справжній баг знайдено лише коли адвокат дав свіжий
> `case_context.md` + реальні файли, і шлях відтворено КІНЕЦЬ-У-КІНЕЦЬ (§12).
> Урок: не зупиняйся на «спростовано» — доводь причину відтворенням, не теорією.

## 3. Корінь — підтверджено кодом

Ланцюг для **searchable-документа, конвертованого з HTML/DOCX**:

### 3.1. Конвертер обчислює правильний текст — і його викидають

- `htmlToPdf` (`src/services/converter/htmlToPdf.js:88`) витягає `extractedText`
  через `container.innerText` — **чистий правильний текст** (декодований з cp1251).
- `docxToPdf` (`src/services/converter/docxToPdf.js:153`) витягає `extractedText`
  через `mammoth.extractRawText` — теж чистий текст.
- Обидва рендерять PDF через `pdfLibHtmlRenderer.htmlToPdfViaPdfLib`.

**Але `extractedText` нікуди не зберігається.** TASK 4 §7.1 («повна відмова від `.txt`»)
прибрав запис `.txt` для searchable. Прямі докази:
- `converterService.js:31-37` — коментар: *«caller його НЕ зберігає у 02_ОБРОБЛЕНІ»*.
- `ocrService.js:17` — *«`.txt` — мертвий дубль (і запис, і читання прибрано)»*.
- `CaseDossier/index.jsx:2960` — гілка `if (extractedText) { toast.success(...) }`:
  для plain DOCX/HTML у `02_ОБРОБЛЕНІ` **не пишеться нічого** (layout пишеться лише для
  фото-склейки). `extractedText` далі відкидається.

### 3.2. Єдине джерело тексту стало непрацездатним для цих PDF

Для `documentNature === 'searchable'`:
- `getDocumentText` (`ocrService.js:323-329`) → `extractTextLayer` →
  `pdfjsLocal.extract` витягає **текстовий шар самого PDF**.
- `contextGenerator.js:484` викликає `getDocumentText`; якщо порожньо/сміття —
  документ іде в `extractTextBatch` (Document AI) **тільки якщо немає тексту взагалі**;
  при сміттєвому, але непорожньому витягу — сміття потрапляє далі як «текст».

**Чому витяг — сміття:** `pdfLibHtmlRenderer` вшиває шрифти з `{ subset: true }`
(`pdfLibHtmlRenderer.js:1588-1597`) і малює текст через `page.drawText`
(`pdfLibHtmlRenderer.js:1129`). Сабсетований шрифт від pdf-lib не несе надійного
**ToUnicode CMap**, тож `pdf.js` при екстракції повертає кодпойнти гліфів сабсету —
**нечитабельну бінарщину**, яку модель і бачить як «PDF у бінарному форматі».

### 3.3. Чому в'ювер усе одно показує текст

В'ювер **рендерить сам PDF** (контури гліфів малюються правильно — шрифт вшито).
Тобто адвокат бачить **візуальний рендер**, а не результат програмної екстракції тексту.
Візуальний рендер ≠ extractable text layer. Звідси вся асиметрія.

## 4. Чому кодування (cp1251) — НЕ корінь

Декодування cp1251 відбувається при конвертації (`htmlToPdf.js:60`, `decodeHtmlBuffer`).
Якби воно помилилось — PDF **візуально** показував би кракозябри. Але в'ювер показує
текст коректно → декодування спрацювало, рядок у PDF правильний. Сміття виникає
**пізніше**, на рівні екстракції текстового шару з pdf-lib-PDF. Кодування — глуха гілка.

---

## 5. Експозиція DOCX (відповідь на «можливо з док?»)

**Так, ймовірно той самий баг.** `docxToPdf` використовує **той самий**
`pdfLibHtmlRenderer.htmlToPdfViaPdfLib` з тим самим `subset: true`. Отже DOCX→PDF
searchable-документи мають **той самий непрацездатний текстовий шар** і **той самий**
ефект у генераторі. Інтуїція адвоката правильна.

**Треба підтвердити емпірично** (див. §7): знайти на Drive реальний DOCX→PDF
(searchable, без `.txt`/layout) і прогнати генератор / `getDocumentText`.

## 8. Варіанти фіксу (НЕ реалізовувати без узгодження)

**Варіант A (рекомендований) — повернути збереження `extractedText` для конвертованих
searchable.** Конвертер **уже має ідеальний текст**; баг суто в тому, що §7.1 його
викинув. Зберігати цей текст як канонічне джерело для searchable-конвертованих
(окремий артефакт типу `.searchable.txt`, або поле/extended-метадані), і навчити
`getDocumentText`/`getCleanOrRawText` читати його **першим** для searchable. Найдешевший
коректний фікс; не залежить від нутрощів сабсету pdf-lib; 0 токенів OCR.
*Потрібне рішення:* де саме зберігати (артефакт vs метадані) — узгодити, бо торкається
конвенції `02_ОБРОБЛЕНІ` і, можливо, схеми.

**Варіант B — додати коректний ToUnicode CMap** у вшивання шрифтів
`pdfLibHtmlRenderer`. Робить текстовий шар екстрагованим «по-справжньому». Ризикованіше,
залежить від можливостей pdf-lib subset; ширший вплив на рендер.

**Варіант C — fallback у `getDocumentText`:** якщо searchable-екстракція порожня/сміттєва
→ Document AI OCR. Суперечить принципу «searchable = без OCR», марнує токени на чисто
цифровому PDF; лишити як крайній запасний.

**Орієнтир:** A — корінь у викинутому тексті, тож повернути його найчесніше. B/C —
лікують симптом.

## 11. ЕМПІРИЧНА КОРЕКЦІЯ (2026-06-21, друга сесія) — корінь §3 СПРОСТОВАНО

Друга сесія виконала пропущені кроки верифікації §7. Результат: **гіпотеза про
subset-шрифти/ToUnicode хибна; PDF і поточний код справні.**

### 11.1. Три незалежні докази, що текст екстрагується

1. **Мінімальний pdf-lib (1.17.1) + Liberation TTF, subset:true, кирилиця → pdfjs.**
   `getTextContent` повертає рядок **дослівно** (`match: true`). ToUnicode у PDF
   **відсутній** — і це НЕ заважає: pdf.js зворотно мапить гліфи через cmap
   вшитого шрифту. Тобто база гіпотези §3.2 невірна.

2. **Реальний `pdfLibHtmlRenderer.htmlToPdfViaPdfLib` на ЄСІТС-style HTML → pdfjs**
   (vitest jsdom + шим fetch шрифтів). Екстраговано **повний читабельний текст**:
   «УХВАЛА … № 564/2044/26 … ВСТАНОВИВ … УХВАЛИВ …». Рендерер малює по словах
   (`drawText` на слово) — текст екстрагується нормально, пробіли між словами є.

3. **Сам файл на Drive** (та сама ухвала Грицюка, fileId `1tdoxj9L…6onSJ`,
   `application/pdf`, **72 КБ** = текстовий PDF, не скан-картинка): Drive
   `contentSnippet` витягує **весь текст ухвали дослівно** (суддя Снітчук Р.М.,
   Костопільський р/с, аліменти, дати, статті ЦПК). Файл має справний текстовий шар.

`pdfjsLocal.extract` (`src/services/ocr/pdfjsLocal.js`) використовує той самий
pdfjs API → для цього файлу повернув би той самий читабельний текст. Отже
`getDocumentText` (searchable → `extractTextLayer`) **працює** на цих PDF.

### 11.2. Чому гіпотеза §3 впала

§3 побудована на code-reading: «subset → немає ToUnicode → pdf.js дає коди
гліфів = сміття». Емпірика (11.1.1) показує: pdf.js НЕ потребує ToUnicode для
TrueType зі справним cmap — fallback на cmap шрифту дає правильний Unicode.
Кроки §7 («підтвердити, чи ToUnicode відсутній і чи витяг — сміття») **ніколи
не виконувались** у першій сесії (приблизно: «виклик двічі впав на approval»).

### 11.3. Справжній статус симптому

Симптом «текст у прочитному вигляді відсутній (PDF у бінарному форматі)» — це
**слова самої AI-моделі** у `case_context.md` (рядок ніде в коді не генерується).
Оскільки PDF читабельний і поточний код його читає, найімовірніше:

- **стейл `case_context.md`** (згенерований за старіших умов / іншого стану доку), або
- **транзієнт** при генерації (напр. Document AI AUTH-збій → порожньо/плейсхолдер,
  який модель описала як «бінарний PDF»).

**Це НЕ дефект конвертера, шрифтів чи екстракції.**

### 11.4. Рекомендована дія

1. **Перегенерувати `case_context.md`** для справи Грицюк 564/2044/26 і подивитись,
   чи ухвали тепер потрапляють у контекст. Найімовірніше — так (текст читабельний).
2. Якщо симптом **відтворюється наживо** — це окремий, ще не локалізований баг у
   рантайм-шляху читання (worker pdfjs / Document AI AUTH / contextGenerator
   error-handling), а НЕ у шрифтах. Тоді: відкрити нову діагностику з
   **реальним** дампом того, що `getDocumentText` повертає в браузері (а не
   гіпотезою).
3. **НЕ** вводити side-car `.txt`/`.searchable.txt` — суперечить §7.1 і не
   потрібно (PDF самодостатній). Цей варіант було прототиповано і **відкинуто**
   за рішенням адвоката («зробити PDF нормальним, без зайвих файлів» — він уже нормальний).

### 11.5. Урок (для LESSONS / філософії)

Хендоф із діагнозом «корінь підтверджено кодом» без жодної емпіричної перевірки
дав **впевнено сформульований, але хибний** корінь, який ледь не призвів до
зайвого коду (відновлення side-car файлів усупереч §7.1). **Перед фіксом —
відтвори баг емпірично.** Code-reading формулює гіпотезу, не доводить її.

## 12. СПРАВЖНІЙ КОРІНЬ (2026-06-22, третя сесія) — filename-extension misrouting

### 12.1. Як знайдено

Адвокат дав свіжий `case_context.md` (2026-06-22 10:26, Брановський) + сам ЄСІТС-HTML
+ конвертований PDF. У контексті баг ЖИВИЙ: документи 20 (ухвала апеляційна),
35 (постанова), 36 (ДНЗС) — усі «PDF у бінарному форматі». Реєстр (`registry_data.json`)
показав: усі три — `documentNature:'searchable'`, `originalMime:'text/html'`,
`originalName` закінчується на `.html`, `driveId` → PDF у 01_ОРИГІНАЛИ.

### 12.2. Ланцюг

1. `contextGenerator` → `ocrService.getDocumentText(d)` для searchable →
   `extractTextLayer({ id: driveId, name: originalName, mimeType:'application/pdf' })`
   → `pdfjsLocal.extract(...)`. `name` = `"Постанова.html"` (оригінальне ім'я HTML).
2. `pdfjsLocal.extract` класифікує тип за **розширенням імені**:
   `isHtml = ... || lname.endsWith('.html')` → **true** (хоча `mimeType` = PDF!).
3. HTML-гілка: `TextDecoder('windows-1251').decode(arrayBuffer)` на **байтах PDF**
   → `"%PDF-1.7\n%ЃЃЃЃ\n13 0 obj<</Filter /FlateDecode...stream\nxњН..."` (119 478 символів сміття).
4. Це сміття повертається як «текст» → `helperDocs` → промпт → модель описує як
   «PDF у бінарному форматі».

### 12.3. Емпіричне підтвердження

- Прямий `pdfjs.getDocument(bytes)` на тому ж файлі (обходить роутер pdfjsLocal) →
  **24 164 символи чистого тексту** (постанова) / 4 637 (ухвала Грицюк). PDF справний.
- Відтворення HTML-гілки pdfjsLocal на байтах PDF → рівно те сміття «%PDF-1.7 %ЃЃЃЃ…».

Звідси й уся асиметрія: scanned (`.pdf`-ім'я) → Document AI layout → читається; DOCX
(`.docx`-ім'я) → `UNSUPPORTED` → fallback Document AI → читається; лише HTML→PDF
(`.html`-ім'я) → HTML-гілка → сміття.

### 12.4. Фікс

`src/services/ocr/pdfjsLocal.js`: `mimeType` тепер **авторитетніший** за розширення
імені. `isPdfMime = file.mimeType==='application/pdf'` вимикає `isText`/`isHtml`-гілки
за іменем. Stored PDF (у 01_ОРИГІНАЛИ він ЗАВЖДИ PDF) читається текстовим шаром
незалежно від `originalName`. Розширення імені лишається fallback лише коли
`mimeType` не PDF. Тест: `tests/unit/pdfjsLocalPdfMime.test.js` (4).

**Жодних змін у конвертері/PDF/схемі, жодних side-car файлів** — рівно те, що просив
адвокат: «зробити так, щоб текст читався з самого PDF». PDF завжди був нормальний;
читач його неправильно роутив.

### 12.5. Межі

Forward-fix працює одразу для всіх наявних `.html`→PDF документів (нічого не треба
перевидавати — фікс на боці читання, не запису). Стара порада §11.4 «перегенерувати
case_context» лишається: після деплою фіксу перегенерація Брановського/Грицюка
підтягне повний текст ухвал/постанов.
