### Крок 2.4: Поясни різницю `_metadata/` папки

**Питання:** чому у Брановського є `_metadata/documents_extended.json`, а у Кісельової немає?

**Знайди в коді:**
1. Де створюється папка `_metadata/`? (через `gapi.client.drive.files.create` з MIME `application/vnd.google-apps.folder`)
2. Де створюється файл `documents_extended.json`?
3. Чи це робиться при створенні справи, чи при першому додаванні документа з важкими полями, чи при міграції, чи лише в seed?
4. Чи це баг (функція не викликається) чи логіка (так задумано)?

**Можливі причини:**
- Брановський — seed-кейс (з INITIAL_CASES), створювався іншим шляхом ніж Кісельова
- Кісельова — реальна справа додана адвокатом, і flow `addCase` не створює `_metadata/`
- Або міграція v4→v5 створила це для legacy-кейсів які мали важкі поля

**Перевір реально в коді** який з варіантів правда.

### Крок 2.7: Доповнення до знахідок

### Крок 2.8: Додаткова знахідка — MIME type для documents_extended.json

**Адвокат помітив:** при скачуванні з Google Drive `registry_data.json` приходить з розширенням `.json`, а `documents_extended.json` — з розширенням `.txt`. Це означає що Drive поставив `mimeType: 'text/plain'` для другого файлу замість `'application/json'`.

**Перевір у коді:**
1. Знайди функцію яка створює/оновлює `registry_data.json` — який MIME вона використовує?
2. Знайди функцію яка створює/оновлює `documents_extended.json` — який MIME?
3. Чи це баг (забули вказати правильний MIME) чи свідома різниця?

Це дрібна знахідка, але **узгоджена з основним питанням** "чому `_metadata/` створюється не для всіх справ" — імовірно код який створює `documents_extended.json` має якісь особливості або був написаний поспіхом, тому пропустив правильний MIME.

Зафіксуй у звіті як додаткова знахідка з рекомендацією виправити MIME при подальших фіксах.

## КРОК 3 — CLEANUP

**Cleanup тимчасових файлів — обов'язково.** JSON файли в `tmp_diagnostic/` — це **тимчасові production-дані** які адвокат завантажив для діагностики. Вони містять реальну інформацію по справах і **НЕ повинні залишатись у репозиторії**. Після завершення діагностики:

1. Видали папку `tmp_diagnostic/` з усіма файлами:
   ```bash
   rm -rf tmp_diagnostic/
   ```

2. Перевір що файли не закомічені в git:
   ```bash
   git status
   git log --oneline -5
   ```

3. Якщо з якоїсь причини вони потрапили у git history — повідом адвоката окремо, треба буде розглядати спеціальні дії (`git rm --cached` + force push або BFG repo cleaner). Краще щоб взагалі туди не потрапляли.

4. Підтверди в звіті що файли видалені і не закомічені.

5. Закоміть звіт `diagnostic_document_lifecycle_v2.md`:
   ```bash
   git add diagnostic_document_lifecycle_v2.md
   git commit -m "docs: diagnostic v2 with real production data analysis"
   ```
