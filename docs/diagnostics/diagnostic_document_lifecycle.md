## 1. РЕЗЮМЕ

Підтвердились усі шість гіпотез — повністю або частково. Корінь проблеми Брановського/Кісельова — **не Drive auth і не приватність файлів**, як ми підозрювали, а простіше: у відповідних `case.documents[]` сидять записи з **`driveId: null`**. Це orphan-записи в реєстрі (не на Drive).

Два механізми як вони туди потрапляють:
1. **Брановський** — це **seed-кейс із INITIAL_CASES**. Усі 12 документів створені через `createDocument()` без `driveId` (App.jsx:111-124). Це демо-дані, не відображаються тому що файлу на Drive фізично немає.
2. **Кісельова** і нові документи Брановського — найімовірніше додавалися через **AddDocumentModal**, у якому є критичний баг: при невдалому Drive upload (`Failed to fetch`) toast про помилку показується, **але запис у реєстрі все одно створюється з `driveId: null`** (CaseDossier:2804-2849, ділянка 2810-2814 ловить помилку і **не throw**).

DocumentViewer → ScanContent для `driveId === null` показує `<AlertTriangle/>` "Файл не прикріплено до Drive" (DocumentViewerContent.jsx:33-44). Адвокат каже "білий iframe з порожнім вмістом" — це власне і є empty state, який візуально читається як "нічого не показалось" коли він не вчитується в текст іконки.

Видалення в режимі `'full'` **викликає** `deleteDriveFile()` (App.jsx:5429-5434), але огорнуто в `try { } catch { console.warn }` без сигналу користувачу. Якщо `driveId === null` — Drive delete взагалі не виконується (умова `if (doc.driveId)` на 5429). Тобто для документів-orphans з `driveId: null` "повне" видалення видаляє **тільки запис**, а файли в `01_ОРИГІНАЛИ`/`02_ОБРОБЛЕНІ` (якщо вони туди потрапили через інший код-шлях, як ocr cache) залишаються — без посилання з реєстру на них.

Архітектура частково узгоджена. Є канонічна фабрика `createDocument()` і централізований `delete_document` ACTION. Але є три тонкі тріщини:
- AddDocumentModal не fail-fast при upload error;
- DocumentProcessor пише через `updateCase()` напряму, обходячи `add_document` ACTION (audit/permissions недоступні);
- OCR-кеш в `02_ОБРОБЛЕНІ` створюється без запису в реєстрі (sirota by design — кеш не документ, але потенційне джерело "блукаючих" .txt при перезавантаженні документа з новим driveId).

### 2.3 Висновок по карті

- **6 шляхів** створення документа: 4 живих (UI), 1 seed, 1 міграція. Плюс `writeCache` як ортогональний шлях створення файлу-без-запису.

- **Канонічна фабрика `createDocument()` використовується у всіх UI-точках** — добре.
- **3 з 4 UI-точок ходять через ACTION**, DocumentProcessor (A2, A3) — через `updateCase()` напряму. Архітектурний обхід.
- **Видалення централізоване через один ACTION з трьома режимами** — добре.
- **Усі помилки видалення проковтуються** (silent warning) — погано для UX.

## 3. СТАН КІСЕЛЬОВА І БРАНОВСЬКОГО — ЩО МОЖНА ПРОЧИТАТИ З КОДУ

Реальний стан `registry_data.json` у production я не маю — він у Drive адвоката. Але з коду чітко видно дві групи документів:

### 3.1 Брановський (case_4) — seed з INITIAL_CASES

12 документів створених у `INITIAL_CASES` (App.jsx:111-124) через `createDocument()` **БЕЗ передачі `driveId`**:

```js
createDocument({ id: "doc_seed_branovsky_01", procId: "proc_main", name: "Позовна заява",
                 icon: "📄", category: "pleading", author: "ours", isKey: true,
                 addedBy: 'migration', namingStatus: 'manual', folder: '01_ОРИГІНАЛИ' }),
// ... ще 11 аналогічних
```

`createDocument` поверне об'єкт з `driveId: null` (бо параметр відсутній). Поля `category`/`author`/`procId` валідні (є в legend).

**`needsReview` для них:** category/author/procId є → не маркуються ⚠.
**Drive файлу немає взагалі** — це демо-дані, ніхто файл не створював.
**В DocumentViewer:** ScanContent → AlertTriangle "Файл не прикріплено до Drive". TextContent → useEffect повертається на `if (!document.driveId || ...)` :84 → text залишається `null` → "Текст для цього документа ще не розпізнано" + кнопка "Розпізнати зараз" (яка не зможе зробити нічого корисного — driveId-то немає).

**Якщо адвокат завантажує "ту саму" Позовну заяву через AddDocumentModal у Брановського** — створиться **новий другий запис** з реальним driveId (бо `doc_seed_branovsky_01` лишається). Старий orphan-seed і новий документ співіснуватимуть під однаковою назвою.

### 3.2 Кісельова (case_9) — пуста справа в seed

У INITIAL_CASES Кісельова не має жодного документа в seed (App.jsx:130). Усі документи там — додані вручну.

Якщо в Drive за справою Кісельова реально є файли в `01_ОРИГІНАЛИ` і `02_ОБРОБЛЕНІ` (як описує адвокат — file `701_1413_25 кисельова ухвала.pdf` 44 КБ), а в реєстрі є запис без `driveId` — це продукт **AddDocumentModal upload failure**. Або контекст-агент колись клав файли в Drive без створення запису (нагадую — `extractText` через ocrService може писати .txt у `02_ОБРОБЛЕНІ`, але оригінал у `01_ОРИГІНАЛИ` контекст-агент сам не пише; якщо адвокат це бачить — джерело інше, або давній код-шлях якого вже немає, або сам адвокат завантажив через старішу версію DocumentProcessor).

**Як адвокат може швидко перевірити (DevTools у відкритому додатку):**
```js
// У консолі браузера на сторінці http://localhost:5173 (чи production)
const cases = JSON.parse(localStorage.getItem('levytskyi_cases_cache_v1') || '[]');
const branovsky = cases.find(c => c.name === 'Брановський');
const kiselova  = cases.find(c => c.name === 'Кісельова');

console.table((branovsky?.documents || []).map(d => ({
  id: d.id, name: d.name, driveId: d.driveId, addedBy: d.addedBy, status: d.status
})));
console.table((kiselova?.documents || []).map(d => ({
  id: d.id, name: d.name, driveId: d.driveId, addedBy: d.addedBy, status: d.status
})));
```

Очікую побачити:
- Брановський: 12 записів `doc_seed_branovsky_01..12` з `driveId: null`, `addedBy: 'migration'`. Можливо ще кілька з реальним driveId, якщо адвокат завантажував.
- Кісельова: декілька записів. Хоч один з `driveId: null` і `addedBy: 'lawyer_manual'` — це підтвердження AddDocumentModal баг.

## 4. ПЕРЕВІРКА ГІПОТЕЗ

### Гіпотеза 1: Документи Кісельова/Брановський — orphans

**Статус: ПІДТВЕРДЖЕНА (двома різними механізмами).**

**Докази:**
- Брановський: 12 seed-документів з `driveId: null` у INITIAL_CASES (App.jsx:111-124).
- AddDocumentModal upload error не блокує створення запису:
  ```js
  // CaseDossier/index.jsx:2806-2849
  if (file && driveConnected) {
    try {
      const prepared = await prepareFile(file);
      driveId = await uploadFileLocal(prepared, caseData);
    } catch (err) {
      console.error('Drive upload error:', err);
      toast.error('Не вдалось завантажити на Drive', { description: err?.message });
      // ❗ НЕ throw — falls through!
    }
  }
  // далі createDocument з driveId: null і add_document ACTION
  ```
  → Адвокат бачить toast "Не вдалось завантажити на Drive: Failed to fetch", потім toast "Документ додано" :2848. У списку з'являється документ з `driveId: null`.

**Уточнення гіпотези:** проблема НЕ в колізії імен у `01_ОРИГІНАЛИ`. `gapi.client.drive.files.create` (як і Drive upload API) дозволяє файли з однаковими іменами в одній папці — Drive просто створить два файли з однаковою назвою (різний fileId). Тобто гіпотеза "колізія імен → Failed to fetch" — спростована. Реальна причина "Failed to fetch" — мережева/CORS/timeout/auth (`fetch()` кидає `TypeError: Failed to fetch` коли запит навіть не дійшов до сервера).

**Що з reprocess + "Текст оновлено":** для документа без `driveId` reprocess нічого корисного не зробить (нема чого читати з Drive). Але якщо адвокат бачить toast "Текст оновлено" — імовірно reprocess запускається на іншому документі або API таки повертає success на порожньому контенті. Без логів важко сказати точно. Звертаємо увагу — це окрема знахідка.

### Гіпотеза 2: Видалення "повністю" не видаляє файли з Drive

**Статус: ПІДТВЕРДЖЕНА (з нюансами).**

**Докази:**
- Код викликає `deleteDriveFile(doc.driveId)` (App.jsx:5429-5434) — функція реальна, посилає DELETE на Drive API (driveService.js:348-359).
- Виклик огорнутий у try/catch з лише `console.warn` :5432-5434. Помилка не пробивається до користувача і не міняє результат `success: true`.
- 403 Forbidden, 5xx, мережа → файл лишається, операція "вдалася" з точки зору UI.
- Найголовніше: **якщо `doc.driveId === null`** (а саме це у Брановського/Кісельова згідно з гіпотезою 1) — гілка `if (doc.driveId)` :5429 взагалі не виконується. Drive delete не викликається. Файли (якщо вони фізично є на Drive — від іншого процесу) лишаються, запис у реєстрі видаляється.
- `deleteOcrCacheForDocument` :5436-5440 теж тихо проковтує помилки — кеш `02_ОБРОБЛЕНІ` може лишитися.

**Точкове підтвердження симптому адвоката** ("видалив документ з опцією повністю — запис зник, файли в `01_ОРИГІНАЛИ` і `02_ОБРОБЛЕНІ` залишилися"):
- Якщо документ був orphan-записом без `driveId` → файли (з паралельного процесу — наприклад, контекст-агент чи стара версія DP) лишаться.
- Якщо документ мав `driveId`, але видалення провалилось з 403/мережею → файли лишаться, користувач не дізнається.

### Гіпотеза 3: Декілька шляхів створення файлів і записів — не синхронізовані

**Статус: ПІДТВЕРДЖЕНА частково.**

**Докази:** див. таблицю 2.1.

- **6 шляхів створення документа.** Один обхід ACTION (DocumentProcessor через `updateCase()`). Один без fail-fast (AddDocumentModal). Решта 4 шляхи синхронізовані коректно.
- **Канонічну схему через `createDocument()` використовують ВСІ.** Це позитив — TASK 1 (canonical document schema v5) виконано добре.
- **Контекст-агент НЕ додає документи в реєстр.** Він читає файли з Drive і викликає `extractText()`, який пише `02_ОБРОБЛЕНІ/<basename>_<driveId>.txt`. Це кеш-файл, не документ-запис. Тобто гіпотеза "контекст-агент створює orphans у реєстрі" — спростована, контекст-агент orphan'ів у реєстрі не створює, але створює файли в `02_ОБРОБЛЕНІ` без запису (за дизайном, бо це кеш).

**Прихована проблема, не охоплена гіпотезою:** OCR-кеш прив'язаний до `driveId` через `cacheFileName(file) = ${sanitizeBasename(name)}_${file.id}.txt` (ocrService.js:42). Якщо документ перезавантажити (отримає новий `driveId`) — кеш стане orphan з точки зору пошуку (не знайдеться по ключу), хоча файл лишиться в `02_ОБРОБЛЕНІ`. При видаленні документа з новим `driveId` — `deleteOcrCacheForDocument` теж не знайде старого orphan-кешу.

### Гіпотеза 4: PDF rendering через Drive iframe не працює без auth

**Статус: ПІДТВЕРДЖЕНА частково; справжня корінна причина в більшості випадків інша.**

**Докази:**
- `ScanContent` (DocumentViewerContent.jsx:33-69):
  - При `!document.driveId` → AlertTriangle "Файл не прикріплено до Drive" :34-44.
  - Інакше — `<iframe src="https://drive.google.com/file/d/{driveId}/preview"/>` :59-67. **Без onError, без fallback.**
- Drive iframe `/preview` для приватних файлів справді не покаже контент — Google показує власну "Sign in" сторінку всередині iframe. Браузер блокує third-party cookies і це може дати порожній прямокутник.
- АЛЕ у нашому випадку Брановського — driveId=null, тож iframe навіть не рендериться. Адвокат бачить empty state з `AlertTriangle` (DocumentViewer.css:159 задає min-height ~400px → іконка може загубитися візуально, особливо якщо `--color-text-3` тонкий).

**Що показує Viewer для різних станів:**

| Стан | Scan view | Text view |
|------|-----------|-----------|
| `driveId === null` (orphan/seed) | AlertTriangle "Файл не прикріплено до Drive" | "Текст для цього документа ще не розпізнано" |
| `driveId` валідний, доступ є | iframe з preview | OCR-текст з кешу або empty з кнопкою "Розпізнати" |
| `driveId` валідний, файл приватний (не наша Drive-сесія) | білий iframe (silent fail) | в `getCachedText` → catch повертає `null` → "Текст ще не розпізнано" |
| `driveId` валідний, файл видалено з Drive | білий iframe | те саме |

**Дві проблеми UX:**
1. AlertTriangle empty state не дуже впадає в око на темному фоні з тонким сірим текстом — адвокат не помічає, описує як "білий прямокутник".
2. iframe для приватного файлу — silent fail без onError. Жодного fallback на pdf.js + blob через `gapi`.

### Гіпотеза 5: AddDocumentModal date picker не зачеплено

**Статус: ПІДТВЕРДЖЕНА.**

**Докази:**
- `AddDocumentModal.jsx:152-157`:
  ```jsx
  <Input
    label="Дата документа"
    type="date"
    value={state.date}
    onChange={(v) => setState((s) => ({ ...s, date: v }))}
  />
  ```
- `Input` (`UI/Input.jsx`) — passthrough до нативного `<input>`. На Android `type="date"` завжди викликає системний DatePicker. У `src/components/UI/` кастомного DatePicker немає.

TASK 10.1 переписав native `<select>` (через `Select`), але про `<input type="date">` — окремий вектор. Це не баг, а **скоп TASK 10.1 не покривав цей кейс**. Очікувано і логічно — не критичне, але не "брендоване". Це робиться окремим TASK на створення `<DatePicker/>` компонента.

### Гіпотеза 6: Архів — текст карток "з'їжджає"

**Статус: ПІДТВЕРДЖЕНА. Точна причина ідентифікована.**

**Докази:**
- `ArchiveView.css:133-140`:
  ```css
  .archive-card__name {
    font-size: var(--text-sm);
    font-weight: var(--weight-medium);
    white-space: nowrap;       /* ← обмежує до одного рядка */
    overflow: hidden;
    text-overflow: ellipsis;   /* ← "Позовна заява" → "П…" */
    color: var(--color-text);
  }
  ```
- `ArchiveView.jsx:134-168` — flex-структура: Checkbox (24px, `flex-shrink: 0`) + content (flex 1, `min-width: 0`) + actions кнопки.
- Media query `@media (max-width: 768px)` :185-196 робить wrap у 2 рядки. Але **на лівій панелі ширина ~280px і це не мобільний viewport** — media-query не спрацьовує.
- На 280px ширині після Checkbox/icon/actions залишається ~120-150px для назви → "Позовна заява" обрізається до "П…".

**Фікс:** прибрати `white-space: nowrap` для назви (дозволити wrap у 2 рядки) АБО додати media-query на ширину контейнера, не viewport (через container queries чи `max-width: 480px` блок). Це простий CSS-фікс.

## 5. ЗНАХІДКИ ПОЗА ГІПОТЕЗАМИ

### 5.1 (КРИТИЧНО) AddDocumentModal не fail-fast при upload error

**Місце:** `CaseDossier/index.jsx:2806-2849`.

```js
let driveId = null;
if (file && driveConnected) {
  try {
    driveId = await uploadFileLocal(prepared, caseData);
  } catch (err) {
    console.error('Drive upload error:', err);
    toast.error('Не вдалось завантажити на Drive', { description: err?.message });
    // ❗ НЕ throw, НЕ return
  }
}
const doc = createDocument({ ..., driveId, ... });  // driveId === null
await onExecuteAction('dossier_agent', 'add_document', { caseId, document: doc });
toast.success('Документ додано');                    // суперечливий toast
```

**Симптоми для адвоката:**
- Два суперечливі toasts підряд: "Не вдалось завантажити" + "Документ додано".
- Запис в реєстрі без файлу.
- DocumentViewer показує AlertTriangle empty state.
- Видалення в режимі `'full'` не дочистить нічого з Drive (бо `if (doc.driveId)` пропустить гілку).

Це найімовірніше джерело симптомів адвоката. Drag-n-drop у "Огляд" (A4) має правильну логіку — там `if (!result?.success) throw` і запис не створюється при помилці.

### 5.2 DocumentProcessor обходить ACTION

`DocumentProcessor/index.jsx:826` і `:966` викликають `updateCase(caseId, "documents", [...])` напряму, не `add_document` ACTION. Наслідки:
- Не пройдуть `validateDocument` (хоча `createDocument` звичайно повертає валідний об'єкт — теоретичний ризик якщо в майбутньому DP додасть кастомні поля).
- Audit log не пишеться (для add це й нормально — `add_document` не в `AUDIT_ACTIONS`, але коли додамо аудит документів — DP пропустить).
- `checkCaseAccess` обходиться (для одного user не критично, але порушує філософію "ембріон з повним ДНК").

**Це треба виправити в Document Processor v2.** Поки що — задокументовано як борг у `bugs_found_during_billing_foundation.md` стиль.

### 5.3 OCR-кеш orphan після зміни driveId

`ocrService.js:42` — `cacheFileName = sanitizeBasename(name)_${driveId}.txt`. Якщо документ:
1. Створено → driveId=A → кеш `<name>_A.txt`.
2. Видалено разом з файлом → кеш видалено.
3. Адвокат завантажив той самий файл наново → driveId=B → кеш `<name>_B.txt`.

Якщо між кроками 2 і 3 кеш з `_A` НЕ був видалений (наприклад, через 403 при `deleteDriveFile`) — він стане orphan-`.txt` без жодного запису з ним.

Менш критично, але накопичується смітник у `02_ОБРОБЛЕНІ` з часом.

### 5.4 Усі помилки видалення тихо проковтуються

`App.jsx:5413-5417`, `:5432-5434`, `:5436-5440` — три try/catch з `console.warn`. Якщо помилка трапилась — користувач не дізнається. Можна:
- Вертати `success: 'partial'` з полем `warnings: [...]`.
- DeleteDocumentModal показувати warning toast при partial success.

Це покращення UX, не критичний баг.

### 5.5 DeleteDocumentModal не пропонує "видалити з помилок-orphans"

Якщо документ orphan (`driveId: null`), модалка все одно показує дві опції — "Архівувати" і "Видалити повністю". "Повністю" для orphan ефективно дорівнює `registry_only`. Це не баг, але можна додати UI-підказку: "У цього документа немає файлу на Drive — видалення прибере лише запис".

### 5.6 ScanContent для PDF без `mimeType` потрапляє в iframe-гілку

`DocumentViewerContent.jsx:46` — `isImage` визначається через `document.mimeType`. У канонічній схемі поля `mimeType` НЕМАЄ (`CANONICAL_DOCUMENT_FIELDS` не містить його). Тобто для всіх документів `mimeType === undefined`, `isImage` завжди `false`, картинки рендеряться через iframe. Drive iframe для зображень працює, але неоптимально — для JPEG/PNG швидше прямий `<img>`.

`AddDocumentModal` тощо `mimeType` теж не зберігають — це поле випадає з канонічної схеми. **Або треба додати `mimeType` у `CANONICAL_DOCUMENT_FIELDS`, або забрати залежність від нього у Viewer**.

## 6. АРХІТЕКТУРНИЙ АНАЛІЗ — ВІДВЕРТО

### 6.1 Узгодженість

Архітектура **на 80% узгоджена**. Канонічна схема, фабрика, validateDocument, делегатори ACTION — все це чисто і добре зроблено. Але між шарами є **3 невеликі тріщини** які накопичились від різних TASK:

1. AddDocumentModal створено як "брендована заміна" в TASK 10.1, але логіка обробки upload errors там залишилась простою — focused на UI заміні `<select>`, не на надійності upload pipeline. Це регресія порівняно з drag-n-drop, який має правильну error handling.

2. DocumentProcessor живе своїм життям — не використовує ACTION. Це до Document Processor v2 нормально, але треба тримати в полі зору.

3. OCR-кеш живе у `02_ОБРОБЛЕНІ` через окремий код-шлях ocrService. Він не знає про `case.documents[]`, орієнтується на ім'я файлу + driveId. Якщо `case.documents[]` змінюється не атомарно з Drive — utenсильки лишаються.

Це **не "хаос накопичений"**, але **і не повністю узгоджена пайплайна**. Скоріше: добре спроектована основа з кількома живими ділянками де старі і нові підходи зустрічаються.

### 6.2 Скільки реальних шляхів додавання

**Шість**, з них **три активних UI-шляхи** (AddDocumentModal, drop queue в Огляді, DocumentProcessor batch+split). Не синхронізовані повністю:
- AddDocumentModal не fail-fast.
- DocumentProcessor обходить ACTION.
- Drop queue в Огляді — best practice, на нього варто рівнятися.

### 6.3 Що мінімально треба щоб документи Брановського/Кісельова показались

**Треба три речі — від найдешевшого:**

1. **Локально (5 хв):** прибрати seed-документи з Брановського (App.jsx:111-124) → 12 фантомів зникне з матеріалів справи. Це регресія для демо, але демо має сенс лише на пустому tenant'і; на реальних даних seed Брановського — це чисто гра.
   
   *Альтернатива (зберегти демо):* у seed додати `status: 'archived'` для всіх 12 — тоді в Матеріалах їх не буде, в Архіві будуть.

2. **Локально (15 хв):** виправити AddDocumentModal — при failed upload не створювати запис, просто залишити модалку відкритою з error-станом, щоб адвокат міг повторити з тим самим файлом.

3. **Локально (15-30 хв):** для документів які мають реальний `driveId`, але приватні / без preview доступу — додати `<iframe onError>` + fallback "Відкрити в Drive" посилання. Або хоч додати кнопку "Відкрити в Drive" біля iframe щоб адвокат не залишався в порожньому стані.

Цих трьох змін достатньо для зникнення симптомів. **Архітектурно нічого не ламається.**

### 6.4 Локальний фікс vs Document Processor v2

**Моя рекомендація: гібрид.**

- **Прямо зараз** зробити три локальні фікси (#1-#3 з 6.3) + пункти 5.1, 5.2 з знахідок поза гіпотезами + CSS-фікс ArchiveView. Це 1-2 години роботи, прибере всі поточні симптоми, не вплине на майбутній DP v2.
- **DP v2** не варто прискорювати або додавати в нього "пишемо все з нуля". DP v2 і так в roadmap; коли він вийде, він буде новим UI з tool use і собі підтягне правильну error handling. Але **поточні документи живуть у тій самій схемі і таблиці** — DP v2 не змінить структуру `case.documents[]`. Тому **поточні баги треба фіксити локально**, навіть якщо DP v2 на горизонті.
- Кастомний DatePicker — окремий невеликий TASK (півдня), не блокуючий.

## 8. ПОЯСНЕННЯ ДЛЯ АДВОКАТА (без термінології)

Коротко що відбувається.

**Чому документи не показуються у Брановського.**
Брановський — демо-справа. У ній 12 документів які створив код для прикладу, без жодного реального файлу на Drive. Коли ви відкриваєте такий документ — система чесно показує "Файл не прикріплено", але іконка тонка і ви її як "білий екран" сприймаєте. Це не баг рендеру, це чесний empty state.

**Чому документи не показуються у Кісельової.**
Найімовірніше — коли ви додавали документ через "+ Додати документ", файл не залився на Drive (звідси "Failed to fetch"), але **запис у реєстрі все одно з'явився, з порожнім посиланням на файл**. Це баг модалки додавання. У результаті Viewer бачить запис без файлу — і пише "Файл не прикріплено", той самий empty state.

**Чому "Видалити повністю" не видаляє файли.**
Дві причини. Перша: для документів без реального файлу (як вище) — нема чого видаляти, але якщо файли все ж лежать у `01_ОРИГІНАЛИ` (як їх туди привів інший процес), система їх не пов'язує з записом і не чіпає. Друга: коли є справжнє посилання на файл, але Drive повертає помилку (наприклад приватний файл) — система мовчки логує warning і повідомляє вам "видалено", хоча файл лишається. Користувач не дізнається.

**Чому архів стискається до "П.".**
Просто CSS у нової картки архіву поставив `nowrap` для назви. На вузькій панелі (~280px) лишається ~120px для назви, і "Позовна заява" обрізається до "П…". Малий CSS-фікс.

**Чому модалка показує системний календар.**
Поле дати в новій модалці — нативне поле браузера (`<input type="date">`). На Android воно запускає системний DatePicker. TASK 10.1 замінив випадаючі списки, але до полів дати руки не дійшли — це окремий невеликий TASK.

**Що рекомендую.**
Не кидатися переписувати Document Processor (наступний великий TASK). Натомість зробити одну вечірню сесію — 5 локальних фіксів за 1-2 години:
1. У модалці додавання документа — якщо файл не залився, не створювати запис (модалка лишається відкритою, можете спробувати ще раз).
2. У демо-справі Брановського — або прибрати seed-документи, або позначити їх архівованими.
3. У DocumentViewer — повідомлення "Файл не прикріплено" зробити помітнішим + додати кнопку "Відкрити в Drive".
4. У архіві — прибрати `nowrap` щоб довгі назви переносилися в 2 рядки.
5. При видаленні — повертати warning якщо Drive не зміг видалити файл (наразі мовчки приймається).

Це прибере всі поточні симптоми. Document Processor v2 і кастомний DatePicker — окремими TASK, без поспіху.

**Чесна оцінка стану системи.**
Архітектура у хорошому стані. Канонічна схема документів — правильна. Centralized ACTIONs — правильні. Є три невеликі тріщини між шарами (модалка не fail-fast, DP обходить ACTION, OCR-кеш прив'язаний до driveId), але це борги розміром у 1-2 години роботи кожен. Це **не хаос накопичений**, це **чиста архітектура з трьома точковими дефектами**.
