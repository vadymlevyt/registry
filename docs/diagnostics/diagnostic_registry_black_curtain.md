# DIAGNOSTIC — «чорна штора» на списку Реєстру при відкритті документа

**Гілка:** `claude/diag-registry-black`
**Дата:** 2026-06-07
**Статус:** діагностика завершена, фікс НЕ застосовано (чекає звірки з власником)

---

## Симптом (як подано)

- Досьє → вкладка «Матеріали» → режим «Реєстр» (ліва панель = список документів),
  праворуч `DocumentViewer`.
- Тригер: клік по рядку (вибір/відкриття документа) → область СПИСКУ стає
  чорною/порожньою.
- Крос-платформено (десктоп + планшет) → НЕ webkit-reflow планшета.
- Десктоп: зверху лишається мала смужка списку (видно, скролиться), решту накриває
  «чорна штора» знизу певної точки. Планшет: уся область списку чорна.
- Лікування: перемкнути Огляд → назад Матеріали (remount) → список повертається
  з тим самим виділеним документом.

---

## Корінь (з доказами)

### 1. «Чорне» = той самий фоновий токен, не справжній чорний

- `src/styles/tokens.css:20` → `--color-bg: #0f1117;` — майже чорний (темна тема).
- `.document-viewer` (`DocumentViewer.css:7`) → `background: var(--color-bg);`
- `.materials-left-panel` (`CaseDossier.css:23`) → `background: var(--color-bg);`

Тобто фон лівої панелі і фон в'юера — **ідентичні** (#0f1117). Будь-яка
паінт-аномалія на межі цих двох елементів візуально читається як суцільна
«чорна штора»: межі між панеллю і в'юером не видно.

### 2. Вибір документа НЕ міняє структуру лівої панелі

- `selectedDoc` — єдиний стан, що змінюється при кліку
  (`index.jsx:66`; set у `onClick` рядків `index.jsx:2102` дерево / `2207` реєстр).
- `filteredDocs` (`index.jsx:1053`) і `registrySel` (`index.jsx:1063`) від
  `selectedDoc` **не залежать** → список рядків і `BulkActionBar` не
  перебудовуються від вибору.
- Це підтверджує, чому попередня спроба (стабілізація висоти `BulkActionBar`,
  коміт `fccf075`) не допомогла: бар не є тригером. Тригер — **відкриття
  документа праворуч**, а не чекбокс.

### 3. `DocumentViewer` мутується «на місці» + вставляє важкий iframe

- `index.jsx:2290` → `<DocumentViewer document={selectedDoc} ... />` — **без `key`**.
- При першому виборі `selectedDoc: null → doc`:
  - `null` → ранній return порожнього стану
    (`DocumentViewer/index.jsx:178` — `.document-viewer--empty`, центрований flex).
  - `doc` → повний рендер: header + content + footer.
- React зберігає **той самий** кореневий `<div.document-viewer>` і мутує його
  вміст на місці (прибирає `--empty`/центрування, монтує дочірні).
- Для майже всіх документів `content` → `ScanContent`
  (`DocumentViewerContent.jsx:489`) → `PdfRenderer`
  (`DocumentViewerContent.jsx:524`) → **same-origin `<iframe>` повного pdf.js
  viewer** (`PdfRenderer.jsx:92`). Дефолт-режим scanned-без-layout і searchable
  PDF = `scan` → саме ця гілка (`DocumentViewer/index.jsx:119`).

Вставка важкого iframe = поява нового **композитного шару** в браузері. Це
відбувається всередині спільного контексту позиціювання `.materials-layout`
(`CaseDossier.css:14-15` → `overflow: hidden; position: relative;`), де ліва
панель і в'юер — сусіди в одному `position:relative` контейнері.

### 4. Чому remount лікує, а транзишн — ні

- Перемикання вкладки **демонтує** всю піддерево Матеріалів і монтує **свіжо**:
  `DocumentViewer` створюється одразу з `selectedDoc` (повний рендер), iframe
  монтується один раз у чистому піддереві. Стара паінт-картина відкидається.
- Тобто **фінальний DOM/layout транзишн-рендера і remount-рендера ІДЕНТИЧНІ**
  (той самий вибраний документ, той самий iframe). Відрізняється лише шлях:
  *мутація на місці* (empty→content + вставка iframe у переюзаний вузол) проти
  *чистого монтажу*.
- Якби це був суто CSS-колапс layout — він проявлявся б і після remount (бо
  фінальний layout той самий). Він НЕ проявляється → отже це **паінт/композитна
  аномалія саме транзишн-мутації**, не помилка статичного CSS.

### 5. Чому «штора» не зникає при скролі (= оверлей, не пропущений паінт)

Зі звіту: верхня смужка скролиться і видима, але решта лишається чорною навіть
коли список під нею прокручується. Якби це був просто непомальований контент —
скрол відкривав би рядки. Те, що під «шторою» скрол не проявляє рядків, вказує,
що зверху лежить **композитний шар** (бекинг iframe / стале composited-полотно
в'юера з тим самим `#0f1117`), а не порожній фон. Контейнер списку живий (DOM на
місці, верх скролиться) — його просто **перекриває стале темне полотно**.

**Підсумок кореня:** при першому відкритті документа React мутує переюзаний вузол
`.document-viewer` (empty→content) і вставляє важкий same-origin pdf.js `<iframe>`
у спільний `position:relative; overflow:hidden` контейнер `.materials-layout`.
Браузер залишає стале темне композитне полотно поверх сусідньої зони списку лівої
панелі. Оскільки фон панелі і фон в'юера — однаковий `#0f1117`, це читається як
«чорна штора». Чистий remount (перемикання вкладки) відкидає старий шар і все
малюється коректно.

---

## Що НЕ є коренем (перевірено, відкидаю)

- **`BulkActionBar` висота** (попередня гіпотеза, `fccf075`) — бар залежить від
  `filteredDocs.length>0`, не від вибору; від кліку по рядку не з'являється/зникає.
- **`.materials-layout overflow:hidden`** (теж пробували) — це контейнер, не
  джерело темного полотна; layout фінально стабільний (доказ §4).
- **Колапс flex-контейнера списку** — фінальний layout після remount ідентичний,
  а там колапсу немає → отже не layout-колапс.
- **Планшетний webkit-reflow** — баг крос-платформений (десктоп Chromium теж).

## Варіанти фіксу (на вибір; НЕ реалізовано)

### Варіант A — `key` на `DocumentViewer` (прямий, мімікрує лікувальний remount)

`index.jsx:2290`: додати `key={selectedDoc?.id || 'empty'}` (або
`key={selectedDoc ? 'viewer' : 'empty'}`). Тоді при `null→doc` React **повністю
ремонтує** в'юер замість мутації на місці — кожен вибір поводиться як відомий
робочий remount після перемикання вкладки.

- Плюс: детерміновано відтворює вже перевірений «зцілений» шлях; дешево; низький
  ризик.
- Кошт: повний ремонт в'юера на перший вибір. Незначний — зміна документа й так
  щоразу створює новий iframe (новий `driveId`).

### Варіант B — винести ліву панель на власний композитний шар (б'є по причині)

`.materials-left-panel` (`CaseDossier.css:18`): додати `isolation: isolate;`
(новий stacking-контекст) і/або `transform: translateZ(0);` (промоція у власний
GPU-шар). Так паінт панелі стає незалежним від шару iframe-в'юера — стале полотно
в'юера не може лягти поверх списку.

- Плюс: адресує саме композитне перекриття, не чіпає React-дерево.
- Кошт/ризик: ще один постійний composited-шар; перевірити, що `isolation`/
  `translateZ` не ламають `position:absolute` тогл-кнопку колапсу
  (`.materials-collapse-toggle`, z-index:12) і drawer ≤768px.

**Рекомендація:** почати з **A** (детермінований, дзеркалить відомий робочий
шлях), тримати **B** як альтернативу/доповнення, якщо A десь не закриє.
Обидва — після звірки з власником.
