## МЕТА ФАЗИ

Додати у в'ювер режим перегляду **«Точний»** для сканованих документів: текст збирається
**на льоту** з `.layout.json` детермінованим конденсатором (КРОК 1, `layoutToMarkdownDraft` — вже
існує з 3.1), **без AI, без зберігання**. Адвокат перемикається Скан / Текст / **Точний** і
**порівнює** Точний (з layout) із поточним «Текст» (з `.txt`).

**Видимий результат:** у в'ювері скан-документа з'являється опція «Точний» — гарно вирівняні
абзаци/заголовки прямо з layout, дослівно, миттєво.

## СКЛАДОВІ

### V2-A1.1 — опція «Точний» у перемикачі в'ювера
- Додати до `viewMode` третю опцію **«Точний»** — **тільки** коли документ `scanned` І має layout
  (перевірка наявності `.layout.json` через `getCachedLayout`/метадані). Для searchable / без layout —
  опція **не показується**.
- Поточні «Скан» і «Текст» — **без змін** (Текст і далі з `getCleanOrRawText`).

### V2-A1.2 — рендер Точного (live, без зберігання)
- При виборі «Точний»: `const layout = await getCachedLayout(file)` → `layoutToMarkdownDraft(layout)`
  → показати через `MarkdownRenderer`.
- **Нічого не зберігаємо** на Drive (live-обчислення щоразу; дешево, детерміновано).
- Стан завантаження (спінер) поки тягнеться layout; помилка/нема layout → акуратний фолбек
  (сховати опцію або повідомлення «layout недоступний»), НЕ валити в'ювер.
- Якщо `layoutToMarkdownDraft` тягне важкі залежності при імпорті — lazy `import()` (як патерн 3.1).
  (Перевірити: функція чиста; якщо імпорт `cleanTextService` безпечний — звичайний import.)
