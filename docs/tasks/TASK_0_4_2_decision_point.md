## КОНТЕКСТ

TASK 0.4.1 для кримінальних з кількома обвинуваченими робив так: НЕ вгадувати прізвище, `primaryParty = case_no`, додати у `warnings`. Тобто **тихо помічав** проблему.

Це окремий випадок ширшого **принципу точки рішення** (нижче).

## ПРИНЦИП ТОЧКИ РІШЕННЯ АДВОКАТА (новий, фіксується)

> Агент синхронізації працює самостійно до кінця, АБО до точки де далі не може без рішення адвоката. Тоді він НЕ перериває роботу по одному питанню — він доходить до кінця обходу, збирає ВСІ питання що потребують рішення адвоката, і подає їх ОДНИМ СПИСКОМ перед фінальним JSON. Адвокат відповідає одним блоком, агент коригує і видає фінальний JSON.

Це поведінка для всіх майбутніх неоднозначностей, не тільки кримінальних. Зараз єдиний кейс — кілька обвинувачених. Згодом можуть бути інші (невизначена категорія, незрозумілий тип документа тощо) — усі збираються в одну точку рішення.

**Обмеження Track A:** Claude for Chrome — це чат, без інтерактивних галочок/форм. Тому список питань — текстовий, відповідь адвоката — текстова. Повноцінний UI з галочками буде у власному розширенні (Track B). Зараз робимо максимально зручний текстовий формат.

## ЩО ТРЕБА ЗРОБИТИ

### Зміна 1 — Блок 3 (кримінальні з кількома обвинуваченими)

Замінити поточну логіку «тихо в warnings» на «позначити для точки рішення»:

```
═══════════════════════════════════════════════════════════════
КРИМІНАЛЬНІ СПРАВИ З КІЛЬКОМА ОБВИНУВАЧЕНИМИ
═══════════════════════════════════════════════════════════════

Якщо у кримінальній справі (роль "Захисник") у "Інформація про
справу" КІЛЬКА обвинувачених і НЕ ЗРОЗУМІЛО кого саме захищає
Левицький:

- НЕ вгадуй прізвище
- Засідання все одно витягни (вони цінні незалежно від назви)
- primaryParty тимчасово = case_no
- ПОЗНАЧ цю справу як таку що ПОТРЕБУЄ РІШЕННЯ АДВОКАТА
  (запам'ятай: case_no, список обвинувачених)
- Додай питання у список для ТОЧКИ РІШЕННЯ (блок нижче)

НЕ зупиняй обхід заради цього питання — продовжуй далі,
питання збереться у фінальний список.
```

### Зміна 2 — Новий блок «ТОЧКА РІШЕННЯ» (перед фінальним JSON)

Додати перед блоком «ПРОМІЖНІ РЕЗУЛЬТАТИ» / структурою envelope:

```
═══════════════════════════════════════════════════════════════
ТОЧКА РІШЕННЯ АДВОКАТА (перед фінальним JSON)
═══════════════════════════════════════════════════════════════

Коли ти обійшов усі справи — перевір чи накопичились питання що
потребують рішення адвоката (наприклад кримінальні з кількома
обвинуваченими де незрозуміло кого захищає Левицький).

ЯКЩО таких питань НЕМАЄ:
Одразу видавай фінальний JSON.

ЯКЩО питання Є:
НЕ видавай фінальний JSON одразу. Спочатку подай адвокату СПИСОК
усіх питань одним блоком і ЧЕКАЙ його відповідь:

────────────────────────────────────────
Перш ніж завершити — потрібні твої рішення:

1. Справа [case_no] ([суд], кримінальна).
   Обвинувачені: [Прізвище 1], [Прізвище 2].
   Кого з них захищає Левицький?
   Відповідь: [Прізвище 1] / [Прізвище 2] / обох

[2, 3... якщо більше питань]

Відповідай номером і вибором, наприклад: "1 — [Прізвище 1]"
────────────────────────────────────────

Після відповіді адвоката:
- Постав обраного підзахисного як primaryParty для тієї справи
- Якщо "обох" — primaryParty = перший обвинувачений, у notes
  додай "Захист обох обвинувачених: [список]"
- Видай ФІНАЛЬНИЙ JSON з урахуванням відповідей

Це ТОЧКА РІШЕННЯ — єдине місце де ти зупиняєшся спитати. Усі
неоднозначні справи збираються сюди, не перебивають обхід по одній.
```
