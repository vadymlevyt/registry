## 1. СУТЬ І НАВІЩО

Документи з Електронного суду (ЄСІТС) приходять **ZIP-архівами**. Усередині:
- **документи** (часто searchable PDF з текстовим шаром; також HTML — ухвали,
  протоколи, картка руху; скановані PDF; зображення);
- **файли електронного підпису КЕП** (`.p7s`/`.sig`) — НЕ документи.

Завдання: дати адвокату кинути ZIP (один чи кілька, разом з іншими файлами), а система:
1. **розпакує** ZIP у складові файли;
2. **відкине** файли підписів `.p7s`/`.sig` (просто прибрати, нічого не зберігати —
   рішення власника: факт «підписано» наразі не використовується, легко додати потім);
3. усе решта (вміст ZIP **+** окремі файли, додані поряд) пустить **звичайним
   сценарієм `addFiles`** — кожен своїм шляхом (PDF як є, скан стиснути/осіарити,
   HTML/DOCX→PDF, зображення→PDF). Тумблери «без OCR»/«Стиснути» працюють штатно.

**Архітектурно:** це **ще один фронт-крок** перед `addFiles` — точно як крок
стиснення. «Одна труба, кожен файл своїм шляхом»; ZIP просто **розгортається першим**.

**Гарна новина:** ядро вже написане і протестоване — `src/services/documentPipeline/
stages/unpack.js` (`createIntakeWithUnpack` + чисті предикати, `unpack.test.js`,
`dp2-stages.test.js`). Але воно **сплить** (ніде не імпортується). Цей TASK переюзовує
його чисті частини у фронт-кроці для `addFiles` (НЕ активує stage-machine варіант).

## 2. ЩО ВХОДИТЬ

### 2.1 Новий фронт-крок
`src/services/addFiles/unpackArchivesFrontStep.js` — чиста функція
`unpackArchivesFrontStep(files, opts) → { files: File[], report }`:
- на вхід — масив `File` (як їх збирає DP `buildAddAsIsInput` / модалка);
- ZIP → розпакувати у складові `File`; **викинути** `.p7s`/`.sig`; sidecar `.json`
  (якщо раптом є) — поки ігнорувати (див. §5 HTML-метадані);
- RAR/7z → **не розпаковувати**, лишити як один файл + повідомлення;
- не-архів → passthrough незмінним;
- кілька ZIP → кожен; усе зливається в один плоский список;
- `report` = `{ unpacked: [{archive, entryCount}], signaturesDropped: n, archivesKept: [...] }`
  — для тоста/діагностики.

**Переюзати з `unpack.js`** (не дублювати класифікацію — single source):
`isArchive`, `archiveKind`, `isSignatureFile` (вже export). Для самого розпакування —
винести/переюзати `defaultUnzipArchive` (fflate, lazy-import) і `entryToFile`/`guessMime`
(зараз internal — **експортувати** з `unpack.js` або винести у спільний `archiveCore.js`,
щоб і дрімаючий `createIntakeWithUnpack`, і новий фронт-крок тягли з одного місця).

### 2.2 Дротування
- **Document Processor** (основний дім — там батч): у `buildAddAsIsInput` (або одразу
  перед `pipeline.addFiles`) прогнати зібрані файли через `unpackArchivesFrontStep`,
  тоді — у `addFiles`. Порядок: **unpack → (compress всередині addFiles) → addFiles**.
- **Модалка**: архіви НЕ обробляє — показує підказку «через Document Processor» (§6, рішення власника).

### 2.3 UI-сигнал
Після розпакування — тост: «Розпаковано N файлів з архіву; M підписів відкинуто»
(коротко, людською мовою). RAR/7z → «Архів RAR не розпаковується — додано як є».

## 3. АЛГОРИТМ (по кроках)

```
вхід: список File (з пристрою / Drive-пікера / їх суміш)
─────────────────────────────────────────────────────────
для кожного file:
  якщо isArchive(file):
     kind = archiveKind(file)
     якщо kind === 'zip':
        entries = unzip(file)            // fflate, lazy
        для кожного entry (без директорій):
           якщо isSignatureFile(entry.name): ВІДКИНУТИ (лічильник++)
           інакше: → entryToFile(entry) у вихідний список
     інакше (rar/7z): лишити file як є + warning
  інакше:
     file → у вихідний список як є
─────────────────────────────────────────────────────────
вихід: плоский список File → addFiles(input, {ocrMode, compress})
       кожен далі своїм шляхом (convert/compress/OCR per file)
```

## 4. МЕЖІ — ЩО НЕ РОБИТИ

- **НЕ зберігати інформацію про підписи** (`ctx.signatures[]` НЕ робимо — рішення

- **НЕ розпаковувати RAR/7z** (браузер не вміє; лишити як є + повідомлення).
- **НЕ робити вилучення метаданих** (metadata extractor) у цьому TASK — це окрема

- **НЕ активувати дрімаючий `createIntakeWithUnpack`** як INTAKE-stage у Provider

- **НЕ змінювати `addFiles`-сервіс** — він уже приймає готовий список файлів; ZIP
  розгортається ДО нього.

## 5. HTML-МЕТАДАНІ — лише ГАЧОК (не реалізація)

Важлива деталь (думка власника): ЄСІТС-документи часто **HTML**, і при HTML→PDF
структуровані метадані зі сторінки **губляться** (лишається текстовий шар). Єдине
місце дістати їх — **до конвертації**, тобто **на цьому кроці розпакування** (поки
HTML ще HTML).

Цей TASK: **закласти порожній/опційний гачок** `onArchiveEntry(entry)` (або
`extractHtmlMetadata?: (htmlFile) => meta`), куди в МАЙБУТНЬОМУ вкладеться metadata
extractor. Зараз — no-op. **Реалізацію витягу — НЕ робити** (окремий TASK; краще серверно).

## 6. НА ЩО ЗВАЖАТИ

- **Порядок:** unpack ПЕРШИЙ (бо архів містить файли); стиснення/конвертація — далі
  всередині `addFiles` per file.
- **Пам'ять:** розпакування ZIP тримає вміст у пам'яті; великі архіви — за можливостями
  девайса (власник це приймає). fflate розпаковує синхронно — для дуже великих можливий
  пік; не оптимізуємо передчасно, але **не тримати** зайвих копій.
- **Мікс після розпакування:** вміст ЄСІТС-ZIP зазвичай мішаний (PDF+HTML+зображення).
  Це сценарій «**просто додати**» (addFiles), НЕ нарізка. Якщо адвокат не ввімкнув
  «просто додати» і набір мішаний — спрацює наявний guard-тост «увімкніть Просто додати».
  (Тобто для ЄСІТС: тумблер «Просто додати» УВІМК.)
- **Ідентичність із модалкою:** завантаження — через спільний `uploadFileToCaseFolder`
  (вже є); конвертація — `converterService`; нічого не дублюємо.

### Рішення власника: архіви — ТІЛЬКИ Document Processor
**Модалка архіви НЕ обробляє.** Модалка одно-файлова (форма на ОДИН документ), а ZIP
вибухає у багато — це не її UX. Тому:
- Розпакування (`unpackArchivesFrontStep`) дротуємо **лише в Document Processor**.
- Якщо адвокат кинув **архів у модалку** — показати підказку «Архіви додавайте через
  Document Processor» і **не додавати** (не пхати ZIP як один файл).
- Фронт-крок лишається у `services/` (generic), але **споживач один — DP**.

## 7. SAAS IMPLICATIONS

- **Поля сутностей:** нових полів схеми НЕ додаємо (документи з ZIP створюються тим
  самим `createDocument`, ті самі канонічні поля). `source` лишається за вибором каналу
  (для ручного ZIP — `manual`; для майбутнього court_sync-каналу — `court_sync`).
- **Permissions:** окремих ACTIONS не треба — додавання йде через наявний `add_documents`/
  `add_document` (allowlist незмінний). Розпакування — чиста клієнтська функція, без
  модифікації даних.
- **Tenant isolation:** не зачіпається (файли лягають у папку справи через
  `uploadFileToCaseFolder`, що вже tenant-agnostic через caseData).
- **Multi-user:** без змін.

## 8. BILLING IMPLICATIONS

- **Точки інструментації:** саме розпакування — **не білабельна** технічна дія (як
  стиснення/конвертація). Білінг виникає лише на тому, що роблять складові файли далі
  (OCR через `addFiles` пост-крок — уже інструментовано в TASK 4). Окремих точок НЕ
  додаємо.
- **Категорії часу:** без змін.
- **AI usage:** unpack не викликає AI (нуль `ai_usage`). Якщо в майбутньому HTML-метадані
  тягтиме AI — то вже інший TASK зі своєю інструментацією.

## 9. AI-FIRST / VERIFIABILITY

- **AI-aware:** розпакування — детермінована функція, AI не потрібен. Доступність через
  агента: додавання вмісту вже покрите `add_documents` ACTION (агент може додавати
  файли); розпакування — клієнтський фронт-крок UI-входу (як вибір файлу).
- **Verifiable:** результат перевіряється без UI — `unpackArchivesFrontStep` повертає
  плоский список + report; юніт-тест перевіряє склад (підписи відкинуто, ZIP розгорнуто,
  RAR лишено).
