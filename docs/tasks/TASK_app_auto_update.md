# TASK — Авто-оновлення застосунку (без ручного чищення кешу)

**Тип:** спека для сесії-виконавця. Адмін-сесія НЕ реалізує сама.
**Статус:** очікує затвердження → виконавець.
**Дата:** 2026-06-10
**Привід:** після деплою відкрита вкладка лишається на СТАРОМУ коді, поки користувач
вручну не зробить hard-reload / clear cache / анонімне вікно. Це неприйнятно —
застосунок має оновлюватися сам.

---

## 0. Вимога

> Користувач НІКОЛИ не має вручну чистити кеш чи робити hard-reload, щоб отримати
> нову версію. Застосунок сам помічає, що вийшла новіша версія, і пропонує
> (або тихо робить) оновлення.

## 1. Корінь

SPA (Vite на GitHub Pages) тримається у вкладці. Vite хешує імена ассетів — новий
білд має нові імена, тож проблема НЕ в самих ассетах, а в тому що:
1. відкрита вкладка вже завантажила старий JS у пам'ять і **не дізнається** про новий
   деплой, поки її не перезавантажать;
2. `index.html` на GitHub Pages кешується (короткий max-age), тож навіть звичайний
   reload може якийсь час віддавати старий `index.html` → старі ассети.

Service-worker'а/PWA немає (перевірено) — авто-оновлення зараз немає взагалі.

## 2. Рішення — version-check + банер «Оновити»

**2a. Бейкаємо build-id у застосунок.** При білді зашити унікальний ідентифікатор
збірки (git SHA або timestamp) через Vite `define` (напр. `__BUILD_ID__`). Той самий
ідентифікатор записати у файл `version.json` у вихідній збірці (`dist/version.json`),
напр. `{ "buildId": "<sha-or-ts>" }`. (Скрипт prebuild або маленький Vite-плагін, що
пише і define, і version.json з ОДНОГО джерела — правило #11: один build-id, два
місця, одне джерело.)

**2b. `updateChecker.js`** — новий сервіс:
- перевіряє оновлення: при старті, при `window` focus (`visibilitychange`/`focus`),
  і кожні N хв (напр. 5);
- `fetch('version.json?t=' + Date.now(), { cache: 'no-store' })` — гарантовано свіжо,
  повз кеш;
- якщо `fetched.buildId !== __BUILD_ID__` → є новіша версія → емітить подію/колбек;
- стійкий до офлайну/помилок fetch (try/catch, не падати, просто пропустити цикл).

**2c. UI — ненав'язливий банер.** Коли виявлено новішу версію:
- показати банер «Доступна нова версія Legal BMS» + кнопка **«Оновити»** →
  `window.location.reload()`;
- банер не блокує роботу; з'являється згори/знизу, можна відкласти;
- (опційно, на розсуд) авто-reload коли застосунок idle і немає незбережених змін —
  стан персиститься у Drive/localStorage, тож reload безпечний. За замовчуванням —
  банер з кнопкою (не смикати користувача посеред роботи).

**2d. Свіжий `index.html` при reload.** Щоб reload точно підтягнув НОВІ ассети, а не
кешований `index.html`:
- додати в `index.html` `<meta http-equiv="Cache-Control" content="no-cache">`
  (best-effort; браузери трактують по-різному, але разом з version-check допомагає);
- ассети (хешовані) можуть кешуватись довго — їх імена унікальні щодеплою, проблеми
  немає; критичний лише `index.html` і `version.json` (останній — завжди no-store).

> Якщо `index.html`-кеш GitHub Pages усе ж віддає старе ≤10 хв — банер просто
> з'явиться знову, користувач натисне ще раз; у межах хвилин усе зійдеться. Головне:
> **користувач нічого не чистить руками** — застосунок сам помічає і пропонує.

## 3. SaaS / SEMANTIC CLARITY

- `__BUILD_ID__` — один сенс «версія цієї збірки», одне джерело (prebuild) → і define,
  і `version.json`. Не плодити паралельних версій.
- Майбутнє (multi-tenant SaaS): механізм глобальний, tenant-незалежний — оновлення
  коду стосується всіх. Жодних tenant-полів не треба.
- Нічого не зберігається в registry/Drive — це суто клієнтський механізм оновлення.

## 4. Тести (обов'язково, `npm test` зелений)

- `updateChecker`: `buildId` збігається → подія НЕ емітиться; відрізняється → емітиться;
  fetch кидає → не падає, тихо пропускає; перевірка по focus/інтервалу викликає fetch
  з `cache:'no-store'` і cache-bust-параметром.
- Банер: рендериться лише коли є новіша версія; кнопка «Оновити» викликає reload (стаб).
- Build-id інжектиться (define присутній; version.json генерується prebuild-скриптом —
  перевірити що скрипт пише валідний JSON з тим самим id).

## 5. Межі

- НЕ вводити service worker / повний PWA (поза scope; version-check простіший і
  достатній). НЕ автo-reload посеред активної роботи без сигналу (за замовчуванням —
  банер). НЕ чіпати ЄСІТС/білінг/дані.
- Двозначність → ЗУПИНИСЬ і спитай.

## 6. Воркфлоу / здача

- НЕ пушити в main. Запуш СВОЮ гілку, назви її → адмін-звірка діфа → одне-реченнєве
  «ок» адвоката → FF.
- Звіт: `docs/reports/report_task_app_auto_update.md`.

Критерій готовності: після деплою відкрита вкладка протягом кількох хвилин САМА
показує «Доступна нова версія — Оновити»; натискання оновлює до нового коду; жодного
ручного чищення кешу / hard-reload / анонімного вікна не потрібно; `npm test` зелений.
