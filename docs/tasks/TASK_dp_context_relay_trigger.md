## 1. ПРОБЛЕМА (точні рядки)

Генерація `case_context.md` після обробки документів запускається через подію
`DOCUMENT_BATCH_PROCESSED`. Зараз тригер **синхронний і ловить stale-стан**:

- **Публікація** події відбувається синхронно одразу після `await onExecuteAction('add_documents')`:
  - фото-шлях: `src/components/DocumentProcessorV2/index.jsx:488`
  - pipeline: `src/services/documentPipeline.js:351`
  - Обидва місця **вже несуть `documentIds`** доданих документів — це ключ до фікса.
- **Слухач** `dpContextHandlerRef.current` (`src/components/CaseDossier/index.jsx:740-800`)
  у цей момент читає `caseData` із замикання, яке **ще не перерендерилось** із новими
  документами. `add_documents` викликав `setCases` (async, батчиться React), тож на момент
  publish `caseData.documents` — **старий список**.
- `generateCaseContext` бере документи з `caseData.documents` як SSOT
  (`src/components/CaseDossier/services/contextGenerator.js:471`).

**Наслідок:**
- Непорожня справа → нарис регенерується **без щойно доданих документів** (тихий data-bug, найгірше).
- Порожня справа → `NO_FILES`, warning-toast «Нарис справи не оновлено», хоча Drive і код у нормі.

**Що НЕ є проблемою:** сама `generateCaseContext` (вона коректно читає SSOT). Виправляємо
тільки **момент** виклику.

## 2. РІШЕННЯ — ЕСТАФЕТНА МОДЕЛЬ (без таймауту)

Ментальна модель адвоката (узгоджено): **естафета**, не «таймер очікування».

> **Виконавцю — читати уважно.** Метафори нижче можуть мати локальні неточності;
> авторитетний сенс — у цьому абзаці. **Прапорець має ПОДВІЙНИЙ сенс ОДНОЧАСНО:**
> (а) факт «DP добіг свою дистанцію» (завершив persist документів), і
> (б) вказівку «системі належить запустити контекст-генератор У МОМЕНТ, коли оновляться
> метадані за списком документів». Коли цей момент настає (`expectedDocIds` приземлились у
> `caseData.documents`) — **система** запускає генератор. Генератор добігає свою дистанцію і
> **сам прибирає прапорець**. В естафетних термінах: генератор підхоплює паличку, біжить
> разом із нею і викидає на фініші (або передав би наступному — але тут ланцюг закінчується,
> наступного етапу немає).

1. **Document Processor біжить свою дистанцію.** Обробка може тривати довго (10–20 HEIC →
   5–10 хв). Жодного таймауту/«протухання» — це була помилка попереднього дизайну, **прибрати її
   повністю**.
2. **У КІНЦІ свого забігу DP передає естафетну паличку** — ставить прапорець «цей DP-запуск
   просив оновити нарис» (тільки якщо тумблер `updateCaseContext` був увімкнений). Прапорець
   несе `expectedDocIds` (з `payload.documentIds`).
3. **Система «кричить» саме під цей прапорець:** коли `caseData.documents` реально оновився і
   містить **усі** `expectedDocIds` — стартує генератор контексту. Не на будь-яку зміну
   метаданих (перейменування/видалення/ручне додавання через модалку **не** тригерять —
   інакше палили б токени й OCR на кожен чих), а саме під естафетну паличку від DP.
4. **Генератор підхоплює паличку, біжить із нею, і на фініші викидає** — після завершення
   (успіх АБО реальна помилка генерації) прапорець знімається. Жодного нескінченного ретраю.
5. **Падає DP → генератор не стартує.** Якщо `add_documents` не вдався — publish не
   відбувається, прапорець не ставиться, документи не приземляються в метадані → генератор
   мовчки спить. Жодного фальшивого «успіху», жодного окремого toast «не дочекався».

### Чому потрібен прапорець (а не «будь-яка зміна метаданих = тригер»)

`caseData.documents` змінюється з багатьох причин (ручне «+ Додати документ», перейменування,
видалення). Якщо тригерити повну (дорогу: OCR + AI) регенерацію на **кожну** зміну — система
палитиме токени безконтрольно. Прапорець **звужує** тригер рівно до «саме цей DP-запуск із
увімкненим тумблером». Це не порушення принципу «модулі не знають одне про одного»: тут
естафета свідомо спільна — система **має** знати, що дані щойно оновив саме DP. Генератор при
цьому лишається «сплячим»: сам нічого не знає, поки його не смикне прапорець (або кнопка
«Створити контекст» на вкладці «Огляд»).

## 3. РЕАЛІЗАЦІЯ

Усе в `src/components/CaseDossier/index.jsx`. **Без** змін у `contextGenerator.js`,
`documentPipeline.js`, `DocumentProcessorV2/index.jsx` (publish уже коректний — несе
`documentIds`).

### 3.1. Прапорець (естафетна паличка)

Новий React-стан у CaseDossier:

```js
// Естафетна паличка від DP: { caseId, expectedDocIds: string[], scenarioRunId? } | null.
// Ставиться у КІНЦІ DP-забігу (на DOCUMENT_BATCH_PROCESSED з updateCaseContext===true),
// знімається коли генератор контексту добіг свій фініш (успіх або помилка).
const [pendingContextRegen, setPendingContextRegen] = useState(null);
```

Однозначність (правило #11), одне речення: **`pendingContextRegen` = «DP-запуск із
увімкненим тумблером передав естафету генератору контексту; чекаємо доки `expectedDocIds`
приземляться в `caseData.documents`».**

Подвійний сенс прапорця (свідомо, не «АБО» — це одна неподільна паличка): його **наявність**
одночасно засвідчує «DP завершив цей забіг» **і** «генератор має стартувати, щойно метадані
оновляться за `expectedDocIds`». Його **зняття** (`setPendingContextRegen(null)`) означає
«генератор добіг фініш». Жодного третього стану немає — прапорець або переданий (хтось має
бігти), або викинутий (дистанцію пройдено).

### 3.2. Слухач події — ТІЛЬКИ ставить прапорець (не генерує)

Замінити поточний `dpContextHandlerRef.current` (рядки 740-800), який зараз **одразу генерує**,
на лаконічний запис прапорця:

```js
const dpContextHandlerRef = useRef(null);
dpContextHandlerRef.current = (payload) => {
  if (!payload || payload.updateCaseContext !== true) return;
  if (payload.caseId !== caseData?.id) return;
  const ids = Array.isArray(payload.documentIds) ? payload.documentIds : [];
  // Естафета: фіксуємо паличку. Нова подія перезатирає попередню (self-heal стале).
  setPendingContextRegen({
    caseId: payload.caseId,
    expectedDocIds: ids,
    scenarioRunId: payload.scenarioRunId || null,
  });
};
```

Підписка (`useEffect` на mount, рядки 802-808) — лишається як є.

### 3.3. Тригер — `useEffect`, що слухає метадані («місце, що кричить»)

Новий ефект: коли прапорець стоїть і `caseData.documents` містить усі `expectedDocIds` —
запустити генерацію. Це і є «система кричить саме під цей прапорець».

```js
useEffect(() => {
  if (!pendingContextRegen) return;
  if (pendingContextRegen.caseId !== caseData?.id) return;
  if (isCreatingContext) return;                  // генерація вже біжить — не дублюємо

  const have = new Set((caseData?.documents || []).map((d) => d?.id));
  const allLanded = pendingContextRegen.expectedDocIds.every((id) => have.has(id));
  if (!allLanded) return;                          // документи ще не приземлились — спимо далі

  // Паличка отримана, документи в метаданих → біжимо.
  runDpContextRegen(pendingContextRegen);
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, [caseData?.documents, pendingContextRegen, isCreatingContext]);
```

`runDpContextRegen` — винесене тіло старого хендлера (Drive token / folderId / ensureSubFolders
/ `generateCaseContext` / toast'и `messages.context.updated` / refresh `loadCaseContext`).
Відмінності від старого:

- На вході — гарантовано **оновлений** `caseData.documents` (ефект спрацював ПІСЛЯ ре-рендеру).
- У `finally` — **зняти прапорець**: `setPendingContextRegen(null)` (генератор «викинув
  паличку на фініші»), плюс `setIsCreatingContext(false)` / `setContextLoading(false)` як раніше.
- **Прибрати** будь-який таймаут / «не вдалося дочекатись документів» toast — його більше немає.
- Реальні помилки генерації (`NO_API_KEY` / `AUTH` / `EMPTY` / `SAVE_FAILED` / виняток) —
  toast'и лишаються як зараз (рядки 779-795): генератор уже біг, отже про збій звітуємо.
- `NO_FILES` після приземлення документів — теоретично неможливий (усі `expectedDocIds` мають
  `driveId`, бо `add_documents` їх записав); якщо все ж трапився — лишити поточний warning.

### 3.4. Чищення прапорця на старті нового DP-забігу (hardening)

Естафета: новий забіг DP перезатирає паличку. Це вже забезпечено в 3.2 (`setPendingContextRegen`
на кожній новій події перезаписує). Додатковий явний clear на старті DP **не обов'язковий**,
бо publish відбувається тільки ПІСЛЯ успішного `add_documents` — стала паличка від невдалого
забігу неможлива (publish не відбувся). **Не** додавати окрему подію «DP started» заради цього —
YAGNI.

### 3.5. Відома межа (задокументувати в коментарі, не фіксити)

Прапорець живе в state CaseDossier. DP — вкладка **всередині** CaseDossier, тож під час обробки
дисьє змонтоване; перемикання вкладок усередині дисьє паличку зберігає. Якщо адвокат під час
обробки **піде на іншу справу/модуль** — CaseDossier розмонтується, паличка втратиться, нарис не
оновиться автоматично (адвокат зможе оновити вручну кнопкою на «Огляд»). Це прийнятно для MVP;
винесення прапорця в App.jsx — окремий борг (→ `tracking_debt.md`, якщо знадобиться).

## 4. ТЕСТИ (обов'язково, перед коммітом `npm test` зелений)

### 4.1. Інтеграційні (`tests/integration/`)

Розширити/додати сценарій (поряд з `document-processor.test.js`):

1. **Непорожня справа, гонка усунена:** справа з N існуючими документами; емітимо
   `DOCUMENT_BATCH_PROCESSED` з `updateCaseContext:true` і `documentIds` нових M документів
   ДО того як `caseData.documents` оновлено → генерація **не** стартує (прапорець стоїть, чекає).
   Після оновлення `caseData.documents` (N+M) → генерація стартує **рівно один раз** і бачить
   повний набір N+M (а не N).
2. **Падіння DP:** `add_documents` повертає `{success:false}` → publish не відбувається →
   прапорець не ставиться → генерація не стартує, жодного toast.
3. **Тумблер вимкнено:** подія з `updateCaseContext:false` → прапорець не ставиться.
4. **Чужа справа:** `payload.caseId !== caseData.id` → ігнор.
5. **Без дублювання:** повторні ре-рендери `caseData.documents` після старту (`isCreatingContext`)
   не запускають другу генерацію.
6. **Ручне додавання через модалку** (не DP, без `DOCUMENT_BATCH_PROCESSED`) → нарис не тригериться.

### 4.2. Юніт (`tests/unit/`)

Якщо логіку «чи всі expectedDocIds приземлились» винести в чисту хелпер-функцію
(`allExpectedDocsLanded(documents, expectedDocIds)`) — покрити: порожній expected, частковий
збіг, повний збіг, зайві документи (надлишок не заважає).

## 5. SAAS IMPLICATIONS

- Подія `DOCUMENT_BATCH_PROCESSED` уже несе `tenantId`/`userId` у payload (DP публікує). Прапорець
  scoped до `caseId` поточного дисьє; tenant-ізоляція успадковується від `caseData` (відкрита
  справа вже пройшла `checkCaseAccess`). Нових сутностей без `tenantId` не створюємо.

  через `aiUsageSink`) — мульти-тенант шлях не змінюється.

## 6. BILLING IMPLICATIONS

- **Не множити інструментацію.** Кнопкова генерація («Огляд») уже звітує `context_regenerated`
  (`index.jsx:612`). DP-естафетна генерація — це **той самий** акт оновлення нарису; вирішити
  свідомо: або DP-шлях теж репортить `context_regenerated` (один раз, у `runDpContextRegen`),
  або навмисно НЕ репортить (автоматичне продовження обробки, не окрема дія адвоката). Поточний
  код DP-шляху `context_regenerated` НЕ репортить — **зберегти цю поведінку** (не додавати),

## 8. ЕСТАФЕТА ЯК ПЕРЕЮЗОВНИЙ ПАТЕРН (примітка на майбутнє)

Цей механізм — окремий випадок загального патерну «**естафета**» для послідовних процесів,
де етапи виконують **різні модулі**, а наступний має стартувати лише коли попередній реально
завершив і його результат **приземлився у спільний стан** (не просто «викликав setState»):

- **Паличка** = маленький прапорець у спільному стані з (1) фактом «попередній етап завершив» і
  (2) умовою-тригером «що має з'явитися у спільному стані, щоб стартував наступний».
- **Передача** — на фініші етапу A (не на старті — інакше це не естафета, а оптимістичний запуск).
- **Підхоплення** — реактивний слухач спільного стану (тут `useEffect` на `caseData.documents`),
  який чекає виконання умови-тригера БЕЗ таймауту.
- **Викидання** — етап B знімає паличку на власному фініші (успіх або помилка), не лишаючи хвостів.

Перевага над «синхронним викликом наступного одразу після await»: усуває гонку зі стейт-
пропагацією (саме її ми тут і лікуємо). Якщо в майбутньому з'являться інші послідовні крос-
модульні ланцюги (напр. DP → індексація → нотифікація), цей патерн варто винести у спільний
хелпер. **Зараз — НЕ узагальнювати** (YAGNI): реалізувати рівно один ланцюг DP → контекст.
Якщо рішення винести патерн дозріє — занотувати в `tracking_debt.md`, не розширювати цей TASK.
