## ПРОБЛЕМА (звірено по коду)

Після TASK `bulk_delete_unify` видалення документа **повністю** чистить реєстр (`cases[].documents[]`), extended-метадані, Drive і resume-стан — фантома в реєстрі немає (підтверджено тестами). АЛЕ `case_context.md` — це **AI-згенерований знімок** («нарис справи»), що лежить на Drive і **передається чат-агенту досьє** як `## КОНТЕКСТ СПРАВИ` (`CaseDossier/index.jsx:315-316`).

Чинний дизайн (`CaseDossier/index.jsx:838-839`): **ручне видалення НЕ публікує `DOCUMENT_BATCH_PROCESSED` → нарис не перегенеровується**. Наслідок: видалив документ, але не натиснув «Створити контекст» → агент усе ще читає старий нарис, де описано зміст видаленого документа → ризик галюцинацій / посилань на неіснуючий документ.

Реєстр чистий; проблема — **застарілий похідний кеш** (нарис), який агент сприймає як істину про склад справи.

## РІШЕННЯ АДВОКАТА (узгоджено — НЕ переобговорювати)

Зробити **обидві** міри одночасно:
1. **Позначка «нарис застарів» + нудж** — коли склад документів змінився з моменту генерації нарису, показати банер «Оновіть контекст» (без авто-витрат токенів; адвокат тисне «Створити контекст» сам).
2. **Авторитетний живий перелік + застереження в промпті** — навіть якщо адвокат не оновив контекст, агент отримує актуальний склад документів і інструкцію: документ, якого немає в актуальному переліку, вважати видаленим і не посилатись на нього.

## АРХІТЕКТУРА РІШЕННЯ

### A. Чистий хелпер дельти — `contextGenerator.js`

**A.1 `deriveContextDocDelta(generatedIds, currentIds)`** (експорт, чистий, юніт-тест):
- Вхід: масив id документів на момент генерації нарису + поточний масив id (з живого реєстру).
- Вихід: `{ stale, removed, added }`. `removed` = id з generated, яких немає в current (видалені після генерації — головний кейс). `added` = навпаки (нові, ще не в нарисі). `stale = removed.length > 0 || added.length > 0`.
- **Edge: невідомий знімок.** Якщо `generatedIds` порожній/`null` (нарис згенеровано до цієї фічі, або знімок не збережено) → повертати `{ stale:false, removed:[], added:[], unknown:true }`. **НЕ показувати фальшиву застарілість** коли порівнювати нема з чим.

**A.2 `buildAuthoritativeDocsBlock(documents)`** (експорт, чистий, юніт-тест):
- Вхід: масив активних документів справи (`status !== 'archived'`).
- Вихід: рядок для системного промпта — заголовок + перелік `- {name} ({date})` (date опційно) + застереження (текст нижче, §C). Якщо документів нема → «Наразі у справі немає документів.» + те саме застереження.
- Чистий → тестується без DOM.

### B. Збереження знімка + детекція застарілості — `CaseDossier/index.jsx`

**B.1 Знімок id на момент генерації (localStorage, без schema bump):**
- Ключ `context_doc_ids_<caseId>` = JSON-масив `caseData.documents.map(d => d.id)` (УСІ документи реєстру — джерело нарису у `contextGenerator` це `caseData.documents`, §474).
- **Точки запису — на УСПІШНІЙ генерації, обидві:** `handleCreateContext` (кнопка «Створити контекст», `~404+`) і `runDpContextRegen` (DP-естафета, `~872+`). Спільний хелпер `persistContextDocIds(caseId, ids)` (один write, try/catch).
- Чому localStorage а не поле справи: застарілість — похідний UI-сигнал, не дані справи; поле в схему = bump + міграція без потреби (правило #6/#11). Аналог наявних `agent_history_<caseId>`, `${MODE_KEY_PREFIX}${documentId}`.

**B.2 Детекція (derive щорендер, коли нарис завантажено):**
- Читати знімок з localStorage; `contextDelta = deriveContextDocDelta(stored, caseData.documents.map(d=>d.id))`.
- Тримати в `useMemo` по `[caseContext, caseData.documents]` (не на кожен рендер заново парсити localStorage без потреби).

**B.3 Нудж-банер:**
- Показати коли `caseContext && contextDelta.stale`. Текст: «Нарис справи застарів: видалено {removed.length}, додано {added.length} документів з моменту генерації. Оновіть контекст, щоб агент не плутався.» + кнопка «Оновити контекст» → наявний `handleCreateContext`.
- Місце: біля індикатора контексту / у шапці agent-панелі (поряд із `~1468` де зелений `FileText` «Контекст справи створено»). Лише дизайн-токени, без емодзі (правило модуля). Зникає сам після успішної регенерації (знімок перезаписується → delta порожня).

### C. Авторитетний перелік + застереження в системному промпті — `buildAgentSystemPrompt` (`~300-321`)

Після блоку `## КОНТЕКСТ СПРАВИ` (рядок 315-319) **завжди** (незалежно від застарілості) додати блок з `buildAuthoritativeDocsBlock(активні документи)`:

```
## АКТУАЛЬНИЙ СКЛАД ДОКУМЕНТІВ (авторитетне джерело, станом на зараз)
- {name1} ({date1})
- {name2} ({date2})
...

УВАГА: блок «КОНТЕКСТ СПРАВИ» вище — знімок на момент генерації нарису. Якщо
в нарисі згадано документ, якого НЕМАЄ в актуальному складі вище — його
ВИДАЛИЛИ з досьє; не цитуй його, не посилайся на нього, вважай що його не
існує. Авторитетне джерело переліку документів справи — ЦЕЙ блок, а не нарис.
```

- Перелік — лише `status !== 'archived'` (агент працює з активними; архівні поза Viewer'ом).
- Це доповнює наявний рядок `Документів: ${(caseData.documents||[]).length}` (§313) конкретними іменами → агент може крос-звірити згадки нарису з істиною.

## ТЕСТИ
- Unit `deriveContextDocDelta`: removed-only → stale; added-only → stale; ідентичні → not stale; порожній/null generatedIds → `{stale:false, unknown:true}`.
- Unit `buildAuthoritativeDocsBlock`: перелік активних з іменами+датами; порожній список → «немає документів» + застереження присутнє в обох гілках.
- (Опційно) integration: після `delete_documents(full)` без регенерації — derive дає stale; після persist нового знімка — not stale.
