Перший прогін Court Sync MVP на реальному кабінеті Левицького через Claude for Chrome (Sonnet 4.6):

**Що працювало ідеально:**

- Дедуплікація повісток (кілька повісток на одну дату → одне засідання)
- Логіка засідань (Внесення дат слухання + повістки)

**Що виявив тест (проблеми для виправлення):**

1. **Уперлось у Usage limit** після 257 кроків, обробивши 14 з 16 справ. Не встиг сформувати фінальний JSON. Уся робота фактично втрачена (хоча дані були зібрані в логу).

2. **~18 кроків на справу** — забагато. Причини: зайві screenshot для самоперевірки, повторні кліки через промахи по інтерфейсу, навігація туди-назад.

3. **Справи "паразити"** — ЄСІТС підтягнув справи де адвокат фактично не бере участі (через довіреність на клієнта-сторону). Адвоката немає в учасниках. Claude витратив кроки заходячи в них, аж потім зрозумів що адвоката немає.

**Знахідка адвоката (детермінований фільтр):**

У колонці "Мій процесуальний статус" у списку справ:
- **Релевантні справи** мають КОНКРЕТНУ роль: "Представник позивача/відповідача/заявника/потерпілого/скаржника", "Адвокат", "Захисник", або поєднання
- **Нерелевантні справи** мають РІВНО ОДНЕ СЛОВО "Представник" без уточнення кого — це справи підтягнуті через довіреність на клієнта-сторону, адвокат там фактично не працює, у "Інформація про справу" його немає

Цей фільтр дозволяє відсіяти нерелевантні справи **прямо у списку, не заходячи в них** — максимальна економія кроків.

### Блок 1 — Детермінований фільтр ролей (КРИТИЧНО для економії)

Додати у промпт ПЕРЕД заходом у справу — фільтрація прямо в списку справ за двома критеріями одночасно:

```
═══════════════════════════════════════════════════════════════
ФІЛЬТРАЦІЯ СПРАВ У СПИСКУ (до заходу в справу — економить кроки)
═══════════════════════════════════════════════════════════════

У списку "Мої справи" кожен рядок має дві ключові колонки:
- Номер справи (формат NNN/NNNNN/NN-X)
- Мій процесуальний статус (роль адвоката)

Для КОЖНОЇ справи у списку застосуй ОБИДВА фільтри ПЕРШ НІЖ заходити:

ФІЛЬТР 1 — рік у номері справи:
Цифри після ДРУГОГО слешу = рік.
- 450/2275/25 → 25 → проходить
- 367/4744/26 → 26 → проходить
- 761/15469/20-ц → 20 → ПРОПУСК
- 528/1522/24 → 24 → ПРОПУСК
Бери тільки 25 або 26.

ФІЛЬТР 2 — роль у колонці "Мій процесуальний статус":

БЕРИ справу якщо роль КОНКРЕТНА:
- "Адвокат" (самостійна роль — завжди твоя справа)
- "Захисник" (з уточненням чи без — завжди твоя справа, кримінальні)
- "Представник позивача"
- "Представник відповідача"
- "Представник заявника"
- "Представник потерпілого"
- "Представник скаржника"
- "Представник третьої особи"
- Будь-яке ПОЄДНАННЯ ролей ("Адвокат, Представник позивача",
  "Захисник, Скаржник", "Представник позивача, Представник...")

ПРОПУСКАЙ справу якщо роль — РІВНО ОДНЕ СЛОВО "Представник"
без уточнення кого саме.
Це справи підтягнуті через довіреність на клієнта-сторону —
адвокат там фактично не бере участі, у "Інформація про справу"
його немає. НЕ заходь у них (економія кроків).
Додай у skipped: { case_no, reason: "Роль 'Представник' без
уточнення — справа підтягнута через довіреність, адвокат не
учасник" }

ВАЖЛИВО: різниця тільки між голим "Представник" (одне слово —
пропуск) і всім іншим (конкретна роль — береш). "Адвокат" і
"Захисник" — це самостійні конкретні ролі, беруться завжди.

Заходь У СПРАВУ тільки якщо вона пройшла ОБИДВА фільтри.
```

### Блок 2 — Визначення primaryParty за роллю

Закріпити явно (у тесті Claude здогадувався сам, але закріплюємо для стабільності):

```
═══════════════════════════════════════════════════════════════
ВИЗНАЧЕННЯ ОСНОВНОЇ СТОРОНИ (primaryParty для назви справи)
═══════════════════════════════════════════════════════════════

Роль адвоката визначає яку сторону взяти як primaryParty:
- "Представник позивача"     → позивач
- "Представник відповідача"  → відповідач
- "Представник заявника"     → заявник
- "Представник потерпілого"  → потерпілий (кримінальні)
- "Представник скаржника"    → скаржник
- "Представник третьої особи" → третя особа
- "Адвокат"                  → визнач за "Інформація про справу"
                               кого представляє Левицький
- "Захисник"                 → обвинувачений/підозрюваний якого
                               захищає
- Поєднання                  → бери першу конкретну роль

Формат primaryParty: "Прізвище І.П." (Бабенко О.І.)
Для юросіб (ТОВ/ПП/ФОП/АТ) → повна назва без скорочення.
```

### Блок 3 — Кримінальні з кількома обвинуваченими (перепитати, не вгадувати)

```
═══════════════════════════════════════════════════════════════
КРИМІНАЛЬНІ СПРАВИ З КІЛЬКОМА ОБВИНУВАЧЕНИМИ
═══════════════════════════════════════════════════════════════

Якщо у кримінальній справі (категорія criminal, роль "Захисник")
у "Інформація про справу" КІЛЬКА обвинувачених і НЕ ЗРОЗУМІЛО
кого саме захищає Левицький:

- НЕ вгадуй прізвище для назви
- Засідання все одно витягни (вони цінні незалежно від назви)
- primaryParty залиш = case_no (номер справи замість прізвища)
- Додай у warnings: "Справа [case_no]: кілька обвинувачених
  ([прізвища через кому]). Уточніть кого захищає Левицький для
  коректної назви справи."

Адвокат потім виправить назву вручну. Не блокуй імпорт через
невизначеність назви.
```

### Блок 4 — Оптимізація кроків (м'яка, без шкоди якості)

```
═══════════════════════════════════════════════════════════════
ЕКОНОМІЯ КРОКІВ (важливо — є ліміт використання)
═══════════════════════════════════════════════════════════════

Є ліміт на кількість дій у браузері. Працюй ощадливо, АЛЕ
НІКОЛИ не жертвуй точністю заради економії. Якщо сумніваєшся —
краще зайвий крок ніж пропущене засідання.

ПРАВИЛА ОЩАДЛИВОСТІ:

1. Для ЧИТАННЯ ДАНИХ (дати, номери, прізвища, суди, час) —
   використовуй Extract page text як ПЕРШИЙ вибір. Це швидше
   за screenshot.

2. Take screenshot роби коли:
   - Треба зрозуміти СТРУКТУРУ сторінки (де клікати)
   - Перевірити чи відкрився документ/модалка
   - Текст незрозумілий або extract повернув мало
   - Документ показується як скан-зображення
   ЯКЩО extract text дав усе що треба — screenshot НЕ роби.

3. Заходь у "Інформація про справу" ОДИН раз на справу.
   Запам'ятай сторони. Не відкривай повторно.

4. НЕ комбінуй Wait + Screenshot коли достатньо Extract text.
   Wait потрібен тільки якщо сторінка явно ще вантажиться.

5. Для засідань: спочатку шукай "Внесення дат слухання" — там
   дата структурована, читається одним extract. Відкривай
   окремі повістки ТІЛЬКИ якщо "Внесення дат слухання" немає
   або в ньому немає дати.

6. Якщо клік відкрив не те що очікував — це нормальна поведінка
   інтерфейсу. Виправ і продовжуй спокійно, без зайвих
   скріншотів-перевірок.

ПРІОРИТЕТ: точність > економія. Економія — це прибрати ЗАЙВЕ
(дублюючі скріншоти, повторні відкриття), НЕ скоротити
необхідну перевірку даних.
```

### Блок 5 — Checkpoint-парашут

```
═══════════════════════════════════════════════════════════════
ПРОМІЖНІ РЕЗУЛЬТАТИ (страховка від ліміту)
═══════════════════════════════════════════════════════════════

Кабінет може містити багато справ. Щоб не втратити роботу якщо
впрешся в ліміт використання:

ПІСЛЯ КОЖНИХ 5 ОБРОБЛЕНИХ СПРАВ — видавай проміжний JSON у чат
у блоці коду з міткою:

[ПРОМІЖНИЙ РЕЗУЛЬТАТ — N справ]
```json
{ ... envelope з усіма обробленими досі справами ... }
```

Це парашут. Якщо далі впрешся в ліміт — у адвоката буде останній
проміжний JSON з тим що встиг.

КОЛИ ДІЙШОВ ДО КІНЦЯ (усі релевантні справи оброблені):
Видай ОДИН ФІНАЛЬНИЙ JSON з усіма справами:

[ФІНАЛЬНИЙ РЕЗУЛЬТАТ]
```json
{ ... повний envelope ... }
```

У щасливому сценарії адвокат вставляє тільки фінальний.
Проміжні — лише на випадок збою.

Кожен JSON (проміжний і фінальний) — повний валідний envelope
за форматом нижче, не дельта. Дедуплікація в Legal BMS через
ecitsCaseId забезпечує що вставка проміжного + фінального не
створить дублів.
```
