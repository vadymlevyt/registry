## ДІАГНОЗ #11 🔴 (головний) — РЕГРЕС: обрізка/поворот з попапа не зберігаються

### Симптоми (адвокат, обидва модулі — модалка І DP)
Тап на фото → попап → обрізав/виправив рамку/повернув → «✓ Готово» → у сітці фото
**НЕ обрізане**. Повторний тап → у попапі **сире і перевернуте**. **Нестабільно:**
після кількох тапів іноді стабілізується (показує обрізане і в попапі, і в сітці).
Одного разу — **чорний екран** у попапі.

### Гіпотеза радника (перевір!)
**Race condition у керуванні життям object URL** ефекту прев'ю:
- Файл: `src/components/CaseDossier/ImageMergePanel/index.jsx` ефект прев'ю (~183-248)
  і його дзеркало `src/components/DocumentProcessorV2/DpImageMergeEditor.jsx` (~209-276).
- Крихка схема: `previewUrlsToRevokeRef` — **спільний ref** для всіх запусків ефекту +
  `setTimeout(…, 1000)` на відкладену ревокацію.
- При швидких тапах: ефект перезапускається (crop змінив `cropAppliedSet`/`cropOverrides`),
  старий запуск отримує `cancelled=true`, АЛЕ його (або наступного) `setTimeout` ревокує
  URL зі спільної черги наввипередки → інколи ревокується **ще-живий** новий URL →
  **чорний екран** / показ сирого. «Нестабільно» = гонка виграється по-різному за таймінгом.

### Що радник ВИКЛЮЧИВ (перевір що згоден)
- **НЕ** `createPortal` (#4, коміт `0597130`) — лише перенос DOM-вузла, callbacks/стан не чіпає.
- **НЕ** deps-фікс `eb6f797` (прибрання `cropProposals`) — `cropOverrides`/`cropAppliedSet`
  лишились у deps.
- **Wiring цілий**: попап `onCropOverride(idx, rect, {applied})` → `handleCropOverride`
  ставить `cropOverrides`+`cropAppliedSet` → ефект має згенерувати blob. Ланки на місці.

### Якщо підтвердилось — напрямок фіксу (узгодь деталі)
Прибрати крихку «спільний ref + setTimeout 1с». Варіанти (обери найчистіший, обґрунтуй):
- ревокація старих URL **детерміновано в cleanup ефекту** (`return () => …`), не через
  глобальний відкладений таймер;
- АБО per-idx URL з ревокацією рівно при заміні конкретного idx;
- кожен запуск ефекту володіє **власним** набором URL (не пушить у спільний ref).
**Спільне джерело** → фікс лагодить і модалку, і DP одразу (правило #30). Якщо логіка
вже частково дублюється між index.jsx і DpImageMergeEditor — звести у спільний хелпер
(`services/imageDocument/` або хук), щоб не лагодити двічі.

## ДІАГНОЗ #12 🔴 — «Видалити всі дублікати» ігнорує ручний вибір

### Симптом
Розділив групу («це не дублікат») АБО вручну лишив «жовтий» (не-рекомендований) і
видалив зелений → тиснеш «Видалити всі дублікати» → воно **виносить твій жовтий** і
застосовує AI-рекомендацію до розділених груп теж.

### Гіпотеза радника (перевір!)
`handleKeepAllRecommendedDuplicates` (`ImageMergePanel/index.jsx:607-624`) бере **всі**
групи з `sortResult.duplicates`, лишає лише `g.recommended`, **ігноруючи**:
1. `dismissedDuplicateGroupIds` (розділені групи);
2. ручний вибір адвоката (що вже видалено/лишено вручну — `removedIndices`/`orderedIndices`).

### Очікувана поведінка (РІШЕННЯ АДВОКАТА — тверде)
**ПОВАЖАТИ РУЧНИЙ ВИБІР.** «Видалити всі» застосовує AI-рекомендацію **ТІЛЬКИ до
незайманих груп**:
- група у `dismissedDuplicateGroupIds` → **пропустити повністю** (не чіпати);
- група, де адвокат уже вручну видалив/лишив члена → **пропустити** (його вибір головніший);
- лише «чисті» (незаймані) групи → лишити `recommended`, решту видалити.

Перевір чи цей баг є і в DP (`DpImageMergeEditor` — handler-аналог). Фікс — у спільному
місці / однаково в обох.

## ДІАГНОЗ #10 🟡 — DP не малює групову РАМКУ дублів (паритет)

### Симптом
Модалка: дублі однієї групи в **спільній жовтій рамці** (по 2-3 поруч), рекомендований
зеленим всередині. DP (`DpImageMergeEditor`): лише **бейджі** «Дублікат!»/«Рекомендація
залишити» на кожній картці, **без групової рамки** — не видно які картки = одна група.

### Гіпотеза радника (перевір!)
DP передає `duplicateMembership` у grid, але не рендерить **обгортку-рамку групи** — те,
що модалка робить (звірити з `PreviewView`/grid модалки: як саме рамка групи малюється,
які класи/контейнер). Перенести той самий механізм у DP (зі спільного, не дублювати).
