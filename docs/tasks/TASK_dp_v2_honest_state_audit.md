## 0. Контекст і мета

Адвокат — власник цього проекту (Legal BMS, AB Левицького) — за останні
~2 місяці тестував на реальних справах **ТІЛЬКИ функцію нарізки**:
- Брановський (цивільна, 65pp) → ~93% точність
- Нестеренко (крим. том, 273pp) → ~35% точність (борг #19 → щойно
  закритий через TASK_dp_toc_and_enriched_digest)

**Решта функцій DP v2 — невідомо чи реально працюють** на реальних даних
або там тільки скелети/заглушки/passthrough. Архітектурно вони існують
у коді, але минулі досвід (DP-4 рік тому + Phase B нещодавно) двічі
показав клас бага «1400+ юніт-тестів зелені, реальний Provider —
катастрофа». Тому **архітектурна присутність ≠ робоча функція**.

**Мета цього TASK:** дати адвокату чесну карту «що реально працює, що
здається працює, що зламано, що не реалізовано» перед плануванням
наступних 2-3 місяців роботи. Без цієї карти будь-який наступний TASK
ризикує піти на нібито-працюючу базу.

**Чому окрема сесія, не та що писала TASK:** контекст автора цього TASK
багатий знанням коду але засмічений шумом серії (Ф0-Ф3, G0-G6, B1-B3,
P1-P4, ToC). Аудит потребує **чистого систематичного проходу**. TASK
дає тобі точну карту коду — твоя робота систематично пройти по ній і
чесно класифікувати.

## 2. ЩО НЕ РОБИТИ (жорсткі правила)

- **НЕ міняти жодного рядка коду.** Це read-only аудит.
- **НЕ запускати pipeline на реальних даних.** Адвокат сам тестує за
  твоєю картою. Локально можна `npm test` для відомостей які є тести,
  але НЕ ганяти DP на реальному файлі.
- **НЕ виправляти баги.** Тільки фіксувати їх з конкретним описом.
- **НЕ оцінювати оптимістично.** Якщо є тільки юніт-тест без
  Provider-integration — це **жовтий**, не зелений. Це принциповий
  стандарт чесності для цього аудиту.
- **НЕ плодити нові `.md`.** Лише один підсумковий звіт у вказаному
  файлі. Жодних окремих audit/bugs/notes файлів.
- **НЕ розширювати scope.** Якщо знаходиш цікаве але не з твого списку
  — коротка згадка у звіті, не окремий розділ.

## 3. КОЛЬОРОВА СХЕМА ЧЕСНОСТІ (обов'язкова на кожну функцію)

| Колір | Що означає | Критерій |
|---|---|---|
| **🟢 Зелений** | Реалізовано + ДОВЕДЕНО що працює на реальному Provider | Реалізація у коді + Provider-integration тест **ЯКІСНОЇ ПОВЕДІНКИ** (реальний прохід через диригент, не стадія в ізоляції) + адвокат хоч раз бачив це працюючим на реальних даних |
| **🟡 Жовтий** | Реалізовано, але покрито ТІЛЬКИ юніт-тестами | Реалізація є + юніт-тести є, але **немає Provider-integration тесту якісної поведінки**. Клас бага DP-4 не виключений. На реальному Provider може бути зламане. |
| **🟠 Помаранчевий** | Реалізовано, **конкретний ризик** що зламано | Реалізація є, але ти знайшов конкретну причину чому воно ймовірно не працює (відсутній виклик ін'єктора, dead code, неправильне підключення, тощо). Вкажи що саме. |
| **🔴 Червоний** | Не реалізовано / заглушка / passthrough | Функція описана архітектурно (в спеці/CLAUDE.md), але у коді — порожня заглушка, `return ctx;`, TODO коментар, або взагалі немає. |

**КРИТИЧНО:** не писати 🟢 якщо немає Provider-integration тесту якісної
поведінки. Краще 🟡 з нотою «треба перевірити на планшеті» ніж хибне 🟢.

## 6. ОПЕРАЦІЙНІ ПРАВИЛА

- **Git:** працюй на гілці яку видасть harness (`claude/*`). Не пушити
  в main. Doc-only push → адвокат сам merge.
- **npm test** — можна запустити для перевірки які тести проходять,
  але не для діагностики кожної функції (це не runtime audit).
- **Не редагуй** `CLAUDE.md`, `DEVELOPMENT_PHILOSOPHY.md`, `LESSONS.md`,
  `tracking_debt.md`, `ARCHITECTURE_HISTORY.md`. Тільки читай.
- **Один файл на виході:** `docs/diagnostics/diagnostic_dp_v2_state.md`.
  Не плодити audit_*.md / bugs_*.md / discovered_*.md.
- **Якщо знайдеш баг** — фіксуй у звіті як ризик. **НЕ виправляй.**
  Виправлення — окремі майбутні TASK.
- **Чесність важніша за повноту.** Краще написати «не перевірив, бо…»
  ніж вгадати статус.
