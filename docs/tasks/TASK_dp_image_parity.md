## МЕТА

DP-редактор склейки фото (`DpImageMergeEditor`, TASK 1B) **не підтягнув** з модалки два
готові контролі, і має UI-баг накладання:
- **#1** не показує **дублі** (модалка бере в жовту рамку, зеленим — рекомендований, дає вибрати);
- **#9** не дає **контролю обрізки** (модалка: банер «Не обрізати жодну» + per-thumb ✕; DP сам
  обрізає без глобального керування);
- **#4** при відкритті попапа (збільшеного фото) **меню досьє налазить поверх** попапа.

Усі три — **зона image-merge editor**. Контролі #1/#9 **уже існують у модалці** (`PreviewView`
+ спільні `ImageEditor/*`) — їх треба **перенести у DP зі спільного місця**, не дублювати
(правило «Спільний рендер UI»). #4 — z-index/portal у спільному `PreviewPopup`.

## СКЛАДОВІ

### #1 — Дублі + сортування в DP через СПІЛЬНИЙ AI-сервіс (виправлення хибної передумови спеки)

**⚠️ ВАЖЛИВО — спека спершу хибно припускала** що дублі вже є з `sortResult` у
DP-флоу. Це НЕ так (виявлено аудитом виконавця 2026-05-30): DP image-flow робить
`prepareImagesForMerge` (HEIC+OCR+orientation, **без** сортування) → `groupImagesIntoDocuments`
(тільки межі документів). **`sortImages` у DP не викликається взагалі** — він кличеться
лише в модалці через `convertImagesToPdf`. Тому даних про дублі в DP немає, і «без
зайвих AI-викликів» — неправильне припущення.

**Рішення адвоката: А — `sortImages` per-group + зробити цей крок СПІЛЬНИМ сервісом.**

`sortImages` (`services/sortation/imageSortingAgent.js`) визначає **і порядок сторінок
усередині документа, І дублі** (`{order, duplicates, suggestedName, warnings}`, Sonnet,
вже логує AI usage). Це саме та відповідальність що потрібна. Зараз DP його **не кличе**
→ DP не лише не показує дублі, а й **не сортує** сторінки всередині документа. А — лагодить
обидва.

**Спільний сервіс (вимога адвоката):** винести крок «сортування+дублі для групи фото» у
**спільну функцію**, яку кличуть **обидва** споживачі (модалка + DP) і будь-який майбутній
третій. Не дублювати виклик по-різному.
- Файл: `src/services/imageDocument/sortImageDocument.js` (НОВЕ) — тонка спільна обгортка
  над `sortImages`: приймає масив фото однієї логічної одиниці (документа/батча) + опції,
  повертає `{order, duplicates, suggestedName, warnings}`. Логування AI usage всередині
  (один шлях, C7). Таймаут/fallback (як зараз у `multiImageToPdf:237-253`) — у цю обгортку.
- **Модалка:** `multiImageToPdf.convertImagesToPdf` перемикається на спільну обгортку
  замість прямого `sortImages` (behavior-preserving — той самий результат). Модалка
  лишається ІДЕНТИЧНОЮ для адвоката.
- **DP:** у image-flow після `groupImagesIntoDocuments` — для **кожної групи з >1 фото**
  викликати спільну обгортку → отримати `order` (сортування всередині документа) +
  `duplicates`. Групи з 1 фото — не кликати (нема що сортувати/дедупити).

**Прокидання в editor:** обчислити `duplicateMembership` (як `PreviewView:119`) з
`duplicates` груп і передати у `DpImageMergeEditor` (зараз `:567` віддає порожній `new Map()`,
`onKeepRecommendedDuplicate/onDismissDuplicateGroup = null` `:852-853` — компоненти вже
приймають, лише не постачаються дані/handlers). Grid (`Thumbnail`/`RenderItem`) уже малює
жовту рамку/зелений рекомендований — DP лише постачає дані + handlers. Блок «Знайдено N
груп дублікатів» + вибір/зняти — той самий що модалка (зі спільного).

**Вартість (свідомо):** А додає `sortImages`-виклики у DP per-group (>1 фото). Для ухвали
7 фото / 2-3 групи = 2-3 дешеві Sonnet-виклики. Логуються через обгортку (C7). Це
оновлює секцію AI USAGE (нижче) — додаткові виклики прийняті свідомо.

### #9 — Контроль обрізки в DP (паритет)
**Файли:** `DpImageMergeEditor.jsx`.
- Додати **crop-банер** як у `PreviewView:310-323`: «Обрізку буде застосовано до N стор…
  **Не обрізати жодну**» (`onSkipAllCrops`) — коли є cropProposals. Реалізувати `onSkipAllCrops`
  (зняти всі) і, якщо модалка має, `onApplyCropToAll`.
- Per-thumb ✕ вже є (`onToggleCropDisabled`) — переконатись що видимий і працює як у модалці.
- **Звірити дефолт із модалкою:** обрізка за AI-пропозицією застосовується за дефолтом, але
  адвокат може зняти (per-thumb або «Не обрізати жодну»). Поведінка фінальної збірки
  (`rebuildFromOcrResults`) має точно збігатися з модалкою — однакова обрізка → однаковий PDF.

### #4 — Overlay: попап поверх меню досьє
**Файли:** `ImageEditor/PreviewPopup.jsx` (+ можливо CSS).
- Зробити так щоб `PreviewPopup` рендерився **поверх** chrome досьє (агент-панель/меню) у

- **Не зламати модалку** — у модалковому контексті попап має лишитись як є. Спільний фікс

- Пов'язано з боргом #26 (CSS-префікс `image-merge-panel__` у спільному) — якщо торкаєшся
