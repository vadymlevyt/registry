## КОНТЕКСТ — корінь проблеми (три прогони)

Три реальні прогони Court Sync на кабінеті Левицького виявили **галюцинації даних**:
- Бабенко "Олександра Іванівна" (правда) → "Олег Іванович" (галюцинація)
- Янченко "Ілля Ігорович" → "Ірина Іванівна"

**Діагностика (підтверджена):**
- Галюцинації виникають **НЕ через обсяг** (третій прогін мав лише 11 активних справ — теж галюцинував)
- Причина — **переривання сесії** (потухлий екран, ліміт токенів) + **відновлення зі стиснутого summary**
- При фіналізації Claude **реконструює дані з ПАМ'ЯТІ**, яка стиснута compacting'ом → вигадує імена/час
- ecitsCaseId і case_no вціліли (вони фіксувались рано), але **імена/деталі попливли** при фінальній збірці з пам'яті

**Перший прогін був чистий** бо проходив без відновлення зі summary (дочитував з кабінету наживо).

## РІШЕННЯ — зовнішня пам'ять через повідомлення в чаті

**Принцип:** Claude НЕ тримає дані в робочій пам'яті до фіналу (вона стискається). Натомість **виписує кожні 5 справ повним JSON-блоком у чат**. Надіслані повідомлення в чаті **не стискаються** compacting'ом — вони лишаються дослівно. У кінці Claude збирає фінал **перечитуючи власні проміжні повідомлення**, а НЕ з пам'яті.

Чат стає **зовнішнім надійним журналом**. Compacting тисне робочі роздуми, але не вже-надіслані блоки тексту.

**Це ручна імітація того що Track B (розширення) робитиме кодом** — запис кожної справи одразу, без тримання в пам'яті.

### Зміна 1 — Жорстке зобов'язання виписувати проміжні (замінити блок ПРОМІЖНІ РЕЗУЛЬТАТИ)

Поточний блок каже "видавай кожні 5" — але Sonnet це ігнорує (відкладає до кінця). Посилюємо:

```
═══════════════════════════════════════════════════════════════
ЗОВНІШНЯ ПАМ'ЯТЬ — ОБОВ'ЯЗКОВІ ПРОМІЖНІ ЗАПИСИ
═══════════════════════════════════════════════════════════════

КРИТИЧНО ВАЖЛИВО. Твоя робоча пам'ять стискається під час довгого
обходу — деталі (імена, час засідань) можуть спотворитись. Щоб
цього уникнути, ти ведеш ЗОВНІШНІЙ ЖУРНАЛ у чаті.

ПРАВИЛО (обов'язкове, не опція):

Після обробки КОЖНИХ 5 справ — НЕГАЙНО випиши у чат повний блок:

[ЖУРНАЛ — справи N..N+4]
```json
[
  { ... повні дані справи 1: ecitsCaseId, case_no, court,
    category, advocateRole, primaryParty, primaryPartyFullName,
    cabinetUrl, hearings[...] ... },
  ... ще 4 справи ...
]
```

ЦЕ НЕ МОЖНА ПРОПУСКАТИ. НЕ продовжуй обхід наступних справ поки
не виписав журнал за попередні 5. Виписуй ДОСЛІВНО те що прочитав
зі сторінки — імена, по-батькові, час засідань точно як у кабінеті.

Якщо справ менше 5 в кінці — випиши журнал за ті що лишились.

Навіщо: якщо перерветься сесія (ліміт, збій) — журнал у чаті
збережеться. І головне — у КІНЦІ ти збереш фінал З ЦИХ ЗАПИСІВ,
не з пам'яті (див. блок ФІНАЛІЗАЦІЯ).
```

### Зміна 2 — Фіналізація з повідомлень (новий блок, ключовий)

Додати перед СТРУКТУРА ENVELOPE:

```
═══════════════════════════════════════════════════════════════
ФІНАЛІЗАЦІЯ — ЗБИРАЙ З ЖУРНАЛУ, НЕ З ПАМ'ЯТІ
═══════════════════════════════════════════════════════════════

КРИТИЧНО. Коли формуєш фінальний JSON — НЕ покладайся на свою
пам'ять про справи. Твоя пам'ять могла стиснутись під час обходу
і спотворити деталі (імена, час).

НАТОМІСТЬ:
1. ПЕРЕЧИТАЙ свої попередні повідомлення [ЖУРНАЛ — ...] у цьому чаті
2. Збери фінальний JSON САМЕ З НИХ — копіюй дані з журналу дослівно
3. Журнальні записи — твоє надійне джерело правди, бо вони
   зафіксовані як текст і не стискались

Якщо бачиш розбіжність між тим що "пам'ятаєш" і тим що у журналі —
вір ЖУРНАЛУ. Він точніший.

Об'єднай усі журнальні блоки + застосуй рішення з ТОЧКИ РІШЕННЯ
(якщо були) + сформуй фінальний envelope.

НЕ вигадуй і НЕ відновлюй з пам'яті жодне ім'я, по-батькові чи
час — усе має бути з журналу. Якщо чогось немає в журналі — постав
null, не вигадуй.
```

### Зміна 3 — Нагадування у блоці ПОЧИНАЙ

пам'яті. Журнал — джерело правди.
```
