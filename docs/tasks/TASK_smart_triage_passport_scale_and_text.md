## 0. Навіщо це доповнення (корінь)

**Реальна ціль адвоката:** стандартний топ-том судової справи —
**200-250 сторінок** — має оброблятися швидко і якісно. Це не край, це норма.

**Прогалина батьківського TASK.** Ф0 описує паспорт як «текст сторінки +
дайджест геометрії» (TASK_smart_triage §3, §6; consultation §4). Поточна
реалізація `buildStructuralPassport` (`src/services/documentPipeline/
pageMarkers.js`) буквально вкладає **повний `_text` КОЖНОЇ сторінки**:

```
.map((page, i) => `=== СТОРІНКА ${i + 1} ===${line ? `\n${line}` : ''}\n${page._text || ''}`)
```

Старий cap `slice(0,50000)` навмисно прибрано (фікс DP-4 проти галюцинацій
номерів сторінок). Отже промпт Triage = **весь OCR-текст усієї справи**.

**Вартісна модель §6 цього НЕ закриває.** Вона прибрала *image*-токени
(Vision). Вона нічого не каже про **обсяг тексту**. Це окремий, другий
бюджет: **контекстне вікно Haiku**.

**Статична оцінка (без прогону, кирилиця ~2 символи/токен):**

| Підхід | Токенів/стор. | 250 стор. | Haiku 200K вікно |
|---|---|---|---|
| Поточний (повний текст у паспорті) | ~1000-2000 | **~250-500K** | **ПЕРЕПОВНЕННЯ** |
| Компактний паспорт (це доповнення) | ~200-280 | **~50-70K** | у зоні якості (≤~120K) |

Якість Haiku на межах деградує **задовго до** жорсткого ліміту — це той
самий механізм, що дав помилки меж уже на 65 стор. Брановського.

**Тихий режим відмови (критично).** При переповненні API відхиляє запит →
`triageStage.js:201-207` ловить у catch → плану немає → passthrough →
**уся справа лягає одним нерізаним документом**, лише рядок `warnings`. Не
падіння, не помилка на екрані. На великому томі адвокат отримує це мовчки.

## 1. Місія доповнення (що означає «готово»)

1. **Triage отримує компактний паспорт меж**, не повний текст. Том
   200-250 стор. влазить в одному проході Haiku, у зоні якості, швидко.
2. **Повний OCR-текст і його очистка в `.md` НЕ зникають** — вони
   виробляються і персистяться як раніше (для в'ювера), просто **розв'язані**
   з тим, що споживає Triage. Адвокату текст потрібен у Матеріалах справи
   для копіювання/редагування — це лишається.
3. Жодного нового «режиму»/екрана (обмеження батьківського TASK №3, §13).

## 2. Архітектурний принцип розв'язки (правило #11)

Один сенс на ім'я. Сьогодні `resolveBoundaryText` (єдина точка вибору
тексту для Triage, pageMarkers.js) повертає **повний** структурний паспорт.
Ми НЕ нашаровуємо на `buildStructuralPassport` прапор `compact` (це другий
сенс на одне ім'я — точно баг-патерн skipCache з філософії).

**Розв'язка двох незалежних потоків:**

```
                 OCR (streaming chunk, ВЖЕ Є) → pageStructure/_text/.layout.json
                          │                                  │
        ┌─────────────────┘                                  └──────────────┐
        ▼ ВХІД TRIAGE (новий, компактний)            ВХІД ОЧИСТКИ (як є, повний)
  buildCompactTriagePassport()                    extractV3 cleanText (Haiku) →
   = дайджест + краї сторінки                      processedText (.md) →
   = ~200-280 ток/стор.                            splitDocumentsV3 пише
   = тільки сигнали меж                            02_ОБРОБЛЕНІ/<doc>.md (per-doc)
        │                                                  │
        ▼ Haiku Triage → план з .route                     ▼ в'юер: перемикач
                                                            «Текст» читає .md
                                                            за діапазоном документа
```

Те саме джерело (`pageStructure`), **дві різні проекції**. Triage —
проекція «сигнали меж». В'юер — проекція «читабельний текст». Не плутати,
не дублювати.

## 3. ЗМІНА Ф0 — компактний паспорт меж (ядро доповнення)

### 3.1 Нова чиста функція (НЕ розширення наявної)

`src/services/documentPipeline/pageMarkers.js` — **додати**
`buildCompactTriagePassport(layoutJson, expectedPageCount, opts)`.
`buildStructuralPassport` / `buildPagedText` / `isPagedLayout` —
**не чіпати** (їх може споживати щось інше; філософія: додати, не
переписати).

**Один сенс (коментар на місці оголошення, обов'язково):**

```js
// buildCompactTriagePassport — паспорт МЕЖ для Triage: на сторінку лише
// дайджест структури + перші/останні рядки тексту (де живуть сигнали меж).
// НЕ повний readable-текст (той — extractV3 → 02_ОБРОБЛЕНІ, окремий потік).
// Один сенс: «мінімальний достатній сигнал меж документів для AI».
```

**Контракт сторінки в паспорті:**

```
=== СТОРІНКА N ===
[дайджест: orientation | формат | таблиці | заголовок:"..." | футер-№:K | СКИДАННЯ-НУМЕРАЦІЇ]
⟨head: перші HEAD_LINES непорожніх рядків _text, обрізані по HEAD_CHARS⟩
…
⟨tail: останні TAIL_LINES непорожніх рядків _text, обрізані по TAIL_CHARS⟩
```

Дайджест — **уже рахується** в `pageDigest()` (заголовок, футер-№,
скидання нумерації, orientation, формат, tables/formFields). Переюз його
як є — це найсильніші сигнали меж в українській судовій практиці і вони
**на краях сторінки**, не в тілі.

**Налаштовуваний ручник (`opts`, дефолти — стартові точки, не догма):**

| opt | Дефолт | Сенс |
|---|---|---|
| `headLines` | 3 | скільки перших непорожніх рядків сторінки |
| `tailLines` | 2 | скільки останніх непорожніх рядків |
| `headChars` | 400 | жорсткий cap на head-фрагмент |
| `tailChars` | 200 | жорсткий cap на tail-фрагмент |
| `fullTextIfNoSignal` | true | якщо дайджест порожній І сторінка коротка (≤ `ambiguousMaxChars`) — вкласти повний `_text` цієї сторінки (вузький fallback для неоднозначних) |
| `ambiguousMaxChars` | 1200 | межа «короткої» сторінки для fallback вище |

`fullTextIfNoSignal` — це той самий обмежений виняток, що рятує сторінки
без структурних сигналів (квитанція-картинка, чистий штамп) **без**
повернення до повнотекстового паспорта: спрацьовує лише коли дайджест
порожній і сторінка коротка, тому бюджет не вибухає.

### 3.2 Перемкнути єдину точку Triage на компактний паспорт

`resolveBoundaryText(layoutJson, expectedPageCount, plainText)` —
**єдина** точка, яку викликає `triageStage.js:passportOf`. Змінити її
тіло так, щоб для Triage вона повертала **компактний** паспорт:

```
buildCompactTriagePassport(...)  ||  buildPagedText(...)  ||  String(plainText||'')
```

(`buildStructuralPassport` прибирається з цього ланцюга — він лишається
експортованим для будь-якого іншого споживача повного тексту, якщо такий є;
grep підтвердити споживачів перед зміною.)

Жодних змін у `triagePrompt.js`, `triageStage.js` (артефакти, catch-шлях),
диригенті, схемі. Контракт `artifacts[].passport` той самий — змінюється
тільки **щільність** його вмісту.

### 3.3 Бюджетний запобіжник (щоб тихий fallback не повернувся непомітно)

Додати **м'яку перевірку розміру** перед AI-викликом Triage (у
`triageStage.js` перед `await triage(...)` або в транспорті): якщо
сумарний розмір промпта перевищує поріг `TRIAGE_PROMPT_SOFT_LIMIT_CHARS`
(стартово ~450K символів ≈ ~120K токенів кирилиці) — **не мовчати**:
зафіксувати `decision` типу `triage_prompt_oversize` з реальним розміром
і к-стю сторінок (видно у Зоні 3 / звіті), далі або зменшити ручники
паспорта (зрізати head/tail), або явно повідомити. Це прибирає
«мовчазне переповнення → один нерізаний документ».

Один сенс: «поріг, вище якого паспорт точно завеликий — сигналізувати,
не глитати». Не «обрізати дані» — саме **видимий сигнал**.

## 4. РОЗВ'ЯЗКА ТЕКСТУ — повний OCR + очистка `.md` лишаються (для в'ювера)

**Стан коду (підтверджено аудитом — нічого не ламаємо):**

- `streamingExecutor` робить chunk-OCR усіх сторінок (RAM-bounded,
  resumable; `memoryMonitor.DEFAULT_CHUNK_PAGES=25`, jobState коментар
  прямо згадує сценарій «250 стор. → 30 файлів»). **OCR-шар уже під 250
  стор.** Стіна — лише в промпті Triage (§0), не тут.
- `extractV3.js` (DP-3 EXTRACT override): якщо `cleanForReading` →
  ін'єктований `cleanText` (Haiku, `DocumentPipelineContext.jsx:~105-116`,
  maxTokens 8000) чистить OCR-сміття → `processedText`, `textFormat='md'`.
  Інакше — сирий `.txt`.
- `splitDocumentsV3.js` нарізає **паралельний layout per-page по діапазону
  документа** (:~502-558) і пише `02_ОБРОБЛЕНІ/<doc>_<driveId>.txt|.md` +
  `.layout.json` **на фінальний логічний документ**.
- В'юер `DocumentViewer/` показує перемикач Скан/Текст лише для `scanned`;
  «Текст» читає кешований текст через `ocrService.getCachedText` (:~168-174)
  з `02_ОБРОБЛЕНІ` за `driveId`. DOCX/HTML/PDF-з-шаром (`documentNature=
  'searchable'`, `originalMime` ≠ pdf) — перемикач НЕ показується (текст
  уже є з конвертера, `writeExtractedTextArtifact`); це коректно і
  лишається.

**Висновок:** гібрид, який описав адвокат, **структурно вже існує**.
Компактний паспорт (§3) **нічого з цього не торкає** — `extractV3`,
`splitDocumentsV3`, персист тексту, в'юер працюють як раніше. Triage
просто дивиться іншу (компактну) проекцію. Адвокат і далі отримує
повний очищений `.md` кожного логічного документа у Матеріалах справи.

**Доробка цього доповнення тут — мінімальна (DELTA, золота середина):**

1. **Підтвердити тестом**, що per-document `.md` (а не лише `.txt`) лягає
   у `02_ОБРОБЛЕНІ` за діапазоном документа, і в'юер «Текст» його читає.
   Якщо є розрив (split бере сирий `_text` з layout замість `processedText`
   при нарізці — аудит позначив це як integration-note extractV3) — це
   **єдина реальна правка**: при `slice`/`fragment_reconstruct` per-doc
   текст збирати з очищеного `processedText` за діапазоном сторінок, не з
   сирого layout `_text`. Один сенс: «текст логічного документа = очищений
   текст його сторінок».
2. Більше нічого. Перемикач режиму НЕ додаємо (див. §5).

## 5. РІШЕННЯ, ЯКІ ПОТРЕБУЮТЬ АДВОКАТА (не вирішую сам — REVIEW-крок)

### D1 — компактний паспорт окремою функцією
**Пропозиція:** нова `buildCompactTriagePassport`, стара
`buildStructuralPassport` недоторкана (правило #11 + «додати, не
переписати»). **Рекомендація: так.** Потрібна згода на сам підхід §3.

### D2 — перемикач «тільки нарізка / нарізка+текст»
Адвокат просив перемикач на старті DP. Батьківський TASK має зафіксоване
обмеження «один вхід, без режимів і кнопок» (№3, §13) — але воно про
**маршрутизацію** (адвокат не обирає slice/merge — це робить Triage), НЕ
про очистку. Очистка **вже** керується опцією `cleanForReading` (одна з 8
тумблерів `DEFAULT_SETTINGS`, DocumentProcessorV2:33). Тобто бажаний вибір
«документи / документи+текст» = **наявний тумблер `cleanForReading`**, не
новий режим — конфлікту з обмеженням немає.
**Пропозиція:** переюзати наявний тумблер, нового екрана не робити.
**Рекомендація: так.** Питання адвокату нижче.

### D3 — «вибрати які документи розпізнавати/чистити» (per-document)
Це справді нове (post-slice гранулярність, якої в `extractV3` нема — він
file-level до нарізки). Філософія: UI відкладати, DELTA, без
over-engineering; батьківський TASK — «без нових режимів». **Пропозиція:**
**НЕ** в цьому доповненні. Винести в `tracking_debt.md` з явним тригером
(«коли адвокат регулярно не потребує текст частини документів великого
тому»). Зараз: `cleanForReading` діє на весь job (як є).

### D4 — доля `.layout.json` після очистки
Адвокат: після `.md` layout непотрібний, можна викидати. **Аудит
заперечує безумовне видалення:** `.layout.json` читається `extractV3`
(для очистки) і `splitDocumentsV3` (per-doc layout), а в'юер per-page
структуру теж може споживати. До `persist` він **потрібен**. Філософія:
структуру даних не видаляти превентивно «бо здається непотрібним».
**Пропозиція:** **не** видаляти в цьому доповненні. Зафіксувати в
`tracking_debt.md`: «прунити file-level `.layout.json` у `02_ОБРОБЛЕНІ`
після успішного `persist`, тригер — тиск Drive-сховища; per-doc layout
лишати доки в'юер його споживає». Рішення адвоката: лишаємо як є зараз?

### D5 — калібрування тесту (чесно)
Брановський — 65 стор. Він перевіряє **паритет/покращення якості меж** з
компактним паспортом, **НЕ стелю 250 стор.** (він нижче неї). Для стелі
потрібен реальний/синтетичний том ~200-250 стор. Acceptance це враховує
(§9): є дві окремі перевірки.
