## 0. Навіщо це доповнення (корінь)

**Реальна ціль адвоката:** стандартний топ-том судової справи —
**200-250 сторінок** — має оброблятися швидко і якісно. Це не край, це норма.

**Прогалина батьківського TASK.** Ф0 описує паспорт як «текст сторінки +
дайджест геометрії» (TASK_smart_triage §3, §6; consultation §4). Поточна
реалізація `buildStructuralPassport` (`src/services/documentPipeline/
pageMarkers.js`) буквально вкладає **повний `_text` КОЖНОЇ сторінки**:

```
.map((page, i) => `=== СТОРІНКА ${i + 1} ===${line ? `\n${line}` : ''}\n${page._text || ''}`)
```

Старий cap `slice(0,50000)` навмисно прибрано (фікс DP-4 проти галюцинацій
номерів сторінок). Отже промпт Triage = **весь OCR-текст усієї справи**.

**Вартісна модель §6 цього НЕ закриває.** Вона прибрала *image*-токени
(Vision). Вона нічого не каже про **обсяг тексту**. Це окремий, другий
бюджет: **контекстне вікно Haiku**.

**Статична оцінка (без прогону, кирилиця ~2 символи/токен):**

| Підхід | Токенів/стор. | 250 стор. | Haiku 200K вікно |
|---|---|---|---|
| Поточний (повний текст у паспорті) | ~1000-2000 | **~250-500K** | **ПЕРЕПОВНЕННЯ** |
| Компактний паспорт (це доповнення) | ~200-280 | **~50-70K** | у зоні якості (≤~120K) |

Якість Haiku на межах деградує **задовго до** жорсткого ліміту — це той
самий механізм, що дав помилки меж уже на 65 стор. Брановського.

**Тихий режим відмови (критично).** При переповненні API відхиляє запит →
`triageStage.js:201-207` ловить у catch → плану немає → passthrough →
**уся справа лягає одним нерізаним документом**, лише рядок `warnings`. Не
падіння, не помилка на екрані. На великому томі адвокат отримує це мовчки.

## 1. Місія доповнення (що означає «готово»)

1. **Triage отримує компактний паспорт меж**, не повний текст. Том
   200-250 стор. влазить в одному проході Haiku, у зоні якості, швидко.
2. **Повний OCR-текст і його очистка в `.md` НЕ зникають** — вони
   виробляються і персистяться як раніше (для в'ювера), просто **розв'язані**
   з тим, що споживає Triage. Адвокату текст потрібен у Матеріалах справи
   для копіювання/редагування — це лишається.
3. Жодного нового «режиму»/екрана (обмеження батьківського TASK №3, §13).

## 2. Архітектурний принцип розв'язки (правило #11)

Один сенс на ім'я. Сьогодні `resolveBoundaryText` (єдина точка вибору
тексту для Triage, pageMarkers.js) повертає **повний** структурний паспорт.
Ми НЕ нашаровуємо на `buildStructuralPassport` прапор `compact` (це другий
сенс на одне ім'я — точно баг-патерн skipCache з філософії).

**Три незалежні потоки з одного джерела (`pageStructure`):**

```
                 OCR (streaming chunk, ВЖЕ Є) → pageStructure/_text/.layout.json
                          │                                  │
        ┌─────────────────┘                                  └──────────────┐
        ▼ ВХІД TRIAGE (новий, компактний)            СИРИЙ ТЕКСТ (як є, повний)
  buildCompactTriagePassport()                    splitDocumentsV3 пише
   = сукупність дорадчих сигналів меж              02_ОБРОБЛЕНІ/<doc>.txt
   = ~200-280 ток/стор.                                   │
   = жоден сигнал не вирішує сам                          ▼ ОЧИСТКА (опційна,
        │                                          над уже-OCR-текстом, БЕЗ
        ▼ Haiku Triage → план з .route             повторного OCR): Haiku →
                                                   02_ОБРОБЛЕНІ/<doc>.md
                                                   inline АБО окремо/пізніше
                                                   по обраних документах
                                                          │
                                                          ▼ в'юер «Скан/Текст»
                                                   (тільки скани): Текст =
                                                   .md якщо є, інакше .txt
```

Те саме джерело, **три різні проекції**. Triage — «сигнали меж». Сирий
текст — побічний продукт OCR, **існує завжди**. Очистка — окрема операція
**над уже наявним текстом** (ніколи не запускає OCR), inline або
самостійно пізніше. Не плутати, не дублювати.

## 3. ЗМІНА Ф0 — компактний паспорт меж (ядро доповнення)

### 3.1 Нова чиста функція (НЕ розширення наявної)

`src/services/documentPipeline/pageMarkers.js` — **додати**
`buildCompactTriagePassport(layoutJson, expectedPageCount, opts)`.
`buildStructuralPassport` / `buildPagedText` / `isPagedLayout` —
**не чіпати** (їх може споживати щось інше; філософія: додати, не
переписати).

**Один сенс (коментар на місці оголошення, обов'язково):**

```js
// buildCompactTriagePassport — паспорт МЕЖ для Triage: на сторінку лише
// сукупність дорадчих сигналів (дайджест структури + краї тексту), які AI
// зважує РАЗОМ. Жоден сигнал не вирішує сам, жоден не жорстке правило.
// НЕ повний readable-текст (той — окремий потік, 02_ОБРОБЛЕНІ).
// Один сенс: «мінімальна достатня сукупність сигналів меж для AI».
```

**Контракт сторінки в паспорті:**

```
=== СТОРІНКА N ===
[дайджест: сукупність дорадчих сигналів — див. 3.2]
⟨head: перші HEAD_LINES непорожніх рядків _text, обрізані по HEAD_CHARS⟩
…
⟨tail: останні TAIL_LINES непорожніх рядків _text, обрізані по TAIL_CHARS⟩
```

### 3.2 Дайджест = сукупність взаємопов'язаних дорадчих сигналів (D1)

**Принцип (зафіксовано адвокатом):** жоден параметр не має самостійного
значення. Текст, печатки, підписи, стрибок якості, формат — усе
аналізується **в сукупності і взаємозв'язку**. Саме ансамбль цих
різнорідних, але компактних даних (за обсягом у рази менший за повний
текст) дає точну картину де починається і де закінчується документ. AI
зважує їх разом; жоден не детермінований гейт (обмеження батьківського
TASK №3).

Сигнали дайджесту (усі — текстом зі збереженого `pageStructure`, нуль
image-токенів; усі — `null`-safe, відсутність поля = сигнал просто не
додається):

| Сигнал | Джерело Document AI | Статус |
|---|---|---|
| орієнтація | `layout.orientation` | вже в `pageDigest` |
| формат (фото/A4) | `dimension` (w/h) | вже |
| таблиці / поля-форми | `tables` / `formFields` | вже |
| заголовок (короткий центрований 1-й блок) | `blocks` + `boundingPoly` | вже |
| футер-№ + **СКИДАННЯ-НУМЕРАЦІЇ** | останній рядок `_text` | вже (найсильніший) |
| **печатка/підпис на сторінці** | `visualElements` (прапор типу) | **додати** |
| **стрибок якості** vs попередня стор. | `imageQualityScores` | **додати** |
| **розрідженість** (мало блоків / низька щільність) | к-сть `blocks` / довжина `_text` | **додати** |
| **дельта формату/орієнтації** vs попередня | порівняння з prev | **додати** |
| зміна мови | `detectedLanguages` | **додати (слабкий)** |

«Додати» — збагачення `pageDigest()` у ФД-0. Кілька десятків символів на
сторінку — мізер проти економії на тілі тексту. Найцінніше саме на твоїх
багах: квитанція судового збору = стрибок якості + зміна формату + власна
коротка сторінка + власна печатка — чотири сигнали одночасно.

**Налаштовуваний ручник (`opts`, дефолти — стартові точки):**

| opt | Дефолт | Сенс |
|---|---|---|
| `headLines` | 3 | перших непорожніх рядків сторінки |
| `tailLines` | 2 | останніх непорожніх рядків |
| `headChars` | 400 | cap на head-фрагмент |
| `tailChars` | 200 | cap на tail-фрагмент |
| `fullTextIfNoSignal` | true | дайджест порожній І сторінка коротка → повний `_text` цієї сторінки (вузький fallback) |
| `ambiguousMaxChars` | 1200 | межа «короткої» для fallback вище |

### 3.3 Перемкнути єдину точку Triage на компактний паспорт

`resolveBoundaryText(layoutJson, expectedPageCount, plainText)` —
**єдина** точка, яку викликає `triageStage.js:passportOf`. Тіло змінити:

```
buildCompactTriagePassport(...)  ||  buildPagedText(...)  ||  String(plainText||'')
```

`buildStructuralPassport` прибирається з цього ланцюга (лишається
експортованою для інших споживачів повного тексту, якщо такі є — grep
підтвердити перед зміною). Жодних змін у `triagePrompt.js`, `triageStage`
(артефакти/catch), диригенті, схемі. Контракт `artifacts[].passport` той
самий — змінюється тільки **щільність**.

### 3.4 Бюджетний запобіжник (проти повернення тихого fallback)

Перед AI-викликом Triage — м'яка перевірка розміру. Якщо промпт >
`TRIAGE_PROMPT_SOFT_LIMIT_CHARS` (~450K симв. ≈ ~120K токенів кирилиці) —
**не мовчати**: `decision` типу `triage_prompt_oversize` з реальним
розміром + к-сть сторінок (видно у Зоні 3 / звіті). Один сенс: «поріг,
вище якого паспорт точно завеликий — видимий сигнал, не глитати».

## 4. РОЗВ'ЯЗКА ТЕКСТУ — повний OCR + опційна очистка `.md` (для в'ювера)

**Стан коду (підтверджено аудитом — нічого не ламаємо):**

- `streamingExecutor` робить chunk-OCR усіх сторінок (RAM-bounded,
  resumable; `memoryMonitor.DEFAULT_CHUNK_PAGES=25`; jobState коментар
  прямо згадує «250 стор. → 30 файлів»). **OCR-шар уже під 250 стор.**
  Стіна — лише в промпті Triage (§0), не тут.
- `extractV3.js` (DP-3 EXTRACT override): `cleanForReading=true` →
  ін'єктований `cleanText` (Haiku, `DocumentPipelineContext.jsx:~105-116`,
  maxTokens 8000) → `processedText`, `textFormat='md'`. Інакше — сирий
  `.txt`. **Очистка працює над уже-OCR-текстом; OCR вона НЕ запускає.**
- `splitDocumentsV3.js` нарізає паралельний layout per-page по діапазону
  документа (:~502-558) → `02_ОБРОБЛЕНІ/<doc>_<driveId>.txt|.md` +
  `.layout.json` на фінальний логічний документ.
- В'юер `DocumentViewer/` показує перемикач **«Скан / Текст»** (скріншоти
  адвоката, 19.05) **лише для `scanned`**; «Текст» читає кеш через
  `ocrService.getCachedText` (:~168-174) з `02_ОБРОБЛЕНІ` за `driveId`.
  DOCX/HTML/PDF-з-текстовим-шаром (`documentNature='searchable'`,
  `originalMime` ≠ pdf) — **перемикач НЕ показується** (текст копіюється
  прямо з PDF; `writeExtractedTextArtifact` уже зберіг текст). Це
  коректно і лишається.

**Контракт перемикача в'ювера «Скан / Текст» (D2, фінально):**

- Показується **тільки** для сканованих PDF (`documentNature='scanned'`),
  бо лише там текст не вибирається з самого PDF. Призначення — дати
  адвокату **скопіювати частину або весь текст** документа.
- Не показується для PDF з текстовим шаром / DOCX / HTML — там копіювати
  можна прямо.
- Режим «Текст» рендерить: **очищений `.md` якщо він є, інакше сирий
  `.txt`**. Один сенс перемикача: «показати видобутий текст цього
  сканованого документа для читання/копіювання».

**Очистка — опційна, розв'язана з нарізкою (D3, фінально):**

Очистка це **окрема операція над уже персистованим OCR-текстом**, яка
**ніколи не запускає OCR**. Її можна виконати:
- **(a) inline** під час DP — наявний тумблер `cleanForReading` (D2);
- **(b) самостійно пізніше** — над `02_ОБРОБЛЕНІ/<doc>.txt` +
  `.layout.json` обраного адвокатом документа → `<doc>.md`.

На томах 200-250 стор. основним буде шлях (b): спершу швидко нарізати,
потім вибірково чистити лише ті документи, з якими адвокат працює. Це
зручність і гнучкість, і це пряма вимога цілі «швидко на 250 стор.»
(важкий прохід Haiku-очистки на весь том не нав'язується).

**Архітектурна вимога цього доповнення (ембріон з повним ДНК):** ДНК —
зараз; UI — потім. Фактор Haiku-очисника так, щоб **та сама логіка**
викликалась і з `extractV3` (inline), і як самостійна операція над
персистованим текстом одного документа. Шов (seam) закладаємо в цьому
доповненні; вибіркову per-document UI-дію (кнопка «Очистити цей
документ» у в'юері/дереві) — окрема пізніша фаза (UI відкладається за
філософією; шов — ні).

**Перемикач у DP — фінальна семантика (D2):** один тумблер, назва в коді
`cleanForReading` незмінна, один сенс «чи робити inline-прохід очистки в
`.md`». Мітки кажуть правду (вимкнено ≠ «без тексту»):
- **Вимкнено = «Документи (текст сирий)»** — нарізані PDF; текст сирий
  `.txt`; у в'юері «Текст» показує сирий текст; чистити можна пізніше (b).
- **Увімкнено = «Документи + чистий текст»** — нарізані PDF + inline
  Haiku-`.md`; у в'юері «Текст» показує очищений текст.

## 5. ЗАФІКСОВАНІ РІШЕННЯ D1–D5 (адвокат, 19.05.2026)

- **D1 — ПРИЙНЯТО.** Компактний паспорт окремою функцією
  `buildCompactTriagePassport`; стара `buildStructuralPassport`
  недоторкана. Дайджест — сукупність взаємопов'язаних **дорадчих**
  сигналів (§3.2), AI зважує разом, жоден не самостійний, жоден не
  жорсткий гейт.
- **D2 — ПРИЙНЯТО з уточненням.** Два пов'язані перемикачі, кожен — один
  сенс: (1) DP-тумблер `cleanForReading` (назва незмінна, мітки правдиві,
  §4); (2) в'юер «Скан/Текст» лише для сканів, «Текст» = `.md`||`.txt`
  (§4). Жодного нового екрана/режиму.
- **D3 — ПРИЙНЯТО з поправкою (чесна реконсиляція).** Раніше «повністю
  відкласти». Твій сценарій (на 250 стор. чистити **окремо + вибірково**)
  робить розв'язку очистки **проєктним наміром**, не відкладеним:
  **архітектурний шов standalone-очистки над персистованим текстом —
  у цьому доповненні (ФД-3)**; відкладається **лише вибіркова
  per-document UI-дія** → `tracking_debt.md` з тригером «коли адвокат
  регулярно чистить підмножину великого тому».
- **D4 — ПРИЙНЯТО, ПОСИЛЕНО.** `.layout.json` **не видаляти**. Тепер є
  конкретна причина (не лише «безпечніше»): standalone-очистка (D3.b)
  пізніше **потребує** `.layout.json` для очистки. Прун file-level layout
  → `tracking_debt.md`, вузький тригер: «документ уже очищено І адвокат
  не перечищатиме І тиск Drive-сховища».
- **D5 — ПРИЙНЯТО (визнання).** Брановський (65 стор.) валідує **якість
  меж**, не стелю 200-250. Стелю перевіряти окремим томом ~200-250 стор.
  (реальний/синтетичний). Дві окремі перевірки в acceptance (§10).

## 6. Фази (вкладаються в Ф0/Ф1 батьківського TASK, не нові стадії)

Кожна самодостатня, STOP-точки явні, зелені тести на межі.

- **ФД-0 — `buildCompactTriagePassport` + збагачений дайджест** (чиста
  функція; сигнали §3.2 включно з печатка/підпис, стрибок якості,
  розрідженість, дельта, мова; `opts`; один-сенс коментар). Юніти:
  кожен сигнал присутній у дайджесті; head/tail-обрізка;
  fullTextIfNoSignal вузький; маркери цілі; ідемпотентно. *STOP.*
- **ФД-1 — `resolveBoundaryText` → компактний** + grep споживачів
  `buildStructuralPassport`. Юніт: повертає компактний; fallback цілий.
  *STOP.* (ФД-0+ФД-1 уже прибирають стіну 250 стор. — негайна цінність.)
- **ФД-2 — бюджетний запобіжник** (`triage_prompt_oversize` видимий, не
  тихо). Юніт: великий синтетичний layout → decision виставлено. *STOP.*
- **ФД-3 — розв'язка тексту + шов standalone-очистки.** (a) per-document
  **очищений** `.md` за діапазоном у `02_ОБРОБЛЕНІ`, в'юер «Текст» його
  читає (полагодити якщо split бере сирий layout `_text` замість

## 7. SAAS IMPLICATIONS (CLAUDE.md #10)

- Жодної нової сутності зі станом. `buildCompactTriagePassport` — чиста
  функція; паспорт транзитний у `ctx`, не персиститься у схему, не
  tenant-scoped. `tenantId` не додається, `schemaVersion` не bump'иться,
  міграції/бекапу нема (як батьківський TASK §5, §13).
- Standalone-очистка (ФД-3.b шов) оперує файлами справи на Drive —
  успадковує tenant справи; самостійна дія, коли отримає UI, піде через
  `executeAction` з permission-перевіркою (ДНК закладається, активація —
  пізніша фаза). НЕ активувати `metadata_extractor_agent`. НЕ UI ролей.
- Resume/job_state per-case; компактний паспорт не змінює resume-контракт
  (layout-джерело те саме).

## 8. BILLING IMPLICATIONS (CLAUDE.md #10)

- Triage AI-виклик уже інструментований (батьківський TASK §12). Це
  доповнення **не додає точок**, а **зменшує** `inputTokens`/
  `estimatedCostUSD` на job у рази (повний текст → компактний паспорт).
  Baseline до/після — у звіті ФД-4.
- `extractV3` inline-очистка (Haiku) — наявна точка, не змінюється.
- Standalone-очистка (ФД-3.b) коли отримає UI-активацію — **нова
  білінг-точка** (`activityTracker.report` + `logAiUsage`, Haiku-виклик
  над текстом обраного документа). Шов закладаємо тепер (ембріон ДНК),
  інструментацію активної дії — у фазі UI. НЕ дублювати `ai_usage[]` ↔
  `time_entries[]`. Уся телеметрія у try/catch.
