## 1. ПРОБЛЕМА

`passportOptsForBudget(pageCount)` (`pageMarkers.js`) — **бінарний обрив**, не плавність:

| Обсяг | Профіль | Симв/стор |
|---|---|---|
| **≤ 70 стор.** | rich (`RICH_PASSPORT_OPTS`) | head 1500 + tail 1500 = **~3000** |
| **> 70 стор.** | компакт (`COMPACT_DEFAULTS`) | head 400 + tail 200 = **~600** |

На 71-й сторінці щільність падає в **5 разів** одним стрибком. Том на 100-250 стор.
голодує (600 симв/стор), хоча у бюджет влізло б значно більше — Haiku вгадує межі за
обрізками. Це одна з причин ~40% точності на великих томах.

## 2. ЗАМІР (обґрунтування цифр — реальний `count_tokens`, не оцінка)

Документи раніше використовували оцінку «~2-3 симв/токен» без прогону. **2026-06-21
зроблено реальний замір** через Anthropic `count_tokens` (токенайзер `claude-haiku-4-5-20251001`,
модель тріажу) на репрезентативному українському судовому тексті:

| Тип тексту | симв | токени | симв/токен |
|---|---|---|---|
| Проза (ухвала), зразок 1 | 1509 | 669 | 2.26 |
| Проза (ухвала), зразок 2 | 545 | 229 | 2.38 |
| Цифри/посилання на статті | 454 | 234 | 1.94 |
| Розріджена (штамп/виконавчий)* | 162 | 105 | 1.54 |

\* на дуже коротких текстах фіксований overhead повідомлення завищує токени; для
істотних сторінок орієнтир **~2.0-2.2 симв/токен**. **Робочий консервативний коеф. =
2.0 симв/токен** (юр-текст із номерами токенізується важче, ~1.9).

**Наслідок — перерахунок токен-сум (×2.0):**

| | Файли казали (×~3) | Реально (×2.0) |
|---|---|---|
| rich-70 (70×3000 симв) | ~65-70K ток | **~105K ток** (відомо-ДОБРЕ, працює) |
| компакт-250 (250×600 симв) | ~62K ток | ~75K ток |
| degenerate rich-100 (100×3000) | ~95K ток | **~150K ток** (відомо-ПОГАНО, ламає Haiku) |

**Висновок:** безпечна стеля Haiku — між rich-70 (**~105K ток = 210K симв**, працює) і
rich-100 (**~150K ток = 300K симв**, degenerate). Тому **безпечний бюджет = 210K символів
(рівень rich-70)** — доведено-робочий, з комфортним запасом до degenerate.

## 3. ДИЗАЙН — пропорційне заповнення бюджету

Замість «обрати один із двох профілів» — рахуємо символи/стор. з єдиного бюджету:

```
B(pageCount) = clamp( round(CHAR_BUDGET / pageCount), FLOOR, CEIL )
  CHAR_BUDGET = 210000   // = 70 × 3000 (рівень rich-70, відомо-добрий)
  FLOOR       = 600      // = поточний компакт (на гігантських томах не гірше)
  CEIL        = 3000     // = поточний rich (не більше за повне тіло)
```

Розкладка `B` на opts: **head:tail = 1:1** (як rich), `headChars = ceil(B/2)`,
`tailChars = floor(B/2)`; решта opts = `RICH_PASSPORT_OPTS` (headLines/tailLines/
fullTextIfNoSignal/ambiguousMaxChars).

**Чому 1:1, а не компактне 2:1:** щоб ≤70 стор. лишалося **байт-у-байт** як зараз
(rich = 1500/1500). На великих томах split стає 1:1 замість 2:1 — це покращення (більше
хвоста теж корисно), сумарні символи незмінні; компактний профіль `COMPACT_DEFAULTS`
прямо помічений «стартові точки, не остаточні», тож не священний.

**Поведінка (CHAR_BUDGET=210000):**

| Обсяг | B = симв/стор | Зараз | Зміна | Сума симв (≤105K ток) |
|---|---|---|---|---|
| ≤70 стор | 3000 (стеля) | 3000 | **0 — байт-у-байт** ✅ | ~105K ток |
| 70 стор | 3000 (210000/70 рівно) | 3000 | 0 ✅ | 105K |
| 100 стор | 2100 | 600 | ↑3.5× | 105K |
| 150 стор | 1400 | 600 | ↑2.3× | 105K |
| 250 стор | 840 | 600 | ↑1.4× | 105K |
| 350 стор | 600 (підлога) | 600 | 0 ✅ | спадає |
| 700 стор | 600 (підлога) | 600 | 0 ✅ | — |

**Ключові інваріанти дизайну:**
1. **≤70 стор. — байт-у-байт як зараз** (B клампиться у CEIL=3000 = rich). Малі томи
   (100% точність на Брановському) не чіпаються.
2. **Спуск починається точно на 70** (210000/70 = 3000), без розриву.
3. **Сума ніде не перевищує 210K симв (~105K ток)** = рівень відомо-доброго rich-70 →
   degenerate (≥150K ток) НЕ повертається в принципі.
4. **70-250 стор. отримують 1.4-3.5× більше тексту** — головний виграш.
5. Великі (>350) — підлога 600, як зараз.

## 4. ЗМІНА В КОДІ (`src/services/documentPipeline/pageMarkers.js`)

**Замінюється** бінарний `passportOptsForBudget` + поріг `richMaxPages()`. Орієнтовно:

```js
// CHAR_BUDGET — сумарний символьний бюджет паспорта на ВЕСЬ том (рівень rich-70 =
// 70×CEIL, відомо-добрий за заміром 2026-06-21, нижче degenerate-стелі Haiku ~150K ток).
// Один сенс: «скільки символів усього віддаємо Triage; розподіляється рівномірно по
// сторінках, клампиться в [FLOOR, CEIL] на сторінку». Override-хук — для калібровки/тестів.
const PASSPORT_CHAR_BUDGET_DEFAULT = 210000;
const PASSPORT_FLOOR_CHARS = 600;   // = поточний компакт (head 400 + tail 200)
const PASSPORT_CEIL_CHARS  = 3000;  // = поточний rich   (head 1500 + tail 1500)
let _charBudgetOverride = null;
export function _setPassportCharBudget(n) {       // тести / майбутня tenant-калібровка
  _charBudgetOverride = (typeof n === 'number' && n > 0) ? n : null;
}
function passportCharBudget() {
  return _charBudgetOverride ?? PASSPORT_CHAR_BUDGET_DEFAULT;
}

function passportOptsForBudget(pageCount) {
  if (!(pageCount > 0)) return {};   // невідомий обсяг → компакт-дефолти (як зараз для 0)
  const perPage = Math.min(
    PASSPORT_CEIL_CHARS,
    Math.max(PASSPORT_FLOOR_CHARS, Math.round(passportCharBudget() / pageCount)),
  );
  return {
    ...RICH_PASSPORT_OPTS,
    headChars: Math.ceil(perPage / 2),
    tailChars: Math.floor(perPage / 2),
  };
}
```

`RICH_PASSPORT_MAX_PAGES_DEFAULT` / `richMaxPages()` / `_setRichPassportMaxPages` —
**видалити** (поріг 70 тепер імпліцитний у бюджеті: 210000/3000 = 70). Якщо на них є
посилання в тестах (`triageStage.test.js` / passport-тести) — оновити на
`_setPassportCharBudget`. Грепнути перед видаленням.

**Перевірити байт-ідентичність ≤70:** при pageCount ∈ [1,70] функція має повертати
`{...RICH_PASSPORT_OPTS, headChars:1500, tailChars:1500}` — тобто рівно `RICH_PASSPORT_OPTS`.

## 6. ТЕСТИ (`tests/unit/` — разом з кодом)

Юніт на `passportOptsForBudget` (експортувати для тесту або тестувати через `resolveBoundaryText`):
- **Байт-ідентичність ≤70:** pageCount ∈ {1, 35, 70} → `headChars=1500, tailChars=1500` (= rich).
- **Спуск після 70:** pageCount=100 → perPage 2100 (head 1050/tail 1050); 150 → 1400; 250 → 840.
- **Підлога:** pageCount=350 → 600 (head 300/tail 300); 700 → 600.
- **Монотонність:** perPage(pageCount) не зростає з ростом pageCount.
- **Клампи:** ніколи < 600, ніколи > 3000.
- **Невідомий обсяг:** pageCount=0 → компакт-дефолти (як зараз).
- **Override-хук:** `_setPassportCharBudget(300000)` змінює спуск; `null` повертає дефолт.
- Усі попередні passport/triage-тести лишаються зеленими (оновити лише ті, що зверталися
  до `_setRichPassportMaxPages`).

`npm test` повністю зелений перед комітом; CI блокує деплой при red.

## 8. КАЛІБРУВАННЯ (після приземлення — не код цього TASK)

Дефолт `PASSPORT_CHAR_BUDGET=210000` — **відомо-добрий старт** (рівень rich-70). Далі на
реальних томах через `_setPassportCharBudget` (або майбутній оператор) піднімати бюджет і
дивитися, де Haiku починає втрачати межі (degenerate). Паралельно через меню ⚙ перемкнути
`qiParserDocument` на Sonnet і порівняти — Sonnet має вищу стелю якості, тож витримає
більший бюджет. Це визначить поріг «з якого обсягу вмикати Sonnet» (важіль A4-г, окремо).
