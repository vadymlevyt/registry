# TASK — Видалення справи: персист на Drive + негайне «створити наново» при імпорті

**Тип:** спека для сесії-виконавця. Адмін-сесія НЕ реалізує сама.
**Статус:** очікує затвердження → виконавець.
**Дата:** 2026-06-10
**Пріоритет:** баг + продуктова вимога (реальний ЄСІТС-імпорт).
**Закриває:** tracking_debt #60.

---

## 0. Вимога адвоката (2026-06-10)

> (1) Видалена справа має **створюватися наново** при імпорті (а не оновлюватись),
> **негайно**, **без F5**. Незалежно від того, скільки справ видалено за раз.
> (2) Видалення не лишає **жодного хвоста** активної справи: контент і кеші
> прибираються повністю. Аудит і білінг **зберігаються**, але як інертні
> «сироти з позначкою [видалена справа]» — на систему не впливають,
> можуть бути прибрані пізніше (питання модуля білінгу).
> (3) Storage-guard ЛИШАЄТЬСЯ (реальний інцидент: стара localStorage з іншого
> пристрою затерла дані) — фікс його не послаблює; навмисне видалення проходить
> через явний сигнал, усе випадкове блокується як зараз.

## 1. Корінь (доказово з коду)

**1a. Write-guard блокує persist видалення кількох справ.**
`src/services/registryWriteGuard.js` (`evaluateRegistryWriteGuard`, ~рядок 47):
```js
if (casesPrev > 0 && casesNew < casesPrev - 1) return 'cases_count_decreased';
```
Дозволено зменшення `cases` **лише на 1** за один запис. Видалення ≥2 справ за
один збережувальний цикл (`App.jsx` save-useEffect, deps `[cases, …]`, ~рядок
4515-4568) → `writeRegistry` повертає `guard_blocked` → **запис на Drive
блокується** → видалення не персиститься → після reload справи повертаються.

Guard зроблено свідомо (захист від затирання повної історії порожнім масивом —
коментар у файлі), але він **надмірно блокує навмисне видалення кількох справ**.

**1b. Видалення не оновлює `casesRef` синхронно.**
`deleteCasePermanently` (`App.jsx:4879`) і агентський `destroy_case` роблять
**сирий** `setCases(prev => filter)`, не `setCasesWithRef`. Тому живий `casesRef`
оновлюється лише через пост-commit `useEffect` — для негайного повторного імпорту
в тій самій сесії читач може ще бачити «привид».

## 2. Зміна A — навмисне видалення персиститься завжди

Дозволити **навмисному (user-initiated) видаленню** записатись на Drive
незалежно від кількості видалених справ, **зберігши** захист guard від
**випадкового** затирання (порожній/майже-порожній масив без сигналу про намір).

Підхід (деталі — на розсуд виконавця в межах цього наміру):
- провести «очікуваний shrink справ» від місця видалення до guard. Напр.
  module-scoped `expectIntentionalCasesShrink(n)` у `registryWriteGuard.js` (або
  параметр у `driveService.writeRegistry`), який `deleteCasePermanently` виставляє
  ПЕРЕД видаленням, а guard **споживає на наступному записі**: дозволяє зменшення
  до `prev - (1 + n)`, потім сигнал скидається.
- guard і далі блокує `cases_count_decreased` для **неочікуваного** масового
  падіння (без сигналу) — захист від випадкового затирання лишається.
- Інші поля guard (ai_usage/auditLog/users/tenants) — НЕ чіпати.

> Альтернатива на розсуд: оновлювати `prev.cases`-baseline guard'а синхронно при
> навмисному видаленні. Головне — намір: **навмисне видалення завжди персиститься,
> випадкове затирання — ні.**

## 3. Зміна B — видалення через `setCasesWithRef`

`deleteCasePermanently` і агентський `destroy_case` (`actionsRegistry.js`, гілка
що фільтрує cases) → провести через `setCasesWithRef` (синхронне оновлення
`casesRef.current` в updater'і). Тоді негайний повторний імпорт (без F5) бачить,
що справи нема → `processCase` іде у `create_case` → **створює наново**.

> Наслідок: дедуп по `case_no` після повного видалення справу не знаходить →
> імпорт її **створює наново**. Жодних змін на боці імпорту не потрібно — лише
> довести видалення до кінця (A+B+C).

## 3.5. Зміна C — каскадне очищення + tombstone-індекс (рішення адвоката 2026-06-10)

Видалення прибирає весь **контент і кеші** справи, а **ledger-записи зберігає,
позначені як видалені** (інертні, не впливають на систему, покладено purge на
майбутній модуль білінгу).

**Прибрати повністю при видаленні справи `caseId`:**
- сама справа з `cases[]` (Зміна A/B);
- нотатки справи (`levytskyi_notes` bucket — записи цієї справи);
- agent-history: `localStorage agent_history_<caseId>` + Drive `agent_history.json`
  (per-case) + `case.agentHistory`;
- Drive: папка справи (`driveFolderId`) з документами + `.metadata/documents_extended.json`
  (вже частково в `deleteCasePermanently` — довести до кінця);
- `caseAccess[]` запис цього `caseId` (заглушка-індекс);
- дзеркало `localStorage levytskyi_cases` (перезаписується станом без справи — ок).

**Зберегти, але позначити як видалені (ledger-сироти, інертні):**
- `auditLog[]` (включно із записом `destroy_case`);
- `time_entries[]` цього `caseId` (білінг);
- `ai_usage[]` з `context.caseId` (телеметрія — та сама природа).

**Tombstone-індекс — SSOT «які справи видалені»:** нове поле реєстру
`deletedCases[]` (адитивне, default `[]`; лінива ініціалізація на load або
міграція — на розсуд, але має потрапити у save-об'єкт `App.jsx` і не ламати
write-guard, бо РОСТЕ). Запис: `{ caseId, case_no, name, deletedAt, deletedBy }`.
`deleteCasePermanently` пушить сюди ПЕРЕД видаленням.

**Ізоляція (критично — «ні на що не впливають»):** будь-який споживач ledger'ів
(білінг/звіти/дашборд/`timeEntriesQuery`) трактує `caseId ∈ deletedCases[]` як
видалену справу: НЕ показує серед активних, НЕ рахує в активних метриках, не
падає на відсутній live-case. Дедуп ЄСІТС — по `cases[]` (справи нема) → не
зачеплений. Активні в'ю ітерують `cases[]` → видаленої не бачать автоматично.

> #11: `deletedCases[]` — один сенс «надгробок видаленої справи» (не плутати з
> `caseAccess`/`team`). Ledger лишає свій `caseId`; маркер «видалена» — централізовано
> в `deletedCases[]`, не дублюється прапором на кожному записі.

## 4. Тести (обов'язково, `npm test` зелений)

- `registryWriteGuard`: без сигналу — блок при `cases < prev-1` (як зараз); з
  сигналом `expectIntentionalCasesShrink(n)` — дозвіл до `prev-(1+n)`, далі сигнал
  скинуто (наступний звичайний запис знову строгий); порожній масив без сигналу —
  блок (захист лишається).
- Видалення оновлює `casesRef` синхронно (стаб deferred setCases → після
  `setCasesWithRef` casesRef.current уже без видаленої справи).
- Інтеграційно: створити справу → видалити → (та сама сесія, без reload) повторний
  ЄСІТС-імпорт того ж `case_no` → `casesCreated` (не `casesUpdated`), без помилок.
- **Каскад/tombstone:** після видалення — `cases[]`/нотатки/`caseAccess`/agent-history
  очищені; `time_entries`/`ai_usage`/`auditLog` цього caseId **лишились**;
  `deletedCases[]` має запис `{caseId, case_no, name, deletedAt, deletedBy}`;
  споживач ledger'ів виключає `caseId ∈ deletedCases[]` з активних.
- Існуючі тести зелені.

## 5. Межі / SEMANTIC CLARITY

- НЕ послаблювати guard для випадкового затирання (порожній масив без наміру —
  лишається заблокованим). #11: «навмисне видалення» і «випадкова втрата» —
  різні наміри, розрізняються явним сигналом, не послабленням порога для всіх.
- НЕ чіпати v12-контракт / FIX-IDENTITY / race-фікс / дедуп.
