## 1. Правильний алгоритм (слова адвоката — це контракт)

### 1.1 Сценарій «Просто додати документи» — окремий, самодостатній

- Це **окремий сценарій, окремий процес, окремий код/файл**. **Ніяк не
  зав'язаний на Document Processor** (нарізку/склейку). Інша труба, інший кран.
- Вмикається тумблером **«Додати просто документи»**.
- Кількість файлів значення **не має** — 1 чи N обробляються однаково.
- **Цей сценарій УЖЕ Є в модалці «+ Додати документ» і працює.** Завдання —
  **виокремити його у СПІЛЬНИЙ сервіс** і дати **і модалці, і Document
  Processor** користуватися ним. Саме це означало «спільність», про яку йшлося.

### 1.2 Опція «без OCR» (в межах сценарію «просто додати»)

Тумблер «без OCR» / «без розпізнавання» означає:
- **НЕ** викликається Google **Document AI**.
- **НЕ** задіюється планка Picatinny (OCR провайдер-патерн).
- Документ **не осіариться** — проходить без розпізнавання тексту.
- Записується **просто в `01_ОРИГІНАЛИ`**. Без артефактів у `02_ОБРОБЛЕНІ`. Все.
- **Метадані НЕ потрібні.** Vision-метадані зараз — джерело проблем; у першій
  ітерації їх НЕМАЄ.
- *Максимум, як варіант на майбутнє (НЕ зараз):* 1 сторінка з кожного файлу
  читається Claude Vision → метадані. «Докрутиться потім, коли все працюватиме,
  якщо працюватиме.» Див. «Відкриті питання».

### 1.3 Тумблер «Стиснути файли» — теж окремий, фронт-крок

- Живе **окремо**, не прив'язаний ні до нарізки, ні до «просто додати».
- Логіка: **СПОЧАТКУ** стискаються ті файли, що можуть стиснутися. Стиснення —
  це **перший крок**, який нічого іншого не запускає. Коли файли стиснулись —
  **це тригерить наступний сценарій** (залежно від того, який тумблер обрано).

### 1.4 Комбінований приклад (як має працювати)

Адвокат додає за раз: **HTML** (ухвала суду) + **DOC** + **великий сканований
PDF** + **текстовий PDF з текстовим шаром**. Вмикає **«Стиснути»** + **«без OCR»**.

По типах файлів:
- **DOC, HTML** → конвертуються в PDF (converterService, TASK A).
- **Текстовий PDF (з текстовим шаром)** → просто записується в `01_ОРИГІНАЛИ`,
  **без артефактів** (OCR не треба — текст уже в шарі).
- **Сканований PDF** → бо тумблер «Стиснути» стоїть, **спершу стискається**,
  потім записується в `01_ОРИГІНАЛИ`. Бо «без OCR» — **без Vision, без
  метаданих, без артефактів**.
- **Зображення** (1 чи кілька) → кожне записується в PDF.
  ⚠️ **Суперечність у диктовці** — адвокат сказав і «усиариться / мають
  артефакти», і «без OCR = без розпізнавання, без артефактів». **Уточнити**
  (Відкрите питання 1).

Кінцевий стан при «без OCR»: тільки конвертація (де треба) + стиснення
(сканований + тумблер) + запис в `01_ОРИГІНАЛИ` без артефактів. Без Vision,
без метаданих.

## 4. Цільова архітектура (для обговорення → плану)

```
Зона старту в UI (модалка АБО Document Processor):
  тумблер «Стиснути» ON?  →  [ФРОНТ-КРОК] стиснути ті файли, що можуть
                              (imageCompressor, один прохід)  ──┐
                                                                 │
  далі маршрут за тумблером:                                     ▼
   • «Просто додати документи» → СПІЛЬНИЙ сценарій addFiles (НОВЕ, окремий
        модуль, нуль зв'язку з нарізкою):
            для кожного файлу:
              конвертувати в PDF (DOC/HTML/зображення) якщо треба
              «без OCR» ON?  → записати в 01_ОРИГІНАЛИ без артефактів,
                                без Vision, без метаданих
              «без OCR» OFF? → OCR через ocrService (планка) + артефакти у 02
   • (нарізка/склейка) → існуючі труби, БЕЗ вшитого стиснення
```

- Спільний сценарій `addFiles` — **один модуль** (напр. `src/services/addFiles/`),
  чиста DI-фабрика (як `createActions`). **І модалка, і DP** його кличуть.
- Жодного `createDocumentPipeline`, жодного `ingest`-фасаду, жодного спільного
  `runOptionsRef` між «додати» і «нарізкою».
