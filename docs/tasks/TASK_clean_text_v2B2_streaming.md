## МЕТА ФАЗИ

AI-режими (Чистий/Конспект) **течуть у в'ювер по мірі генерації** — адвокат бачить текст, що
наростає, а не порожній спінер 2-4 хв.

**Видимий результат:** натиснув «Згенерувати» → текст починає з'являтися майже одразу і наростає
до повного; відчуття як у claude.ai.

---

## ДВІ ОПЦІЇ РЕАЛІЗАЦІЇ (виконавець обирає після аналізу)

**Опція A — справжній токен-стрім (бажано, «як claude.ai»):**
- API-виклик у режимі `stream:true` → читати SSE (`response.body` reader) → акумулювати дельти →
  `onDelta(partialText)` у в'ювер → інкрементальний рендер.
- Для пачок: кожна пачка стрімить; в'ювер **дописує** (пачка1 + пачка2 + …).

**Опція B — прогресивно по ПАЧКАХ (простіший фолбек):**
- Без SSE. `cleanDocument` емітить **кожну готову пачку** одразу (а не чекає всі) → в'ювер дописує.
- Багатосторінкові показують прогрес (шматки з'являються); односторінкові — все одно одним блоком.

> Рекомендація: спершу оцінити складність A; якщо ризик великий — B (вона вже близька до наявного
> пачкового циклу + onProgress). Узгодити з адвокатом перед кодом, якщо обираєш B.

---

## КРИТИЧНІ ПРОЕКТНІ ТОЧКИ (для Опції A)

1. **OPT-IN, не ламати інших.** `callAPIWithRetry` спільний (triage та ін.). Стрім — ОКРЕМА гілка/
   функція (напр. `callAPIStreaming` або опція `{ stream:true, onDelta }`), вмикається ТІЛЬКИ
   clean_text. Дефолтний нестрімовий шлях для решти — без змін.
2. **Idle-timeout замість total-timeout.** Наш `requestTimeoutMs` абортить за ЗАГАЛЬНИМ часом —
   під час стріму це вб'є довгу (але активну!) генерацію. Для стріму: таймаут **скидається на кожній
   дельті** (idle-timeout, напр. 60с без токенів = abort). Поки токени течуть — не абортимо.
3. **Ретрай зі стрімом.** Якщо стрім обірвався на півдорозі — **відкинути часткове, повторити пачку**
   з нуля (не «склеювати» половинки). Зберегти backoff як у callAPIWithRetry.
4. **Throttled рендер.** НЕ ре-рендерити весь Markdown на кожен токен (перф). Дроселювати (напр.
   оновлення раз на ~150-300мс або на N токенів). MarkdownRenderer на проміжному тексті — ок
   (незакриті `==`/`**` тимчасово — припустимо під час стріму; фінальний рендер коректний).
5. **Долі артефактів — без змін.** Зберігаємо `.md` лише при ПОВНОМУ успіху (як V2-A2). Стрім — лише
   показ під час генерації; фінал зберігається тим самим шляхом.
6. **Стрім тече в DI/споживача.** `onDelta` прокидається через ядро (`cleanDocument`→`polishToMarkdown`)
   → ACTION/в'ювер. Тести стабують стрім.

## СКЛАДОВІ

### V2-B2.1 — стрімовий API-шлях
- `callAPIStreaming` (або `callAPIWithRetry(..., {stream:true, onDelta})`) — `stream:true`, читання SSE,
  акумуляція `content_block_delta` → `onDelta(chunkText, accumulated)`. Повертає фінальний текст+usage
  (як нестрімовий). Idle-timeout (скид на дельті). Ретрай пачки на обрив. **Лише для clean_text** (opt-in).

### V2-B2.2 — ядро прокидає стрім
- `polishToMarkdown` приймає `onDelta`; коли є — кличе стрімовий шлях, емітить дельти.
- `cleanDocument`: для кожної пачки стрімить; акумулює між пачками; емітить у консумера (через `onProgress`/новий `onStreamDelta`).
- C7-логування (usage) — як було (на пачку); стрім не міняє білінг.

### V2-B2.3 — в'ювер: інкрементальний показ
- Генерація (handleGenerate / onGenerateVariant) передає `onDelta` → стан «стрімить» → показ
  наростаючого тексту в тілі таба (throttled рендер). По завершенні — звичайний показ збереженого `.md`.
- Прогрес «Очищаю пачку i/N» лишається як індикатор поверх (або зникає на користь живого тексту).
