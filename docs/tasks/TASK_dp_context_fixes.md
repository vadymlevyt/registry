## МЕТА

Реальне тестування виявило що контекст-генератор **псує розуміння справи агентом**:
у `case_context.md` нараховано 89-91 документ замість реальних 43, з «45 помилками».
А для **фото** (image-merge) тумблер «Оновити case_context» взагалі мовчки не працює.
Цей TASK лагодить контекст-зону цілком (5 пунктів), щоб агент бачив правду і оновлення
працювало на обох шляхах (нарізка PDF + склейка фото).

**Усі 5 пунктів — в одній зоні коду** (`services/contextGenerator.js` + event-wiring DP),
яку ця сесія щойно будувала в TASK 2. Зона image-merge editor (#1/#9/#4) — **НЕ цей TASK**,
окрема сесія.

  (канонічний SSOT), не сканування папок Drive (тягне артефакти/дублі).
- **AI-first** — агент отримує правдивий контекст; оновлення працює на всіх шляхах DP.
- **Однозначність (#11)** — «кількість документів» = рівно canonical documents, одне значення.
- **Додавати, не переписувати** — правимо джерело збору + додаємо публікацію події; промпт-форму
  не ламаємо (лише додаємо дату+час).

## СКЛАДОВІ

### #7 🔴 — джерело документів = реєстр, не folder-scan (КОМІТ 1, першим)
**Файл:** `src/components/CaseDossier/services/contextGenerator.js`.
- Замінити збір через `collectFromFolder(01)+collectFromFolder(02)` на побудову списку
  з **`caseData.documents`** (канонічний SSOT). Кожен документ → `{ id: driveId, name,
  subFolders }` для `ocrService.extractTextBatch`.
- Текст береться як зараз — через `ocrService` (кеш `.txt` за `driveId`, fallback OCR;
  `pdfjsLocal` витягує текстовий шар searchable-PDF). Тобто **зберігаємо** властивість
  «джерело = документи, не лише .txt» (text-layer PDF не губиться).
- Результат: лік = `caseData.documents.length` (43), нуль `.layout.json`, нуль дублів
  01/02, нуль layout-помилок.
- **Зачіпає обидва шляхи** (Огляд + DP) — це одна функція. Це **корекція коректності**
  (лік міняється на правильний) — оновити тести під реальний лік.
- Якщо у `caseData.documents` бракує якогось driveId — пропустити з warning у stats
  (не вигадувати).

### #5 🔴 — фото-шлях публікує подію (КОМІТ 2)
**Файл:** `src/components/DocumentProcessorV2/index.jsx` (`handleImageMergeSubmit`).
- Після успішного `add_documents` для image-merge — **публікувати** `DOCUMENT_BATCH_PROCESSED`
  тим самим `eventBus`/`topics`, з payload `{ caseId, updateCaseContext: settings.updateCaseContext === true, documents: [...] }` (форма як у `emitStage`, `documentPipeline.js:338-348`).
- Тоді слухач CaseDossier (TASK 2) спрацьовує однаково для фото → оновлення + сигнал.
- Перевірити що поле `updateCaseContext` береться зі стану тумблера тієї ж DP-сесії.

### #2 🟡 — архів контексту в DP-тригері (КОМІТ 3 разом з #3/#6, або в #5)
- Переконатись що `generateCaseContext` **сам** архівує попередній `case_context.md`
  (як в Огляді) — і DP-тригер кличе той самий шлях. Якщо архівація була в Огляд-обгортці
  поза сервісом — перенести у сервіс, щоб обидва шляхи архівували.
- Перевірити на PDF-шляху (де контекст ран) чи архів пишеться; якщо ні — полагодити в сервісі.

### #3 🟡 — чіткіша сигналізація фонового оновлення
**Файл:** `CaseDossier/index.jsx` (слухач DP-події).
- Зробити сигнал помітним і завершеним: при старті — toast «Оновлюю нарис справи…»
  (вже є), при завершенні — чіткий «✓ Нарис справи оновлено» (а не той самий що «оброблено»).
- Не дублювати з DP-тостом «Оброблено N документів» — це різні події (нарізка ≠ контекст).
- Якщо генерація йде довго — лишити видимий індикатор (не зникає мовчки).

### #6 🟢 — дата+час у нарисі
**Файл:** `services/contextGenerator.js` (`buildCaseMetadata`/`fillSystemPrompt`) і/або промпт.
- У `case_context.md` писати **дату І час** генерації (зараз лише дата) — щоб відрізняти
  створення від оновлення і відстежувати зміни.
- Не змінювати структуру/розділи промпту — лише поле дати розширити до дати+часу.
