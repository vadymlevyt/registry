## 1. МЕТА І ФІЛОСОФІЯ

### 1.1 Що це і навіщо
Document Processor — найбільший і найскладніший модуль системи. Його стан **рухається**
(паралельні сесії), документація **місцями розходиться з кодом** (підтверджено 2026-06-14:
роадмап казав «CONVERT/EXTRACT порожні» — насправді живі; «детектор опису тому» нібито є —
насправді зник). Перш ніж робити нові спеки доопрацювання, потрібен **єдиний звірений з кодом
знімок реального стану** — по ВСІХ сценаріях DP, не лише по нарізці.

### 1.2 Діагностика-перш, без змін коду
Цей TASK **нічого не виправляє і не рефакторить**. Він тільки **читає, звіряє і фіксує**.
Будь-яка знахідка → у звіт, НЕ у патч. Виправлення — окремі майбутні спеки, що народяться
з цих звітів. Це свідома межа: спершу карта, потім дороги (рішення власника 2026-06-14 —
«боюсь, що DP розуміється надто загально, щоб писати спеки»).

### 1.3 Лінза продакшн / SaaS / серверна архітектура
Кінцева ціль DP — піти у SaaS **після переїзду всієї системи на серверну архітектуру**.
Тому КОЖНА ділянка аудиту оцінюється ще й під цією лінзою: що зараз зав'язане на браузер
(Web Workers, pdf-lib у клієнті, Drive-OAuth з клієнта, виклики Document AI/Anthropic з
клієнта, localStorage-кеші) і що з цього **мусить переїхати на сервер** при SaaS-міграції.
Це не «переписати зараз», а **інвентаризувати блокери міграції** для майбутнього планування.

### 1.4 НАСКРІЗНА КІНЦЕВА МЕТА — чистий, стрункий код без зайвого
Окрім усіх заявлених завдань, аудит служить одній наскрізній цілі (рішення власника
2026-06-14): **довести код Document Processor до стану, де він чіткий, стрункий і де немає
нічого зайвого.** DP роками нарощувався поколіннями (склейка, нарізка, паспорт, Triage,
кілька поколінь пошуку меж), і в коді співіснують **живі шляхи, дрімаючі заготовки і мертві
рудименти** одночасно. Кінцевий продукт аудиту — не просто опис стану, а **карта того, що
прибрати**, щоб лишилось рівно те, що працює і потрібне.

Тому аудит ЯВНО полює на «зайве» як на окремий клас знахідок (не побіжно, а системно — §3.4):
дублі, надлишкові покоління, осиротілі функції, тіні-дефолти, мертві гілки, claim'и
універсальності, що не реалізуються, паралельні шляхи що роблять одне. Кожна така знахідка
отримує вирок **живий / дрімає-приберегти / мертве-видалити** з доказами `file:line` хто
реально кличе в проді. Конкретний приклад (старий аналізатор меж, борг #66) — лише ОДИН
екземпляр цього класу; завдання аудиту — знайти ВСІ подібні.

**Що виконавець зробить (нічого не змінюючи в коді — лише читає і фіксує).** (1) Намалює

## 2. ЗВІРЕНА КАРТА ПОТОЧНОГО ПОТОКУ НАРІЗКИ (baseline)

Складена 2026-06-14 трьома незалежними проходами по коду (чанки/tmp/OCR · паспорт ·
межі/різ/збереження) і затверджена власником як вихідна точка. Аудит починається з
**перевірки цієї карти на актуальність** і добудовує аналогічні для інших сценаріїв.

```
╔═══════════════════════════════════════════════════════════════════╗
║   ДОКУМЕНТ-ПРОЦЕСОР — ПОВНИЙ ПОТОЧНИЙ ПОТІК НАРІЗКИ (як є)         ║
╚═══════════════════════════════════════════════════════════════════╝

0. ВВІД
   Адвокат кидає у DP один/кілька файлів. Тумблер режиму:
   «просто додати» (повз нарізку) АБО нарізка (повний потік нижче).
   [опц.] якщо ввімкнено стиснення — том стискається перед роботою.

1. ОРИГІНАЛ → ТИМЧАСОВА ТЕКА
   Кожен оригінал заливається у тимчасову теку на Drive:
       _temp/<справа>_<прогін>/orig_<файл>.pdf
   Після заливки оригінал викидається з пам'яті.

2. ЧАНКУВАННЯ (бо в Document AI ліміт сторінок)
   Том ріжеться на ЧАНКИ, вони теж лягають у tmp:
       _temp/<...>/chunk_<файл>_NNN.pdf
   • розмір чанка адаптивний: ~25 стор. за умовч., межі 5–40,
     під розмір файлу (~40 МБ) і вільну пам'ять.
   • ВЕРХНІЙ рівень нарізки (дисципліна пам'яті/tmp).
   • НИЖНІЙ рівень: усередині Document AI кожен чанк ще ріжеться
     по 15 стор. на запит (жорсткий ліміт Google: 15 стор. / 40 МБ).

3. ЧИТАННЯ (Document AI), чанк за чанком
   Байти ОДНОГО чанка → Document AI → ТЕКСТ + ЛАЙАУТ кожної сторінки
   (геометрія: заголовок, таблиці, печатки/підписи, орієнтація, формат
    + текст самої сторінки). Байти чанка одразу з пам'яті геть.
   Стан прогону пишеться у tmp після кожного чанка → resume.
   (У пам'яті лайаут повний; при ЗБЕРЕЖЕННІ викидаємо найважче —
    картинки сторінок і токени, ~7 МБ → ~400 КБ.)

4. ЗБІРКА
   Усі чанки прочитані → склеюємо в один текст файлу + посторінковий
   лайаут. На вихід на КОЖЕН файл: текст + лайаут по сторінках + к-ть стор.

5. ПАСПОРТ  ← ДЕТЕРМІНОВАНИЙ код, 0 токенів, без AI
   Стислий погляд згори: на кожну сторінку маркер + сигнали (заголовок,
   № у футері, скидання нумерації, орієнтація, формат, таблиці, печатка)
   + трохи тексту сторінки. ДВА ПРОФІЛІ щільності:
       • «ЩІЛЬНИЙ» — том ≤ 70 стор.: майже повний текст сторінок.
       • «ХУДИЙ»   — том > 70 стор.: лише краї (3 перші + 2 останні рядки).
   Перемикач — ЖОРСТКИЙ поріг 70 стор. (не плавний). ⟵ роадмап: зробити адаптивним.

6. ПОШУК МЕЖ (Triage)  ← єдиний AI-крок (Haiku)
   AI читає ПАСПОРТ → ПЛАН: список «віртуальних документів», кожен =
   з яких сторінок яких файлів складається (фрагменти) + спосіб збірки
   (взяти як є / вирізати / склеїти фото / зібрати з фрагментів / відкинути).
   Це лише ПРОПОЗИЦІЯ. Запобіжник: том ≥70 стор. + «весь том одним документом»
   = провал → СТОП «потрібна ручна нарізка».

7. ПІДГОТОВКА ТЕКСТУ (тонкий крок) — чіпляє текст+лайаут, НЕ чистить.

8. ПІДТВЕРДЖЕННЯ — зараз АВТОМАТИЧНЕ. Екрану правки ЩЕ НЕМА. ⟵ найбільша майбутня частина.

9. РІЗ + ЗБЕРЕЖЕННЯ (на КОЖЕН документ плану)
   • ріжемо/склеюємо сторінки PDF у воркері → готовий PDF-шматок
     (джерело з tmp; кожен файл парситься раз на всі діапазони).
   • лайаут ріжемо по сторінках документа — ДЕТЕРМІНОВАНО, без AI.
   • зберігаємо ДВА файли на документ:
         PDF-оригінал шматка → 01_ОРИГІНАЛИ
         JSON-лайаут         → 02_ОБРОБЛЕНІ
     (.txt НЕ пишемо — вірний текст з лайауту на вимогу.)
   • канонічний запис документа в реєстр (через штатний шар: аудит/білінг/права).
   • зайві сторінки → 03_ФРАГМЕНТИ.

10. ПРИБИРАННЯ
    Успіх → tmp-теку прогону повністю видаляємо.
    Збій/скасування → tmp НЕ чистимо (resume). Авто-GC старих tmp — НЕМА.

   ✅ РЕЗУЛЬТАТ: був том у tmp → N документів, кожен = PDF (01) + лайаут (02) + запис у реєстрі.
```

## 3. СКОУП АУДИТУ

### 3.1 Чотири сценарії DP (кожен — окремий самодостатній шлях)
Для КОЖНОГО — побудувати таку саму карту «від вводу до результату» (як §2), звірену з кодом:

1. **НАРІЗКА сканованих PDF** — карта §2 є; перевірити її актуальність, знайти прогалини,
   зафіксувати реальні vs документовані розбіжності, мертві/живі станції диригента.
2. **СКЛЕЙКА фото** (image merge) — окремий шлях (не через pdf-нарізку): як фото
   групуються, сортуються, склеюються в PDF, де OCR, як зберігається результат.
3. **«ПРОСТО ДОДАТИ» файли** (add-files сервіс) — per-file додавання без нарізки:
   конвертація форматів, стиснення-фронт-крок, точка заливки, «без OCR» = без Vision.
4. **РОЗПАКУВАННЯ ZIP** (ЄСІТС-інгест) — фронт-крок перед add-files: розпак, відкидання
   підписів `.p7s`/`.sig`, маршрутизація вмісту, межа «архіви лише в DP».

### 3.2 Наскрізні теми (через усі чотири сценарії)
- **tmp / Drive lifecycle:** які теки/файли створюються, коли чистяться, осиротілі tmp,
  відсутність авто-GC; кирилиця в `q=` (правило #8); структура `_temp` (борг C6).
- **OCR провайдер-патерн:** Document AI / Claude Vision / pdfjsLocal — хто коли, fallback,
  кеш, imageless mode, layout-strip (image/tokens).
- **Білінг / ai_usage інструментація:** усі AI-точки через `callAgent` (B1) чи лишились
  ручні? Чи не дублюється облік? `billAsUserAction` для фону. Інструментація 25 точок.
- **Error handling:** ErrorBoundary на модулі; try/catch у async (правило #4 blank page);
  resume/cancel; поведінка при 401 Drive; помилки конвертації/OCR/Triage.
- **Канонічна схема документа:** усі точки створення через `createDocument()`; addedBy↔source;
  легкі vs важкі поля; що пишеться у 01/02/03/.metadata.
- **Мертвий код / претензія на універсальність:** генерик-«диригент», passthrough-станції
  (CLASSIFY/PROPOSE_METADATA), persistStage-тінь, осиротілий розпак (#57), діра `.doc`-як-PDF.
- **Лінія поколінь пошуку меж — КАНДИДАТИ НА ВИДАЛЕННЯ (явна перевірка):** пошук меж
  документів пережив кілька поколінь, усі досі в коді, але живе одне. Аудит мусить по
  КОЖНОМУ винести вирок «живий / дрімає-приберегти / мертве-видалити», з доказами `file:line`
  хто РЕАЛЬНО кличе в проді:
  - `documentBoundary/analyzeViaToolUse.js` (`analyzeBoundariesViaToolUse`, через фасад
    `documentBoundary/index.js` → `detectBoundaries`) — **СТАРИЙ**: шле весь PDF у модель як
    base64 Document Block (дорогі image-токени, модель `documentProcessor`). Попередня
    перевірка (2026-06-14): **живого виклику в потоці DP НЕ знайдено** — лишився фасад + тести,
    свідомо «приберігся на майбутнє як виконавець маршрутів». **Кандидат на видалення** —
    підтвердити аудитом, що жоден live-шлях його не кличе, і вирішити: лишити заготовкою чи знести.
  - `stages/detectBoundariesV2.js`, `stages/detectBoundariesV3.js` — проміжні покоління стадій;
    у живому Provider інжектиться `createTriageStage` (Ф2), не V2/V3. Перевірити, чи V2/V3 ще
    десь у проді, чи теж дрімають.
  - `stages/triageStage.js` → `analyzeTriageViaToolUse.js` (Smart Triage, паспорт-вхід, Haiku) —
    **ЖИВИЙ** поточний шлях. Baseline для звірки.
  Окремо: ці брати «дзеркалять структуру» — саме через це чек `data.error` розійшовся (B1 P2,
  закрито централізацією в `callAgent`); перевірити, чи нема ще такого дрейфу між поколіннями.
- **Тести:** покриття DP-шляхів у `tests/`; що НЕ покрито; де golden-фікстури можливі.
- **Серверна міграція (блокери):** Web Workers (pdf-lib split/merge у браузері), Drive-OAuth
  з клієнта, виклики Document AI/Anthropic з клієнта, localStorage-кеші, 3-tier agentHistory —
  що з цього прив'язане до браузера і потребує переїзду на сервер під SaaS.

### 3.3 Звірка з роадмапом (A1…A7)
Для кожного пункту роадмапу Вісі A — зафіксувати: що з нього **вже в коді**, що **частково**,
що **відсутнє** (як зниклий детектор опису тому). Окремо: чи точні твердження «чесного стану DP».

### 3.4 ПОЛЮВАННЯ НА ЗАЙВЕ (системно, не побіжно) — головний інструмент мети §1.4
Окремий наскрізний прохід по ВСЬОМУ коду DP (усі сценарії + диригент + стадії + сервіси +
воркери) із ЄДИНОЮ метою — знайти все зайве, що заважає коду бути чітким і струнким. Це не
підпункт «мертвого коду», а **повноцінна інвентаризація** з вироком по кожній знахідці. Шукати
системно за класами:

1. **Надлишкові покоління / суперседнуте.** Кілька реалізацій однієї роботи, де живе одне
   (приклад: лінія пошуку меж §3.2 / борг #66). По кожному поколінню — хто кличе в проді.
2. **Дублі та паралельні шляхи.** Дві+ функції/модулі, що роблять те саме різними словами;
   код, що «дзеркалить» іншого і дрейфує (як `data.error` B1 P2). Звести або видалити.
3. **Осиротілі функції/файли/експорти.** Визначені, але ніким не імпортовані/не викликані в
   проді (лише тести або взагалі ніхто). `grep` по кожному експорту → є live-споживач?
4. **Тіні-дефолти і недосяжні гілки.** Дефолт, що завжди перекритий (як `persistStage`);
   `if`-гілки, умова яких ніколи не істинна; `else` після гарантованого return.
5. **Claim'и універсальності, що не реалізуються.** «Приймаю будь-що», порожні passthrough-
   станції (CLASSIFY/PROPOSE_METADATA), параметри/прапори, які ніхто не передає інакше дефолту.
6. **Залишки після реверту/міграцій.** Код, згаданий у коментарях як «приберегли на майбутнє»,
   але без споживача роками; закоментовані блоки; TODO без власника; `_v2/_v3/_old/_legacy`-суфікси.
7. **Інструментація-дублі.** Подвійний облік (ai_usage/activityTracker) у двох місцях на одну
   подію; логування, що ніхто не читає.

Для кожної знахідки: вирок **живий / дрімає-приберегти (з явним тригером) / мертве-видалити**,
докази `file:line` хто кличе в проді, і — для «видалити» — оцінка ризику (тести, що тримають,
приховані споживачі). Результат — пріоритезований **«список на схуднення»** DP (що саме знести,
що звести, що лишити), готовий як вхід для майбутніх cleanup-спек і боргів.

### 3.5 ДОДАТКОВІ ТЕМИ АУДИТУ (доповнення адмін-сесії 2026-06-15)
З контексту реструктуризації roadmap і наскрізного розбору DP. Розширюють §3.1-§3.4,
не заміняють. Усі — read-only діагностика, з доказами `file:line`.

**A. Наскрізне методологічне правило — вирок по станціях ПО-ШЛЯХОВО, не глобально.**
Кожен вирок про станцію диригента (жива / passthrough / мертва) виноситься ОКРЕМО для
кожного з 4 сценаріїв (§3.1): одна станція може бути passthrough в одному шляху і живою
в іншому. Підстава: попереднє узагальнення «CONVERT/EXTRACT порожні» було хибним саме бо
глобальне — насправді `convertStage` passthrough на стрім-шляху нарізки (файли вже PDF у
tmp), але жива на «просто додати»/device-файлах (`convertStage`→`convertToPdf` для
`item.raw`). Жодного глобального «жива/порожня» без розбивки по шляхах.

**B. Вхідний роутер / диспетчер сценаріїв (шов МІЖ 4 сценаріями — зараз падає між §3.1.1-4).**
Аудитувати ЛОГІКУ ВИБОРУ, який сценарій запуститься. Точка: `DocumentProcessorV2/index.jsx`
`startProcessing` — «двері»: `isAllImagesInput()`→склейка; `skipPdfSlicing`→«просто додати»;
мікс фото+PDF→toast-завернути (#27); інше→нарізка. Для КОЖНОЇ дверей: що ловить і що
ПРОПУСКАЄ. Відома діра (підтвердити `file:line`): завертання спрацьовує лише на суміш З
ФОТО (`hasAnyImage && hasAnyNonImage`); суміш PDF+DOCX воріт НЕ має → не-PDF (DOCX/RTF)
проїжджає на стрім-нарізку, що чекає лише PDF → ймовірний злам. Зафіксувати точну умову і наслідок.

**C. Якість пошуку меж — діагностика (не лише ХТО шукає §3.2, а НАСКІЛЬКИ добре; фундамент A4).**
Виміряти й зафіксувати:
- утилізація контекстного вікна паспортом на малому/середньому/великому томі — скільки
  токенів займає паспорт vs ліміт вікна моделі. Гіпотеза до перевірки: на ~250 стор. худий
  паспорт займає малу частку вікна (вікно НЕ вузьке місце — є запас на щільніший паспорт);
- запобіжник виродженого плану (`triageStage.js` `isDegeneratePlan`): ТОЧНА умова
  спрацювання + що ПРОПУСКАЄ. Відомо (підтвердити): ловить лише «рівно 1 документ покриває
  100% сторінок при ≥порогу»; НЕ ловить «підозріло мало документів на великий том»;
- поріг щільності паспорта: підтвердити жорсткий поріг (rich ≤N / худий >N), значення N,
  квантувати обрив (head/tail рядки і символи rich vs худий);
- резолв моделі для Triage: який `agentType`, що повертає `resolveModel`, чи ДОСЯЖНИЙ
  Sonnet/Opus для великих томів через ієрархію (чи лише Haiku).

**D. Гранулярність контролю + готовність до per-документного «важеля» (під A7/§7.3, борг #56).**
Зафіксувати: `ocrMode`/стиснення — на весь батч чи по документу? Чи існує ПЕРСИСТЕНТНЕ
поле документа «рівень участі в контексті» (повний/вижимка/не включати), чи це лише
рантайм/інференс/`isKey`? Інвентар готовності під A7 (екран правки) і §7.3 («важіль» = ДВА
ортогональні значення: стан розпізнавання + рівень участі, правило #11).

**E. Точний B2-інвентар міграції AI-точок (B1 щойно приземлився).**
B1 переніс на `callAgent` лише Triage. Скласти перелік КОЖНОЇ AI-точки DP і суміжних
(App.jsx, Dashboard, contextGenerator, claudeVision, фото-агенти) з класифікацією: `на
callAgent` / `прямий fetch (=робота B2)` / `чат-агент (свідомий виняток)`. Готовий список
для B2 (борг #55 Частина Б). Заодно перевірити дубль-облік (§3.2 білінг).

**F. «Перевірено лише моками» — як пріоритет для e2e (зв'язок з §5).**
Клас багів #20 (Phase B) / DP-4: юніт-тести зелені, реальний Provider — катастрофа.
Позначити поведінки DP, перевірені ТІЛЬКИ моками і ніколи наскрізно через реальний
Provider — саме вони пріоритет для Playwright-e2e.

**G. Відомі екземпляри «зайвого/дрейфу» для вироку (поряд з #66, додати до §3.4).**
Борг #38 (ДВІ системи прогрес-індикації — job-based `jobProgressStore` vs component-based
`ProcessingProgress`; #11 — одна модель прогресу); борг #44 (застарілий
`aiUsageService.MODEL_PRICING` → `estimatedCostUSD` в `ai_usage[]` неточний).

**H. Легкий профіль вартості по сценаріях (під SaaS-економіку).**
Для кожного сценарію — які моделі задіяні і порядок токенів на типовий том (груба оцінка,
не бенчмарк). Врахувати, що #44 робить поточні `estimatedCostUSD` хибними. Вхід для
SaaS-планування вартості.
