## МЕТА

У кожної справи є `case_context.md` — стислий «нарис справи» (сторони, суть, статус,
ключові документи), який читають AI-агенти, щоб **знати справу** при розмові з адвокатом.

Зараз його генерація:
- **працює тільки в одному місці** — вкладка «Огляд» CaseDossier, кнопка «Створити
  контекст», логіка **зашита inline** у компоненті (`index.jsx:972-~1236`, ~260 рядків);
- **тумблер «Оновити case_context.md» у DP — порожній** (є в UI, нікуди не підключений).

TASK 2 робить нарис справи **єдиним і робочим у двох місцях**:
1. Виносить генерацію у **спільний сервіс** `CaseDossier/services/contextGenerator.js`.
2. Вкладка «Огляд» тягне звідти (поведінка ідентична).
3. DP-тумблер стає **робочим**: після обробки документів нарис справи оновлюється автоматично.

Так DP і досьє починають «годувати» одне одного: обробив документи → система сама
освіжила розуміння справи для агентів.

## PHILOSOPHY CHECK

- **Single Source of Truth / Rule of Three** — генерація context живе в ОДНОМУ місці,
  два споживачі (Огляд + DP) тягнуть звідти. Третій споживач (DP) активує винос.
- **AI-first** — нарис справи годує агентів; точка де DP підсилює досьє.
- **Додавати, не переписувати** — Огляд лишається behavior-preserving; DP отримує
  робочий тумблер через ту саму функцію.
- **Однозначність (#11)** — сервіс має чисту відповідальність (згенерувати нарис),
  UI-стан (повідомлення/лоадер) лишається в компоненті.
- **Тести разом з кодом** — винесений сервіс покривається unit-тестом.

## SEMANTIC CLARITY CHECK (#11)

**`contextGenerator.js`** (новий сервіс):
> Чиста(ішa) функція генерації `case_context.md`: приймає документи/нотатки справи +
> залежності (OCR, AI-виклик, Drive, прогрес-callback), повертає згенерований текст
> нарису і зберігає на Drive. НЕ тримає React-стан. Відповідає ТІЛЬКИ за «сформувати
> і зберегти нарис». UI-повідомлення/лоадер — у компоненті-споживачі.

Не змішувати в сервісі генерацію з UI-станом — інакше він не reusable для DP.

**Тумблер `updateCaseContext`** (вже існує, сенс уточнити на місці):
> `updateCaseContext` — після обробки документів у DP **перегенерувати** `case_context.md`
> справи. Одне рішення: оновити нарис із урахуванням нових документів.

## РІШЕННЯ АДВОКАТА (узгоджено 2026-05-29)

1. **Тригер DP → перегенерація = ВАРІАНТ A (через подію).** DP після persist публікує
   `DOCUMENT_BATCH_PROCESSED` (вже публікує). CaseDossier слухає (wiring є, 3021) і
   **якщо в payload `updateCaseContext === true`** для поточної справи — викликає
   `contextGenerator`. DP **не лізе** в нутрощі досьє (слабке звʼязування, брати-споживачі).
   Потрібно: додати `updateCaseContext` у payload події з DP-налаштувань.
   Варіант B (прямий імпорт DP→сервіс) — відхилено (звʼязування вбік).

2. **Регенерація — ПОВНА (не інкрементальна).** Нарис щоразу будується з **повного
   поточного набору документів справи** (DP додає нові через `add_documents` ПЕРЕД
   генерацією → нові вже всередині). Покриває обидва кейси: немає файлу → створює; є →
   перезаписує свіжим. Окремої логіки злиття «старий контекст + лише нові» НЕ робити.
   Інкрементальне оновлення — **майбутній кандидат** (борг, див. нижче), не цей TASK.

3. **Одна сесія / один деплой у кінці.** Внутрішньо можна два коміти (винос → підключення)
   для чистоти історії, але **push/деплой — ОДИН**, наприкінці. Адвокат тестує руками
   обидва боки (Огляд + DP-тумблер) за один деплой.

4. **Джерело тексту — критично (наслідок 1C):** генератор бере **самі документи** з
   `01_ОРИГІНАЛИ`/`02_ОБРОБЛЕНІ` (виключаючи `.txt`) і проганяє через
   `ocrService.extractTextBatch`. OCR сам витягує текстовий шар searchable-PDF
   (`pdfjsLocal`), а `.txt` — лише кеш. **НЕ перемикати на «читати тільки .txt»** —
   інакше text-layer PDF (яким 1C не пише `.txt`) випадуть з контексту. Зберегти
   поточну логіку «документи + extractTextBatch» один-в-один.
