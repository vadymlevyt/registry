## КОНТЕКСТ — що показав повний тест

Реальний прогін новим промптом (0.4.1+0.4.2) на ПОВНОМУ кабінеті виявив проблему:

**Симптом:** імена/по-батькові в результуючому JSON **галюциновані** (Бабенко «Олександра Іванівна» в кабінеті → «Олег Іванович» у JSON), час засідань зник (`null`), номери проваджень зникли.

**Першопричина:** промпт містив рядок *"Якщо справа без засідань 2026 — все одно включи її (з порожнім hearings)"*. Через це Claude:
1. Не міг відсіяти неактивні справи (завершені, спрощені без виклику)
2. Заходив і повністю обробляв КОЖНУ справу що пройшла фільтр року+ролі (22 справи замість ~10 активних)
3. Контекст переповнювався
4. Спрацьовував compacting
5. Деталі справ оброблених раніше **розмивались → галюцинації** при фіналізації

**Висновок:** зайвий обхід неактивних справ → переповнення → галюцинації. Прибравши зайві справи, прибираємо першопричину.

**Підтвердження:** перший прогін (старий промпт, уперся в ліміт на 14 справах, менший обсяг) дав ЧИСТІ імена. Другий (повний обхід 22 справ) — галюцинації. Різниця — обсяг обробленого.

## ПРАВИЛЬНА ЛОГІКА ФІЛЬТРА

Справа потрапляє у `cases[]` якщо виконуються ВСІ умови:

1. **Рік у номері справи** — 25 або 26 (без змін з 0.4.1)
2. **Конкретна роль** — не голе "Представник" (без змін з 0.4.1)
3. **НОВЕ — активність за засіданнями:**
   - Рік справи **26** → беремо **ЗАВЖДИ** (свіжа справа, засідання можуть ще не бути призначені)
   - Рік справи **25** → беремо **ТІЛЬКИ якщо є хоча б одне засідання 2026** (будь-яка інстанція — перша, апеляція, касація; будь-яке провадження)

Якщо справа 25 року без засідань 2026 → у `skipped[]`, НЕ в `cases[]`.

**Інстанція не має значення.** Засідання 2026 в будь-якому провадженні справи 25 року робить її релевантною.

### Зміна 2 — Оновити КРОК 4 (засідання) логікою активності

Перебудувати порядок обробки справи що пройшла фільтр року+ролі:

```
═══════════════════════════════════════════════════════════════
КРОК 4 — Засідання 2026 та рішення про активність справи
═══════════════════════════════════════════════════════════════

Зайшовши у справу, СПОЧАТКУ визнач чи вона активна:

ЯКЩО рік справи = 26 (другий сегмент номера):
  Справа СВІЖА — беремо ЗАВЖДИ, навіть якщо засідань ще немає.
  Збери засідання 2026 якщо є (може бути порожньо).

ЯКЩО рік справи = 25:
  Треба перевірити чи є засідання 2026.
  - Спочатку глянь "Внесення дат слухання" (структурована дата,
    один extract). Якщо немає — глянь повістки.
  - Є хоча б одне засідання 2026 (будь-яка інстанція/провадження)
    → справа АКТИВНА, обробляй повністю.
  - НЕМАЄ жодного засідання 2026 → справа неактивна (завершена,
    спрощене провадження без виклику, або стара).
    Додай у skipped: { case_no, reason: "справа 2025 без засідань
    2026 — неактивна" }. НЕ витрачай кроки на повні реквізити,
    переходь до наступної справи.

Для АКТИВНИХ справ далі збирай засідання 2026:

Йди по хронології документів ВПЕРЕД. Спочатку "Внесення дат
слухання", потім повістки якщо треба.

Для КОЖНОГО документа про засідання:
1. Витягни дату, час, зал, провадження
2. Перевір чи дата вже є у списку для цієї справи
3. Нова — додай. Вже є — ПРОПУСТИ (дублікат повістки)

ПРИКЛАД дедуплікації:
- Документ #1: повістка на 1 квітня → додай
- Документ #2: повістка на 1 квітня → ПРОПУСТИ (вже є)
- Документ #3: повістка на 7 квітня → додай

Беремо ТІЛЬКИ засідання 2026 (минулі і майбутні в межах року).
Засідання інших років — пропусти.

Поля засідання: date (ISO), time, court, hearingRoom,
proceedingNumber, cabinetUrl, noticeType.
```

### Зміна 3 — Порядок витягування реквізитів

Зараз КРОК 3 (реквізити) йде ПЕРЕД засіданнями. Але для справ 25 року без засідань ми не хочемо витрачати кроки на реквізити. Тому:

**Перебудувати порядок для справ 25 року:**
1. Зайшов у справу
2. Швидко глянув засідання (Внесення дат / повістки)
3. Нема 2026 і рік 25 → skipped одразу (реквізити НЕ витягуємо)
4. Є 2026 (або рік 26) → тепер витягуємо повні реквізити + засідання

Для справ 26 року — реквізити витягуємо завжди (свіжа береться без перевірки засідань).

Сформулюй це в промпті природно — спочатку швидка перевірка активності, потім повна обробка тільки активних.
