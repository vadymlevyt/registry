## КОНТЕКСТ

Мікро-TASK 1 показав чесною реакцією тоста що `ocrService.writeCache` мовчки не записує кеш на Drive. Адвокат натискає "Розпізнати зараз" → текст витягається → toast.warning "Текст розпізнано, але не вдалось зберегти кеш". Підтверджено експериментально на справі Кісельова, документ "Ухвала".

## ПРИЧИНА (з diagnostic_document_lifecycle_v2.md розділ 7.1)

`ocrService.js` функція `listFolderFilesByName` (рядки 45-53) використовує `q= name='${name}'` де `name` містить кирилицю (наприклад `"701_1413_25 кисельова ухвала_<driveId>.txt"`). Це порушення правила #8.

Через ненадійність кирилиці в `q=` filter Drive API:
- `checkCache` не знаходить існуючий кеш — Viewer показує empty state навіть коли файл є
- `writeCache` не може коректно перевірити чи кеш уже існує перед записом

## ПРАВИЛЬНИЙ ПАТЕРН

Вже використовується в `driveService.js:381-388` функції `deleteOcrCacheForDocument`:

```javascript
// q= БЕЗ name filter (тільки folder + trashed)
const q = `'${folderId}' in parents and trashed=false`;
const res = await driveRequest(...);
// фільтрація по name відбувається в JavaScript
const file = data.files.find(f => f.name === cacheFileName);
```

Тобто Drive API дає всі файли папки, JavaScript знаходить потрібний по точному імені. Працює надійно з будь-якими символами включно з кирилицею.

   - Якщо documentNature невідомий — як визначає що робити?
