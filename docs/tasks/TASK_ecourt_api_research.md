**Дата:** 2026-04-28

**Виконавець:** засновник особисто (адвокатська і ділова робота)

## КОНТЕКСТ

ДП «Центр судових сервісів» має офіційне **АРІ ЕС** v1.9 — REST API для ЄСІТС.

**Еволюція розуміння в обговореннях:**

1. Перша гіпотеза — використати АРІ для основної практики. Помилка.
2. Виправлення — АРІ розраховане на юр.осіб які **самі є** стороною (банки, корпорації). Адвокатські кабінети ЄСІТС реєструються на РНОКПП фіз.особи. Соло-адвокатура поза АРІ.
3. **Особисте відкриття при дослідженні інтерфейсу** — адвокати об'єднання/бюро можуть **підключати свої справи до кабінету юр.особи**. Це робить АРІ ЕС придатним для адвокатських об'єднань і бюро через корпоративний кабінет.

**Стратегія — двоканальна:**
- **Канал 1 — Court Sync через RPA** для масового ринку (соло, малі бюро) — окремий ARCHITECTURE
- **Канал 2 — АРІ ЕС офіційно** для enterprise (середні і великі АО, юрфірми)

Цей TASK — для Каналу 2.

## ЕТАП 1 — ДОМАШНЯ РОБОТА (без зовнішніх контактів)

Перш ніж писати в ДП ЦСС — дослідити інтерфейс ЄСІТС як юр.особа.

### 1.1 Перевірити кабінет юр.особи

`cabinet.court.gov.ua` — чи є кабінет на ЄДРПОУ 40434074 (АБ Левицького). Якщо ні — пройти реєстрацію (це окрема процедура від реєстрації фізособи).

### 1.2 Дослідити функції корпоративного кабінету

- Чи є вкладка «справи де представником є наші адвокати»?
- Чи можна **підключити** справу адвоката-фізособи до кабінету юр.особи? Через що — окрема дія, ордер, заява?
- Чи всі справи автоматично «бачні» юр.особі коли адвокат — її співробітник?
- Чи є розділ управління співробітниками-адвокатами?
- Чи є модуль довіреностей всередині кабінету юр.особи?

### 1.3 Скріни ключових екранів

В папці `_research/ecourt_corp_cabinet/` на Drive. Доказова база для ДП ЦСС і архітектури.

### 1.4 Список конкретних питань

На основі дослідження. Орієнтовно:

1. Як адвокат підключає справу до кабінету юр.особи?
2. Чи отримує юр.особа через АРІ доступ до підключених справ?
3. Чи можна програмно подавати документи через АРІ з підписом конкретного адвоката?
4. Як працює модуль довіреностей в АРІ?
5. Чи можна тестово підключитись без договору?
6. Скільки коштує підключення тестового і промислового?
7. Чи є SLA, технічна підтримка?

## ЕТАП 3 — ЩО ДАЄ АРІ ЕС ЧОГО НЕ ДАЄ RPA

| Функція | RPA (Канал 1) | АРІ ЕС (Канал 2) |
|---|---|---|
| Сесія КЕП | в браузері адвоката | на сервері юр.особи |
| Моніторинг | при активності адвоката | 24/7 фоновий |
| Залежність від UI ЄСІТС | висока | відсутня |
| Програмна подача документів | складно | через API |
| Електронні довіреності | вручну | модуль PoA |
| Аудит дій | обмежений | повний |
| Швидкість великих обсягів | хвилини | мілісекунди |
| SLA від ДП ЦСС | відсутній | гарантований |

### UI відмінності для enterprise

- **Центральний моніторинг** — керівник бачить все без логіну адвокатів
- **Програмна подача** — кнопка «Подати в суд через АРІ» в досьє
- **Модуль довіреностей** — створення/скасування/перевірка електронних довіреностей
- **Аудит** — повний журнал дій всіх адвокатів
- **API для клієнтів** — клієнтський портал з реальними даними з АРІ

### ROI для enterprise

Юр.фірма з 30 адвокатами і 200 справами:
- **Без Legal BMS Enterprise:** ~30 годин/тиждень адмінроботи + ризики пропуску → ~200-300 тис. грн/міс «втрачених»
- **З Legal BMS Enterprise за $1500/міс ($60К грн):** ~5 годин/тиждень → економія ~150-200 тис. грн/міс
- **Net benefit:** ~120-150 тис. грн/міс
- **Окупність:** один-два місяці

## РИЗИКИ

### Ризик 1 — ДП ЦСС може відмовити

«АРІ ЕС не для адвокатської спільноти». Тоді:
- Чекаємо появи API для адвокатів-фізосіб
- Працюємо тільки з RPA (Канал 1)
- Тиснемо ринок щоб ДП ЦСС зрозумів попит

### Ризик 2 — Ціна може бути захмарною

Якщо $5000 за тестове + $20000 річна підтримка — блокер для нашого розміру. Тоді:
- Партнер-юрфірма яка покриє витрати
- Або відкладаємо

### Ризик 3 — Технічна складність Hawk Auth

Hawk + HMAC-SHA256 нестандартне. У браузерному JS треба бібліотеки. Серверна інтеграція (Node.js) — стандартні криптомодулі.

### Ризик 4 — Legal BMS зараз single-tenant

Перш ніж enterprise — перебудувати:
- Поле `tenant_id` всюди
- Ізоляція даних
- Credentials per tenant
- RBAC в межах tenant

Це **окремий великий sprint**.
