## СКЛАДОВІ

### 3.2.1 — Кнопка «Очистити тексти» в Огляді (retroactive, N док.)
- Поряд зі «Створити контекст».
- Сканує документи справи; **фільтр** (parent скоуп): тільки `scanned` з `textFormat!=='md'`.
  Пропускає `searchable` і вже-`.md`.
- Цикл `cleanDocument` по черзі, прогрес «Чищу N з M» (дорого — N AI-викликів).
- ResultCard: очищено N, пропущено M (searchable/вже-md), помилок K, згруповані `attentionNotes`.
- UI-стан у компоненті; логіка — ядро. `billAsUserAction:true` (дія адвоката).

### 3.2.2 — Кнопка «Очистити документ» у Viewer (один)
- На панелі DocumentViewer (header/footer), активна лише для `scanned`.
- Один `cleanDocument`. Після — viewer перечитує і показує свіжий `.md` (3.1 дав `getCleanOrRawText`).
- `attentionNotes` показати поряд (плашка/toast).

### 3.2.3 — ACTION `clean_document_text` (AI-first)
- `actionsRegistry.js`: `clean_document_text({caseId, documentId})` → handler кличе `cleanDocument`.
  `audit:false` (не критична дія). `billAsUserAction:true`.
- PERMISSIONS: `dossier_agent` отримує `clean_document_text`.
- Промпт агента досьє: додати в список tools «`clean_document_text` — очистити текст
  скан-документа у гарний Markdown (тільки scanned)».
- Так адвокат: «очисти цей документ» / «почисти всі тексти справи» (агент ітерує).

### 3.2.4 — Підсвітки уваги + чип/панель у в'ювері (узгоджений дизайн)

**Звідки беруться підсвітки (контракт із ядром 3.1):** AI-поліш обгортає підозрілі фрази
маркером `==текст==` прямо в `.md` (зміст усередині — дослівно, це форматна позначка, не
зміна тексту). `MarkdownRenderer` рендерить `==x==` → `<mark class="attention" data-mark="N">x</mark>`
(порядковий `data-mark` за порядком у тексті). Причини зберігаються в `attentionNotes`
(`documents_extended.json`) — **без номера сторінки** (пачкування робить його ненадійним;
прив'язка — суто за порядком мітки в документі).

**Єдиний вхід — чип у в'ювері, лише в режимі ТЕКСТ.** Мітки — властивість текстового
рендеру, тому в режимі **Скан** (оригінал) чипа НЕМАЄ; перемкнувся на **Текст** — з'являється
(якщо `markCount > 0`). Умова рендеру: `viewMode === 'text' && markCount > 0`.

```
┌─ в'ювер, режим ТЕКСТ (хедер/футер, біля «Очистити текст») ─┐
│  [⚠ 5 поміток]   ← ЄДИНИЙ елемент; у режимі Скан відсутній  │
└────────────────────────────────────────────────────────────┘
        │ тап
        ▼
┌─ панель поміток ───────────────────────────────────────────┐
│  [✓] Підсвічувати в тексті        ← вкл/викл (CSS, миттєво)  │
│  ───────────────────────────────────────────────────────── │
│  1. «сума в розписці…»   — звір з оригіналом            →   │  ← клік = скрол+пульс
│  2. «дата 12.03.2024…»   — можлива OCR-помилка          →   │
│  3. …                                                        │
│  ───────────────────────────────────────────────────────── │
│                 [ Зняти всі назавжди ]                       │  ← стрип `==` з .md, re-save
└──────────────────────────────────────────────────────────────┘
```

- **Мердж кнопок:** один вхід-чип, усередині панелі — і перемикач показу, і список, і «зняти
  всі». На тулбарі рівно одна штука (НЕ дві окремі кнопки toggle+delete).
- **Клік-навігація:** пункт списку ↔ `data-mark` за **порядком** (1↔1). Клік →
  `scrollIntoView({behavior:'smooth', block:'center'})` + тимчасовий клас «пульс». Без номерів
  сторінок. Ми володіємо рендером (`.md` ≠ Mozilla-PDF-в'ювер) — тривіально.
- **«Зняти всі назавжди»:** стрипнути всі `==` з `.md`, re-save через `cleanDocument`-шви
  (saveMarkdown), очистити `attentionNotes` в extended → чип зникає.
- **Дані «і пізніше»:** чип читає мітки з `.md` + причини з `extended.attentionNotes` (обидва
  persisted у 3.1) → відкрив документ хоч через місяць, список знову живий. Нічого додатково не зберігаємо.
- **Картка результату** після кнопки (3.2.1/3.2.2) показує підсумок і **веде в той самий
  в'ювер-чип** — НЕ дублює список окремим UI.
- **Колір:** жовтий/янтарний = «увага» (НЕ зелений — той читається як «ок»).

## СПІЛЬНІСТЬ ДИЗАЙНУ (ОБОВ'ЯЗКОВО — наскрізне для 3.2 і 3.3)

Біль із минулого: CSS/стилі розповзлись по модулях, кожен тягне своє → морока зводити.
**Робити одразу спільним, не «потім рознесемо».** Для ВСЬОГО нового UI 3.2 (чип, панель,
`<mark>`, плашки, кнопки) і 3.3 (компонент вибору):

- **CSS / кольори / відступи / радіуси / тіні** — ТІЛЬКИ через наявні **design-токени**
  (CSS-перемінні проєкту, напр. `--text3` тощо). НЕ хардкодити hex/px по місцю, НЕ заводити
  нову колір-палітру. Колір підсвітки — новий **токен** (напр. `--attention-bg`), не літерал.
- **Шрифти** — наявні системні/проєктні, без нових сімей.
- **Компоненти** — якщо елемент повторюється (чип, плашка, кнопка-дія, список-панель) —
  **спільний компонент** у `components/shared/` (або наявному спільному місці), НЕ локальна копія
  в модулі. Компонент вибору 3.3 — той самий принцип (parent §Вісь B).
- **Логіка/функції** — нуль дублювання: парсинг міток, стрип `==`, scroll-to-mark — спільні
  хелпери, переюзні і кнопками, і агентом (борг #47), і майбутнім редактором.
- **Перевірка перед комітом:** жодного нового `.css`-острова з власними літералами кольору/розміру;
  все веде до спільних токенів/компонентів. Якщо токена бракує — додати **в спільне місце**, не локально.

> Правило-орієнтир (#11 + Rule of Three): якщо стиль/елемент/функція з'являється у 2+ місцях —
> він спільний з народження. Це economy зведення майбутніх фаз і узгодженого вигляду.
