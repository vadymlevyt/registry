## PREMISE (доменний факт — обґрунтовує агресивний downscale)

Фото в системі — це **викидна сировина**, НЕ оригінал. Адвокат тягне фото з пристрою/Drive
лише щоб система зробила з них **PDF + OCR-текст**; після цього фото більше не потрібні
(лишаються там, звідки прийшли — то відповідальність користувача, не системи). «Оригінал»
для Legal BMS — це **оброблений PDF** у папці `01_ОРИГІНАЛИ`. Справжні оригінали (папір /
цифрові з е-підписом) — поза Legal BMS, для них інше місце зберігання.

**Наслідок:** берегти повну роздільність фото немає причин. Downscale до роздільності,
достатньої для OCR і читабельного PDF, нічого цінного не втрачає.

## ЧАСТИНА 1 — Downscale фото (головна, найбільший виграш)

### Механізм (НЕ стиснення, а зменшення пікселів)
Фото з телефона ~12 МП (4032×3024). Для читання документа достатньо **~2000–2500px по
довшій стороні** (≈3 МП). Зменшення роздільності = викидаємо ~75–80% непотрібних пікселів:
- файл: 2.5 МБ → ~300–600 КБ (5–8×);
- **розпакований растр у RAM** (= W×H×4 байти, головний тиск на пам'ять): 48 МБ → ~12 МБ (~4×).

### Куди (ОДНА спільна точка — правило #30)
`src/services/imageDocument/prepareImagesForMerge.js` — спільна phase-1 для **і** модалки
(`ImageMergePanel`), **і** DP image-merge. Зараз 3 кроки: HEIC→JPEG → OCR → orientation.
Вставити **крок 1.5 «downscale»** одразу ПІСЛЯ HEIC→JPEG і ПЕРЕД OCR:
- замінити `normalizedFiles[i]` на зменшений blob → і OCR, і подальша збірка PDF беруть
  уже легку версію (один шов, обидва споживачі виграють).

### Як (нативний Canvas, ~20 рядків, без саморобного стискача)
Новий спільний util, напр. `src/services/imageDocument/downscaleImage.js`:
```
downscaleImage(blob, { maxDim = 2400, quality = 0.82 }) → Promise<Blob>
```
- decode → намалювати в `<canvas>` зменшеного розміру (`drawImage`) → `canvas.toBlob('image/jpeg', quality)`.
- **Guard за РОЗДІЛЬНІСТЮ (пікселі), НЕ за вагою файлу (КБ):** якщо фото вже ≤ maxDim по
  обох сторонах — повернути як є (no-op, БЕЗ перекодування → нуль втрати якості/CPU).
  Причина: тиск на пам'ять = W×H×4 байти, незалежно від ваги JPEG. Перевірка за КБ хибна —
  «легкий» (600 КБ) але hi-res (4032×3024) кадр у RAM = ~48 МБ і його ТРЕБА зменшити;
  а реально мала за пікселями картинка (1500×1100) пропускається, навіть якщо файл «важкий».
- (опц. запобіжник) якщо зменшений blob раптом ≥ оригіналу — лишити оригінал.
- `maxDim`/`quality` — **константи у файлі** (не UI), легко тюнити.

### КРИТИЧНИЙ нюанс — EXIF-орієнтація (ризик регресії)
Перемальовування в canvas **скидає EXIF-orientation**. Якщо браузер авто-обертав фото за
EXIF, після redraw воно може «лягти боком». Рішення: при малюванні **врахувати EXIF-поворот**
(вивести upright-пікселі, без залежності від EXIF-тега), щоб подальший крок orientation
(`prepareImagesForMerge` крок 3 / `orientationCorrector`) працював на однорідній базі.
**Обов'язково тест на орієнтацію** (фото з EXIF-rotation → після downscale upright, не боком).

### Якість / OCR
~2000–2500px по довшій стороні (≈170 DPI для A4) — Document AI обробляє без проблем, текст
читабельний. Цільову роздільність **звірити на реальному скані** перед фіксацією константи
(не опускати надто низько — текст має лишатись чітким). Фінальний PDF — у цій роздільності
(прийнятно, бо фото = сировина, не оригінал; див. PREMISE).

### Швидкість
Downscale одного фото — десятки мс (нативний Canvas), на 5–10 фото <1с; вішається на крок,
що вже декодує картинку → майже задарма. Усе downstream (OCR-аплоад, crop/поворот/прев'ю)
стає **швидшим** на менших даних. Чистий виграш по швидкості, не вартість.

## ЧАСТИНА 2 — Блоб-гігієна (доповнення)

### Проблема
`URL.createObjectURL(blob)` тримає дані в пам'яті, **поки адресу не відкликано**
(`URL.revokeObjectURL`). Браузер сам не звільняє. Невідкликані URL накопичуються між фото →
пам'ять росте.

### Завдання
Аудит **усіх** `createObjectURL` у тракті image-editor / DP / модалки → переконатись, що
кожен має парний `revokeObjectURL` при заміні / видаленні / unmount.
- `usePreviewUrls` (борг B1) — preview-URL вже коректно відкликає (зразок дисципліни).
- Перевірити решту: `thumbUrlsRef` у `DpImageMergeEditor`, мініатюри модалки, будь-які
  інші `createObjectURL` (grep по `src/components/`).

### Як перевірити
`git grep -n "createObjectURL" src/components/` → для кожного знайти парний `revokeObjectURL`
(на swap і на unmount). Де нема — додати. Тест/перевірка: відкликання при unmount.

### Зв'язок із Частиною 1
Downscale робить кожен «бак» малим; блоб-гігієна не дає старим бакам стояти повними. Разом:
малий розмір кожного × відсутність накопичення = пам'ять тримається низько.

## НЕ робити (межі)
- ❌ Стиснення PDF (перепак вже стисненого → 3–5%; для чужих готових PDF — окремо, сервіс типу iLovePDF). Для НАШИХ документів downscale фото ДО збірки → PDF народжується малим, проблема знімається сама.
- ❌ Web Worker (оптимізація #4) — відкладена (найімовірніше переїде на сервер; не робимо завчасно).
- ❌ Контекст-генератор, інші модулі — не чіпати.
- ❌ Не чіпати справжню логіку OCR/групування/сортування — лише вставити downscale + гігієну.

## SAAS implications
Downscale-перед-обробкою — клієнтський, tenant-нейтральний; при майбутньому серверному
аплоаді зменшує bandwidth + навантаження бекенда (цінний і там, не викидна робота).
Блоб-гігієна — пер-сесійна пам'ять браузера, tenant-нейтральна.

## BILLING implications
Менші зображення → менші OCR-payload'и в Document AI → потенційно нижча OCR/AI-вартість.
Нових точок інструментації не треба; наявне C7-логування не змінюється.
