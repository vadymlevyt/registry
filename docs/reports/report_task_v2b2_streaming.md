## ОБРАНА ОПЦІЯ — «Спосіб C» (роздільник) + справжній токен-стрім (рішення архітектора)

Замість двох опцій спеки (A — JSON-парсер часткового виходу; B — прогрес по пачках) **архітектор обрав третій, елегантніший шлях** і він реалізований:

**Корінь проблеми.** Промти Чистий/Конспект повертали **JSON-обгортку** `{markdown, attentionNotes}`. Справжній токен-стрім (Опція A) тоді стрімить JSON-токени — щоб показати «як claude.ai», довелося б інкрементально витягати поле `markdown` з НЕПОВНОГО JSON (буферизація escape/UTF-8 на межах SSE-шматків). Це і є «найскладніша частина», якою спека лякала.

**Спосіб C прибирає проблему в корені.** Змінено **лише ФОРМАТ ВИХОДУ** промтів (не правила вірності):
- Модель віддає **спершу голий Markdown** (з інлайн-мітками `==…==` як у V2-C),
- потім окремий рядок-роздільник **`---ПОМІТКИ---`**,
- потім **JSON-масив** поміток `[{ "note": "…" }]`.

Тоді стрім **тривіальний**: під час SSE просто акумулюємо текст і показуємо все **до роздільника** як готовий Markdown — JSON-хвіст ніколи не потрапляє на екран. Жодного інкрементального парсингу JSON.

**Долі артефактів і правила вірності НЕ змінені.** `.md` зберігається лише при ПОВНОМУ успіху (як V2-A2); `.layout`/`.txt` цілі; обрив → джерела недоторкані, нічого не збережено. Залізні правила Чистого (rules 1–9) у промті лишились дослівно — переписано **тільки секцію «ФОРМАТ ВІДПОВІДІ»**.

**Парсер — graceful degrade.** `parsePolishOutput`: роздільника нема → **весь вихід = markdown, нотатки = []**. Тобто навіть якщо модель забуде роздільник, вірність тексту збережена (лише без чипів уваги) — це і є вбудований фолбек, що знімає головний ризик Способу C.

**Фолбек A (на майбутнє, НЕ реалізовано).** Якщо роздільник на практиці виявиться ненадійним (модель ставить `---ПОМІТКИ---` всередині контенту / систематично пропускає), повертаємось на JSON-обгортку + інкрементальний парсер поля `markdown` з часткового JSON (з буферизацією неповного escape на межі SSE). Поки в цьому немає потреби — graceful degrade Способу C безпечний.

## ЩО ЗМІНЕНО (по файлах)

### 1. `src/services/toolUseRunner.js` — стрімовий шлях (OPT-IN)
- **`createSSEAccumulator()`** — ЧИСТИЙ парсер Anthropic SSE (без I/O). `.push(chunkStr)` склеює рядки, акумулює `text_delta` у текст, ловить `usage`/`stop_reason`. Виокремлено як чисту функцію → юніт-тест годує шматки з довільними межами (подія розрізана між читаннями).
- **`callAPIStreaming(params, options)`** — ОКРЕМА від `callAPIWithRetry` функція (нестрімовий шлях інших споживачів НЕ зачеплено):
  - `body` отримує `stream:true`, відповідь читається як SSE через акумулятор;
  - **idle-timeout замість total-timeout**: лічильник скидається на КОЖНІЙ дельті (`resetIdle()` у циклі читання). Поки токени течуть — не абортимо; `idleTimeoutMs` тиші (дефолт 60с) = `controller.abort()` → транзитивно → backoff+retry;
  - **ретрай пачки з нуля при обриві**: новий акумулятор щоспроби → часткове відкидається, половинки не склеюються; backoff/429/5xx/401/400 — дзеркало `callAPIWithRetry`;
  - повертає **той самий shape** `{ content:[{type:'text',text}], usage, stop_reason }` → downstream-парсинг ідентичний.

### 2. `src/services/cleanTextService.js` — формат C + прокидання стріму
- Константа **`ATTENTION_SEPARATOR = '---ПОМІТКИ---'`** + `STREAM_IDLE_TIMEOUT_MS = 60000`.
- Промти: секція виходу `buildVerbatimPrompt`/`buildDigestPrompt` переписана на Спосіб C (правила вірності 1–9 — дослівно як були). Digest → голий Markdown без роздільника/JSON.
- **`extractJsonArray` + `parsePolishOutput`** — розбір `markdown [+ роздільник + JSON-масив]`; роздільника нема → весь текст = markdown, notes=[]. Толерантно приймає й хвіст `{attentionNotes:[…]}`.
- **`streamingMarkdownView(accumulated)`** (export) — показ ДО роздільника; гасить ХВОСТОВИЙ частковий роздільник (поріг ≥ «`---П`», щоб плоский `---` між сторінками НЕ обрізався).
- `polishToMarkdown` — нові DI-шви `onStreamDelta` + `callAIStream`. `onStreamDelta` функція → стрімовий шлях (idle-timeout, емітить `streamingMarkdownView`); інакше нестрімовий `callAI` (total-timeout) як раніше. Парсинг виходу — через `parsePolishOutput` (JSON-обгортка прибрана).
- `cleanDocument` — нові DI-шви `onStreamDelta` + `callAIStream`. Емітить markdown **цілого документа що наростає**: `completedPrefix` (готові пачки) + жива пачка (через `PAGE_SEP`). `completedPrefix` оновлюється у циклі після кожної завершеної пачки. Halving-retry на truncation теж стрімить (мінорний візуальний «стрибок» лише на рідкісному обрізанні — фінал коректний).

### 3. `src/services/actionsRegistry.js` — ACTION прокидає стрім
- `clean_document_text` приймає `onStreamDelta` (UI-ін'єкція у params, НЕ в tool-схемі — модель функцію не передає) і прокидає у `cleanDocument`. Агентний/tool_use виклик → `onStreamDelta` відсутній → нестрімово.

### 4. `src/components/CaseDossier/index.jsx` — обгортка
- `handleGenerateVariant(doc, mode, onStreamDelta)` — прокидає `onStreamDelta` у params ACTION (лише коли є).

### 5. `src/components/DocumentViewer/index.jsx` + `DocumentViewerContent.jsx` — інкрементальний показ
- Стан `streamingText` + **throttled** емітер (`STREAM_RENDER_THROTTLE_MS = 200мс`): `pushStreamDelta` тримає latest у ref, планує оновлення стану не частіше за інтервал → MarkdownRenderer НЕ ре-рендериться на кожен токен. Скид на старті/завершенні генерації, зміні документа, демонтажі.
- `handleGenerate` передає `pushStreamDelta` третім аргументом у `onGenerateVariant`.
- `VariantContent` (режим generating): є стрім-текст → показ markdown що наростає + нейтральна плашка «{режим} генерується…»; токенів ще нема → спінер «Очищаю…». Підсвітки/чип (V2-C) — лише на ФІНАЛЬНОМУ тексті (ready), не під час стріму.
- CSS: `--streaming` модифікатор плашки (нейтральний колір, лише design-токени).
