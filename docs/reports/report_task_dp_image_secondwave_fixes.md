### #11 🔴 (головний) — ⚠️ ДІАГНОЗ РАДНИКА НЕ ПІДТВЕРДИВСЯ → знайдено інший корінь
Гіпотеза радника: race спільного `previewUrlsToRevokeRef` + `setTimeout(1000)` у
ефекті прев'ю **сітки** ревокує живий URL.

Чому НЕ тримається (статичний аналіз): URL потрапляє в чергу ревокації рівно в
момент, коли його заміняють (`setPreviewUrls(prev => { штовхаємо урли prev; повертаємо
newUrls })`) — у чергу завжди йде **попередній** набір, який цим оновленням
перестає бути показаним. Плюс `cancelled`-гард не дає скасованому запуску
закомітити. Затримка 1с гарантує paint. **Показаний у сітці URL у цю чергу не
потрапляє ніколи.** Схема некрасива (лік пам'яті, orphan-таймери), але не ревокує
живий URL і не пояснює симптоми.

**Справжній корінь (підтверджено адвокатом+радником, гіпотезу сітки знято):**
ефект `displayUrl` у `PreviewPopup` (рядки 125-192) ревокує свій per-run
`createdUrl` у cleanup. Натиск ↻ (`handleRotateInPopup`) ставить
`popupRotationLockRef=true`; ефект перезапускається, але **рано виходить** (lock) —
не створює новий blob і не міняє `displayUrl`. При цьому cleanup попереднього
запуску вже ревокнув `createdUrl`, на який `displayUrl` досі вказує → `<img>`/cropper
тримає мертвий blob URL → чорний/сире-перевернуте. «Нестабільно» = браузер інколи
тримає декодований кадр (тоді «прокатує»), інколи ні.

За STOP-умовою спинився і доповів; адвокат+радник підтвердили мій діагноз
(сильніший за версію про чергу сітки) і дали Варіант 1: **reproduce-first** →
детермінований падаючий тест → мінімальний фікс. Сітку не чіпати (не корінь;
переписування ризикує регресом).

## 2. #11 — корінь і механізм усунення (reproduce-first)

**Падаючий тест-сторож (спершу):** `tests/unit/previewPopupRotateRevoke.test.jsx`
— рендерить `PreviewPopup`, чекає поки ефект згенерує об'єктний URL, тисне ↻
(lock-гілка), і перевіряє що активний `displayUrl` **не ревокнутий**. До фіксу
падав детерміновано: `expected ['blob:obj-0'] to not include 'blob:obj-0'`
(активний URL ревокувався). Без fake DOM-таймінгу — ревокація синхронна у commit'і.

**Фікс (мінімальний, у спільному `PreviewPopup.jsx`):** володіння об'єктним URL
винесено у `ownedDisplayUrlRef`. Попередній власний URL ревокується **лише коли
його реально замінили** новим `displayUrl` (а не у per-run cleanup ефекту).
Cleanup тепер тільки ставить `cancelled=true`. Додано окремий unmount-ефект для
фінальної ревокації власного URL.

**Чому стабільно:** активний показаний URL більше ніколи не ревокується
«наввипередки» — ні на lock-гілці (рано виходить, нічого не чіпає), ні при
скасованому запуску (`cancelled` не дає закомітити/ревокнути). Ревокується тільки
вже-замінений (раніше показаний) URL, який браузер уже декодував.

**Спільне джерело:** `PreviewPopup` — той самий компонент для модалки
(`PreviewView`) і DP (`DpImageMergeEditor`). Один фікс лагодить обидва (правило #30).

Сітку (`previewUrls` effect + `previewUrlsToRevokeRef`) свідомо **не чіпав** — не
корінь, переписування дало б ризик без користі (філософія «додавати, не
переписувати»).

## 3. #12 — нова логіка (поважання вибору)

Нова спільна чиста функція `src/services/imageDocument/duplicateSelection.js` —
`selectRecommendedDuplicateRemovals(duplicates, { dismissedGroupIds, isMemberPresent })`.
Повертає Set індексів до видалення, застосовуючи AI-рекомендацію **тільки до
незайманих** груп. Група пропускається повністю якщо:
- вона у `dismissedGroupIds` (розгрупована «це не дублікати»); АБО
- адвокат уже вручну торкнувся групи — будь-який член відсутній серед активних
  (`isMemberPresent(idx)===false`). Покриває обидва ручні сценарії: «видалив
  зелений, лишив жовтий» і «видалив частину вручну».

Обидва споживачі кличуть ту саму функцію зі своїм предикатом присутності:
- **Модалка:** `isMemberPresent = orderedIndices.has(idx)` (видалені прибрані з
  `orderedIndices`).
- **DP:** `isMemberPresent = groups.flatMap(pageIndices).has(idx)` (видалені
  прибрані з `pageIndices`).

Жодного дублювання логіки — один файл, два предикати.

## 4. #10 — як додано рамку

DP-сітка (`GroupSection`) тепер розкладає `pageIndices` на сегменти
(`buildDuplicateSegments`): суміжний run членів одного активного дубль-`groupId`
(≥2 поруч) стає обрамленим сегментом, який рендериться у **тих самих спільних
класах** `.image-merge-panel__dup-group` / `-header` / `-label` / `-dismiss` /
`-body`, що й модалка. Поодинокі/розділені члени лишаються звичайними картками.

Картки всередині рамки отримують `inGroup=true` (прокинуто через
`DpSortableItem` → `RenderItem` single-гілку → `Thumbnail`): прибирається власна
жовта рамка картки (вона тепер на рамці групи), зелена для recommended лишається —
як у модалці. Зміна `RenderItem` зворотно-сумісна (модалкові single-елементи
`item.inGroup` не ставлять → `false`, поведінка не змінюється).

DnD не зачеплено: кожна картка лишається власним `useSortable` (рамка — лише
візуальна обгортка). CSS: `.dp-image-merge-editor__dup-group { grid-column: 1/-1 }`
— рамка займає весь рядок auto-fill сітки.
