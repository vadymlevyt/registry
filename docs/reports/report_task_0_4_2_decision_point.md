## 1. ЩО ЗМІНЕНО

### Зміна 1 — Блок 3 «КРИМІНАЛЬНІ СПРАВИ З КІЛЬКОМА ОБВИНУВАЧЕНИМИ»

Замість тихого `warnings: "...Уточніть кого захищає..."` тепер активна
позначка справи як такої що **ПОТРЕБУЄ РІШЕННЯ АДВОКАТА** + явна
інструкція **НЕ зупиняти обхід** заради одного питання. Поведінка:

- `НЕ вгадуй прізвище` (без змін)
- `Засідання все одно витягни` (без змін)
- `primaryParty тимчасово = case_no` (раніше: «залиш = case_no»)
- Новий пункт: «ПОЗНАЧ цю справу як таку що ПОТРЕБУЄ РІШЕННЯ АДВОКАТА (запам'ятай: case_no, суд, список обвинувачених)»
- Новий пункт: «Додай питання у список для ТОЧКИ РІШЕННЯ»
- Завершальна інструкція: «НЕ зупиняй обхід заради цього питання — продовжуй далі. Усі неоднозначності збираються в одну точку…»

### Зміна 2 — Новий блок «ТОЧКА РІШЕННЯ АДВОКАТА»

Доданий між блоками «ПРОМІЖНІ РЕЗУЛЬТАТИ» і «СТРУКТУРА ENVELOPE».
Структура:

- Принцип: працюй до кінця обходу АБО до фінальної точки рішення.
- Перевірка наприкінці: чи накопичились питання.
- Гілка «питань НЕМАЄ» → одразу `[ФІНАЛЬНИЙ РЕЗУЛЬТАТ]`.
- Гілка «питання Є» → НЕ видавай фінальний JSON, спочатку подай адвокату
  СПИСОК усіх питань ОДНИМ блоком, ЧЕКАЙ відповідь.
- Шаблон питання з прикладом відповіді (`1 — [Прізвище 1]` / `обох`).
- Обробка відповіді:
  - звичайна → `primaryParty = обраний підзахисний` у форматі «Прізвище І.П.»
  - `«обох»` → `primaryParty = перший обвинувачений`, у `notes` справи
    додається «Захист обох обвинувачених: [список через кому]»
- Майбутньостійкість: «якщо з'являться інші типи неоднозначностей —
  вони так само збираються у цей список».

### Зміна 3 — Блок «ПОЧИНАЙ»

Додано один абзац-нагадування у кінець:

> Якщо натрапиш на справи що потребують рішення (кілька обвинувачених тощо)
> — не зупиняйся, збери їх. Перед фінальним JSON подай одним списком усі
> питання і дочекайся однієї відповіді — це ТОЧКА РІШЕННЯ.

### Решта без змін

Стиль роботи, фільтр ролей (Блок 1 з 0.4.1), мапінг ролей у primaryParty
(Блок 2), КРОК 3 (поля справи), КРОК 4 (засідання + дедуплікація),
ПРОМІЖНІ РЕЗУЛЬТАТИ (checkpoint кожні 5 справ), СТРУКТУРА ENVELOPE,
БЕЗПЕКА — усе з 0.4.1 збережено без правок.

### Хедер-коментар у `promptBuilder.js`

Додано короткий розділ TASK 0.4.2 (принцип точки рішення + причина
вибраного розміщення блоку). Експертна автономія: блок ТОЧКА РІШЕННЯ
розміщено **після** ПРОМІЖНІ РЕЗУЛЬТАТИ і **перед** СТРУКТУРА ENVELOPE,
а не «перед ПРОМІЖНІ» як був один з двох варіантів у специфікації.
Логіка: ПРОМІЖНІ РЕЗУЛЬТАТИ закінчується інструкцією видати
`[ФІНАЛЬНИЙ РЕЗУЛЬТАТ]`, а ТОЧКА РІШЕННЯ — це шлюз перед цим видачею.
Виходить природний порядок «чекпойнти → шлюз → формат JSON».

### 4.1. Розміщення блоку (експертна автономія)

### 4.2. Уточнення формулювань

Дрібні правки заради чіткості для Sonnet:

- «primaryParty залиш = case_no» → «primaryParty **тимчасово** = case_no»
  (підкреслює що це не фінальне значення, а маркер для виправлення).
- У переліку запам'ятати додано «**суд**» (поза case_no і списком
  обвинувачених — суд потрібен для тексту питання адвокату).
- Шаблон питання містить «[суд]» у дужках — щоб питання було повністю
  самодостатнє для адвоката який не пам'ятає всі справи напам'ять.
- У вказівці про відповідь «обох» уточнено формат `notes`-рядка:
  «Захист обох обвинувачених: [список через кому]».
- ПОЧИНАЙ: «дочекайся **однієї** відповіді» — підкреслює що адвокат
  відповідає одним блоком (а не по одному питанню).

### 4.3. Інших знахідок немає

Структури даних, ACTIONS, scenarioProcessor, envelope — без змін.
