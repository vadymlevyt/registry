## 1. ПОЯСНЕННЯ ДЛЯ ВЛАСНИКА (проста мова)

Document Processor — це перехрестя з **чотирма дорогами**: нарізати том, склеїти фото,
просто додати файли, розпакувати архів. Кожна має їхати окремо. Цей TASK прибрав до
чистоти **тільки одну дорогу — нарізку**, не торкаючись трьох інших.

Зроблено три речі:

1. **Знесли старі іржаві деталі (Частина B).** Пошук меж документів колись переписувався
   кілька разів; три старі покоління досі лежали мертвим вантажем. Прибрали — близько
   **800 рядків** коду, який жоден робочий шлях не викликав (доведено пошуком).

2. **Викинули муляжі універсальності (Частина D).** Нарізку обслуговувала стара
   «універсальна машина», яка вдавала, ніби вміє і Word конвертувати, і класифікувати, і
   метадані пропонувати — а насправді половина цих станцій були порожні. Тепер диригент —
   **чистий конвеєр нарізки**: прийняв PDF → прочитав → знайшов межі → порізав → розклав у
   справу. Крапка. Ще близько **200 рядків** геть.

3. **Поставили шлагбаум на в'їзді в нарізку (Частина A — ця частина).** Раніше, якщо в
   режимі нарізки кинути Word — машина намагалась порізати його як PDF і **тихо ламалась**.
   А цифровий PDF (із вже готовим текстом) марно ганявся через дороге розпізнавання. Тепер
   нарізка приймає **рівно один вид входу — сканований PDF** (сфотографовані/відскановані
   сторінки без текстового шару). Усе інше система відразу ввічливо завертає: «це інша
   дорога, увімкніть „Просто додати файли"». Жодних тихих зламів.

**Результат:** мінус понад тисяча рядків коду, нуль тихих зламів, і дорога «нарізка»
нарешті робить рівно одне. Це прибирання **перед ремонтом** — наступні роботи (якість
нарізки великих томів, метадані, екран правки) будуватимуться на чистому.

## 2. ЩО ЗРОБЛЕНО — ЧАСТИНА A (ворота входу нарізки)

Ця сесія виконала **Частину A** (Частини B і D зроблено раніше: коміти `bc8e675`, `e27d009`).

### Ворота — два рівні, ПЕРЕД `pipeline.run`

У `src/components/DocumentProcessorV2/index.jsx`, `startProcessing`, у гілці нарізки
(`!useAddAsIs`), одразу після наявного guard'а «мікс фото+PDF» і **перед** `setRunning(true)`:

1. **Синхронний рівень (за типом, нуль I/O):** якщо серед файлів нарізки є хоч один
   не-PDF → `toast.warning` + `return`:
   > «Нарізка приймає лише сканований PDF. Для Word / інших форматів увімкніть
   > „Просто додати файли"».

2. **Асинхронний рівень (текстовий шар):** для кожного PDF-кандидата — дешевий **peek
   перших ~3 сторінок** через pdf.js: рахуємо середню к-сть символів/стор. Якщо
   `avgChars >= 200` → це **цифровий PDF** (має текстовий шар) → `toast.warning` + `return`:
   > «Цей PDF має текстовий шар (цифровий) — він не для нарізки. Для додавання як є
   > увімкніть „Просто додати файли"».
   Якщо текстового шару нема (avgChars < 200) → **скан** → пускаємо у `pipeline.run` як раніше.

### Де живе логіка

Окремий слайс-скоупний хелпер — **`src/components/DocumentProcessorV2/sliceInputGate.js`**
(не роздуваємо `index.jsx` inline-логікою). Контракт:

```
sliceInputGate(files, { driveRequest, peekPdf? })
  → { allow, reason, message }
  reason ∈ 'ok' | 'non_pdf' | 'digital_pdf' | 'drive_auth' | 'gate_error'
```

`startProcessing` будує `gateFiles` зі `selected` + `inboxSelected` і викликає хелпер;
на `allow:false` показує відповідний toast і завертає (до `pipeline.run` не доходить).

### Поведінка fail-open / 401 (правила #4, #8)

- **FAIL-OPEN:** якщо peek кинув (битий PDF / не вдалось визначити) — **НЕ блокуємо**,
  пускаємо у нарізку + `console.warn` (краще пустити легітимний скан, ніж заблокувати том).
- **Байти для peek:** device-файл — з `raw` (File уже в пам'яті); Drive-source (лише
  `driveId`) — підкачуємо через `driveRequest(alt=media)`, peek парсить тільки перші
  сторінки (не подвоюємо повний прохід OCR).
- **ВИНЯТОК 401/403 Drive** під час підкачки → НЕ мовчазний крах, а
  `toast.error("Сесію Drive завершено — перепідключіться")` + `return` (правило #8).
- `peekPdf` ін'єктується (динамічний `import('pdfjs-dist')` за замовчуванням) — модуль не
  тягне pdfjs, поки peek не потрібен; тести підставляють фейк (детермінізм, без worker'а).
- Усе в try/catch; виклик у `startProcessing` обгорнуто остаточною fail-open сіткою.

## 6. §4.3 — НОТАТКА ПРО `ecitsInboxWatcher` (зафіксувати, НЕ виправляти тут)

Ворота Частини A стоять **лише в UI `startProcessing`** — на дорозі адвоката. Шлях
автоматичного інбокс-вотчера (`ecitsInboxWatcher.runPipeline` → `run()`, слайс-шлях через
`DocumentPipelineContext`) **воріт НЕ проходить** — він кличе диригент нарізки напряму, в
обхід `startProcessing`.

Це **свідома межа A1** (узгоджено спекою §4.3): ворота ставимо на ручний вхід адвоката, не
на watcher. Якщо ЄСІТС колись подасть у слайс-шлях не-PDF або цифровий PDF — це **питання
Вісі A6** (передача з е-суду), не A1. На сьогодні ЄСІТС подає у вотчер лише очікувані
скановані PDF, тож активної діри тут немає; зафіксовано як вхід для A6.
