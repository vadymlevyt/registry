## Що зроблено

Замінено крихкий **синхронний** тригер генератора `case_context.md` на **естафетну модель**
(без таймауту). Раніше подія `DOCUMENT_BATCH_PROCESSED` публікувалась синхронно після
`add_documents`, а слухач у CaseDossier читав ще-не-перерендерений `caseData.documents` →
генерація бачила старий список (тихий data-bug на непорожній справі, `NO_FILES` на порожній).

Тепер три такти:

1. **Слухач (3.2)** — лише **ставить** естафетну паличку `pendingContextRegen`
   `{ caseId, expectedDocIds, scenarioRunId }`, нічого не генерує. Нова подія перезатирає
   попередню (self-heal). Ручне додавання/перейменування/видалення не публікують подію → нарис
   не чіпають.
2. **Тригер (3.3)** — `useEffect` на `caseData.documents`: коли **всі** `expectedDocIds`
   реально приземлились у метадані → стартує генерація з повного (вже оновленого) SSOT.
3. **Фініш** — `runDpContextRegen` на завершенні (успіх **АБО** помилка) у `finally` сам знімає
   паличку (`setPendingContextRegen(null)`). **Жодного таймауту** — обробка може йти 10+ хв.

### Жорсткі межі (дотримано)

- ✅ Жодного таймауту в коді тригера.
- ✅ Падіння DP → publish не відбувається → паличка не ставиться → генерація не стартує, без
  фальшивих toast.
- ✅ Ручне додавання / перейменування / видалення НЕ тригерять регенерацію (звужено паличкою).
- ✅ Білінг не подвоєно: DP-шлях свідомо **НЕ** репортить `context_regenerated` (зафіксовано
  коментарем у `runDpContextRegen`); токени і так йдуть у `ai_usage[]` через `aiUsageSink`.
- ✅ Без schema bump.
