# ЧАСТИНА 2 — АРХІТЕКТУРНА КОНСУЛЬТАЦІЯ

## 1. Чому виникли три різні поведінки

### Симптом А — Витяг з ЄДР, Рішення суду — сирий текст без перемикача

**Це не регресія від мікро-TASK 5.** Уважно дивлячись на `DocumentViewer/index.jsx`: для документів з `documentNature='searchable'` `effectiveMode` завжди `'text'`, перемикач Скан/Текст ховається — це поведінка яка існувала до мікро-TASK 5 і існує зараз після revert.

Найімовірніша причина чому адвокат побачив зміну:
- Раніше ці документи мали `documentNature=null` (legacy v4 без класифікації) → `inferNatureFromFile` повертав null → `defaultNatureForUI` віддавав `'scanned'` за замовчуванням → перемикач був видимий, режим Скан показував iframe.
- На якомусь етапі (після виправлень мікро-TASK 3 чи 4 з `onReprocess` що коректує `documentNature`, або після фонової класифікації PDF) — `documentNature` цих документів став `'searchable'`. Це **правильна** класифікація (текстовий PDF), але побічний ефект — Viewer переключився на text-only режим.
- Адвокат тепер бачить «сирий текст» з 02_ОБРОБЛЕНІ через `<pre>{text}</pre>` у `TextContent`. Перемикача немає бо документ searchable. Це нова поведінка яка з'явилась поступово, з мікро-TASK 5 не пов'язана прямо.

**Дія:** не лікувати в мікро-TASK 5. Це питання UX рендеру searchable-режиму — окремий мікро-TASK (див. Опція В нижче).

### Симптом Б — Рахунок: pdf.js рендерить, але масштаб зламаний, ліва частина обрізана

**Це справжній баг мікро-TASK 5 у CSS.** У `PdfJsViewer.css` контейнер сторінок має:

```css
.pdfjs-viewer__pages {
  display: flex;
  flex-direction: column;
  align-items: center;
  overflow: auto;
}
```

Це класична CSS-помилка з flex + overflow. Коли елемент-сторінка ширший за контейнер (а A4 у scale=1 ≈ 595 px, тоді як на mobile viewport контейнер часто 360-430 px) — `align-items: center` зсуває elem'ент так що його лівий край опиняється **за межами scroll-області** і доступний тільки правий бік. Це візуально виглядає як "обрізана зліва" сторінка.

Друга помилка: scale `1.0` означає 100% від PDF natural size (як у Acrobat), а не fit-to-width. Адвокат натуально очікував "100% = вся ширина екрану" — це не так у PDF.js. Треба було обчислювати `initialScale = containerWidth / firstPageViewport.width` і брати його за дефолт.

**Дія:** при наступній спробі pdf.js — використати `align-items: flex-start` плюс inner wrapper з `margin: 0 auto` (центрує тільки коли вміщається), і обчислювати fit-to-width scale від `ResizeObserver` на контейнері.

### Симптом В — Ухвала додаткове рішення — працює правильно

Найімовірніше:
- Сторінки цього документа мають базовий розмір ≤ ширина контейнера (наприклад скан стиснутий до 400 px), тому `align-items: center` не викликає overflow зліва.
- Або користувач переглядав на ширшому екрані де A4 595 px вміщається.

Добре що працювало — підтверджує що **архітектура pdf.js+textLayer сама по собі робоча**, проблема в CSS layout та scale-стратегії, а не в принципі.

## 2. Чи pdf.js взагалі підходить

**Так, технічно підходить.** pdf.js — це той самий рушій який Mozilla використовує всередині Firefox. Це не «третя сторона з flaky API». Усі подібні reader-додатки (Notion, Linear, Confluence) використовують pdf.js під капотом.

**Але інтеграція нетривіальна:**

| Підводний камінь | Що ламається в наївній реалізації |
|------------------|------------------------------------|
| **Scale + container width** | "100%" в pdf.js = natural size, не fit-to-width. Треба `ResizeObserver` + перерахунок scale. |
| **Layout shift при lazy-render** | Без pre-fetch viewport-розмірів для всіх сторінок placeholder скаче коли реальний рендер замінює його. (У моїй версії я це частково зробив через pre-fetch, але pre-fetch блокує статус 'ready' на N запитах getPage.) |
| **HiDPI + canvas memory** | На Retina iPad канвас займає `width × height × 4 × 4` байт (output scale 2x). 30-сторінковий PDF легко з'їдає 500+ MB → OOM crash. Треба destroy canvas коли сторінка не у viewport. |
| **Шрифти** | pdf.js рендерить вбудовані шрифти. Якщо вони не embed (поширене у скан-PDF) — fallback на standard. Це може зрушити textLayer span'и відносно canvas, виділення стає неточним. |
| **Encrypted/password-protected PDF** | Без UI для пароля — `getDocument` зависає або повертає помилку. |
| **PDF з форм-полями (acroforms)** | textLayer їх не покриває, виділення працює на текст але не на поля. |
| **Touch selection на mobile** | iOS Safari потребує спеціальних touch handlers щоб long-press → виділення працювало. Просто `user-select: text` недостатньо. |
| **textLayer і transform CSS змінні** | Pdfjs пише `--total-scale-factor` на root container. Якщо забути встановити цю змінну — span'и наїжджають один на одного. |

Мій PdfJsViewer обробляв 4 з цих 8 пасток. 4 інші — лишилися дірками. Тому **симптом Б — закономірний**, а не один невдалий момент.

## 3. Альтернативи

### Опція А — react-pdf (`@react-pdf/renderer` — НЕ те, треба `react-pdf` від Wojtek Maj)

**Так:** wrapper над pdf.js з готовими хуками `<Document>`, `<Page>`, `<Outline>`. Обробляє `getDocument`, lazy-loading через React Suspense, fit-to-width через `width` prop.

| Плюс | Мінус |
|------|-------|
| ~70% наших edge cases вже вирішено | Додатковий ~30 KB в bundle |
| API стабільне (5+ років, 8K+ stars) | textLayer є, але selection styling треба додати самому (CSS) |
| Мобільна підтримка з коробки | Не вирішує проблему canvas memory на 100+ сторінкових PDF — все одно треба destroy logic |
| Активна підтримка | Залежить від pdfjs-dist версії, інколи лагає за upstream |

**Оцінка:** найреалістичніший шлях якщо хочемо canvas+textLayer. Економить ~70% часу на повторну спробу.

### Опція Б — PDF.js Viewer (full Mozilla viewer вбудований через iframe з нашого hosting)

Mozilla підтримує повний Viewer (зі sidebar, pagination, search, print) — це те що ви бачите коли Firefox відкриває PDF напряму. Можна задеплоїти його окремою папкою у `dist/pdfjs-viewer/` і вбудовувати через `<iframe src="/pdfjs-viewer/web/viewer.html?file=blob:...">`.

| Плюс | Мінус |
|------|-------|
| Найповніший UI — sidebar з thumbnails, пошук, друк, виділення | +1.5 MB у бандлі (Mozilla viewer великий) |
| Selection і copy працюють "з коробки" | iframe ізоляція — комунікація з нашим React через postMessage |
| Mozilla підтримує — оновлення безкоштовні | Стилі не наші, треба окремо дизайнити |
| Працює на мобільному (Firefox/Chrome тестують на iOS) | Складніше передати наш контекст (selectedDoc state, footer кнопки) |

**Оцінка:** good if we want lawyer to also get search, thumbnails, full page navigation. Overkill if тільки виділення тексту.

### Опція В — pdfobject + iframe Drive fallback

`pdfobject.js` — тонкий wrapper що генерує `<embed>` з PDF, який браузер рендерить нативно (Chrome PDF viewer, Safari PDFKit). Selection і copy працюють через нативний viewer.

| Плюс | Мінус |
|------|-------|
| 5 рядків коду, ~3 KB | На iPad Safari нативний viewer іноді кешується агресивно |
| Selection нативний, працює як Acrobat | Не контролюємо UI взагалі |
| Будь-який PDF — рендерить | Деякі мобільні браузери не підтримують `<embed>` — треба fallback |

**Оцінка:** найдешевше для "lawyer wants to copy text from PDF". Втрачаємо контроль над UI, але отримуємо фізично working selection. Гідна ставка.

### Опція Г — Конвертація PDF → HTML на сервері (pdf2htmlEX, Pandoc)

Серверна тулза що з PDF робить HTML з абсолютно позиціонованими div'ами + embedded fonts. На клієнті — просто `<iframe srcdoc=...>` або `<div dangerouslySetInnerHTML>`.

| Плюс | Мінус |
|------|-------|
| HTML — нативне виділення | Потрібен сервер (зараз його немає — ми клієнт-лише) |
| Format-preserving | Конвертер може спотворювати layout складних PDF |
| Один раз сконвертували — тримаємо на Drive | Кожен документ → +один файл у Drive |

**Оцінка:** відкладаємо до моменту коли матимемо backend (не зараз).

### Опція Д — Document AI metadata + HTML рендер ⭐ (окремий розгляд нижче)

### Опція Е — Сторонні AI API для PDF (Marker, Mistral OCR PDF, LlamaParse)

API які беруть PDF і повертають Markdown зі збереженням структури (заголовки, таблиці, abzац). Якісно, але платно ($0.01-0.05/сторінка).

| Плюс | Мінус |
|------|-------|
| Найкращий рендер для адвокатської роботи | Платно, треба бюджет |
| Markdown зручний для пошуку, агентів | Залежність від ще одного провайдера |

**Оцінка:** чудово якби бюджет дозволяв. Поки відкладаємо.

## 4. Окремий розгляд — Document AI metadata + HTML

**Це найбільш вигідний для адвокатської роботи варіант.** Ось чому:

### Що Document AI вже повертає

`processors/2cc453e438078154` (наш OCR процесор у `europe-west2`) повертає не тільки `text`, а й `pages[]` з:

```json
{
  "pages": [{
    "pageNumber": 1,
    "dimension": { "width": 595, "height": 842, "unit": "points" },
    "paragraphs": [{
      "layout": {
        "boundingPoly": {"vertices": [{"x":50,"y":80},...]},
        "textAnchor": {"textSegments": [{"startIndex":0,"endIndex":120}]}
      }
    }],
    "lines": [...],
    "tokens": [...],
    "blocks": [...],
    "tables": [...],
    "formFields": [...]
  }]
}
```

Зараз ми зберігаємо тільки `text` як plain `.txt` у `02_ОБРОБЛЕНІ` і викидаємо всю структуру.

### Що можна зробити

**Архітектура «метадата на Drive»:**

1. У `02_ОБРОБЛЕНІ` зберігати **два файли** на документ:
   - `<doc>.txt` — як зараз (для агентів, пошуку, плейн-копіювання)
   - `<doc>.layout.json` — компактна структура: `{pages:[{w,h,paragraphs:[{bbox,text,style}]}]}` (тільки потрібне, без tokens/blocks)

2. У Viewer для **searchable PDF**:
   - Рендерити `.layout.json` як HTML: кожен paragraph — абсолютно позиціонований `<p>` з координатами з `boundingPoly`. Шрифт — стандартний body, розмір приблизний з `style.fontSize`.
   - Оригінальний PDF на тлі через canvas/iframe — **не потрібно**. Адвокат бачить «той самий документ» але як HTML.
   - Виділення тексту — нативне веб-виділення.

3. Для **scanned PDF**:
   - HTML overlay поверх canvas-зображення сторінки. Layout.json дає координати слів — створюємо невидимі `<span>` поверх кожного слова. Принципово той самий патерн що pdf.js textLayer, але з нашими даними замість pdf.js.
   - Це зберігає візуальний скан з печаткою/підписами + дає вибірний текст.

### Плюси

- **Отримаємо те чого не дає Drive iframe** — нативне виділення тексту.
- **Один формат рендеру для всіх PDF** — searchable і scanned. Логіка проста: layout.json є → рендеримо, немає → fallback iframe.
- **Дані вже збираються Document AI** — треба лише перестати їх викидати.
- **Майбутні можливості:** маркер/нотатки прив'язуються до paragraph id, цитування з посиланням на сторінку, агент-цитування з координатами.
- **Серверний rendering не потрібен** — все клієнтське.
- **Фактично побічний продукт OCR** — інфраструктура (Document AI, 02_ОБРОБЛЕНІ, ocrService) уже є.

### Мінуси / ризики

- **Layout не piксel-perfect.** boundingPoly від Document AI приблизний — інколи слова на 1-2 mm зміщені від візуального положення. Для виділення це OK (span зсунутий — selection все одно ловить). Для **візуальної копії оригіналу** — недостатньо.
- **Шрифти — наближення.** Document AI не повертає embed-шрифти PDF. Документ виглядатиме «як HTML», не «як PDF».
- **Треба переробити формат `02_ОБРОБЛЕНІ`** — з .txt на .txt + .layout.json. Це міграція. Існуючі документи треба перерозпізнати щоб отримати layout (або зберегти як текст-only до наступного OCR).
- **Збільшення вартості OCR (мінімально):** Document AI бере одну й ту саму суму незалежно від того чи ми зберігаємо layout. Збільшується тільки розмір файла на Drive (~20-50 KB на сторінку проти 5 KB plain).
- **Розмір .layout.json для 50-сторінкового PDF** — десь 1-2 MB. Це OK для Drive але треба думати про lazy-load (як `documents_extended.json`).
- **Робота складніша ніж pdf.js.** Це не "взяв бібліотеку і встромив" — це **розробка свого рендеру** на 2-3 дні.

### Реалістичність

**Так, реалістично.** Це найбільший виграш для адвокатської роботи. Але це **окремий мікро-TASK на 2-3 дні**, а не за один день.

## 5. Рекомендація

### Якщо адвокат хоче результат «швидко й просто, аби виділялось»:

**Опція В (pdfobject + native browser PDF viewer)** — 30 хвилин роботи. Втрачаємо контроль UI, але selection працює завдяки нативному viewer'у Chrome/Safari. Підходить як **тимчасовий фікс**.

### Якщо адвокат хоче зробити **правильно один раз**:

**Опція Д — Document AI metadata + HTML рендер.** Окремий мікро-TASK на 2-3 дні. План:

1. **День 1.** Перевизначити формат `02_ОБРОБЛЕНІ` — додати `.layout.json` поряд з `.txt`. Адаптувати `documentAi.js` провайдер щоб зберігав layout. Адаптувати `ocrService` щоб lazy-завантажував layout при запиті Viewer.
2. **День 2.** Написати компонент `LayoutViewer` що рендерить layout.json як HTML з абсолютно позиціонованими paragraphs. Скан-режим — поверх canvas-зображення з Drive thumbnail API (не pdf.js, а готові thumbnails per page).
3. **День 3.** Інтеграція з DocumentViewer (заміна iframe для searchable, overlay для scanned), тести, плавна міграція existing документів (не ламати ті що ще без layout).

**Чому НЕ повторна спроба pdf.js (опція А react-pdf):**
- react-pdf вирішить ~70% bugs, але не дасть **семантичного покращення**. Адвокат отримає той самий PDF з selection — без структури документа, без можливостей агента посилатись на paragraph, без bookmark системи.
- Document AI metadata path відкриває можливості (маркер, цитування агента, навігація по структурі) які pdf.js не дає взагалі.
- Час інвестування сумірний (3 дні react-pdf із всіма edge cases vs 3 дні Document AI HTML), результат принципово різний.

### Не рекомендую зараз:

- **Не повторювати наївну pdf.js інтеграцію** — самі розгребати 8 edge cases без бібліотеки = повторно отримаємо ті самі симптоми.
- **Не браться за серверний pdf2htmlEX** — нема сервера, бюджет не дозволяє.
- **Не платити за Marker/Mistral PDF API** — дорого, залежність від нового провайдера.
