## 2. ЩО ЗМІНЕНО В `promptBuilder.js`

### Блок 1 — Детермінований фільтр ролей (КРИТИЧНО для економії)

Перед заходом у справу — два паралельні фільтри у списку «Мої справи»:

- **Фільтр 1** — рік у `case_no` (за TASK 0.4, без змін: 25 або 26).
- **Фільтр 2** — роль у колонці «Мій процесуальний статус»:
  - **Беремо** конкретну роль: `Адвокат`, `Захисник`, `Представник позивача/відповідача/заявника/потерпілого/скаржника/третьої особи`, будь-яке поєднання.
  - **Пропускаємо** РІВНО ОДНЕ СЛОВО `Представник` без уточнення кого — це довіреність на клієнта-сторону, адвоката там фактично немає.

`Адвокат` і `Захисник` явно позначені як **самостійні конкретні ролі** (беруться завжди). Це усуває двозначність з реального тесту (Sonnet здогадувався сам, але не стабільно).

Пропущені справи додаються в `skipped[]` із конкретною причиною: `"Роль 'Представник' без уточнення — справа підтягнута через довіреність, адвокат не учасник"`.

### Блок 2 — Мапінг ролі → primaryParty

Явна таблиця: `"Представник позивача"` → позивач, `"Захисник"` → обвинувачений/підозрюваний, поєднання → перша конкретна роль. Формат: `"Прізвище І.П."`, для юросіб — повна назва без скорочення.

### Блок 3 — Кримінальні з кількома обвинуваченими

Якщо у `criminal` справі (`Захисник`) кілька обвинувачених і неясно кого захищає Левицький:
- НЕ вгадувати прізвище для назви.
- Засідання все одно витягнути (вони цінні незалежно від назви).
- `primaryParty = case_no` (номер замість прізвища).
- Додати в `warnings[]`: `"Справа [case_no]: кілька обвинувачених ([прізвища]). Уточніть кого захищає Левицький для коректної назви справи."`

Адвокат потім виправить назву вручну. Не блокує імпорт.

### Блок 4 — Економія кроків

Правила ощадливості (extract text як першочерговий, screenshot тільки коли треба структура/перевірка/скан, «Інформація про справу» — один раз на справу, для засідань спершу шукати «Внесення дат слухання»). Завершується явним пріоритетом: **точність > економія**.

### Блок 5 — Checkpoint кожні 5 справ

Після кожних 5 оброблених справ — проміжний JSON у блоці коду з міткою `[ПРОМІЖНИЙ РЕЗУЛЬТАТ — N справ]`. Фінальний JSON у кінці з міткою `[ФІНАЛЬНИЙ РЕЗУЛЬТАТ]`. Кожен — повний envelope (не дельта); дедуплікація через `ecitsCaseId` забезпечує що адвокат може вставити проміжний АБО фінальний без дублювання.

## 5. ПОБІЧНІ ЗНАХІДКИ ТА ПОКРАЩЕННЯ (ЕКСПЕРТНА АВТОНОМІЯ)

### 5.1. Архітектурний порядок блоків — стиль ВИЩЕ за крокові інструкції

TASK перерахував 5 блоків як список без обов'язкового порядку. Я розклав їх так:

```
1. Вступ                                  (задача коротко)
2. СТИЛЬ РОБОТИ (Блок 4)                  ← правила гри ДО старту
3. ФІЛЬТРАЦІЯ СПРАВ (Блок 1)              ← економія перед першим заходом
4. КРОК 3 — Витягни поля справи           (за TASK 0.4)
5. ВИЗНАЧЕННЯ ОСНОВНОЇ СТОРОНИ (Блок 2)   ← одразу після витягування полів
6. КРИМІНАЛЬНІ З КІЛЬКОМА (Блок 3)        ← спецкейс відразу після Блоку 2
7. КРОК 4 — Засідання                     (з підказкою «Внесення дат слухання»)
8. ПРОМІЖНІ РЕЗУЛЬТАТИ (Блок 5)           ← перед envelope
9. СТРУКТУРА ENVELOPE
10. БЕЗПЕКА
11. ПОЧИНАЙ
```

Логіка: Sonnet засвоює стиль роботи і фільтр ПЕРШ НІЖ читає крокові інструкції. Не доводиться «редагувати поведінку посередині» — модель стартує одразу в правильному режимі. Запатентовано тестами `СТИЛЬ РОБОТИ розташований ПЕРЕД ФІЛЬТРАЦІЄЮ` і `ФІЛЬТРАЦІЯ ПЕРЕД КРОК 3`.

### 5.2. Блок «Стиль роботи» з явним пріоритетом

Завершальний абзац Блоку 4 експліцитно говорить: **«точність > економія. Економія — це прибрати ЗАЙВЕ, НЕ скоротити необхідну перевірку даних.»** — щоб Sonnet не «оптимізував» себе у пропуск засідань. Це покрите тестом і додано на основі того що в реальному тесті Sonnet іноді ризикував коректністю заради швидкості.

### 5.3. `skipped[]` приклади у структурі envelope — два конкретні reason

В описі envelope додав ДВА приклади `skipped[]` запису з різними причинами:
```json
"skipped": [
  { "case_no": "...", "reason": "year_filter" },
  { "case_no": "...", "reason": "Роль 'Представник' без уточнення — справа підтягнута через довіреність, адвокат не учасник" }
]
```

Це показує Sonnet формат і дає шаблон reason-стрічки для обох видів пропуску одночасно — без додаткових інструкцій.

### 5.4. `stats.filtered` оновлено — «за роком+роллю»

В TASK 0.4 envelope мав `stats.filtered: <скільки відфільтровано за роком>`. Тепер: `<скільки відфільтровано за роком+роллю>`. Дрібна, але важлива зміна семантики поля — Sonnet повинен рахувати після ОБОХ фільтрів. Без перейменування поля (зворотна сумісність з scenarioProcessor забезпечена — це лише `stats`, processor його не споживає).

### 5.5. Підказка «Внесення дат слухання» — у КРОЦІ 4 а не тільки в Блоку 4

Згадку про «спочатку шукай «Внесення дат слухання»» я ПРОДУБЛЮВАВ ще і в КРОЦІ 4 (одразу перед циклом засідань). Дублювання навмисне — Sonnet міг прочитати Блок 4 на початку, але до моменту збору засідань могло «вивітритись». Друга згадка в правильному контексті закріплює.

### 5.6. ПОЧИНАЙ — нагадування про checkpoint

Фінальний блок «ПОЧИНАЙ» завершується явно: **«Після кожних 5 справ — проміжний JSON. Фінальний JSON — після останньої справи.»** — щоб модель не забула про checkpoint у момент старту роботи.

### 5.7. Жодних змін у `extractYearFromCaseNo` / `isAcceptedCaseYear`

Утилітарні функції залишаються незмінними. TASK 0.4.1 — суто текстовий, ніяких нових утиліт не потрібно (Блоки 1/2/3 — інструкції для Claude for Chrome, не валідація на нашому боці).

### 5.8. Семантичних ризиків НЕ виявлено

Нові концепти (`year_filter`/`role_generic_representative` reason-стрічки, `warnings` про невизначену сторону) — це **значення в існуючих полях**, не нові структури. Жодного зіткнення з правилом #11.

- "Представник скаржника"    → скаржник
- "Представник третьої особи" → третя особа
- "Адвокат"                  → визнач за "Інформація про справу"
                               кого представляє Левицький
- "Захисник"                 → обвинувачений/підозрюваний якого
                               захищає
- Поєднання                  → бери першу конкретну роль

Формат primaryParty: "Прізвище І.П." (Бабенко О.І.)
Для юросіб (ТОВ/ПП/ФОП/АТ) → повна назва без скорочення.

═══════════════════════════════════════════════════════════════
КРИМІНАЛЬНІ СПРАВИ З КІЛЬКОМА ОБВИНУВАЧЕНИМИ
═══════════════════════════════════════════════════════════════

Якщо у кримінальній справі (категорія criminal, роль "Захисник")
у "Інформація про справу" КІЛЬКА обвинувачених і НЕ ЗРОЗУМІЛО
кого саме захищає Левицький:

- НЕ вгадуй прізвище для назви
- Засідання все одно витягни (вони цінні незалежно від назви)
- primaryParty залиш = case_no (номер справи замість прізвища)
- Додай у warnings: "Справа [case_no]: кілька обвинувачених
  ([прізвища через кому]). Уточніть кого захищає Левицький для
  коректної назви справи."

Адвокат потім виправить назву вручну. Не блокуй імпорт через
невизначеність назви.

═══════════════════════════════════════════════════════════════
КРОК 4 — Збери засідання 2026 року
═══════════════════════════════════════════════════════════════

Зайди в хронологію документів справи (документи від суду). Йди ВПЕРЕД у часі.

Спочатку шукай "Внесення дат слухання" — там дата структурована,
читається одним extract. Окремі повістки відкривай ТІЛЬКИ якщо
"Внесення дат слухання" немає або в ньому немає дати.

Для КОЖНОГО документа про засідання (повістка, ухвала про виклик, внесення
дат слухання):
1. Витягни дату, час, зал, провадження
2. Перевір ЧИ ВЖЕ Є ця дата у твоєму списку для цієї справи
3. Якщо нова — додай
4. Якщо вже є — ПРОПУСТИ (дублікат повістки, не нове засідання)

ПРИКЛАД дедуплікації:
- Документ #1: повістка на 1 квітня → додай
- Документ #2: повістка на 1 квітня → ПРОПУСТИ (вже є)
- Документ #3: повістка на 7 квітня → додай
- Документ #4: повістка на 7 квітня → ПРОПУСТИ (вже є)

Беремо ТІЛЬКИ засідання 2026 року — минулі і майбутні в межах
року. Засідання інших років (минулих або наступних) — пропусти.

Для кожного засідання витягни:
- `date`         — ISO дата "2026-MM-DD"
- `time`         — "HH:MM"
- `court`        — повна назва суду
- `hearingRoom`  — номер залу
- `proceedingNumber` — номер провадження (наприклад "6-392/26")
- `cabinetUrl`   — посилання на повістку
- `noticeType`   — тип документа звідки витягли (наприклад
                    "Судова повістка про виклик в суд")

ПРОМІЖНІ РЕЗУЛЬТАТИ (страховка від ліміту)
═══════════════════════════════════════════════════════════════

Кабінет може містити багато справ. Щоб не втратити роботу якщо
впрешся в ліміт використання:

ПІСЛЯ КОЖНИХ 5 ОБРОБЛЕНИХ СПРАВ — видавай проміжний JSON у чат
у блоці коду з міткою:

[ПРОМІЖНИЙ РЕЗУЛЬТАТ — N справ]
```json
{ ... envelope з усіма обробленими досі справами ... }
```

Це парашут. Якщо далі впрешся в ліміт — у адвоката буде останній
проміжний JSON з тим що встиг.

КОЛИ ДІЙШОВ ДО КІНЦЯ (усі релевантні справи оброблені):
Видай ОДИН ФІНАЛЬНИЙ JSON з усіма справами:

[ФІНАЛЬНИЙ РЕЗУЛЬТАТ]
```json
{ ... повний envelope ... }
```

Кожен JSON (проміжний і фінальний) — повний валідний envelope
за форматом нижче, не дельта. Дедуплікація в Legal BMS через
ecitsCaseId забезпечує що вставка проміжного + фінального не
створить дублів.
