## 0. TL;DR (для прослуховування)

1. **Головна знахідка — втрата даних, не просто аналіз.** Живий `registry_data.json` містить **лише 1 запис `ai_usage`**. Уся попередня токен-історія (≈1.8 МБ, ймовірно кілька тисяч записів) **була тихо затерта** під час одного збереження на Drive.
2. **Чому затерлась.** `ai_usage[]` не мав захисту від «усадки», який є в `cases[]`. Сесія з порожнім in-memory `ai_usage` перезаписала повну історію на Drive. **Виправлено** в цьому TASK: shrink-guard тепер прикриває `ai_usage`, `auditLog`, `users`, `tenants` (а `time_entries` свідомо НЕ чіпає — він легітимно худне при місячній ротації).
3. **Бекапи з історією недоступні через інструмент.** 7 знімків від 05:21 містять історію, але кожен ~7.23 МБ — це понад стелю завантаження Drive MCP (живий файл 5.4 МБ — гранично проходить; бекапи падають із «session expired»). Тому точну розбивку по операціях відновити зараз **не вдалось**.
4. **Що замість точних цифр.** Аналіз спирається на: (а) 1 уцілілий запис як зразок юніт-вартості; (б) `time_entries[]` як проксі активності AI (вони вціліли — 5329 записів); (в) **ваш наземний факт ≈ $5-6 списано** за період; (г) аудит повноти логування.
5. **Сліпі плями логування знайдено.** Найважливіша: **злиття фото через модалку** (`converterService.convertImagesToPdf`) **НЕ пише токени `image_sorter` у `ai_usage`** — лише `images_merged` у `time_entries`. За період — **80 сесій злиття**, чиї Sonnet-токени в обліку відсутні. Тобто будь-яка цифра з `ai_usage` **занижена**.
6. **Рекомендація тарифу (попередня, на основі $5-6 ground truth):** закладати в собівартість AB активного користувача **~$15-25/міс Anthropic** для важкого профілю (активна DP-нарізка), **~$5-10/міс** для нормального. Document AI OCR — **окрема стаття** (Google Cloud, ~$1.5/1000 стор), не входить у ці суми. **Точні числа дасть наступний місяць** — guard тепер береже історію.

## 2. КРИТИЧНА ЗНАХІДКА — втрата `ai_usage` і її першопричина

### 2.1 Симптом
Живий `ai_usage[]` = 1 запис, timestamp `2026-05-31T12:54:34.119Z` (== час модифікації файлу). Це **новий** запис генерації контексту, доданий під час останнього збереження. Уся попередня історія відсутня.

### 2.2 Механізм (root cause)
`driveService.writeRegistry` (App.jsx) мав shrink-guard **тільки для `cases`**:
```js
if (prevCount > 0 && newCount < prevCount - 1) return { ok:false, status:'guard_blocked', ... };
```
Для `ai_usage` (а також `auditLog`/`users`/`tenants`) **аналогічного захисту не було**. Тому сесія, що стартувала з порожнім in-memory `ai_usage` (Drive-значення вже скинуте ранішою сесією, або `registry.ai_usage` не був масивом → `setAiUsage` на load пропустився → лишився localStorage-seed `[]`), при наступному save **тихо перезаписала** повну історію на Drive порожнім масивом. `cases` був цілий → guard пропускав запис.

**Міграції — чисті.** `migrateToVersion7/8/9` усі роблять `...registry` spread (App.jsx-виклики + `migrationService.js:767,836,898`) — `ai_usage` зберігається. Це **не** міграційний баг.

### 2.3 Виправлення (зроблено в цьому TASK)
Новий чистий модуль `src/services/registryWriteGuard.js` + проводка в `App.jsx`:
- **`ai_usage`, `auditLog`** — монотонні логи: блок при колапсі до 0 або падінні значущої історії (≥20) більш ніж удвічі.
- **`users`, `tenants`** — критичні дрібні масиви: блок при повному спорожненні.
- **`cases`** — без змін (−1 дозволено: закриття/видалення).
- **`time_entries` — СВІДОМО поза guard'ом** (легітимно худне 1-го числа при ротації у `_archives/`; наївний guard хибно блокував би нормальний запис).
- Лічильники `lastRead*Count` пишуться на read (`readRegistryStatus`), синхронізуються після успішного write і при restore-from-backup (`_syncGuardCounters`).
- Тести: `tests/unit/registryWriteGuard.test.js` (16 кейсів) — блокує усадку `ai_usage`; НЕ блокує ротацію `time_entries`; свіжий старт (`prev=0`) не блокується.

### 2.4 Чого guard НЕ робить (борг #35)
Guard — **сітка безпеки** (ловить наслідок). Першопричина — запис із недогідрейченого/застарілого стану — потребує runtime-логу послідовності load→save і не усувається статично. Save-ефект уже має gate `if (!driveHydrated) return`, тож формально hydration завершувався; точний тригер не відтворюється без телеметрії. Занесено в `tracking_debt.md #35` з тригером активації.

### 2.5 Відновлення історії
**Неможливе наявними інструментами**: бекап не читається (стеля завантаження), а перезапис відновленого ~7.2 МБ файлу через MCP теж недоступний (`create_file` потребував би ~9.6 млн символів у параметрі; `update_file` відсутній). Відновлення — лише у середовищі застосунку адвоката (браузер тягне бекап через OAuth без цієї стелі), і за рішенням адвоката (компроміс: повний restore-from-backup відкотить також `cases`/`time_entries` до стану 05:21). За погодженням обрано: **fix-guard + звіт; 2-денну історію вважаємо втраченою**.

### 4.1 Сліпі плями (через них `ai_usage` занижений)

**A. Злиття фото через модалку «+ Додати документ» — НЕ логує токени `image_sorter`.**
Шлях `AddDocumentModal → converterService.convertImagesToPdf → multiImageToPdf → sortImageDocument` **не передає `billing.aiUsageSink`** (`multiImageToPdf.js:237-244`, прямий коментар на 235-236: «Модалка НЕ передає billing»). `converterService` пише лише `activityTracker.report('images_merged')` (→ `time_entries`, тобто **час**), а **токени Sonnet-сортувальника в `ai_usage` не потрапляють**. **B. `imageSortingAgent` сам не пише `ai_usage`** — залежить від того, чи caller прокинув sink. У модальному шляху (A) не прокидає → 0.

- `alerts.warnAt:80 / blockAt:100` — застосувати до місячного $-бюджету AI на користувача (борг #31 — уніфікований UX «ліміт вичерпано»).
- Тарифний план має закладати **маржу поверх COGS** + буфер на сліпі плями (поки §4-A не виправлено, реальна вартість вища за лог).
- Зв'язок з `entitlements` (борг #22): DP-нарізка — кандидат на окремий лічильник/ліміт як найдорожчий сценарій.

### 8.3 Передумова достовірних цифр наступного циклу
Shrink-guard тепер береже `ai_usage`. **Наступний місяць реальної роботи дасть повну, недоторкану історію** → можна повторити цей аналіз із точною розбивкою по `operation/model/день/справа`, як і задумано в TASK. Для повної точності варто також: (1) закрити сліпу пляму §4-A (sink у модальному злитті фото); (2) звести борг #32 (LIFO 50000/100000) на одну константу.
