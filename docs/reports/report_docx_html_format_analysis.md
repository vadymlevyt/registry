# Аналіз варіантів — збереження форматування DOCX/HTML при конвертації

**Дата:** 2026-05-12
**Контекст:** Після TASK A адвокат побачив що pdf-lib pipeline зберігає текст, але втрачає форматування (таблиці у стовпчик, заголовки і жирний втрачені, герби у HTML ухвалах з ЄСІТС зникли).

**Документи адвокатської практики не мають складного форматування** — це переважно:
- Звичайні абзаци з вирівнюванням по ширині
- Шапка документа вужча за основний текст (вирівнювання справа)
- Іноді жирний/курсив для виділення окремих слів або норм права
- Прості таблиці на 2-3 стовпчики
- Герби у деяких ухвалах з ЄСІТС (вбудовані як зображення у HTML)

Це не презентації, не складна графіка, не таблиці у таблицях, не кольорова інфографіка. Просто структурований текст з простими елементами.

Адвокат поставив питання: який варіант легше реалізувати щоб зберегти таке форматування? Нижче — чесний аналіз без рекомендації.

---

## Суттєвий контекст з попередніх повідомлень

У попередньому micro-TASK адвокат сам підтвердив що **DocxRenderer працює правильно** (mammoth → HTML inline у Viewer, рендерить документ із форматуванням). Це відкриває варіант D якого спочатку не було у списку.

---

## ВАРІАНТ A — розширити pdf-lib з форматуванням і зображеннями

### Що треба написати:
- Парсер HTML від `mammoth.convertToHtml` (DOMParser, обхід дерева)
- Логіка рендеру для `<h1>-<h6>` (більший шрифт, відступи), `<b>/<strong>`, `<i>/<em>` — потребує 3 окремих файли шрифту: Regular, Bold, Italic, BoldItalic (LiberationSans-Bold/Italic уже є у `node_modules/pdfjs-dist/standard_fonts/`)
- Вирівнювання:
  - Лівий — є зараз
  - Правий — простий (порахувати ширину тексту, малювати в `(pageWidth - rightMargin - textWidth)`)
  - Justify — складно (треба самостійно розподіляти space-width між словами на кожному рядку)
- Таблиці — мінімальний layout-engine:
  - Ширина колонок (рівномірна або зі стилів)
  - Wrap тексту в комірках
  - Висота рядка по найвищій комірці
  - Page-break коли рядок не влазить
- Зображення: mammoth повертає base64 в `<img src="data:image/png;base64,...">`. pdf-lib має `embedPng()`/`embedJpg()`. Витягти base64, embed, малювати з масштабуванням до ширини контенту

### Точність: ~80%
Бо ми реалізуємо ПІДМНОЖИНУ HTML/CSS вручну. Деякі речі обовʼязково випадуть (вкладені таблиці, складна типографія).

### Час:
- 2-3 дні роботи щоб «достатньо добре» (бази B/I, заголовки, прості таблиці без page-break, зображення, шапка справа)
- 1-2 тижні щоб «майже як Word»

### Герби:
- **HTML з ЄСІТС:** герби часто це `<img src="data:image/png;base64,...">` — це embed'иться нормально. Якщо `src="https://court.gov.ua/herb.png"` — потрібен fetch і CORS може заблокувати.
- **DOCX:** mammoth has `options.convertImage` що повертає base64. Embed працює.

### Bundle:
+280 КБ (Bold + Italic шрифти).

## ВАРІАНТ B — повернутись до html2pdf.js і виправити порожній PDF

### Можливі гіпотези чому канвас порожній на планшеті (упорядковано за ймовірністю):

1. **`width: 210mm` + position `left:-10000px`** — html2canvas рахує `clientWidth` на момент захоплення, у мобільному viewport мм-юніти можуть давати 0 або негативний width
2. **Шрифт `'Times New Roman'` не доступний на iOS Safari** (там SF Pro або похідні) → текст рендериться як `tofu` або взагалі пропадає
3. **`requestAnimationFrame` на мобільному Safari тригериться раніше ніж браузер закінчив layout** — потрібна довша затримка (`setTimeout(0)` × 2 або `await document.fonts.ready`)

### Що треба пробувати:
Кожна гіпотеза — окрема ітерація. Деплой → адвокат тестує → повідомляє «знов порожньо». **Без iPad-дебагера це сліпий пошук.**

### Точність якщо запрацює: 100% візуально
Але **PDF буде raster** (html2canvas → JPEG → embed у PDF). Текст у Viewer не виділяється, не копіюється, Ctrl+F не шукає. Файли важчі (1-3 МБ за документ).

### Час:
Від 30 хв (якщо перша гіпотеза вгадає) до 2-3 днів (якщо нічого не допомагає і треба міняти підхід). Без можливості Claude провести у DevTools планшета — це лотерея.

### Герби:
Працюють «безкоштовно» бо html2canvas рендерить як зображення. CORS-обмеження для зовнішніх URL лишається.

## ВАРІАНТ C — гібрид (спочатку html2pdf, fallback pdf-lib)

Доводиться підтримувати **обидва шляхи**. Розгалуження логіки у коді конвертера. Удвічі більше тестів. Унікальна користь над «вибрати один» — нульова. Не виявляє нової інформації.

### Час:
A + B + інтеграція ~3-5 днів.

---

## ВАРІАНТ D — не конвертувати в PDF взагалі, використати існуючі рендерери

Адвокат сам у попередньому повідомленні підтвердив що **DocxRenderer працює правильно** (mammoth → HTML inline у Viewer). HtmlRenderer теж існує і працює (charset detection + iframe srcdoc).

### Що треба змінити:
- У `converterService.js` для DOCX/HTML повернути `passthrough` (як було до TASK A)
- `driveId` → DOCX/HTML оригінал (не PDF)
- Виходить `wasConvertedToPdf` логіка у Viewer стає непотрібною — повертаємо стару маршрутизацію по MIME
- `.txt` extraction (`mammoth.extractRawText`, `innerText`) лишається — це окремий шлях для AI/search кешу
- Прибрати `pdfLibRenderer.js`, `LiberationSans-Regular.ttf`, `@pdf-lib/fontkit`
- Прибрати залежність `html2pdf.js`, `heic2any` (зображення теж під питанням — див. нижче)

### Точність: Найкраща
DocxRenderer використовує **той самий mammoth** з styleMap для вирівнювання — отже зберігає bold, italic, headers, justify, tables, images.

### Герби:
Працюють у HtmlRenderer (звичайний iframe рендерить `<img>`). У DocxRenderer теж — mammoth ембедить картинки як base64 у HTML.

### Час: 1-2 години
Більшість роботи — REVERT того що додавали у TASK A.

### Трейд-офф:
- Втрачаємо «єдиний PDF формат». PDF у системі залишається тільки для PDF/image
- `pageCount` для DOCX/HTML невідомий до рендеру (як було раніше)
- Майбутній модуль «AI Очищення тексту» з координатами слів — не працюватиме для DOCX/HTML (бо це не PDF з pageStructure). Але цей модуль ще не існує і працює тільки для скан-PDF з Document AI

### Що зробити з зображеннями?
Зображення зараз теж конвертуються у PDF через jsPDF. Це окремий випадок — там html2canvas НЕ використовується, лише Canvas API + jsPDF.addImage. Зображення конвертація імовірно працює нормально (адвокат не повідомляв про порожні зображення-PDF). Можна залишити як є — конвертувати тільки image→PDF, а DOCX/HTML passthrough.

## ВАРІАНТ E (новий) — серверна конвертація через LibreOffice headless

Зараз у системі **нуля бекенду** — все на Drive і клієнті. Цей варіант потребує:
- Окремий сервер (Cloud Run / VPS) з `libreoffice --headless --convert-to pdf`
- Або 3rd-party API (CloudConvert, ConvertAPI — платно)
- Latency: 2-5 секунд на конвертацію

### Точність: 99%
LibreOffice розуміє Word format нативно.

### Час: 1-2 дні якщо є інфра
Без інфри — спочатку її треба підняти. Для одного адвоката-засновника на безкоштовному GitHub Pages — overengineering. Релевантніше при SaaS-комерціалізації коли є серверна частина.

## ЧЕСНІ ЗАУВАЖЕННЯ

1. **Варіант D — це визнання що TASK A був помилкою для DOCX/HTML.** Image → PDF можна залишити (там Canvas API працює надійно). Але DOCX/HTML конвертація створила більше проблем ніж вирішила: спочатку opacity:0 баг, потім порожні PDF, тепер втрата форматування. Існуючі рендерери уже працювали. Це боляче визнати після 5 фіксів, але це найдешевший вихід.

2. **Варіант B — це 90% шанс що адвокат пробує 3-4 рази і знов щось не так.** Без можливості Claude бачити DevTools на iPad — будь-який фікс це здогадка. Опис гіпотези → деплой → адвокат тестує → повідомляє. Один цикл — 10 хв адвокатського часу + 5 хв деплою. Якщо знайти з 1-2 спроби — гарно, якщо ні — фрустрація.

3. **Варіант A дає selectable text що корисно у Viewer (виділення цитат, Ctrl+F).** Це перевага. Але треба буде писати ~300-500 рядків коду парсера HTML→pdf-lib з нюансами які тяжко передбачити.

4. **Зображення для DOCX→PDF через варіант A потребують додаткового кроку** у mammoth: `options.convertImage: mammoth.images.imgElement(...)`. Стандартний convertToHtml ВЖЕ повертає `<img>` з data: URL. Тобто з коробки працює.

5. **Існує гібрид D + конвертація зображень:** DOCX/HTML — passthrough як D, JPG/PNG/HEIC — як зараз (jsPDF, працює). Це найбільш консервативний варіант.
