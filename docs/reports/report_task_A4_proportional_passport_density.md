## 1. ПОЯСНЕННЯ ДЛЯ ВЛАСНИКА (проста мова)

Перед нарізкою великого тому система складає «паспорт» — стислу витяжку з кожної
сторінки (початок + кінець тексту), щоб Haiku за нею знайшла межі між документами.
Скільки тексту брати з кожної сторінки — це і є **щільність паспорта**.

**Як було:** жорсткий обрив. До 70 сторінок — щедро (≈3000 символів/стор.). На
71-й сторінці щільність падала в **5 разів** одним стрибком — до 600 символів/стор.
Том на 100–250 сторінок голодував: Haiku вгадувала межі за мізерними обрізками. Це
одна з причин низької точності на великих томах.

**Як стало:** плавно. Є **єдиний бюджет символів на весь том** (210 000 — рівень,
який точно вкладається у вікно Haiku, виміряно реальним лічильником токенів). Він
розподіляється рівномірно по сторінках:
- **≤70 сторінок** — нічого не змінилось, той самий щедрий паспорт (3000/стор.).
  Малі томи (Брановський, 65 стор.) працюють **байт-у-байт як раніше**.
- **70–250 сторінок** — отримують у **1.4–3.5 раза більше** тексту, ніж голодні 600.
- **дуже великі (>350)** — підлога 600/стор. (не гірше, ніж було).

Жодного обриву на 71-й сторінці більше немає — щільність спускається плавно.

## 2. BEFORE / AFTER — таблиця щільності (символів на сторінку)

`perPage = clamp(round(210000 / pageCount), 600, 3000)`; split head:tail = 1:1.

| Обсяг (стор.) | Було (симв/стор) | Стало (симв/стор) | Зміна | head/tail було → стало |
|---|---|---|---|---|
| **70** | 3000 | **3000** | **0 — байт-у-байт** ✅ | 1500/1500 → 1500/1500 |
| 100 | 600 | **2100** | ↑ 3.5× | 400/200 → 1050/1050 |
| 150 | 600 | **1400** | ↑ 2.33× | 400/200 → 700/700 |
| 250 | 600 | **840** | ↑ 1.4× | 400/200 → 420/420 |
| 350 | 600 | **600** | 0 (підлога) | 400/200 → 300/300 |

**≤70 стор. — байт-у-байт незмінні** (clamp дає стелю 3000 = rich для будь-якого
pageCount ≤ 70: 210000/70 = 3000 рівно, нижче — ще вище → теж клампиться у 3000).

**Примітка про split на підлозі (чесно).** Для томів **>350 стор.** сумарна
щільність ідентична (600/стор., як було), але **розкладка head:tail змінилась**
з компактних 400/200 на 300/300 (1:1). Це свідоме рішення спеки §3: split 1:1
тримає ≤70 байт-ідентичними (rich завжди був 1:1), а `COMPACT_DEFAULTS` прямо
помічений «стартові точки, не остаточні» — більше хвоста на гігантських томах теж
корисно. **Малі/середні томи (≤70) це не зачіпає** — там perPage = стеля, head=tail=1500.

## 3. ЩО ЗМІНЕНО В КОДІ

### `src/services/documentPipeline/pageMarkers.js`

**Знято** (поріг 70 тепер імпліцитний у бюджеті: 210000/3000 = 70):
- `RICH_PASSPORT_MAX_PAGES_DEFAULT = 70`
- `_richMaxPagesOverride`, `richMaxPages()`, `export _setRichPassportMaxPages`

**Додано:**
- `PASSPORT_CHAR_BUDGET_DEFAULT = 210000` (= 70 × CEIL, рівень rich-70, відомо-добрий
  за реальним `count_tokens` 2026-06-21: ~105K ток, нижче degenerate-стелі Haiku ~150K).
- `PASSPORT_FLOOR_CHARS = 600` (= поточний компакт), `PASSPORT_CEIL_CHARS = 3000` (= rich).
- `export _setPassportCharBudget(n)` — override-хук (тести / майбутня tenant-калібровка),
  замінює знятий `_setRichPassportMaxPages`. `passportCharBudget()` (внутрішній резолвер).

**Переписано** `passportOptsForBudget(pageCount)` — з бінарного вибору профілю на
пропорційне заповнення бюджету:
```js
export function passportOptsForBudget(pageCount) {
  if (!(pageCount > 0)) return {};                 // невідомий обсяг → компакт-дефолти
  const perPage = Math.min(
    PASSPORT_CEIL_CHARS,
    Math.max(PASSPORT_FLOOR_CHARS, Math.round(passportCharBudget() / pageCount)),
  );
  return { ...RICH_PASSPORT_OPTS, headChars: Math.ceil(perPage / 2), tailChars: Math.floor(perPage / 2) };
}
```

**Експортовано** (для тестів): `passportOptsForBudget`, `RICH_PASSPORT_OPTS`
(додатково — без зміни поведінки). Оновлено doc-коментар `resolveBoundaryText`
(описує пропорційну, а не бінарну адаптивність).

### Тести

  пропорційний, не компакт; межа 70/71 тепер **неперервна**, а не обрив ×2) + **новий

## 4. ДОВЕДЕННЯ ГОЛОВНОГО ІНВАРІАНТА §0 — МАЛІ ТОМИ НЕ ЗМІНИЛИСЬ

Тест `≤70 стор. → байт-у-байт RICH_PASSPORT_OPTS` (pageCount ∈ {1, 35, 70}) перевіряє
**повну рівність об'єкта** `passportOptsForBudget(pc) === RICH_PASSPORT_OPTS` (head 1500
/ tail 1500). Без нього TASK не зданий — він зелений.

Логіка чому це математично гарантовано: для будь-якого pageCount ≤ 70 маємо
`210000/pageCount ≥ 3000`, отже `min(3000, max(600, ≥3000)) = 3000` = CEIL; head=ceil(1500)=1500,
tail=floor(1500)=1500. Паспорт малих/середніх томів **байт-у-байт незмінний** → план
нарізки і весь конвеєр далі теж незмінні. Малі томи фізично не можуть змінитися.

## 7. КАЛІБРУВАННЯ (після приземлення — не код цього TASK)

Дефолт `PASSPORT_CHAR_BUDGET = 210000` — відомо-добрий старт (рівень rich-70). Далі на
реальних томах через `_setPassportCharBudget` піднімати бюджет і дивитися, де Haiku
починає втрачати межі. Паралельно через меню ⚙ перемкнути тріаж на Sonnet (вища стеля
якості) — це визначить поріг «з якого обсягу вмикати Sonnet» (важіль A4-г, окремо).
Сторожі (ФД-2 бюджет, «мало документів») і детектор реєстру — окремими спеками.
