## 1. Що зроблено

`ocrService.listFolderFilesByName` приведено у відповідність до CLAUDE.md правила #8: видалено `name='${name}'` з `q=` filter (там кирилиця — імена кешу типу `701_1413_25 кисельова ухвала_<driveId>.txt`), запит тепер тягне всі файли папки `02_ОБРОБЛЕНІ` тільки за `parents` + `trashed=false`, а потрібний файл знаходиться у JavaScript через `filter(f => f.name === name)`. Цей самий патерн уже застосовано у `driveService.deleteOcrCacheForDocument` (driveService.js:381-388) і вважається канонічним для роботи з кеш-файлами що містять кирилицю. Ні `checkCache`, ні `writeCache` тепер не залежать від ненадійної поведінки Drive API при кириличному `q=`. Інші файли в `src/services/ocr/` (`documentAi.js`, `claudeVision.js`, `pdfjsLocal.js`) Drive q= не використовують взагалі — лише точкові виклики `/files/${id}?alt=media` за driveId. Правило #8 у CLAUDE.md лишається незмінним і обов'язковим. Тести (`npm test`) — 422 з 422 зелені без модифікацій.

## 5. Аналіз логіки searchable / scanned в extractText

### Чи розрізняє `extractText` searchable і scanned PDF?

**Так, розрізняє — але непрямо**, через ланцюжок провайдерів (planka Picatinny) + детекцію текстового шару всередині `pdfjsLocal.extract`. Поле `documentNature` об'єкта документа `extractText` **НЕ читає взагалі**.

### Який код приймає рішення

**Точка 1 — `pickProviderName(file)`** (`ocrService.js:115-133`):
```javascript
function pickProviderName(file) {
  if (file.mimeType === 'application/pdf') return 'pdfjsLocal';   // ← усі PDF спершу через pdfjs
  if (file.mimeType?.startsWith('image/')) return 'documentAi';   // ← зображення → одразу Document AI
  // ... text/plain, text/markdown, text/html, Google Docs → 'pdfjsLocal'
  return null;
}
```

**Точка 2 — `fallbackChain(initial, file)`** (`ocrService.js:136-151`):
```javascript
if (initial === 'pdfjsLocal' && file.mimeType === 'application/pdf') {
  chain.push('documentAi');
  chain.push('claudeVision');
}
```
Тобто для PDF ланцюг: `pdfjsLocal → documentAi → claudeVision`.

**Точка 3 — детекція текстового шару всередині `pdfjsLocal.extract`** (`pdfjsLocal.js:152-157`):
```javascript
const avgChars = numPages > 0 ? trimmed.length / numPages : 0;
const minLength = 200 * Math.min(3, numPages);
if (avgChars < 200 || trimmed.length < minLength) {
  throw makeError('UNSUPPORTED', 'PDF без текстового шару (скан) — потрібен OCR');
}
```

### Що відбувається для PDF з текстовим шаром (searchable)

1. `pickProviderName` повертає `'pdfjsLocal'`.
2. `pdfjsLocal.extract` тягне байти з Drive, парсить через pdfjs, на кожній сторінці викликає `page.getTextContent()`.
3. Середня кількість символів ≥ 200 на сторінку → **повертає текст без виклику Document AI**.
4. `extractText` пише результат у кеш `02_ОБРОБЛЕНІ/<sanitized>_<driveId>.txt`.

**Document AI не викликається. Це швидко і безкоштовно.**

### Що відбувається для сканованого PDF

1. `pickProviderName` повертає `'pdfjsLocal'` (бо мейн просто `application/pdf`).
2. `pdfjsLocal.extract` парсить PDF, але `getTextContent()` повертає мало/нічого.
3. `avgChars < 200` → `pdfjsLocal` кидає `UNSUPPORTED ('PDF без текстового шару (скан)')`.
4. `extractText` ловить помилку, пробує наступного провайдера в ланцюгу — **`documentAi`**.
5. `documentAi.extract` тягне байти повторно, шле в Document AI `/process`. PDF >15 стор. → нарізка через pdf-lib на чанки по 15 сторінок.
6. Повертає текст → пишеться в кеш.

**Document AI викликається. Працює коректно, але є невелика неефективність — Drive download виконується двічі (один раз pdfjs, один раз documentAi).** Це питання продуктивності, не правильності.

### Що відбувається для зображення (jpg / png / heic)

1. `pickProviderName` повертає `'documentAi'` напряму (бо `mimeType.startsWith('image/')`).
2. `documentAi.extract` — один запит, повертає текст.

**Document AI викликається відразу, без проміжного pdfjs.**

### Що відбувається коли documentNature невизначений

`extractText` поле `documentNature` **не читає**. Рішення приймається тільки за `mimeType` + поведінкою pdfjs у рантаймі. Тому невизначений `documentNature` на об'єкті документа не впливає на роботу OCR — система визначає тип файлу самостійно з реального байтового вмісту.

### Висновок щодо логіки

**Логіка розрізнення є**, працює коректно для трьох типів файлів. Але вона **спирається на mimeType + рантайм-детекцію** замість поля `documentNature`. Це:

- **Плюс:** надійніше (mimeType точно є; documentNature може бути неправильно проставлений документ-фактором при додаванні)
- **Мінус:** для сканованих PDF робиться зайвий запит до Drive (pdfjs тягне байти, фейлиться, Document AI тягне ще раз)

### Чи це баг для наступного мікро-TASK?

**Невеликий баг продуктивності, не коректності.** Можлива оптимізація для майбутнього мікро-TASK:

> **Рекомендація для окремого мікро-TASK** «extractText respects documentNature»:
> 1. Перед запуском ланцюга провайдерів читати `file.documentNature`.
> 2. Якщо `documentNature === 'scanned'` для PDF — починати ланцюг із `documentAi`, а не `pdfjsLocal` (зекономить ~1-3 секунди на скан).
> 3. Якщо `documentNature === 'searchable'` — лишити поточну поведінку (pdfjs первинний).
> 4. Якщо `documentNature` undefined/null — поточна поведінка (детекція рантайм). Зворотна сумісність зберігається.
>
> Виконувати після того як `documentNature` стабільно проставляється в усіх точках створення документа (мікро-TASK 3 — AddDocumentModal pipeline).

У цьому мікро-TASK 2 логіку **НЕ змінено** — фокус виключно на правилі #8.

1. Відкрий справу з документом-PDF що має текстовий шар (більшість PDF з ЄСІТС, текстовий копі-пейст).

6. Перевір на Drive у папці справи `02_ОБРОБЛЕНІ` — має бути файл `<назва>_<driveId>.txt`.

### Тест 2 — Сканований PDF

1. Відкрий справу з відсканованим PDF (фотокопія паперового документа).
2. Натисни на документ → DocumentViewer.
3. Натисни **«Розпізнати зараз»**.
4. **Очікувано:** упродовж 5-30 секунд (залежно від кількості сторінок) з'являється текст; зелений toast «Текст розпізнано і збережено».
5. У DocumentViewer побачиш перемикач **[🖼 Скан] [📝 Текст]** — Текст показує OCR-результат, Скан показує самий PDF через Drive iframe.
6. Закрий і знов відкрий → миттєво з кешу.

### Тест 3 — Файл зображення (.jpg / .png / .heic)

1. Відкрий справу з документом-зображенням.
2. Натисни на документ → DocumentViewer.
3. Натисни **«Розпізнати зараз»**.
4. **Очікувано:** 3-10 секунд → текст + зелений toast.
5. Перемикач **[🖼 Скан] [📝 Текст]** — Скан показує саму картинку.
6. Закрий і знов відкрий → миттєво з кешу.

### Що змінилось у поведінці порівняно з мікро-TASK 1

- **Раніше:** після «Розпізнати зараз» з'являвся **помаранчевий** toast «Текст розпізнано, але не вдалось зберегти кеш». Текст бачив один раз — при наступному відкритті документа знову порожньо.
- **Зараз:** з'являється **зелений** toast «Текст розпізнано і збережено». Кеш реально записується. Повторне відкриття — миттєво.

Якщо помаранчевий toast усе ще з'являється — значить є інша причина (немає прав на запис у `02_ОБРОБЛЕНІ`, немає підпапки, проблема з Drive токеном). Скріншот → передай у наступний мікро-TASK.

1. **Мікро-TASK 3 — AddDocumentModal pipeline.** Поза скопом. Адвокат як і раніше має натискати «Розпізнати зараз» один раз на кожен новий документ, доданий через модалку «+ Додати документ». Автоматичне OCR при додаванні — окремий мікро-TASK.

Це **окреме порушення** у legacy-коді, що потенційно пояснює дублювання структури папок Drive. Прямо вказано «НЕ чіпай інші ділянки коду» — фікс відкладено в окремий мікро-TASK «driveService rule #8 compliance».

Рекомендовано окремий мікро-TASK після мікро-TASK 3 коли `documentNature` буде гарантовано проставлятись у всіх точках створення документа.

Якщо колись будуть додавати юніт-тести для `ocrService` — варто змокати `driveRequest` і перевірити що `q=` не містить кирилиці. Ця рекомендація — для окремого мікро-TASK «ocrService unit coverage».
